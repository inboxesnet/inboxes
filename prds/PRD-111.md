# PRD-111: CI Build — Go Binary Not Built

## Status
Draft

## Priority
P2-Medium

## Category
Infrastructure

## Source
PRE-LAUNCH-AUDIT #30

---

## Problem Statement
The CI pipeline runs `go test` but never runs `go build`. This means linker errors, missing imports, and invalid build tags are not caught until the Docker build stage. A broken binary can pass CI green and only fail during deployment.

## Current Behavior
In `.github/workflows/test.yml`, the backend job (lines 9-18) only runs:
```yaml
- run: go test ./... -race -count=1 -timeout 120s
  working-directory: backend
```
There is no `go build` step. The `backend/Dockerfile` (line 6) runs `CGO_ENABLED=0 go build -o /inboxes ./cmd/api` but this only executes during Docker image creation, not during CI test runs. Linker errors (e.g., referencing a deleted function) would pass `go test` but fail `go build`.

## Expected Behavior
CI should include a `go build` step that compiles the full binary, catching linker and import errors before the PR is merged. The build output should be discarded (written to `/dev/null`) since we only need the compilation check, not the artifact.

## Files to Modify
- `.github/workflows/test.yml:17` — Add a `go build -o /dev/null ./cmd/api` step after the test step.

## Impact Analysis

### What This Affects
- CI pipeline duration (adds ~10-15 seconds for compilation).
- Developer confidence that green CI means a deployable binary.

### What Affects This
- Any changes to Go source files, imports, or build tags.
- Go version used in CI (currently 1.23, matching Dockerfile).

### Related PRDs
- PRD-106: Env Vars — missing env var documentation could cause build-time config issues.
- PRD-107: Config Validation — startup validation catches runtime issues that build checks cannot.

## Implementation Notes
Add the build step immediately after the test step in the `backend` job. Use `-o /dev/null` to avoid creating an artifact:
```yaml
- run: go build -o /dev/null ./cmd/api
  working-directory: backend
```
Consider also adding `go vet ./...` for static analysis.

## Acceptance Criteria
- [ ] `.github/workflows/test.yml` includes a `go build -o /dev/null ./cmd/api` step in the backend job.

- [ ] Existing tests still pass in the pipeline.
- [ ] Build step runs after tests so test failures are reported first.
