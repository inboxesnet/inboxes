# PRD-033: Scroll Position Lost

## Status
Done

## Priority
P2-Medium

## Category
Search & Navigation

## Source
PRE-LAUNCH-AUDIT #19

---

## Problem Statement
When a user views a thread in the split-pane or full-page thread view and then navigates back to the thread list, the scroll position resets to the top. If the user was scrolled 50 threads down, they lose their place and must scroll back to where they were. This is especially frustrating when triaging a long inbox because every open-and-close cycle resets progress.

## Current Behavior
The `ThreadListPage` component in `thread-list-page.tsx` renders a thread list in a scrollable `<div>` (line 275). When the user clicks a thread, the `selectedThreadId` state changes (line 146), which shows the `ThreadView` in the right pane (lines 325-335). On desktop, the list pane shrinks to 350px wide (line 320) but stays rendered. On mobile, the list pane is hidden entirely (`hidden md:block`, line 320).

The issue manifests in two scenarios:
1. **Mobile**: The list pane unmounts (`hidden` class) when a thread is selected. When the user taps "Back", the list re-renders from scratch, scroll position at 0.
2. **Desktop split-pane**: The list pane changes width from `flex-1` to `w-[350px]`, which can cause a reflow, and the scroll container may or may not preserve its position depending on the browser. Additionally, if the thread list query re-fetches (e.g., from a background refetch), the entire list re-renders and scroll resets.

There is no scroll position preservation mechanism -- no ref tracking the scroll offset, no `scrollRestoration`, and no `scrollIntoView` on the previously selected thread.

## Expected Behavior
1. When navigating back from a thread view (both mobile and desktop), the thread list should restore scroll position to where the user was.
2. The previously viewed thread should be visible in the viewport (ideally highlighted or at the same scroll offset).
3. On desktop split-pane, selecting a thread should not cause the list to scroll.

## Files to Modify
- `frontend/components/thread-list-page.tsx:275` -- Add a `ref` to the scrollable container div. Store `scrollTop` before showing a thread and restore it when coming back.
- `frontend/components/thread-list-page.tsx:144-149` -- In `handleThreadClick`, save the current `scrollTop` of the list container before changing `selectedThreadId`.
- `frontend/components/thread-list-page.tsx:320` -- On mobile, instead of using `hidden` (which unmounts the DOM and loses scroll), use `invisible` or `overflow-hidden h-0` to keep the element in the DOM but hidden.
- `frontend/components/thread-list.tsx` -- Ensure the list component does not force scroll reset on re-render.

## Impact Analysis

### What This Affects
- All users who browse long thread lists -- inbox triage, reviewing sent emails, checking archive.
- Mobile users are more affected because the list fully unmounts.
- Desktop users are affected when background refetches occur.

### What Affects This
- React Query background refetches -- `useThreadList` may trigger re-renders that reset scroll.
- The `selectedThreadId` state change -- causes the list pane to resize on desktop.
- Mobile responsive layout -- the `hidden`/`block` toggle destroys DOM state.
- Browser scroll restoration behavior -- varies by browser.

### Related PRDs
- PRD-034 (Pagination Controls) -- pagination interactions also affect scroll position
- PRD-031 (Empty Search Results) -- search mode resets scroll independently

## Implementation Notes
- **Scroll position ref approach**:
  ```tsx
  const scrollRef = useRef<HTMLDivElement>(null);
  const savedScrollRef = useRef(0);

  const handleThreadClick = useCallback((threadId: string) => {
    if (scrollRef.current) {
      savedScrollRef.current = scrollRef.current.scrollTop;
    }
    setSelectedThreadId(threadId);
  }, []);

  // Restore scroll when returning from thread view
  useEffect(() => {
    if (!selectedThreadId && scrollRef.current) {
      scrollRef.current.scrollTop = savedScrollRef.current;
    }
  }, [selectedThreadId]);
  ```
- **Mobile preservation**: Replace `hidden md:block` with a CSS approach that keeps the element in the DOM:
  ```tsx
  className={`${selectedThreadId ? "max-h-0 overflow-hidden md:max-h-none md:block ..." : "flex-1"}`}
  ```
  This avoids unmounting the list while still hiding it visually on mobile.
- Consider using `scrollIntoView` to scroll the selected thread into view when returning, as a fallback if exact scroll position restoration is imprecise.

## User Notes
- Power users triaging large inboxes (50+ threads) will immediately notice this improvement.
- The expected behavior matches Gmail, Outlook, and most email clients that preserve list position.

## Acceptance Criteria
- [ ] On mobile: tapping a thread, viewing it, and pressing Back restores the scroll position to within 10px of the original
- [ ] On desktop: clicking a thread in split-pane mode does not cause the list to scroll
- [ ] After viewing a thread and closing it, the thread list shows the previously selected thread visible in the viewport
- [ ] Background query refetches do not reset the scroll position
- [ ] Changing pages (pagination) correctly scrolls to the top of the new page
- [ ] The scroll position is reset when navigating to a different folder (e.g., Inbox to Sent)
