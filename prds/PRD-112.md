# PRD-112: PUBLIC_URL Missing from docker-compose

## Status
Done

## Priority
P2-Medium

## Category
Infrastructure

## Source
PRE-LAUNCH-AUDIT #31

---

## Problem Statement
The `PUBLIC_URL` environment variable is required for webhook registration with Resend (the backend uses it to construct callback URLs), but it was previously missing from the compose file. Although it has since been added on line 42, the `.env.example` file (line 26) defaults it to `http://localhost:8080`, which is incorrect for production Docker deployments. Operators who copy `.env.example` verbatim will register webhooks pointing to localhost, causing all inbound email processing to silently fail.

## Current Behavior
In `docker-compose.yml` (line 42), `PUBLIC_URL` is set to `https://${DOMAIN}`, which is correct when `DOMAIN` is set. However:
- `.env.example` (line 26) sets `PUBLIC_URL=http://localhost:8080` — suitable for local dev but dangerous if copied to production.
- There is no validation that `PUBLIC_URL` is a reachable external URL.
- If `DOMAIN` is not set in the environment, the compose interpolation produces `https://` (empty host), which silently breaks webhook registration.

## Expected Behavior
- `docker-compose.yml` should use `PUBLIC_URL=${PUBLIC_URL:?Set PUBLIC_URL to your externally reachable URL}` to fail fast with a clear error if unset.
- `.env.example` should have a comment making it clear that `PUBLIC_URL` must be changed for production and that `localhost` will not work.
- The backend should validate at startup that `PUBLIC_URL` starts with `https://` and contains a hostname (not `localhost`) when running in production mode.

## Files to Modify
- `docker-compose.yml:42` — Change to use `${PUBLIC_URL:?Set PUBLIC_URL}` or add a comment about required configuration.
- `.env.example:26` — Add prominent warning comment about changing this for production.
- `backend/internal/config/config.go` — Add optional validation warning if `PUBLIC_URL` contains `localhost` and `API_PORT` is not the default dev port.

## Impact Analysis

### What This Affects
- Webhook registration in `backend/internal/handler/onboarding.go` where `PublicURL` is used to construct the Resend webhook callback URL.
- Domain webhook re-registration in `backend/internal/handler/domains.go`.
- Any operator deploying via `docker compose up` for the first time.

### What Affects This
- The `DOMAIN` environment variable which is used to derive `PUBLIC_URL` in compose.
- DNS configuration pointing the domain to the server's IP.

### Related PRDs
- PRD-106: Env Vars — documents missing environment variables from `.env.example`.
- PRD-107: Config Validation — validates critical config values at startup.
- PRD-114: DB Password — similar pattern of dangerous defaults in `.env.example`.

## Implementation Notes
The safest approach is the `${VAR:?message}` shell parameter expansion in docker-compose, which causes `docker compose up` to fail immediately with a clear error message if the variable is unset or empty. This is already used for `ENCRYPTION_KEY` on line 39.

## User Notes
From FULL-SUMMARY.md: "Required for webhook registration but easy to forget since it's not in the compose file." The variable is now present in compose but the default in `.env.example` is still localhost.

## Acceptance Criteria
- [ ] `docker compose up` fails with a clear error if `PUBLIC_URL` is not set or empty.
- [ ] `.env.example` contains a comment warning that `PUBLIC_URL` must be externally reachable in production.
- [ ] Backend logs a warning at startup if `PUBLIC_URL` contains `localhost` or `127.0.0.1`.
- [ ] Webhook registration uses the correct external URL in a Docker deployment with `DOMAIN` set.
