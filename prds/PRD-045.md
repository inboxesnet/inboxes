# PRD-045: Duplicate Threads

## Status
Draft

## Priority
P1-High

## Category
Race Conditions

## Source
GAP-FINDER R3

---

## Problem Statement
When two draft sends fire simultaneously without a thread ID, both create new threads with identical subjects and participants. The thread creation in `drafts.go:368-384` and `emails.go:210-226` uses a simple INSERT without any check-then-insert guard or unique constraint. This can result in duplicate threads for the same conversation, confusing users and splitting email history.

## Current Behavior
- In `backend/internal/handler/drafts.go:368-384`, when `threadID` is nil or empty, the handler creates a new thread:
  ```go
  if threadID != nil && *threadID != "" {
      finalThreadID = *threadID
  } else {
      // INSERT INTO threads ... RETURNING id
  }
  ```
- In `backend/internal/handler/emails.go:210-226`, the same pattern exists for direct sends when `req.ReplyTo` is empty.
- Both handlers wrap their operations in a transaction (`tx`), but the thread INSERT has no uniqueness guard on (org_id, subject, participant_emails).
- Scenario: User clicks "Send" twice quickly. The first request creates thread A. The second request, arriving before the first completes (or after the first completes but before the client receives the response and sets the thread ID), also creates thread B with the same subject and participants.
- This is related to but distinct from the draft double-send issue (PRD-029). PRD-029 is about idempotency of the send itself; this PRD is about the thread creation race.

## Expected Behavior
- Concurrent sends for the same draft should not create duplicate threads.
- If two sends race for the same conversation context, only one thread should be created.
- The second send should either reuse the first thread or be rejected.

## Files to Modify
- `backend/internal/handler/drafts.go:368-384` -- Add deduplication logic for thread creation
- `backend/internal/handler/emails.go:210-226` -- Same deduplication logic
- Potentially `backend/internal/db/migrations/` -- Add a unique partial index or advisory lock

## Impact Analysis

### What This Affects
- New thread creation for outbound emails (compose new)
- Does NOT affect replies (which already have a thread ID)
- Email history coherence -- duplicate threads split conversation history

### What Affects This
- The draft double-send race (PRD-029) -- fixing double-send at the draft level would also prevent most duplicate thread creation
- The inbound email threading logic in `process_fetch.go` which has its own thread-matching logic
- Database constraints on the threads table

### Related PRDs
- PRD-029 (Double-Send) -- preventing double-send would prevent most duplicate thread creation; the two fixes are complementary
- PRD-044 (Label Transactions) -- transactions in thread creation should encompass the full create-thread + create-email + create-job sequence
- PRD-028 (Duplicate Drafts) -- similar race condition pattern with draft auto-save

## Implementation Notes
- Option A: Draft-level idempotency. When sending a draft, mark it as "sending" in the database before creating the thread. A second send of the same draft would see the "sending" status and either wait or reject. This requires adding a `status` column to drafts.
  ```sql
  UPDATE drafts SET status = 'sending' WHERE id = $1 AND user_id = $2 AND status = 'draft'
  RETURNING id
  ```
  If zero rows affected, the draft is already being sent.

- Option B: Thread deduplication at creation. Use an advisory lock keyed on (org_id, subject hash) to serialize concurrent thread creation:
  ```sql
  SELECT pg_advisory_xact_lock(hashtext($1 || $2))
  ```
  Then check for a recently-created thread with the same subject and participants before inserting.

- Option C: Frontend prevention. Disable the send button immediately on click and prevent re-submission. This is the simplest approach but doesn't protect against concurrent browser tabs.

- Recommended approach: Combine Option A (backend) with Option C (frontend). The draft status approach is the most robust because it makes the send operation idempotent at the source.

## User Notes
- Users should not encounter duplicate threads for the same email.
- If a duplicate thread is somehow created, it will appear as a separate conversation in the thread list.

## Acceptance Criteria
- [ ] Sending the same draft twice simultaneously does not create two threads
- [ ] The second concurrent send either reuses the first thread or returns an error/redirect to the first
- [ ] Direct sends (non-draft) with the same subject/recipients don't create duplicates within a short window
- [ ] Normal compose-and-send flow is unaffected
- [ ] Reply sends (which have a thread ID) are unaffected
- [ ] No orphaned email records (every email belongs to exactly one thread)
- [ ] The fix works across concurrent browser tabs, not just double-clicks
