# PRD-012: BCC Privacy Leak via participant_emails

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S10

---

## Problem Statement
BCC (blind carbon copy) addresses are added to the thread's `participant_emails` JSON field, which is returned to the frontend and visible to all thread participants. This defeats the entire purpose of BCC — recipients can infer who was BCC'd by comparing the participant list with the To/Cc headers. This is a privacy violation that could expose sensitive business relationships or leak confidential recipients.

## Current Behavior
In `backend/internal/queue/process_fetch.go`, during inbound email processing:
- Line 200-203: BCC addresses are marshaled into JSON:
  ```go
  toJSON, _ := json.Marshal(emailData.To)
  ccJSON, _ := json.Marshal(emailData.CC)
  bccJSON, _ := json.Marshal(emailData.BCC)
  ```
- Line 280: When creating a new thread, participants include the From and To addresses:
  ```go
  participants, _ := json.Marshal(append([]string{emailData.From}, emailData.To...))
  ```
  - This does NOT include BCC. Good.
- Line 316-326: When updating an existing thread, participant_emails is merged:
  ```go
  UPDATE threads SET ... participant_emails = (
      SELECT jsonb_agg(DISTINCT val) FROM (
          SELECT jsonb_array_elements(participant_emails) AS val
          UNION
          SELECT jsonb_array_elements($3::jsonb) AS val
      ) sub
  ) WHERE id = $1`, threadID, snippet, toJSON,
  ```
  - This merges `toJSON` (To addresses only) into the existing participant list. Good — BCC not included here.
- Line 346-360: Alias label stamping includes BCC addresses:
  ```go
  for _, addr := range append(append(emailData.To, emailData.CC...), emailData.BCC...) {
  ```
  - Comment on line 347: `// (BCC included for alias visibility, but NOT added to participant_emails for privacy)`
  - This is correct — BCC is used for alias routing labels but not participant_emails.

**However**, the email record itself stores BCC addresses:
- Line 297-311: The email INSERT includes `bcc_addresses`:
  ```go
  INSERT INTO emails (..., bcc_addresses, ...) VALUES (..., $10, ...)
  ```
- The `bcc_addresses` field is stored on the email record and could be exposed to the frontend when loading thread emails.

**The actual leak path**: When the frontend loads a thread via `GET /api/threads/{id}`, the response includes emails with their `bcc_addresses` field. Any participant viewing the thread can see who was BCC'd on any message.

## Expected Behavior
- BCC addresses should NEVER be returned to the frontend in thread/email responses.
- They should remain stored in the database for audit/delivery purposes but be stripped from API responses.
- The `participant_emails` on threads should never include BCC addresses (currently correct, but needs to stay that way).
- When returning email objects, the `bcc_addresses` field should be omitted or filtered to only show the viewer's own address if they were BCC'd.

## Files to Modify
- `backend/internal/handler/threads.go` — In the `Get` handler (which returns thread + emails), strip `bcc_addresses` from each email before returning:
  - Option A: Don't SELECT `bcc_addresses` in the query.
  - Option B: Set `bcc_addresses` to empty array in the response map.
- `backend/internal/handler/emails.go` — In the `Search` handler, similarly strip BCC addresses from results.
- `backend/internal/queue/process_fetch.go:280` — Verify that BCC addresses are NOT included in the participants list when creating new threads (currently correct — only From + To).
- `backend/internal/queue/process_fetch.go:316-326` — Verify that the participant merge only uses To addresses (currently correct — uses `toJSON`).

## Impact Analysis

### What This Affects
- Thread view in the frontend — BCC addresses will no longer appear.
- Email search results — BCC addresses will no longer appear.
- Stored data remains unchanged (BCC still in DB for audit).
- Outbound emails — when a user sends with BCC, the BCC field is sent to Resend API but should not be stored visibly.

### What Affects This
- The outbound send flow (`queue/process_send.go`) — BCC addresses for outbound emails should also be handled carefully. They're passed to the Resend API for delivery but should not be stored in a way that other thread participants can see.
- Frontend rendering in `thread-view.tsx:440-443` — currently shows To and Cc but not BCC (no BCC rendering code exists), but the data is in the API response.

### Related PRDs
- PRD-013: Tracking Pixels — Both relate to privacy concerns in email handling.
- PRD-014: Malicious CSS — Part of the broader "untrusted email content" security surface.

## Implementation Notes
- The safest approach is to never SELECT `bcc_addresses` in API-facing queries. Internal processing queries (webhook handler, email worker) can still access it.
- For the thread detail endpoint, the email query likely selects all columns. Change it to explicitly list columns, excluding `bcc_addresses`.
- If BCC display is ever needed (e.g., sender sees their own BCC), add logic to only show BCC to the email's sender:
  ```go
  if email.direction == "outbound" && email.user_id == claims.UserID {
      // Show BCC to the sender
  } else {
      email.bcc_addresses = []
  }
  ```
- For MVP, simply omit BCC from all API responses. Add sender-visible BCC later if needed.
