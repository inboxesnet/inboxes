# PRD-034: Pagination Controls

## Status
Done

## Priority
P2-Medium

## Category
Search & Navigation

## Source
PRE-LAUNCH-AUDIT #20

---

## Problem Statement
The thread list has functional pagination infrastructure (page state, limit constant, page change handler, and a toolbar that shows "1-100 of 350" with prev/next arrows), but the pagination controls are not prominently visible and may be overlooked by users. The controls are small text in the toolbar area, and there are no bottom-of-list pagination indicators, no "load more" prompt, and no "Page X of Y" label. Users who scroll to the bottom of a 100-thread page may think they have seen all their emails.

## Current Behavior
Pagination is implemented in `thread-toolbar.tsx` (lines 322-343):
1. The toolbar shows `{start}-{end} of {total}` in small text (line 326).
2. Previous/next page buttons are ChevronLeft/ChevronRight icons (lines 328-341).
3. The buttons correctly disable at the first and last pages.
4. The LIMIT is 100 threads per page (`thread-list-page.tsx`, line 19).

However:
- The pagination controls are only in the toolbar at the top -- there is nothing at the bottom of the list to indicate more pages exist.
- The text "1-100 of 350" is very small (`text-xs text-muted-foreground`) and easy to miss.
- There is no "Page 1 of 4" indicator.
- On mobile, the toolbar may be cramped, making the pagination controls even harder to find.
- Search results bypass pagination entirely -- `searchResults.length` is used directly with no pagination.

## Expected Behavior
1. Keep the existing top-of-toolbar pagination (it works well for power users).
2. Add a bottom-of-list pagination indicator: when the user scrolls to the end of the list, show a clear "Page 1 of 4 -- Next page" prompt.
3. Consider adding "Page X of Y" text next to the range display in the toolbar.
4. Ensure mobile layout has visible, tappable pagination controls.

## Files to Modify
- `frontend/components/thread-toolbar.tsx:322-343` -- Enhance the pagination display to include "Page X of Y" alongside the existing range text.
- `frontend/components/thread-list-page.tsx:275-296` -- Add a bottom-of-list pagination component after the ThreadList. Show "Page X of Y" with Previous/Next buttons when `totalPages > 1`.
- `frontend/components/thread-list-page.tsx:19` -- Consider whether 100 is the right page size. Most email clients use 25-50 for better perceived performance.

## Impact Analysis

### What This Affects
- Users with large inboxes (100+ threads) who need to navigate to older threads.
- Discovery of pagination -- users who do not notice the top toolbar controls may think the app only shows 100 threads total.

### What Affects This
- The LIMIT constant (100) -- determines when pagination kicks in.
- Total thread count returned by the API -- drives the page count calculation.
- Search mode -- search results are currently unpaginated.

### Related PRDs
- PRD-033 (Scroll Position Lost) -- pagination page changes should reset scroll to top
- PRD-124 (Select All Across Pages) -- pagination is a prerequisite for cross-page selection

## Implementation Notes
- **Bottom pagination component**:
  ```tsx
  {totalPages > 1 && (
    <div className="flex items-center justify-center gap-3 py-3 border-t text-sm text-muted-foreground">
      <button
        onClick={() => onPageChange(page - 1)}
        disabled={page <= 1}
        className="px-3 py-1 rounded hover:bg-muted disabled:opacity-30"
      >
        Previous
      </button>
      <span>Page {page} of {totalPages}</span>
      <button
        onClick={() => onPageChange(page + 1)}
        disabled={page >= totalPages}
        className="px-3 py-1 rounded hover:bg-muted disabled:opacity-30"
      >
        Next
      </button>
    </div>
  )}
  ```
- Place this component after the `<ThreadList>` component inside the scrollable container, so it appears at the bottom of the list.
- The toolbar range text can be enhanced: `{start}-{end} of {total} (page {page}/{totalPages})`.
- Consider making the bottom pagination sticky at the bottom of the viewport if the user is at the end of the scroll container.
- Do not add infinite scroll -- explicit pagination is better for email clients because it provides clear boundaries and works well with keyboard navigation.

## User Notes
- Users with 100+ threads in any folder will see pagination controls.
- The toolbar already shows pagination -- this PRD makes it more prominent with a bottom indicator.

## Acceptance Criteria
- [ ] When a folder has more threads than the page limit, pagination controls are visible at the bottom of the thread list
- [ ] The bottom pagination shows "Page X of Y" with Previous/Next buttons
- [ ] The toolbar pagination range is still shown at the top
- [ ] Previous button is disabled on page 1; Next button is disabled on the last page
- [ ] Clicking Next/Previous updates the thread list and scrolls to the top
- [ ] Pagination controls are not shown when the total is less than or equal to the page limit
- [ ] Mobile layout has tappable pagination buttons with adequate touch targets (min 44x44px)
