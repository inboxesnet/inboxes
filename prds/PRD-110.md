# PRD-110: WebSocket Localhost Fallback in Production

## Status
Draft

## Priority
P2-Medium

## Category
Configuration & Environment

## Source
GAP-FINDER C5

---

## Problem Statement
The frontend WebSocket URL derivation function in `notification-context.tsx` falls back to `ws://localhost:8080` when neither `NEXT_PUBLIC_WS_URL` is set nor `window` is available (server-side rendering). In a misconfigured production deployment, this fallback causes the browser to attempt WebSocket connections to `localhost:8080` instead of the production backend, leaking connection attempts and causing silent real-time update failures.

## Current Behavior
In `frontend/contexts/notification-context.tsx:17-24`, the `getWsUrl()` function:
1. Returns `NEXT_PUBLIC_WS_URL` if set (line 18) -- correct for Docker/Coolify deployments
2. If in browser (`typeof window !== "undefined"`), derives from page origin (line 20-21) -- correct for same-origin deployments behind a reverse proxy
3. Falls back to `ws://localhost:8080` (line 23) -- problematic in production SSR or misconfigured deployments

The fallback at line 23 is reached during Next.js server-side rendering (where `window` is undefined) and when `NEXT_PUBLIC_WS_URL` is not set. While the SSR case is harmless (WebSocket is only opened client-side), the fallback string is used as the initial value and could be evaluated before the client-side check runs.

In `docker-compose.yml:53`, `NEXT_PUBLIC_WS_URL` is set to `wss://${DOMAIN}` -- this is correct for Docker deployments. But bare-metal deployments may forget to set this variable.

## Expected Behavior
- The fallback should not be `ws://localhost:8080` -- it should either:
  - Return an empty string and skip WebSocket connection until a valid URL is available
  - Throw/log a clear error when no valid WebSocket URL can be determined in production
  - Always derive from `window.location` on the client side (no hardcoded fallback needed)
- The SSR path should return `null` or empty string, and the WebSocket should only connect after client-side hydration

## Files to Modify
- `frontend/contexts/notification-context.tsx:17-24` -- Replace localhost fallback with safe behavior
- `frontend/contexts/notification-context.tsx:73-78` -- Add guard to skip WebSocket connection when URL is null/empty

## Implementation Notes
- Simplest fix: remove the `ws://localhost:8080` fallback and return empty string:
  ```tsx
  function getWsUrl(): string {
    if (process.env.NEXT_PUBLIC_WS_URL) return process.env.NEXT_PUBLIC_WS_URL;
    if (typeof window !== "undefined") {
      const proto = window.location.protocol === "https:" ? "wss:" : "ws:";
      return `${proto}//${window.location.host}`;
    }
    return ""; // SSR -- will be set on client-side hydration
  }
  ```
- Guard the WebSocket connection:
  ```tsx
  const connect = useCallback(() => {
    const url = getWsUrl();
    if (!url) return; // No URL available yet (SSR)
    // ... rest of connection logic
  }, []);
  ```
- The `window.location` derivation (line 20-21) is actually the correct behavior for most deployments -- the issue is only the dead-code fallback for SSR
- Since the `NotificationProvider` is rendered inside a client component tree (layout.tsx), `window` will always be defined by the time `connect()` runs in `useEffect`
- Consider adding a console warning in development when the fallback is hit:
  ```tsx
  if (process.env.NODE_ENV === "development") {
    console.warn("WebSocket URL not configured. Set NEXT_PUBLIC_WS_URL for production.");
  }
  ```

## User Notes
This is a deployment safety fix. The localhost fallback could cause confusing behavior in production where WebSocket connections silently fail.

## Acceptance Criteria
- [ ] The `ws://localhost:8080` hardcoded fallback is removed from `getWsUrl()`
- [ ] SSR returns empty string or null, preventing server-side WebSocket connection attempts
- [ ] Client-side derivation from `window.location` continues to work correctly
- [ ] `NEXT_PUBLIC_WS_URL` env var continues to work when explicitly set
- [ ] WebSocket connection is skipped gracefully when URL is empty (no errors thrown)
- [ ] A development-mode console warning appears when no explicit WebSocket URL is configured
- [ ] Production deployments behind a reverse proxy (same origin) work without `NEXT_PUBLIC_WS_URL`
