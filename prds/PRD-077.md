# PRD-077: GDPR Hard Delete

## Status
Done

## Priority
P2-Medium

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L6

---

## Problem Statement
The application only supports soft-delete for users and orgs. Personal data (emails, names, IP addresses in logs, email content) persists indefinitely in the database. Email addresses remain locked by UNIQUE constraints even after soft-deletion, preventing re-invitation. There is no mechanism for GDPR Article 17 "Right to Erasure" compliance, which requires the ability to permanently remove personal data upon request.

## Current Behavior
In `backend/internal/handler/users.go`:
- Lines 184-237: User disable sets `status = 'disabled'` but all personal data remains: `email`, `name`, `password_hash`, `notification_preferences`.
- The `email` column has a global UNIQUE constraint (migration 001, line 30), so a disabled user's email address cannot be used for a new invitation.

In `backend/internal/handler/orgs.go`:
- Lines 119-185: Org deletion sets `deleted_at = now()` on the org, disables all users, and marks domains as deleted. But all org data remains in the database: threads, emails, aliases, discovered addresses, attachments, sync jobs, email jobs.

Database tables with personal data that would need hard-delete support:
- `users`: email, name, password_hash, notification_preferences
- `emails`: from_address, to_addresses, cc_addresses, bcc_addresses, body_html, body_plain, headers
- `threads`: participant_emails, subject, snippet
- `attachments`: data (BYTEA), filename
- `discovered_addresses`: address, local_part
- `alias_users`: junction records
- `aliases`: address, name

## Expected Behavior
- Add a hard-delete endpoint (admin-only or owner-only) that permanently removes all personal data for a user or an entire org.
- For user hard-delete:
  1. Anonymize the user record: replace `email` with a hash, clear `name`, clear `password_hash`, clear `notification_preferences`.
  2. Reassign or anonymize their threads and emails (replace sender/recipient addresses with anonymized values or delete content).
  3. Remove `alias_users` and optionally anonymize `aliases`.
  4. Remove attachments uploaded by the user.
- For org hard-delete:
  1. Delete all threads, emails, and attachments.
  2. Delete all aliases and discovered addresses.
  3. Delete all users (or anonymize them).
  4. Delete the org record itself.
- Free the email UNIQUE constraint so the address can be reused.
- Provide a confirmation step requiring the admin to type the org name or user email.

## Impact Analysis

### What This Affects
- Data retention and GDPR compliance.
- Database storage: hard-deleting orgs with many emails and attachments frees significant storage.
- The UNIQUE constraint on `users.email` will no longer permanently lock deleted email addresses.
- Audit trail: hard-delete removes evidence, so consider keeping a minimal audit log entry ("user X data purged on DATE by admin Y").

### What Affects This
- The soft-delete pattern used throughout the application -- hard-delete is a separate, destructive operation.
- Cascade delete relationships: the schema uses `ON DELETE CASCADE` for `emails` (from `threads`), but not for most other tables.
- PRD-072 (Domain Deletion Cleanup) should run before or as part of hard-delete.
- PRD-074 (Domain Uniqueness) -- domain records also need cleanup.

### Related PRDs
- PRD-072: Domain Deletion Data Cleanup — Soft-delete cleanup is a prerequisite for hard-delete.
- PRD-076: Disabled Users' Aliases — Alias cleanup patterns overlap.
- PRD-121: Cascade Deletes — Foreign key cleanup is essential for hard-delete.
- PRD-074: Domain Uniqueness — Hard-deleting domains should free the uniqueness constraint.
- PRD-122: BYTEA Attachments — Attachments are stored in DB; hard-delete frees significant storage.

## Implementation Notes
- Hard-delete is a dangerous, irreversible operation. Require explicit confirmation (e.g., request body must include `{"confirm": "DELETE my-org-name"}`).
- Consider a two-phase approach: first mark for deletion with a 30-day grace period, then actually delete after the grace period. This prevents accidental data loss.
- For user anonymization: replace `email` with `deleted-{uuid}@anonymized.local`, `name` with `[Deleted User]`, clear `password_hash` and `notification_preferences`.
- Run the entire hard-delete operation in a transaction to prevent partial deletion.
- Log the hard-delete action to a separate audit table that does NOT contain personal data.
- For the initial implementation, focus on org-level hard-delete (which includes all users). Individual user hard-delete can follow.
