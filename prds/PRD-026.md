# PRD-026: Password+Redis Failure

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
GAP-FINDER S14

---

## Problem Statement
When a user changes their password, the system revokes all existing tokens via Redis and issues a fresh token for the current session. However, if Redis is unavailable at that moment, the token revocation silently fails. The user is told their password was changed successfully (and other sessions were invalidated), but in reality all old sessions remain valid for up to 7 days. This is especially dangerous because the user may be changing their password precisely because they suspect their account is compromised — the very scenario where token revocation matters most.

## Current Behavior
In `backend/internal/handler/users.go:388-441`, the password change flow proceeds as follows:

```go
func (h *UserHandler) UpdatePassword(w http.ResponseWriter, r *http.Request) {
    claims := middleware.GetCurrentUser(r.Context())
    ...
    // Verify current password (lines 406-416)
    if bcrypt.CompareHashAndPassword([]byte(hash), []byte(req.CurrentPassword)) != nil {
        writeError(w, http.StatusUnauthorized, "current password is incorrect")
        return
    }

    // Update password in database (lines 424-429)
    if _, err := h.DB.Exec(r.Context(),
        `UPDATE users SET password_hash = $1, updated_at = now() WHERE id = $2`,
        string(newHash), claims.UserID); err != nil {
        writeError(w, http.StatusInternalServerError, "failed to update password")
        return
    }

    // Revoke all old tokens (lines 431-433)
    pwBlacklist := service.NewTokenBlacklist(h.RDB)
    pwBlacklist.RevokeAllForUser(r.Context(), claims.UserID)

    // Issue fresh token (lines 435-438)
    newToken, err := middleware.GenerateToken(h.Secret, claims.UserID, claims.OrgID, claims.Role)
    if err == nil {
        middleware.SetTokenCookie(w, newToken, h.AppURL)
    }

    w.WriteHeader(http.StatusNoContent)
}
```

The `RevokeAllForUser` method in `backend/internal/service/token_blacklist.go:39-46` silently returns when Redis is unavailable:

```go
func (b *TokenBlacklist) RevokeAllForUser(ctx context.Context, userID string) {
    if b == nil || b.rdb == nil {
        return  // silent return
    }
    if err := b.rdb.Set(ctx, "token:rv:"+userID, time.Now().Unix(), 7*24*time.Hour).Err(); err != nil {
        slog.Warn("token_blacklist: failed to revoke all for user", "user_id", userID, "error", err)
    }
}
```

The handler returns `204 No Content` regardless of whether token revocation succeeded. There is no indication to the user that session invalidation failed.

The same pattern exists in:
- `backend/internal/handler/auth.go:397-398` — Password reset also calls `RevokeAllForUser`
- `backend/internal/handler/users.go:206,215,229` — User disable also calls `RevokeAllForUser`

## Expected Behavior
1. `RevokeAllForUser` should return an error instead of silently ignoring failures.
2. The password change handler should check the error from `RevokeAllForUser`.
3. If revocation fails, the handler should still complete the password change (the password is already updated in the database) but include a warning in the response body indicating that other sessions may not have been invalidated.
4. The frontend should display this warning to the user so they know to manually log out from other devices.
5. Log the failure at Error level for monitoring.

## Files to Modify
- `backend/internal/service/token_blacklist.go:39-46` — Change `RevokeAllForUser` to return `error`
- `backend/internal/handler/users.go:431-440` — Check error from `RevokeAllForUser` and include warning in response
- `backend/internal/handler/auth.go:397-398` — Check error from `RevokeAllForUser` in password reset
- `backend/internal/handler/users.go:206,215,229` — Check error from `RevokeAllForUser` in user disable

## Impact Analysis

### What This Affects
- Password change response — may include a warning field when session revocation fails
- Password reset flow — same warning pattern
- User disable flow — admin is informed if the disabled user's sessions may still be active
- Frontend error handling — needs to check for and display the session revocation warning

### What Affects This
- Redis availability — this issue only manifests when Redis is down
- PRD-019 (Token Blacklist Fail-Open) — this is a specific instance of the broader fail-open problem
- `RevokeToken` (single token revocation for logout) has the same silent failure pattern

### Related PRDs
- PRD-019: Token Blacklist Fail-Open — The root cause; this PRD addresses the specific user-facing scenario of password change
- PRD-017: Redis Auth — Unsecured Redis could be deliberately disrupted to exploit this vulnerability

- PRD-022: Change-Password Rate Limit — The password change endpoint needs both rate limiting and reliable revocation

## Implementation Notes
The change to `RevokeAllForUser` return signature requires updating all call sites. Here is the recommended approach:

1. Change signature:
```go
func (b *TokenBlacklist) RevokeAllForUser(ctx context.Context, userID string) error {
    if b == nil || b.rdb == nil {
        return fmt.Errorf("redis client not available")
    }
    if err := b.rdb.Set(ctx, "token:rv:"+userID, time.Now().Unix(), 7*24*time.Hour).Err(); err != nil {
        slog.Error("token_blacklist: failed to revoke all for user", "user_id", userID, "error", err)
        return err
    }
    return nil
}
```

2. In the password change handler, change the response from `204 No Content` to a JSON response when revocation fails:
```go
if err := pwBlacklist.RevokeAllForUser(r.Context(), claims.UserID); err != nil {
    slog.Error("users: session revocation failed during password change", "user_id", claims.UserID, "error", err)
    writeJSON(w, http.StatusOK, map[string]interface{}{
        "warning": "password changed but other sessions may still be active",
    })
    return
}
w.WriteHeader(http.StatusNoContent)
```

3. Similarly, update `RevokeToken` to return an error for consistency.

4. The frontend should check for the `warning` field in the password change response and display it as a toast notification.

Note: The password change should NOT fail if revocation fails. The password is already changed in the database — rolling that back would be worse than having stale sessions. The approach is to succeed with a warning.
