# PRD-039: Password Rules

## Status

## Priority
P2-Medium

## Category

## Problem Statement
The frontend auth forms only enforce `minLength={8}` on password fields via the HTML attribute. The backend requires passwords to be 8-128 characters with at least one uppercase letter, one lowercase letter, and one digit (enforced in `validatePassword()` at `helpers.go:44-63`). Users entering passwords like "password" or "simple12" pass frontend validation but are rejected by the backend with a confusing error.

## Current Behavior
- In `frontend/app/(auth)/signup/page.tsx:169`, the password input has `minLength={8}` but no client-side complexity validation.
- In `frontend/app/(auth)/reset-password/page.tsx:82,98`, same issue on both new-password fields.
- In `frontend/app/(auth)/claim/page.tsx:140`, same issue on the claim form password.
- In `frontend/app/(auth)/setup/page.tsx:327`, same issue on the setup form password.
- The signup page at line 172-174 shows a hint "Must include uppercase, lowercase, and a number", but this is purely informational -- no JavaScript validation runs before submission.
- `backend/internal/handler/helpers.go:44-63` defines `validatePassword()` which requires `hasUpper`, `hasLower`, and `hasDigit`. On failure, it returns: "password must be 8-128 characters with at least one uppercase letter, one lowercase letter, and one digit".
- When a user submits a weak password, the form POSTs to the backend, which returns a 400 error. The error message appears in the form's error div, but only after a round trip.

## Expected Behavior
- All password fields should validate complexity rules client-side before submission.
- A shared `validatePassword()` utility function should enforce: minimum 8 characters, maximum 128 characters, at least one uppercase letter, at least one lowercase letter, at least one digit.
- Validation should run on form submit (not on every keystroke) to avoid annoying the user mid-typing.
- The error message should match the backend's message for consistency.

## Files to Modify

- `frontend/app/(auth)/signup/page.tsx:46-76` -- Call `validatePassword()` before API call in `handleSubmit`

## Implementation Notes
- Create a shared utility function in `frontend/lib/utils.ts`:
  ```typescript
  export function validatePassword(password: string): string | null {
    if (password.length < 8 || password.length > 128) {
      return "Password must be 8-128 characters with at least one uppercase letter, one lowercase letter, and one digit";
    }
    if (!/[A-Z]/.test(password)) return "Password must include at least one uppercase letter";
    if (!/[a-z]/.test(password)) return "Password must include at least one lowercase letter";
    if (!/[0-9]/.test(password)) return "Password must include at least one digit";
    return null;
  }
  ```
- Consider providing granular error messages (e.g., "Missing uppercase letter") rather than the full rule set every time, for better UX.
- The backend validation remains the authoritative check -- client-side validation is a UX improvement only.

## User Notes
- Users will now see immediate feedback when their password does not meet complexity requirements.
- The rules are displayed below the password field on all forms: "Must include uppercase, lowercase, and a number."
