# PRD-002: Content-Disposition Header Injection

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
PRE-LAUNCH-AUDIT #2

---

## Problem Statement
The attachment download endpoint interpolates the filename directly into the `Content-Disposition` HTTP header without sanitizing special characters. A filename containing double quotes, newlines, or non-ASCII characters could break header parsing, enable response splitting, or cause unexpected browser behavior.

## Current Behavior
In `backend/internal/handler/attachments.go`, the `Download` method (line 83-103):
- Line 100: `w.Header().Set("Content-Disposition", fmt.Sprintf(`attachment; filename="%s"`, filename))`
- The `filename` variable comes directly from the database (line 87, scanned from the `attachments` table).
- Filenames are stored as-is during upload (line 68: `header.Filename` used directly in INSERT).
- A filename like `test".html` would produce `attachment; filename="test".html"`, breaking the header.
- A filename with `\r\n` characters could enable HTTP response splitting (header injection).

## Expected Behavior
- Filenames in `Content-Disposition` headers must be sanitized: strip or escape double quotes, remove control characters (CR, LF), and handle non-ASCII via RFC 5987 encoding (`filename*=UTF-8''...`).
- The sanitization should happen at download time (defense-in-depth), and optionally also at upload time.
- The `Content-Type` for downloads should be forced to `application/octet-stream` to prevent serving malicious HTML/SVG inline (see also PRD-020 in future work for MIME bypass).

## Files to Modify
- `backend/internal/handler/attachments.go:100` — Replace the naive `fmt.Sprintf` with a sanitized filename. Implement a `sanitizeFilename()` helper that:
  1. Strips or replaces `"`, `\r`, `\n`, `\0` characters.
  2. Truncates to a reasonable max length (e.g., 255 bytes).
  3. Optionally adds RFC 5987 `filename*` parameter for Unicode support.
- `backend/internal/handler/attachments.go:99` — Force `Content-Type: application/octet-stream` for all downloads as belt-and-suspenders.

## Impact Analysis

### What This Affects
- Attachment downloads via `GET /api/attachments/{id}`.
- Browser download dialog will show the sanitized filename.
- Inbound email attachments stored via `process_fetch.go:497-501` — filenames from external senders are also stored unsanitized.

### What Affects This
- The upload handler (`Upload` on line 19-81) stores the original filename; sanitization at download time is the primary defense.
- `process_fetch.go:458-508` (`downloadAndStoreAttachment`) stores filenames from Resend API responses without sanitization.

### Related PRDs
- PRD-014: Malicious CSS — Both relate to untrusted content handling from external senders.
- PRD-013: Tracking Pixels — Part of the broader "untrusted email content" surface area.

## Implementation Notes
- Use `mime.FormatMediaType` from Go's standard library or build a simple sanitizer:
  ```go
  func sanitizeFilename(name string) string {
      // Remove control characters and quotes
      clean := strings.Map(func(r rune) rune {
          if r < 32 || r == '"' || r == '\\' {
              return '_'
          }
          return r
      }, name)
      if len(clean) > 255 {
          clean = clean[:255]
      }
      if clean == "" {
          clean = "download"
      }
      return clean
  }
  ```
- For full Unicode support, use RFC 5987: `filename*=UTF-8''` + percent-encoded filename. Go's `url.PathEscape` can help.
- Consider sanitizing at upload time too (line 68), but download-time sanitization is the critical path since inbound attachments bypass the upload handler.

## Acceptance Criteria

- [ ] Filenames with CR/LF/null bytes are sanitized (no header injection possible).
- [ ] Non-ASCII filenames are handled via RFC 5987 encoding or stripped to ASCII.
- [ ] Download endpoint sets `Content-Type: application/octet-stream` regardless of stored type.
