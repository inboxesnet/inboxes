# PRD-051: JSON Errors

## Status

## Priority
P1-High

## Category

## Problem Statement
Over 30 `json.Marshal()` calls silently discard errors using the `_` pattern. If marshalling fails (due to unsupported types, cyclic references, or corrupt data), the resulting `nil` or malformed bytes are stored in the database or sent to external APIs like Resend. This can cause corrupted email records, broken thread participant lists, and failed email deliveries with no diagnostic trail.

## Current Behavior
The following `json.Marshal()` calls ignore errors:

**emails.go:**
- Line 180: `resendPayloadJSON, _ := json.Marshal(resendPayload)` — Resend API payload for send job
- Line 215: `participants, _ := json.Marshal(...)` — thread participant list
- Line 229-232: `toJSON, _ := json.Marshal(req.To)`, `ccJSON`, `bccJSON`, `refsJSON` — email address arrays stored in DB

**drafts.go:**
- Line 130-133: `toJSON, _ := json.Marshal(req.To)`, `ccJSON`, `bccJSON`, `attJSON` — draft address/attachment arrays
- Line 195-225: Multiple `json.Marshal` calls in Update for dynamic fields
- Line 305-307: `json.Unmarshal(toAddr, &to)`, `json.Unmarshal(ccAddr, &cc)`, `json.Unmarshal(bccAddr, &bcc)` — draft Send unmarshal errors ignored
- Line 344: `json.Unmarshal(attachmentIDsRaw, &attachmentIDs)` — attachment IDs unmarshal ignored
- Line 352: `resendPayloadJSON, _ := json.Marshal(resendPayload)` — send job payload
- Line 373: `participants, _ := json.Marshal(...)` — thread creation
- Line 387-389: `toJSON, _ := json.Marshal(to)`, `ccJSON`, `bccJSON` — email insert

**process_fetch.go (queue):**

**helpers.go:**
- Line 18: `json.NewEncoder(w).Encode(data)` in `writeJSON` — HTTP response encoding, error not checked

## Expected Behavior
All `json.Marshal()` calls should check for errors:
1. For data being stored in the database: return an HTTP 500 or mark the job as failed.
2. For data being sent to external APIs (Resend): fail the operation with a clear error message.
3. For HTTP response encoding (`writeJSON`): log the error (client already received headers, so we can't change the status code, but we should at least log it).
4. For `json.Unmarshal()` of data from the database: log a warning and use empty defaults rather than nil.

## Impact Analysis

### What This Affects
- Email records in the database (corrupted JSON columns)
- Resend API calls (malformed payloads causing silent send failures)
- Thread participant lists (empty/corrupt participant_emails column)
- Email job queue (corrupt webhook_data preventing processing)
- HTTP responses (truncated or malformed JSON to frontend)

### What Affects This
- Inbound email data quality (malformed headers, unusual characters)
- User input containing special characters or extremely long values

### Related PRDs
- PRD-049 (Unchecked Queries) — related pattern of ignoring errors
- PRD-050 (Fire-and-Forget Writes) — related pattern of ignoring errors
- PRD-052 (Redis Webhook) — if marshal fails for webhook data, the job cannot be processed later

## Implementation Notes
- For `writeJSON` (helpers.go:18), the error cannot change the HTTP status since `WriteHeader` was already called. Log the error at WARN level.
- For marshal errors in email/draft send flows, these should abort the transaction and return HTTP 500.
- For marshal errors in `process_fetch.go`, mark the job as failed with the marshal error message.
- Consider creating a `mustMarshal` helper that logs and returns an error, to reduce boilerplate while ensuring no error is silently lost.
- The `json.Unmarshal` calls on draft Send (lines 305-307, 344) are especially important: if the draft's address JSON is corrupt, the send should fail with a clear error rather than sending to zero recipients.

## User Notes
Users will not notice a difference in the happy path. In edge cases where JSON encoding fails (e.g., sending an email with unusual characters that somehow break encoding), users will see a clear error message instead of silent data corruption.

## Acceptance Criteria
- [ ] All `json.Marshal()` calls in handler and queue packages check for errors
- [ ] All `json.Unmarshal()` calls on user-facing data check for errors and return descriptive messages
- [ ] `writeJSON` logs encoding errors at WARN level
- [ ] Email send flow aborts if any marshal call fails
- [ ] Fetch worker marks job as failed if marshal/unmarshal errors occur
- [ ] No silent nil bytes are stored in database JSON columns
