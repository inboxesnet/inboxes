## Problem Statement
When a search query fails (network error, server error, timeout), the search results area shows a loading spinner indefinitely. There is no `isError` check on the search query, no error message, and no retry mechanism. The user cannot distinguish between "still loading," "found nothing," and "search is broken."

## Current Behavior
In `frontend/components/thread-list-page.tsx:63-70`:
```tsx
const { data: searchData, isFetching: searchFetching } = useQuery({
    queryKey: queryKeys.search.results(domainId, searchQuery),
    queryFn: () =>
      api.get<{ threads: Thread[] }>(
        `/api/emails/search?q=${encodeURIComponent(searchQuery)}&domain_id=${domainId}`
      ),
    enabled: searchQuery.length > 0,
});
```

The `useQuery` hook provides `isError` and `error` properties, but they are not destructured or used.

In the render section (line 244-249), the search results area only checks `searchFetching`:
```tsx
{searchFetching ? (
    <div className="flex items-center justify-center h-32">
        <Spinner className="h-6 w-6" />
    </div>
) : ...
```

If the query fails, `searchFetching` becomes `false`, but `searchData` remains undefined, so `searchResults` is an empty array. The UI then shows the "empty search results" state — but there is no explicit "No results found" message either (the empty state just shows nothing).

So the actual problem is twofold:
1. Search errors show an empty list instead of an error message.
2. Successful searches with zero results show an empty list with no "No results found" message (this overlaps with FULL-SUMMARY item #31).

## Expected Behavior
1. **Search error**: Show "Search failed. Please try again." with a retry button.
2. **Search success with zero results**: Show "No results found for '[query]'".
3. **Search loading**: Show the spinner (already works).
4. **Search success with results**: Show the results (already works).

## Files to Modify

- `frontend/components/thread-list-page.tsx` — Add "No results found" empty state for zero-result searches

## Implementation Notes
- Destructure `isError` and `error` from the search `useQuery` call.
- Add error handling in the search results section:
  ```tsx
  {isSearching ? (
    isSearchError ? (
      <div className="flex flex-col items-center justify-center h-32 gap-2">
        <p className="text-sm text-destructive">Search failed. Please try again.</p>
        <Button variant="outline" size="sm" onClick={handleRefresh}>Retry</Button>
      </div>
    ) : searchFetching ? (
      <Spinner />
    ) : searchResults.length === 0 ? (
      <div className="flex items-center justify-center h-32">
        <p className="text-sm text-muted-foreground">No results found for &quot;{searchQuery}&quot;</p>
      </div>
    ) : (
      <ThreadList ... />
    )
  ) : ...}
  ```
- The retry mechanism can use `queryClient.invalidateQueries` or the existing `handleRefresh` callback.
- React Query will also automatically retry failed queries (default 3 retries with exponential backoff), so transient failures will self-heal. The error state should only appear after all retries are exhausted.

## User Notes
Currently, a failed search shows an empty list with no message. After this fix, users will see a clear error message with a retry button when search fails, and "No results found" when the search succeeds but returns nothing.
