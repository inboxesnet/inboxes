# PRD-024: Last Admin Protection

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
GAP-FINDER S9

---

## Problem Statement
An admin can disable the only other admin in the organization, leaving a single admin as the only point of failure. If that remaining admin loses access to their account (forgotten password with no system email configured, or their own account is compromised), the entire organization is locked out with no recovery path. There is no check ensuring at least one active admin remains when disabling users or changing roles.

## Current Behavior
In `backend/internal/handler/users.go:184-237`, the Disable handler blocks self-disable but does not check whether the target is the last remaining admin:

```go
func (h *UserHandler) Disable(w http.ResponseWriter, r *http.Request) {
    claims := middleware.GetCurrentUser(r.Context())
    if claims.Role != "admin" {
        writeError(w, http.StatusForbidden, "admin required")
        return
    }

    userID := chi.URLParam(r, "id")
    if userID == claims.UserID {
        writeError(w, http.StatusBadRequest, "cannot disable yourself")
        return
    }
    // ... proceeds to disable without checking remaining admin count
```

In `backend/internal/handler/users.go:513-558`, the ChangeRole handler similarly does not check if demoting an admin would leave the org with fewer than 2 admins:

```go
func (h *UserHandler) ChangeRole(w http.ResponseWriter, r *http.Request) {
    ...
    if userID == claims.UserID {
        writeError(w, http.StatusBadRequest, "cannot change your own role")
        return
    }
    ...
    // Blocks changing owner's role (line 543-546), but does NOT check admin count
    tag, err := h.DB.Exec(r.Context(),
        `UPDATE users SET role = $1, updated_at = now() WHERE id = $2 AND org_id = $3`,
        req.Role, userID, claims.OrgID)
    ...
}
```

The self-disable check on line 192-194 prevents an admin from disabling themselves, but does nothing to prevent Admin A from disabling Admin B when B is the only other admin. Similarly, the owner role check on lines 534-546 prevents changing the owner's role but not reducing the admin count below a safe threshold.

## Expected Behavior
1. Before disabling a user who has the `admin` role, count the number of active admins in the org. If the result would drop below 2, reject the operation with a clear error message.
2. Before demoting an admin to `member`, perform the same check.
3. The owner (marked by `is_owner = true`) should be considered as one of the admins in the count.
4. Return a clear error message: "Cannot disable/demote: at least 2 active admins are required."

## Files to Modify
- `backend/internal/handler/users.go:184-237` — Add admin count check before disabling an admin user
- `backend/internal/handler/users.go:513-558` — Add admin count check before demoting an admin to member

## Impact Analysis

### What This Affects
- Admin user management — prevents accidental lockout by maintaining a minimum admin count
- Org recovery — ensures there is always a backup admin who can manage the org
- The self-hosted owner (marked by `is_owner = true`) is additionally protected by the owner check in ChangeRole, but this PRD adds protection for non-owner admins too

### What Affects This
- User invitation flow — new admins must be invited before existing ones can be removed
- Owner flag — the `is_owner` field in the users table provides an additional layer of protection for the instance owner

### Related PRDs
- PRD-019: Token Blacklist Fail-Open — If the last admin's token revocation fails, combined with this issue, the org could be in an unrecoverable state
- PRD-025: Concurrent Sessions — Unlimited sessions on a single admin account compounds the risk if that admin is compromised

## Implementation Notes
Add a helper function that counts active admins for an org:

```go
func countActiveAdmins(ctx context.Context, db *pgxpool.Pool, orgID string) (int, error) {
    var count int
    err := db.QueryRow(ctx,
        `SELECT COUNT(*) FROM users WHERE org_id = $1 AND role = 'admin' AND status = 'active'`,
        orgID,
    ).Scan(&count)
    return count, err
}
```

Call this in both the Disable and ChangeRole handlers before proceeding:

In Disable:
```go
// Check if target is an admin
var targetRole string
db.QueryRow(ctx, `SELECT role FROM users WHERE id = $1 AND org_id = $2`, userID, claims.OrgID).Scan(&targetRole)
if targetRole == "admin" {
    count, _ := countActiveAdmins(ctx, h.DB, claims.OrgID)
    if count <= 2 { // disabling would leave only 1
        writeError(w, http.StatusBadRequest, "cannot disable: at least 2 active admins required")
        return
    }
}
```

In ChangeRole, when `req.Role == "member"`:
```go
if req.Role == "member" {
    var currentRole string
    db.QueryRow(ctx, ...).Scan(&currentRole)
    if currentRole == "admin" {
        count, _ := countActiveAdmins(ctx, h.DB, claims.OrgID)
        if count <= 2 {
            writeError(w, http.StatusBadRequest, "cannot demote: at least 2 active admins required")
            return
        }
    }
}
```

The threshold of 2 ensures that after the operation, at least 1 admin remains. Consider making the threshold configurable or documenting why 2 was chosen (the caller is also an admin, so 2 admins before = 1 after disable/demote, which is the minimum).

Edge case: the `reassignAndDisable` path (line 209) should also perform this check.
