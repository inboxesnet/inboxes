# PRD-058: deleted_at Index

## Status
Done

## Priority
P2-Medium

## Category
Database Performance

## Source
PRE-LAUNCH-AUDIT #26

---

## Problem Statement
Every thread list query filters on `t.deleted_at IS NULL` to exclude soft-deleted threads. Without an index on this column, PostgreSQL performs a sequential scan on the entire threads table for every thread list request. As the table grows (deleted threads accumulate forever since there is no hard delete), this filter becomes increasingly expensive.

## Current Behavior
All thread list queries in `backend/internal/handler/threads.go` include `AND t.deleted_at IS NULL`:
- Archive query (line 90)
- Trash/spam query (line 107)
- Default label query (line 121)
- Unread count queries (line 959, 976)
- Search query in `emails.go:318`

The trash collector (`backend/internal/worker/trash_collector.go:44`) also filters on `t.deleted_at IS NULL`.

Current indexes on the threads table (from migrations):
- `idx_threads_user_domain_folder` on `(user_id, domain_id, folder)` — legacy, likely unused after label migration
- `idx_threads_domain_folder` on `(domain_id, folder)` — legacy
- `idx_threads_last_message` on `(last_message_at DESC)` — does not include deleted_at
- `idx_threads_user_domain_active` on `(user_id, domain_id)` from migration 018 — does not include deleted_at

There is **no index** that covers the `deleted_at IS NULL` filter.

## Expected Behavior
Add a partial index on `threads(deleted_at)` that covers the vast majority of queries. Since most threads are not deleted (`deleted_at IS NULL`), a partial index is efficient:

```sql
CREATE INDEX idx_threads_not_deleted ON threads(id) WHERE deleted_at IS NULL;
```

Or, for the trash collector which needs `deleted_at IS NOT NULL`:
```sql
CREATE INDEX idx_threads_deleted_at ON threads(deleted_at) WHERE deleted_at IS NOT NULL;
```

Both indexes should be added. The first helps all list queries; the second helps the trash collector and any future cleanup queries.

## Files to Modify

## Impact Analysis

### What This Affects
- All thread list query performance (inbox, sent, archive, trash, spam, starred, search)
- Unread count query performance
- Trash collector query performance
- Overall database load

### What Affects This

- Vacuum frequency (index bloat from frequent soft-deletes)
- Query planner statistics

## Implementation Notes
- Use a new migration file: `032_threads_deleted_index.sql`.
- Two indexes:
  ```sql
  -- +goose Up
  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_threads_not_deleted
    ON threads(org_id, last_message_at DESC) WHERE deleted_at IS NULL;

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_threads_deleted_at
    ON threads(deleted_at) WHERE deleted_at IS NOT NULL;

  -- +goose Down
  DROP INDEX IF EXISTS idx_threads_not_deleted;
  DROP INDEX IF EXISTS idx_threads_deleted_at;
  ```
- The first index (`idx_threads_not_deleted`) combines org_id + last_message_at DESC + the partial condition, effectively covering the most common query pattern. This overlaps with PRD-063 (Composite Index) — if both are implemented, they should be consolidated into a single index.
- Use `CONCURRENTLY` to avoid locking the table during index creation on production.
- Run `EXPLAIN ANALYZE` on the thread list query before and after to confirm the index is used.

## User Notes
Thread list pages will load faster, especially for organizations with many soft-deleted threads. The improvement is proportional to the ratio of deleted to active threads.
