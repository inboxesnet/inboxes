# PRD-064: FK Indexes

## Status
Done

## Priority
P2-Medium

## Category
Database Performance

## Source
GAP-FINDER D7

---

## Problem Statement
Several tables with foreign key columns are missing indexes on those columns. PostgreSQL does not automatically create indexes on foreign key columns (unlike some other databases). Without these indexes, JOIN operations and cascade lookups require sequential scans on the child tables, which degrades performance as tables grow.

## Current Behavior
Tables with foreign key columns that lack indexes:

**email_jobs table** (migration 013):
- `user_id` — FK to users, no index. Used in `WHERE user_id = $1` during user reassignment (users.go:300-303).
- `domain_id` — FK to domains, no index. Used in `markDomainDisconnected` (email_worker.go:332-334).
- Has indexes on `(org_id, status)` and `(status, heartbeat_at)` but none on `user_id` or `domain_id`.

**alias_users table** (migration 001):
- `user_id` — part of a composite key `(alias_id, user_id)`, so the index covers `alias_id` as leading column but NOT `user_id` alone. Queries like `WHERE user_id = $1` (in reassignment at users.go:269-273, 281) must scan the full table.
- No standalone index on `user_id`.

**discovered_addresses table** (migration 001):
- `domain_id` — has `idx_discovered_domain` on `(domain_id)` — this one IS covered.
- `user_id` — FK to users, no index. Used in `WHERE user_id = $1` during user reassignment (users.go:307-310).

**drafts table** (migration 007):
- Has `idx_drafts_user_domain` on `(user_id, domain_id)` — covers user_id queries.
- `org_id` — no standalone index, but most queries include `user_id` which is the leading index column.

**sync_jobs table** (migration 008):
- `user_id` — FK to users, no index. Used in `WHERE user_id = $1 AND org_id = $2` during user reassignment.

Current indexes confirmed from migrations:
- `idx_email_jobs_org_status` on `(org_id, status)` — does not cover user_id or domain_id
- `idx_thread_labels_thread` on `(thread_id, label)` — covers thread_id lookups
- `idx_discovered_domain` on `(domain_id)` — covers domain_id for discovered_addresses

## Expected Behavior
Add B-tree indexes on foreign key columns that are used in queries but lack indexes:
1. `email_jobs(user_id)`
2. `email_jobs(domain_id)`
3. `alias_users(user_id)`
4. `discovered_addresses(user_id)`
5. `sync_jobs(user_id)`

## Impact Analysis

### What This Affects
- User disable/reassignment performance (queries multiple FK tables by user_id)
- Domain disconnect detection (email_jobs by domain_id)
- Alias management (alias_users by user_id)
- Overall JOIN performance across the application

### What Affects This
- Table sizes (email_jobs, alias_users, discovered_addresses grow with usage)
- User management operations frequency
- Index maintenance overhead on INSERT/UPDATE/DELETE

## Implementation Notes
- Migration file:
  ```sql
  -- +goose Up
  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_email_jobs_user_id
    ON email_jobs(user_id);

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_email_jobs_domain_id
    ON email_jobs(domain_id);

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_alias_users_user_id
    ON alias_users(user_id);

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_discovered_addresses_user_id
    ON discovered_addresses(user_id) WHERE user_id IS NOT NULL;

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_sync_jobs_user_id
    ON sync_jobs(user_id);

  -- +goose Down
  DROP INDEX IF EXISTS idx_email_jobs_user_id;
  DROP INDEX IF EXISTS idx_email_jobs_domain_id;
  DROP INDEX IF EXISTS idx_alias_users_user_id;
  DROP INDEX IF EXISTS idx_discovered_addresses_user_id;
  DROP INDEX IF EXISTS idx_sync_jobs_user_id;
  ```

- The `discovered_addresses.user_id` index uses a partial index (`WHERE user_id IS NOT NULL`) because many rows will have NULL user_id (addresses not yet assigned to users).
- These are all simple B-tree indexes with low maintenance overhead.
- Use `CONCURRENTLY` for all index creation to avoid table locks.
- The `alias_users.user_id` index is especially important for the `reassignAndDisable` function (users.go:239-341) which runs `DELETE FROM alias_users WHERE user_id = $1` and `INSERT INTO alias_users ... FROM alias_users WHERE user_id = $2`.
