# PRD-086: Emails After Deletion

## Status
Done

## Priority
P1-High

## Category
Billing & Stripe

## Source
GAP-FINDER B6

---

## Problem Statement
When an org is deleted, already-queued email jobs in the Redis `email:jobs` queue are not drained or discarded. The email worker continues to process these jobs, sending emails on behalf of a deleted org. This can cause emails to be sent from addresses the org no longer controls, create confusion for recipients, and potentially violate user expectations about data deletion.

## Current Behavior
In `backend/internal/handler/orgs.go:119-185`, the `Delete` handler:
1. Cancels the Stripe subscription (line 135-148)
2. Soft-deletes the org: `UPDATE orgs SET deleted_at = now()` (line 151-156)
3. Disables all users (line 158-162)
4. Marks all domains as deleted (line 164-168)
5. Revokes all tokens (line 170-181)

However, it does NOT:
- Remove pending email jobs from the Redis queue (`LREM email:jobs <jobID>`)
- Mark pending email_jobs as `failed` or `cancelled` in the database
- Check org.deleted_at in the email worker before sending

The email worker in `backend/internal/queue/email_worker.go:59-119` processes jobs without checking whether the org still exists. The `processSend` function in `process_send.go:23-101` checks domain status (line 37-47) but not org status.

## Expected Behavior
1. On org deletion, all pending email jobs for that org should be cancelled:
   - Mark `email_jobs` with `status = 'cancelled'` where `org_id = <deleted_org> AND status IN ('pending', 'running')`.
   - Remove these job IDs from the Redis queue.
2. The email worker should check `org.deleted_at IS NULL` before processing any job as a defense-in-depth measure.
3. Emails already in-flight (being sent to Resend API) cannot be recalled, but this is acceptable.

## Files to Modify
- `backend/internal/handler/orgs.go:119-185` -- Add job cancellation logic after soft-deleting the org
- `backend/internal/queue/email_worker.go:59-119` -- Add org existence check in `processJob` before dispatching to send/fetch
- `backend/internal/queue/process_send.go:23-47` -- Add org deleted_at check alongside domain status check

## Impact Analysis

### What This Affects
- Email delivery for deleted orgs (should stop)
- Data consistency between org lifecycle and email queue
- User trust that deleting an org stops all activity

### What Affects This

- Email worker processing speed (determines how many queued emails are already in-flight)
- Org deletion flow in `orgs.go`

## Implementation Notes
- In `orgs.go:Delete`, after soft-deleting the org:
  ```go
  // Cancel all pending email jobs
  rows, err := h.DB.Query(ctx,
      `UPDATE email_jobs SET status='cancelled', error_message='org deleted', updated_at=now()
       WHERE org_id = $1 AND status IN ('pending', 'running')
       RETURNING id`, claims.OrgID)
  if err == nil {
      defer rows.Close()
      for rows.Next() {
          var jobID string
          rows.Scan(&jobID)
          h.RDB.LRem(ctx, "email:jobs", 0, jobID)
      }
  }
  ```
- In `email_worker.go:processJob`, after loading the job, add:
  ```go
  var orgDeletedAt *time.Time
  w.pool.QueryRow(ctx, "SELECT deleted_at FROM orgs WHERE id=$1", orgID).Scan(&orgDeletedAt)
  if orgDeletedAt != nil {
      slog.Warn("email worker: skipping job for deleted org", "job_id", jobID, "org_id", orgID)
      w.pool.Exec(ctx, "UPDATE email_jobs SET status='cancelled', error_message='org deleted' WHERE id=$1", jobID)
      return
  }
  ```

## User Notes
- When an org is deleted, any emails still waiting to be sent will be cancelled immediately.
- Emails that were already in the process of being delivered at the moment of deletion may still go through.
