# PRD-126: Draft Attachments Not Restored When Reopening

## Status
Draft

## Priority
P3-Low

## Category
Missing Features

## Source
FULL-SUMMARY.md #126 / GAP-FINDER "Missing Features" table

---

## Problem Statement
When a user saves a draft with attachments and later reopens it, the attachment IDs are stored in the database but the attachments are not displayed in the compose window. The user sees an empty attachment list and may re-upload files that are already stored, leading to duplicate attachments and a confusing experience.

## Current Behavior
- `backend/internal/handler/drafts.go:36-38` — The draft list query includes `COALESCE(attachment_ids, '[]')` in the SELECT, so attachment IDs are returned in the API response.
- `frontend/lib/types.ts:157` — The `Draft` interface includes `attachment_ids?: string[]`.
- `frontend/contexts/email-window-context.tsx:8-23` — The `ComposeData` interface does NOT include an `attachmentIds` or `attachments` field.
- `frontend/contexts/email-window-context.tsx:47-58` — The `openDraft` function maps draft fields to `ComposeData` but does not map `attachment_ids`:
  ```typescript
  const openDraft = useCallback((draft: Draft) => {
    setComposeData({
      draftId: draft.id,
      subject: draft.subject,
      fromAddress: draft.from_address,
      // ... other fields
      // attachment_ids is NOT included
    });
  }, []);
  ```
- `frontend/components/floating-compose-window.tsx:56` — Attachments state is initialized as empty: `const [attachments, setAttachments] = useState<AttachmentMeta[]>([])`.
- `frontend/components/floating-compose-window.tsx:90-108` — The initialization effect reads from `composeData` but has no attachment restoration logic.

## Expected Behavior
When a draft with attachments is reopened:
1. The `openDraft` function should pass attachment IDs in the `ComposeData`.
2. The compose window initialization should fetch attachment metadata (filename, size, content_type) from the backend for each ID.
3. The attachment list in the compose window should display the previously uploaded attachments with their filenames and sizes.
4. The user should be able to remove or add more attachments as normal.

## Files to Modify
- `frontend/contexts/email-window-context.tsx:8-23` — Add `attachmentIds?: string[]` to the `ComposeData` interface.
- `frontend/contexts/email-window-context.tsx:47-58` — Map `draft.attachment_ids` to `composeData.attachmentIds` in `openDraft`.
- `frontend/components/floating-compose-window.tsx:90-108` — In the initialization effect, if `composeData.attachmentIds` is non-empty, fetch metadata for each attachment and populate the `attachments` state.
- `backend/internal/handler/attachments.go` — Add a metadata endpoint (e.g., `GET /api/attachments/{id}/meta`) that returns filename, size, and content_type without the binary data. Alternatively, add a batch metadata endpoint.
- `backend/internal/router/router.go` — Register the new metadata route.

## Impact Analysis

### What This Affects
- Draft editing workflow — users can now see and manage previously uploaded attachments.
- Compose window load time — reopening a draft with attachments requires additional API calls.
- Attachment deduplication — without restoration, users re-upload the same files, creating duplicates.

### What Affects This
- The `attachments` table structure (PRD-122) — metadata queries should not load the `data` column.
- The `drafts.attachment_ids` JSONB column (migration 028, line 15) — this is the source of truth for which attachments belong to a draft.
- Draft save logic — when saving with attachments, the `attachment_ids` must be preserved.

### Related PRDs
- PRD-122: BYTEA Attachments — metadata queries must avoid loading binary data; long-term move to S3.
- PRD-118: Attachment Download — the download endpoint (currently unused) will also be needed when users want to download draft attachments.
- PRD-028: Duplicate Drafts — race conditions during draft save could affect attachment ID consistency.
- PRD-027: Compose Draft Loss — if the save fails, attachment references may be lost.

## Implementation Notes
Two approaches for fetching attachment metadata:

**Option A: Individual metadata endpoint**
```
GET /api/attachments/{id}/meta → { id, filename, content_type, size }
```
Simple but requires N requests for N attachments. Add to `attachments.go`.

**Option B: Batch metadata endpoint**
```
POST /api/attachments/meta → { ids: ["id1", "id2"] } → [{ id, filename, content_type, size }, ...]
```
Single request regardless of attachment count. More efficient.

In the compose window initialization:
```typescript
useEffect(() => {
  if (composeState === "open" && composeData?.attachmentIds?.length) {
    // Fetch metadata for each attachment ID
    Promise.all(
      composeData.attachmentIds.map(id =>
        api.get<AttachmentMeta>(`/api/attachments/${id}/meta`)
      )
    ).then(setAttachments).catch(() => {
      toast.error("Failed to load draft attachments");
    });
  }
}, [composeState, composeData]);
```

Important: The metadata query must NOT select the `data` column:
```sql
SELECT id, filename, content_type, size FROM attachments WHERE id = $1 AND org_id = $2
```

## Acceptance Criteria
- [ ] Opening a draft with `attachment_ids` displays the attachments in the compose window with filename and size.
- [ ] The metadata endpoint returns attachment info without loading binary data.
- [ ] Users can remove previously uploaded attachments from a reopened draft.
- [ ] Users can add new attachments alongside restored ones.
- [ ] Saving the draft preserves both restored and newly added attachment IDs.
- [ ] If metadata fetch fails, the compose window still opens with an error message (not a blank crash).
- [ ] No regression in the new-compose workflow (compose without draft data still works).
