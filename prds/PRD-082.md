# PRD-082: Paywall Flash

## Status
Done

## Priority
P1-High

## Category
Billing & Stripe

## Source
GAP-FINDER B2

## Problem Statement
After completing Stripe checkout, the user is redirected back to the app with `?billing=success` in the URL. However, the Stripe webhook that updates the org's plan to `pro` may not have arrived yet. For 3-5 seconds, `GET /api/billing` returns `plan: "free"`, causing the `PaymentWall` component to flash, blocking the user from accessing the app right after they just paid.

## Current Behavior
1. User clicks "Upgrade to Pro" in `frontend/components/payment-wall.tsx:26-37`, which calls `POST /api/billing/checkout`.
2. `backend/internal/handler/billing.go:32-119` creates a Stripe checkout session with `successURL` containing `?billing=success`.
3. After payment, Stripe redirects the user to the success URL.
4. The frontend loads and calls `GET /api/billing` (line 167-209 in billing.go), which queries the DB for `plan`.
5. The webhook (`checkout.session.completed`, line 230-252) hasn't arrived yet, so the DB still shows `plan: "free"`.
6. `RequirePlan` middleware in `backend/internal/middleware/auth.go:148-186` returns 402, triggering the `payment-required` event and showing `PaymentWall`.
7. Seconds later, the webhook arrives, updates the plan, and a page refresh fixes it -- but the user already saw a confusing paywall.

## Expected Behavior
When the user returns from Stripe checkout with `?billing=success`:
1. The frontend should optimistically treat the session as having an active plan for a short window (e.g., 30 seconds).
2. The frontend should poll `GET /api/billing` every 2 seconds for up to 30 seconds until the plan shows `pro`.
3. If polling confirms `pro`, dismiss any paywall state.
4. If 30 seconds pass without confirmation, show a message: "Your payment is being processed. Please refresh in a moment."

Alternatively, the backend `Checkout` handler can use `checkout.session.completed` data to immediately update the plan by verifying the session status via the Stripe API before redirecting.

## Files to Modify
- `frontend/components/payment-wall.tsx:26-37` -- Suppress paywall when `?billing=success` is in the URL; add polling logic
- `frontend/app/(app)/d/[domainId]/inbox/page.tsx` -- Detect `billing=success` query param and pass to billing context
- `frontend/lib/api.ts` -- Optionally suppress 402 handling when `billing=success` is active
- `backend/internal/handler/billing.go:32-119` -- Alternative: verify checkout session status with Stripe API and update plan immediately before returning the redirect URL

## Impact Analysis

### What This Affects
- User experience immediately after checkout completion
- First impression of the paid product
- Support requests from confused users who think payment failed

### What Affects This
- Stripe webhook delivery latency (typically 1-5 seconds, can be longer)
- Network conditions between Stripe and the backend
- Frontend page load time after redirect

### Related PRDs
- PRD-080 (Stripe Dedup) -- polling may trigger multiple billing API calls
- PRD-081 (Stripe Event Handlers) -- `checkout.session.completed` handler correctness
- PRD-083 (Double Checkout) -- prevents the confusion that leads to repeated checkout attempts

## Implementation Notes
- Simplest approach: detect `?billing=success` in the URL and poll `/api/billing` with a 2-second interval for up to 30 seconds. During this window, suppress 402 responses.
- Better approach: in `Checkout()`, after creating the session, store the session ID. Add a `GET /api/billing/check-session?session_id=xxx` endpoint that calls `checkoutSession.Get()` on the Stripe API and updates the org plan if the session is complete.
- Clean the `?billing=success` param from the URL after processing (using `window.history.replaceState`).

## User Notes
- Users will no longer see a brief paywall flash after completing checkout.
- If webhook delivery is delayed beyond 30 seconds, users see a clear "processing" message instead of a confusing paywall.

## Acceptance Criteria
- [ ] User returning from Stripe checkout with `?billing=success` does not see the PaymentWall component
- [ ] If plan is confirmed as `pro` within 30 seconds, normal app access resumes without manual refresh
- [ ] If plan is not confirmed within 30 seconds, a user-friendly message is displayed
- [ ] The `?billing=success` query parameter is cleaned from the URL after processing
- [ ] No regression in PaymentWall behavior for users without active plans
