# PRD-028: Duplicate Drafts

## Status
Done

## Priority
P1-High

## Category
Drafts & Compose Issues

## Source
PRE-LAUNCH-AUDIT #18

---

## Problem Statement
When a user types rapidly in the compose window, multiple auto-save requests can fire before the first one returns a draft ID. Since the `draftId` state is still `null` when the second save fires, both requests hit `POST /api/drafts` (create) instead of `PATCH /api/drafts/:id` (update), resulting in duplicate draft rows in the database. The user ends up with orphaned drafts they cannot see or manage.

## Current Behavior
In `floating-compose-window.tsx`, the `saveDraft` callback (lines 196-234) checks `if (draftId)` to decide between create (POST) and update (PATCH). The debounced `scheduleSave` (lines 238-248) uses a 3-second timer and aborts in-flight saves via `saveAbortRef`, but there is a race window: if the first save completes the POST request and the response is in transit, a second save can still see `draftId === null` and fire another POST. The `setDraftId(res.id)` on line 227 updates state asynchronously, leaving a gap where concurrent saves both go through as creates.

## Expected Behavior
Only one `POST /api/drafts` (create) should ever execute per compose session. All subsequent saves should use `PATCH /api/drafts/:id` (update). Concurrent saves should be serialized or deduplicated so no duplicate draft rows are created.

## Files to Modify
- `frontend/components/floating-compose-window.tsx:196-234` -- Add a `savingRef` (useRef boolean) mutex. Before starting a save, check if another save is already in-flight. If so, mark that a retry is needed and return. When the in-flight save completes, check the retry flag and re-trigger if needed.
- `frontend/components/floating-compose-window.tsx:238-248` -- Update `scheduleSave` to respect the mutex. Instead of just aborting the previous request, queue the next save to run after the current one completes.
- `frontend/components/floating-compose-window.tsx:54` -- Consider converting `draftId` from state to a ref (`draftIdRef`) so it is available immediately after the first create succeeds, without waiting for a React re-render cycle.

## Impact Analysis

### What This Affects
- The drafts list page -- duplicate drafts clutter the list and confuse users.
- Storage -- orphaned draft rows waste database space and are never cleaned up.
- The draft send flow -- if the user sends one draft, the duplicate remains and could be sent separately.

### What Affects This
- Typing speed and network latency determine how likely the race condition is to trigger.
- The `scheduleSave` debounce timer (3 seconds) -- shorter timers increase the risk.
- The `saveAbortRef` mechanism -- aborting a request does not prevent the server from creating the draft if the request already reached the backend.

### Related PRDs
- PRD-027 (Compose Draft Loss) -- both involve the save lifecycle in compose
- PRD-029 (Double-Send) -- a similar race condition but on the send path
- PRD-030 (Auto-Save Invisible) -- the save status indicator needs to reflect the mutex state

## Implementation Notes
- Preferred approach: use a `useRef` for draftId in addition to (or instead of) state. Refs update synchronously, closing the race window between POST response and the next save call.
  ```tsx
  const draftIdRef = useRef<string | null>(null);
  // In saveDraft:
  if (draftIdRef.current) {
    await api.patch(...);
  } else {
    const res = await api.post(...);
    draftIdRef.current = res.id;
    setDraftId(res.id); // for UI display
  }
  ```
- Alternative: add a `savePendingRef` boolean to skip concurrent saves entirely. After the in-flight save resolves, check if content changed and re-save if needed.
- Backend defense-in-depth: consider adding a `UNIQUE(user_id, domain_id, thread_id)` constraint on drafts with `kind = 'compose'`, or a deduplication check in the create handler.
- The `saveAbortRef.abort()` on line 243 only aborts the fetch on the client side. If the request reached the server, the draft is already created. The ref-based draftId approach is the real fix.

## User Notes
- Users who type continuously (triggering rapid debounce cycles) on slow connections are most affected.
- Visible symptom: multiple identical drafts in the drafts list.

## Acceptance Criteria
- [ ] Typing rapidly in the compose window for 30+ seconds results in exactly one draft row in the database
- [ ] The `draftId` is set synchronously (via ref) after the first successful create, preventing a second POST
- [ ] Subsequent auto-saves use PATCH, not POST
- [ ] If two save requests fire concurrently, only one creates a new draft
- [ ] The drafts list page shows no duplicate entries after rapid compose editing
- [ ] Save status indicator correctly shows "Saving..." and "Saved" without flickering during rapid edits
