# PRD-125: Browser Notification Permission Never Requested

## Problem Statement
The backend supports desktop notification preferences and the WebSocket infrastructure delivers real-time events, but the frontend never proactively requests browser notification permission. Users must manually navigate to Settings > Account > Notifications and toggle the checkbox to trigger the permission prompt. Most users will never discover this setting, missing out on desktop notifications for new emails.

## Current Behavior
- `frontend/components/settings-modal.tsx:1030-1058` — The Settings modal has a "Notifications" card with a checkbox. When checked, it calls `Notification.requestPermission()` (line 1044) and saves the preference via `api.patch("/api/users/me/preferences", { desktop_notifications: e.target.checked })`.
- `frontend/hooks/use-ws-sync.ts:338-339` — When an inbound email event arrives, the code checks `Notification.permission === "granted"` and creates a `new Notification(title, { body: subject })`. This works correctly but only if the user previously granted permission.
- `frontend/contexts/notification-context.tsx` — The WebSocket connection is established but never triggers a permission request.
- There is no onboarding prompt, first-visit tooltip, or automatic permission request anywhere in the app.

## Expected Behavior
The app should prompt the user to enable desktop notifications at an appropriate moment:
1. **On first login after onboarding**: Show a non-intrusive banner or toast: "Enable desktop notifications to get alerted about new emails?" with "Enable" and "Not now" buttons.
2. **On first email receive**: If notifications are not yet granted and an inbound email arrives via WebSocket, show a one-time prompt suggesting the user enable notifications.
3. **In the settings modal**: Keep the existing toggle as the permanent controls.

The prompt should only appear once (tracked via localStorage or user preferences). If the user dismisses it, do not ask again.

## Files to Modify
- `frontend/hooks/use-ws-sync.ts:338-339` — Add a check that triggers a notification prompt on first inbound email if permission is `"default"` (not yet asked).
- `frontend/components/notification-listener.tsx` — Add a notification permission prompt component that appears as a banner/toast.
- `frontend/contexts/notification-context.tsx` — Optionally track permission state and expose it to components.
- `frontend/app/(app)/d/[domainId]/layout.tsx` — Mount the notification prompt component.

## Impact Analysis

### What This Affects
- User engagement — desktop notifications significantly improve email responsiveness.
- First-run experience — the prompt adds friction but provides value.
- Browser compatibility — `Notification.requestPermission()` returns a Promise in modern browsers.

### What Affects This
- Browser notification policies — browsers require a user gesture to trigger the permission prompt. Calling `requestPermission()` without a gesture (e.g., on page load) is silently blocked in most browsers.
- The existing notification toggle in Settings must remain in sync with browser permission state.
- `Notification.permission` can be `"default"`, `"granted"`, or `"denied"`.

### Related PRDs
- PRD-103: Toasts — toast notifications need `aria-live` regions for accessibility; the notification prompt should also be accessible.
- PRD-035: Session Expiry — session management interacts with notification state.
- PRD-037: Keyboard Shortcuts — discoverability pattern (showing users features they don't know exist).

## Implementation Notes
Due to browser security policies, `Notification.requestPermission()` must be called in response to a user interaction (click/tap). The implementation should:

1. On first login, render a dismissible banner: "Get notified about new emails? [Enable] [Not now]"
2. The "Enable" button calls `Notification.requestPermission()` in the click handler.
3. Store `notification_prompt_dismissed: true` in localStorage to prevent showing again.
4. If the user clicks "Not now", set the localStorage flag and hide the banner.
5. The Settings toggle should reflect the actual `Notification.permission` state.

Handle the `"denied"` state gracefully — if the user previously denied in the browser, the Settings toggle should be disabled with a tooltip explaining how to re-enable in browser settings.
