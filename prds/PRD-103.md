# PRD-103: Toast Notifications in Live Region

## Status
Done

## Priority
P3-Low

## Category
Accessibility

## Source
GAP-FINDER A9

---

## Problem Statement
Toast notifications displayed by the Sonner toaster component are not wrapped in an ARIA live region. Screen readers do not automatically announce toasts when they appear, meaning assistive technology users miss important feedback like "Email sent", "Draft saved", "Failed to archive", etc. These transient notifications disappear after a few seconds, so users who cannot see them may never know they occurred.

## Current Behavior
In `frontend/app/layout.tsx:26`, the `<Toaster>` component from Sonner is rendered with `richColors` and `position="bottom-right"`. The Sonner library may or may not include live region attributes depending on the version. Toast calls throughout the codebase use `toast.success()`, `toast.error()`, and `toast()` from Sonner (imported in 15+ components). The toast container does not have explicit `role="region"`, `aria-live="polite"`, or `role="alert"` attributes visible in the configuration.

## Expected Behavior
- Toast notifications should be announced by screen readers when they appear
- Success/info toasts: `aria-live="polite"` (announced after current speech finishes)
- Error toasts: `aria-live="assertive"` or `role="alert"` (announced immediately)
- The live region should be present in the DOM at all times (not dynamically inserted)

## Files to Modify
- `frontend/app/layout.tsx:26` -- Configure the Sonner `<Toaster>` with ARIA live region attributes (check Sonner API for built-in support)
- If Sonner does not support ARIA natively: create a wrapper component that renders a persistent `aria-live` region and mirrors toast content into it

## Impact Analysis

### What This Affects
- Screen reader users' awareness of all toast notifications
- WCAG 2.1 Level A compliance (Success Criterion 4.1.3: Status Messages)
- User confidence that actions succeeded or failed

### What Affects This

## Implementation Notes
- Check Sonner's documentation for `toastOptions` or `aria` configuration:
  ```tsx
  <Toaster
    richColors
    position="bottom-right"
    toastOptions={{
      role: "status",
      "aria-live": "polite",
    }}
  />
  ```
- If Sonner >= v1.0 already includes `role="status"` on toasts, verify this and document
- For error toasts specifically, consider using `role="alert"` for immediate announcement
- Alternative approach: render a visually hidden `<div aria-live="polite" id="toast-live-region">` in the root layout and use a custom toast handler that also writes text content to this region
- Test with VoiceOver on macOS and NVDA on Windows to confirm announcements
