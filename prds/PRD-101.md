# PRD-101: Accessible Settings Modal Tabs

## Status
Done

## Priority
P3-Low

## Category
Accessibility

## Source
GAP-FINDER A7

---

## Problem Statement
The settings modal uses a custom tab interface without proper WAI-ARIA tab pattern roles. The tab buttons and tab panels have no `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-selected`, or `aria-controls` attributes. Screen readers cannot announce which tab is active, how many tabs exist, or navigate between them using standard tab pattern keyboard shortcuts (arrow keys).

## Current Behavior
In `settings-modal.tsx`, the tab type is defined as a union type at line 26: `type Tab = "profile" | "domains" | "team" | "aliases" | "labels" | "organization" | "billing" | "system" | "jobs"`. The tab buttons are rendered as plain `<button>` elements with conditional styling to indicate the active state. The content for each tab is conditionally rendered based on the `activeTab` state. There are no ARIA attributes relating the tab buttons to their content panels. Keyboard navigation between tabs requires Tab key (moving through all tabs) rather than arrow keys as per ARIA spec.

## Expected Behavior
- Tab button container: `role="tablist"`, `aria-label="Settings"`
- Each tab button: `role="tab"`, `aria-selected="true/false"`, `aria-controls="panel-{tab}"`, `id="tab-{tab}"`, `tabIndex={active ? 0 : -1}`
- Each tab panel: `role="tabpanel"`, `aria-labelledby="tab-{tab}"`, `id="panel-{tab}"`, `tabIndex={0}`
- Arrow keys (Left/Right) should move between tabs within the tablist
- Home/End keys should go to first/last tab

## Files to Modify
- `frontend/components/settings-modal.tsx` -- Add ARIA tablist/tab/tabpanel roles to the tab navigation and content areas
- `frontend/components/settings-modal.tsx` -- Add arrow key navigation handler for tab switching
- `frontend/components/settings-modal.tsx` -- Add `aria-selected`, `aria-controls`, `aria-labelledby`, and `id` attributes

## Impact Analysis

### What This Affects
- Screen reader navigation within the settings modal
- WCAG 2.1 Level A compliance (Success Criterion 4.1.2: Name, Role, Value)
- Keyboard interaction model for settings

### What Affects This
- The Dialog component from shadcn/ui wrapping the modal already provides some accessibility (focus trap, Escape to close)
- Admin-only tabs (system, jobs) are conditionally rendered -- tablist should only include visible tabs
- Tab count changes based on user role -- ARIA attributes must be dynamic

### Related PRDs
- PRD-094 (ARIA Labels) -- button labels within settings tabs
- PRD-095 (Focus Trap) -- modal focus management (already handled by Dialog component)
- PRD-054 (Settings Spinners) -- error states within tab panels

## Implementation Notes
- Follow the [WAI-ARIA Tabs pattern](https://www.w3.org/WAI/ARIA/apg/patterns/tabs/)
- Implement "roving tabindex" for the tab buttons: only the active tab has `tabIndex={0}`, others have `tabIndex={-1}`
- Arrow key handler:
  ```tsx
  function handleTabKeyDown(e: React.KeyboardEvent, tabs: Tab[]) {
    const currentIndex = tabs.indexOf(activeTab);
    if (e.key === "ArrowRight") {
      setActiveTab(tabs[(currentIndex + 1) % tabs.length]);
    } else if (e.key === "ArrowLeft") {
      setActiveTab(tabs[(currentIndex - 1 + tabs.length) % tabs.length]);
    }
  }
  ```
- Tab panels should receive focus when the tab is activated
- Consider using an existing accessible tab component from shadcn/ui if available

## Acceptance Criteria
- [ ] Tab container has `role="tablist"` with `aria-label="Settings"`
- [ ] Each tab button has `role="tab"`, `aria-selected`, and `aria-controls` linking to its panel
- [ ] Each tab panel has `role="tabpanel"` and `aria-labelledby` linking to its tab button
- [ ] Left/Right arrow keys navigate between tabs when a tab is focused
- [ ] Only the active tab has `tabIndex={0}`; inactive tabs have `tabIndex={-1}`
- [ ] VoiceOver announces "tab, 1 of 7, Profile, selected" when navigating to a tab
- [ ] Tab panel content is focusable and receives focus when its tab is activated
