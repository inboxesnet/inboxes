# PRD-046: Star/Unstar Sync

## Status
Draft

## Priority
P1-High

## Category
Race Conditions

## Source
GAP-FINDER R4

---

## Problem Statement
Rapid star/unstar clicks on a thread can receive API responses out of order, causing the frontend's optimistic cache state to diverge from the actual server state. The `useStarThread` mutation in `use-threads.ts:44-121` uses optimistic updates that toggle the label locally, but if multiple mutations fire before the first completes, the final cache state depends on network timing rather than the user's last intended action.

## Current Behavior
- In `frontend/hooks/use-threads.ts:44-121`, `useStarThread()` uses React Query's `onMutate` to optimistically toggle the "starred" label in the cache.
- The toggle function `toggleStarredLabel()` at lines 30-35 checks if "starred" is in the labels array and adds/removes it.
- Scenario: User clicks star (click 1), then immediately clicks unstar (click 2).
  1. Click 1: `onMutate` adds "starred" to cache. API request 1 fires.
  2. Click 2: `onMutate` removes "starred" from cache (correct). API request 2 fires.
  3. API response 1 arrives. The `onError` handler is not triggered (success), so the cache stays with "starred" removed.
  4. API response 2 arrives. Also success.
  5. Final state: cache shows unstarred, server shows unstarred. This works by luck.
- But if responses arrive in reversed order (response 2 before response 1 due to network jitter), or if one fails:
  1. Click 1 fails: `onError` invalidates all queries, refetching from server which now has the state from click 2. This creates a flicker.
  2. The `onError` handler at line 115-119 calls `qc.invalidateQueries()` which refetches all thread lists, causing unnecessary network requests and visual flicker.
- The backend Star handler at `threads.go:700-742` uses a check-then-toggle pattern (`hasLabel` then `addLabel`/`removeLabel`) which is itself a race condition (PRD-044). The backend always toggles relative to its current state, so rapid clicks can result in the server toggling back and forth while the frontend's optimistic updates assume each click toggles from the last optimistic state.

## Expected Behavior
- Rapid star/unstar clicks should always converge to the user's last intended state.
- The frontend should debounce star toggles or use a "set to value" API instead of a "toggle" API.
- The UI should not flicker on error recovery.

## Files to Modify
- `frontend/hooks/use-threads.ts:44-121` -- Debounce star mutations or track the intended final state
- `backend/internal/handler/threads.go:700-742` -- Consider a "set starred" API (PATCH with `{ starred: true/false }`) instead of a toggle
- `backend/internal/router/router.go` -- If changing the API, update the route

## Implementation Notes
- **Option A: Debounce at the frontend.** Delay the star mutation by 300ms. If the user clicks again within that window, cancel the pending mutation and start a new timer. This ensures only the final intended action fires.
  ```typescript
  const starDebounceRef = useRef<Record<string, NodeJS.Timeout>>({});
  function debouncedStar(threadId: string) {
    if (starDebounceRef.current[threadId]) {
      clearTimeout(starDebounceRef.current[threadId]);
    }
    starDebounceRef.current[threadId] = setTimeout(() => {
      starMutation.mutate(threadId);
      delete starDebounceRef.current[threadId];
    }, 300);
  }
  ```
  The optimistic update still applies immediately for visual feedback.

- **Option B: Replace toggle API with set-value API.** Change the backend to accept `PATCH /api/threads/:id/star` with body `{ starred: true }` or `{ starred: false }`. The frontend sends the desired end state, not a toggle. This makes the operation idempotent.

- **Option C: Mutation serialization.** Use React Query's `mutationKey` to serialize star mutations per thread ID so they run sequentially.

- Recommended: Combine Option A (debounce) with Option B (set-value API) for maximum robustness. The debounce prevents unnecessary API calls, and the set-value API ensures idempotency.

- Apply the same fix to `useMuteThread()` which has an identical pattern.
