# PRD-072: Domain Deletion Data Cleanup

## Status
Done

## Priority
P1-High

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L1

---

## Problem Statement
When a domain is soft-deleted, only the domain's `status` is set to `'deleted'` and `hidden` is set to `true`. No cleanup of child records (aliases, threads, emails, discovered addresses) is performed. These records become orphaned -- invisible but occupying storage, holding uniqueness constraints, and potentially being returned in queries that do not check domain status.

## Current Behavior
In `backend/internal/handler/domains.go`:
- Lines 434-453: The `Delete` handler performs a single UPDATE:
  ```go
  h.DB.Exec(r.Context(),
    `UPDATE domains SET status = 'deleted', hidden = true, updated_at = now() WHERE id = $1 AND org_id = $2`,
    domainID, claims.OrgID)
  ```
- No cleanup of related tables:
  - `aliases` table: aliases for the deleted domain remain active. Their `address` values hold UNIQUE constraints (migration 001, line 130).
  - `alias_users` table: junction records for the domain's aliases remain.
  - `threads` table: threads with `domain_id` pointing to the deleted domain remain visible unless individually trashed.
  - `emails` table: emails with `domain_id` pointing to the deleted domain remain.
  - `discovered_addresses` table: discovered addresses with `domain_id` remain and their `UNIQUE(domain_id, address)` constraint still holds.

In `backend/internal/handler/orgs.go`:
- Lines 150-168: Org deletion marks domains as `'deleted'` but also does not clean up child records for those domains.

## Expected Behavior
- Domain deletion should cascade soft-delete behavior to child records within a transaction:
  1. Soft-delete all aliases for the domain (or mark them inactive).
  2. Move all threads for the domain to a `deleted` state (set `deleted_at = now()`).
  3. Optionally remove `discovered_addresses` for the domain (these are not user data, just analytics).
  4. Remove `alias_users` junction records for the domain's aliases (so disabled aliases don't appear in user alias lists).
- All of this should happen within a database transaction to ensure atomicity.
- The domain's webhook should be unregistered from Resend (see PRD-078 for org-level cleanup).

## Files to Modify
- `backend/internal/handler/domains.go:434-453` — Expand the `Delete` handler to perform cascading cleanup within a transaction:
  - `UPDATE aliases SET deleted_at = now() WHERE domain_id = $1`
  - `DELETE FROM alias_users WHERE alias_id IN (SELECT id FROM aliases WHERE domain_id = $1)`
  - `UPDATE threads SET deleted_at = now() WHERE domain_id = $1 AND deleted_at IS NULL`
  - `DELETE FROM discovered_addresses WHERE domain_id = $1`

## Impact Analysis

### What This Affects
- Domain deletion workflow.
- Users will no longer see threads and aliases from deleted domains.
- Discovered addresses from deleted domains will be removed, freeing uniqueness constraints.
- Frontend domain sidebar already hides `hidden=true` domains.

### What Affects This
- The `aliases` table may need a `deleted_at` column (currently does not have one).
- The `threads` table already has `deleted_at`.
- PRD-073 (Deleted Domains Still Receive Emails) -- if cleanup is done properly, webhook processing for deleted domains is also addressed.

### Related PRDs
- PRD-073: Deleted Domains Still Receive Emails — Without cleanup, the webhook continues processing emails for deleted domains.
- PRD-077: GDPR Hard Delete — Soft-delete cleanup is a prerequisite for eventual hard-delete support.
- PRD-078: Webhook Cleanup for Deleted Orgs — Similar cascading cleanup pattern at the org level.
- PRD-121: Cascade Deletes — Overlapping concern about foreign key cleanup.
- PRD-086: Emails After Org Deletion — Org deletion needs similar domain cleanup.

## Implementation Notes
- Consider adding a `deleted_at` column to the `aliases` table via a new migration if one does not exist.
- Run all cleanup SQL within a single transaction to prevent partial cleanup on failure.
- The Resend domain itself should NOT be deleted from Resend's side -- the user may want to re-add it later. Only local records are cleaned up.
- For large domains with many threads/emails, the transaction could be slow. Consider batching for very large datasets, but for typical usage (< 10K threads), a single transaction is fine.

## User Notes
From FULL-SUMMARY.md: "Deleting a domain doesn't clean up its data. Aliases, threads, emails, and discovered addresses are left orphaned when a domain is soft-deleted."

## Acceptance Criteria
- [ ] Domain deletion cascades soft-delete to all aliases for that domain.
- [ ] Domain deletion sets `deleted_at` on all threads for that domain.
- [ ] Domain deletion removes discovered addresses for that domain.
- [ ] Domain deletion removes `alias_users` records for the domain's aliases.
- [ ] All cleanup operations run within a single database transaction.
- [ ] If the transaction fails, the domain status is not changed.
- [ ] Integration test: after deleting a domain, queries for that domain's aliases, threads, and discovered addresses return empty results.
