# PRD-106: Missing Environment Variables in .env.example

## Status
Done

## Priority
P2-Medium

## Category
Configuration & Environment

## Source
GAP-FINDER C1

---

## Problem Statement
Several environment variables that are actively used in the codebase are missing from `.env.example`, making it easy for operators to deploy with missing configuration. `TRASH_COLLECTOR_ENABLED` is read in `main.go:97` but not documented anywhere. `BACKEND_URL` may be referenced in frontend config. `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_WS_URL` are in `docker-compose.yml` (lines 52-53) but not in `.env.example`. `EVENT_RETENTION_DAYS` is read in `config.go:47` but not documented. Operators deploying outside Docker will not know these variables exist.

## Current Behavior
The `.env.example` file contains 14 variables across Required, Database, Redis, Optional, and Commercial sections. However:
- `TRASH_COLLECTOR_ENABLED` is read at `backend/cmd/api/main.go:97` as `os.Getenv("TRASH_COLLECTOR_ENABLED") == "true"` -- defaults to disabled, but operators don't know it exists
- `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_WS_URL` are in `docker-compose.yml:52-53` but not in `.env.example`
- `EVENT_RETENTION_DAYS` has a default of 90 in `config.go:47` but is undocumented
- `APP_URL` is in `config.go:37` with a `localhost:3000` default but not clearly explained in `.env.example`
- `DOMAIN` is in `.env.example` line 3 and used by docker-compose but its relationship to `APP_URL`/`PUBLIC_URL` is unclear

## Expected Behavior
All environment variables used in the codebase should be documented in `.env.example` with:
- A clear description of what each variable does
- Default values where applicable
- Whether the variable is required or optional
- Which deployment modes need it (Docker, bare metal, development)

## Files to Modify
- `.env.example` -- Add missing variables with descriptions and defaults:
  - `TRASH_COLLECTOR_ENABLED` (optional, default: false)
  - `NEXT_PUBLIC_API_URL` (required for frontend in Docker)
  - `NEXT_PUBLIC_WS_URL` (required for frontend in Docker)
  - `EVENT_RETENTION_DAYS` (optional, default: 90)
  - `APP_URL` (optional, default: http://localhost:3000)

## Impact Analysis

### What This Affects
- New deployments -- operators will have complete configuration documentation
- Self-hosted users who deploy without Docker
- Debugging time when features don't work due to missing config

### What Affects This
- Any new environment variables added in future features must also be added here
- Docker-compose.yml variable mappings should match .env.example

## Implementation Notes
- Group variables by deployment context:
  ```
  # ── Backend Workers ─────────────────────────────────────────────────
  # TRASH_COLLECTOR_ENABLED=true    # Enable automatic trash purge (default: false)
  # EVENT_RETENTION_DAYS=90         # Days to keep WebSocket events (default: 90)

  # ── Frontend (required outside Docker) ──────────────────────────────
  # NEXT_PUBLIC_API_URL=https://mail.yourdomain.com
  # NEXT_PUBLIC_WS_URL=wss://mail.yourdomain.com

  # ── App URLs ────────────────────────────────────────────────────────
  # APP_URL=http://localhost:3000   # Frontend URL (used for email links)
  ```
- Add comments explaining when each variable is needed vs. when defaults suffice
- Cross-reference with docker-compose.yml to ensure parity

## User Notes
This is a documentation and developer experience improvement. Missing env vars cause silent failures that are hard to debug in production.
