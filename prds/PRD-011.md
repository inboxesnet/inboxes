# PRD-011: LIKE Wildcard Injection in Contact Search

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S7

---

## Problem Statement
The contact suggestion endpoint uses SQL `ILIKE` with user-provided input but does not escape LIKE metacharacters (`%`, `_`). A user entering `%` as their search query returns every contact address in their org — a data enumeration vector. While the data is scoped to the user's own org (not a cross-org leak), it bypasses the intended search behavior where results should match the typed characters.

## Current Behavior
In `backend/internal/handler/contacts.go`, the `Suggest` function (line 16-60):
- Line 26-37: The query uses `ILIKE` with unsanitized input:
  ```go
  rows, err := h.DB.Query(r.Context(),
      `SELECT DISTINCT address, COUNT(*) as cnt
       FROM (
         SELECT from_address as address FROM emails WHERE org_id = $1
         UNION ALL
         SELECT jsonb_array_elements_text(to_addresses) as address FROM emails WHERE org_id = $1
       ) sub
       WHERE address ILIKE $2
       GROUP BY address
       ORDER BY cnt DESC
       LIMIT 10`,
      claims.OrgID, "%"+query+"%")
  ```
- The `query` variable (line 18) comes directly from the URL query parameter `q`:
  ```go
  query := strings.TrimSpace(r.URL.Query().Get("q"))
  ```
- If `query = "%"`, the WHERE clause becomes `WHERE address ILIKE '%%%'` which matches everything.
- If `query = "_"`, it matches any single character, also returning broad results.

## Expected Behavior
- LIKE metacharacters (`%`, `_`, `\`) should be escaped before interpolation into the ILIKE pattern.
- The search should only match literal characters typed by the user.
- The `LIMIT 10` provides some protection against returning enormous result sets, but the enumeration risk remains (an attacker can iterate with `%a`, `%b`, etc.).

## Files to Modify
- `backend/internal/handler/contacts.go:37` — Escape LIKE metacharacters before building the pattern:
  ```go
  // Escape LIKE metacharacters
  escaped := strings.NewReplacer(
      `\`, `\\`,
      `%`, `\%`,
      `_`, `\_`,
  ).Replace(query)

  rows, err := h.DB.Query(r.Context(),
      `...WHERE address ILIKE $2...`,
      claims.OrgID, "%"+escaped+"%")
  ```
- Alternatively, extract a `escapeLIKE()` helper function in `helpers.go` for reuse:
  ```go
  func escapeLIKE(s string) string {
      s = strings.ReplaceAll(s, `\`, `\\`)
      s = strings.ReplaceAll(s, `%`, `\%`)
      s = strings.ReplaceAll(s, `_`, `\_`)
      return s
  }
  ```

## Impact Analysis

### What This Affects
- Contact autocomplete/suggestion in the compose window (`frontend/components/recipient-input.tsx`).
- Searching for contacts in the UI.
- No cross-org data leakage (queries are already scoped by `org_id`), but within-org enumeration is prevented.

### What Affects This
- The `Lookup` endpoint (`contacts.go:62-108`) uses exact match (`= $2`) and `@>` operators, not LIKE, so it is not affected.
- Any other LIKE/ILIKE queries in the codebase should also be audited.

### Related PRDs
- PRD-007: Case-Sensitive Emails — Contact search uses `ILIKE` which is already case-insensitive, so email normalization doesn't affect this query directly, but consistent lowercase storage helps.
- PRD-008: Verification Brute-Force — Both relate to input validation/sanitization of user-provided values.

## Implementation Notes
- PostgreSQL ILIKE uses `\` as the escape character by default. The replacement order matters: escape backslashes first, then `%` and `_`.
- The `LIMIT 10` already prevents large result sets, but the fix is still important because:
  1. An attacker can enumerate by combining wildcards with known characters.
  2. The principle of least surprise — searching for `%` should not return all contacts.
- Consider adding a minimum query length (e.g., 2 characters) as an additional UX improvement, though this is not strictly a security fix.
- Audit other ILIKE/LIKE usage in the codebase:
  - Search in `emails.go` (`Search` handler) may have similar issues.
  - Check `thread_list` queries if they use LIKE.
