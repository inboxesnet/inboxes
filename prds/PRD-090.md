# PRD-090: Token Expiry

## Status
Done

## Priority
P2-Medium

## Category
WebSocket & Real-Time

## Source
GAP-FINDER W2

---

## Problem Statement
WebSocket connections check token validity only every 5 minutes (`tokenCheckEvery = 5 * time.Minute` in `client.go:23`). A revoked or disabled user continues to receive real-time events for up to 5 minutes after their token is blacklisted or their account is disabled. This creates a security window where sensitive data (new emails, thread updates) is delivered to a user who should no longer have access.

## Current Behavior
In `backend/internal/ws/client.go:18-24`:
```go
const (
    // ...
    tokenCheckEvery = 5 * time.Minute
)
```

The `writePump` function (line 158-209) uses a `validationTicker` that fires every 5 minutes:
```go
case <-validationTicker.C:
    // Check if token has expired
    if time.Now().After(c.TokenExp) {
        // close connection
    }
    // Check if token has been revoked
    revoked := blacklist.IsRevoked(revokeCtx, c.JTI, c.UserID, c.IssuedAt)
    if revoked {
        // close connection
    }
```

This means:
- Admin disables a user -> user's WebSocket stays active for up to 5 more minutes
- User logs out (token blacklisted) -> if they have another WS connection, it stays active for up to 5 minutes
- Org deletion revokes all tokens -> WS connections remain for up to 5 minutes, receiving events for a deleted org

## Expected Behavior
1. Reduce the token check interval to 1 minute (balances security vs. Redis/DB load).
2. Add a push-based revocation mechanism: when a token is revoked, publish a Redis message that the Hub can react to immediately, closing the specific user's connections.
3. For immediate revocation (admin disable, org deletion), the backend should publish a "force-disconnect" event through Redis that the Hub processes to close all connections for the affected user(s).

## Files to Modify
- `backend/internal/ws/client.go:23` -- Reduce `tokenCheckEvery` from `5 * time.Minute` to `1 * time.Minute`
- `backend/internal/ws/hub.go:76-122` -- Add subscription to a `ws:disconnect` Redis channel; on message, close all connections for the specified user
- `backend/internal/service/token_blacklist.go` -- After revoking, publish `ws:disconnect` message with user ID
- `backend/internal/handler/users.go` -- On user disable, publish disconnect event
- `backend/internal/handler/orgs.go:170-181` -- On org delete, publish disconnect events for all org users

## Impact Analysis

### What This Affects
- Security window for revoked/disabled users receiving real-time events
- Redis pub/sub message volume (one additional channel)
- Token blacklist query frequency (5x increase if reducing to 1 min)

### What Affects This
- Redis availability (blacklist checks and disconnect events depend on Redis)
- Token revocation frequency (admin actions, logouts, password changes)
- Number of active WebSocket connections (each polls for revocation)

### Related PRDs
- PRD-089 (WS Connection Limit) -- fewer connections means less polling overhead
- PRD-091 (Alias Filter Fail) -- both involve per-connection periodic DB queries
- PRD-086 (Emails After Deletion) -- org deletion should disconnect users immediately

## Implementation Notes
- **Phase 1 (simple):** Reduce `tokenCheckEvery` to `1 * time.Minute`. This is a one-line change and reduces the window from 5 minutes to 1 minute with minimal additional load (5x more blacklist checks, but they're simple Redis lookups).
- **Phase 2 (push-based):** Add a `ws:disconnect` Redis pub/sub channel. When token revocation happens:
  ```go
  // In token_blacklist.go after revocation:
  rdb.Publish(ctx, "ws:disconnect", userID)
  ```
  In `hub.go`, subscribe to this channel alongside `ws:events`:
  ```go
  disconnectSub := h.rdb.Subscribe(ctx, "ws:disconnect")
  // On message, find all clients for the userID and close them
  ```
- The push-based approach provides near-instant disconnection without increasing polling frequency.
- Make `tokenCheckEvery` configurable via environment variable for deployment flexibility.

## User Notes
- After being logged out or disabled, the user's WebSocket connections will be closed within 1 minute (Phase 1) or near-instantly (Phase 2).
- Active browser tabs will show a disconnection state and prompt re-authentication.

## Acceptance Criteria
- [ ] Token validity check interval is reduced to 1 minute
- [ ] Revoked users are disconnected within 1 minute of revocation
- [ ] (Phase 2) A `ws:disconnect` Redis channel is used for immediate disconnection
- [ ] (Phase 2) Admin disabling a user triggers immediate WS disconnection
- [ ] (Phase 2) Org deletion triggers immediate WS disconnection for all org users
- [ ] Token check interval is configurable via environment variable
- [ ] No increase in Redis errors or connection issues under normal load
