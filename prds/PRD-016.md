# PRD-016: DB SSL

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source

## Problem Statement
The default `DATABASE_URL` in both `.env.example` and `docker-compose.yml` uses `sslmode=disable`. If a deployment uses a remote or cloud-hosted database (e.g., AWS RDS, Supabase, Neon), database credentials and all query data travel over the network in plain text. This is a significant security risk for any non-localhost deployment.

## Current Behavior
In `backend/internal/config/config.go:32`, the default DATABASE_URL is hardcoded with `sslmode=disable`:

```go
DatabaseURL: getEnv("DATABASE_URL", "postgres://inboxes:inboxes@localhost:5432/inboxes?sslmode=disable"),
```

In `.env.example:14`, the comment says `DATABASE_URL is constructed automatically in docker-compose.yml` but provides no guidance about SSL.

In `docker-compose.yml:37`, the backend service environment explicitly sets:

```yaml
DATABASE_URL: postgres://${POSTGRES_USER:-inboxes}:${POSTGRES_PASSWORD:-inboxes}@postgres:5432/${POSTGRES_DB:-inboxes}?sslmode=disable
```

There is no documentation or startup warning about the security implications of running with `sslmode=disable` against a remote database. The config loader in `config.go:30-74` validates several required fields but does not check or warn about the SSL mode.

## Expected Behavior
1. The `.env.example` file should include a prominent comment warning that `sslmode=disable` is only safe for localhost/Docker-internal connections and that production with remote databases MUST use `sslmode=require` or `sslmode=verify-full`.
2. The `docker-compose.yml` should include a comment next to the DATABASE_URL explaining that the `sslmode=disable` is safe only because postgres is in the same Docker network.
3. The config loader should log a warning at startup if `sslmode=disable` is detected and the host is not `localhost`, `127.0.0.1`, or a Docker service name.

## Files to Modify
- `backend/internal/config/config.go:32` — Add a startup warning when sslmode=disable is used with a non-local host
- `.env.example:10-14` — Add prominent comment about production SSL requirements
- `docker-compose.yml:37` — Add inline comment explaining why sslmode=disable is acceptable here

## Impact Analysis

### What This Affects
- Self-hosted deployments using remote databases are at risk of credential interception
- Deployment documentation should reference SSL configuration
- No impact on local development or Docker-internal connections

### What Affects This
- Database hosting configuration — cloud-managed databases (RDS, Cloud SQL) typically require SSL
- Network topology — Docker-internal connections are inherently encrypted by being on the same host

### Related PRDs
- PRD-017: Redis Auth — Same pattern of insecure default configuration for infrastructure services
- PRD-026: Password+Redis Failure — Related to the theme of infrastructure security defaults

## Implementation Notes
The startup warning should parse the DATABASE_URL to extract the host. If the host is not `localhost`, `127.0.0.1`, `postgres` (Docker service name), or a Unix socket, and `sslmode=disable` is present, log a `slog.Warn` at startup. This avoids breaking existing local development setups while alerting production deployments.

Example warning: `"database connection has sslmode=disable with non-local host — credentials may be transmitted in plain text"`.

Do NOT change the default to `sslmode=require` as this would break local development and Docker Compose setups where the bundled postgres does not have SSL configured.
