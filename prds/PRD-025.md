# PRD-025: Concurrent Sessions

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
GAP-FINDER S13

---

## Problem Statement
There is no tracking of active session count per user and no limit on concurrent sessions. A compromised password allows unlimited simultaneous logins from different locations, with no detection mechanism or alert to the legitimate user. The user has no way to see how many sessions are active, where they are from, or revoke individual sessions.

## Current Behavior
In `backend/internal/handler/auth.go:270-275`, each successful login generates a new JWT token without checking how many tokens are already active:

```go
token, err := middleware.GenerateToken(h.Secret, userID, orgID, role)
if err != nil {
    writeError(w, http.StatusInternalServerError, "failed to generate token")
    return
}
middleware.SetTokenCookie(w, token, h.AppURL)
```

In `backend/internal/middleware/auth.go:80-94`, `GenerateToken` creates a new JWT with a unique JTI (UUID) and a 7-day expiry:

```go
func GenerateToken(secret, userID, orgID, role string) (string, error) {
    now := time.Now()
    claims := &Claims{
        RegisteredClaims: jwt.RegisteredClaims{
            ID:        uuid.NewString(),
            IssuedAt:  jwt.NewNumericDate(now),
            ExpiresAt: jwt.NewNumericDate(now.Add(7 * 24 * time.Hour)),
        },
        ...
    }
    ...
}
```

There is no mechanism to:
- Track how many active sessions (JTIs) exist per user
- Limit the maximum number of concurrent sessions
- Alert users when a new login occurs from an unfamiliar location
- List or revoke individual sessions from the UI

The token blacklist service (`backend/internal/service/token_blacklist.go`) supports individual token revocation (`RevokeToken` by JTI) and user-wide revocation (`RevokeAllForUser`), but there is no session registry that tracks which JTIs are active.

On the WebSocket side, `backend/internal/ws/hub.go:16` maintains `clients map[string]map[*Client]bool` (userID to set of clients), which is unbounded. A single user can open hundreds of WebSocket connections without restriction. This is also related to PRD item #89 (WS Connection Limit).

## Expected Behavior
1. Track active sessions per user in Redis (a set of JTI values per user ID).
2. On login, add the new JTI to the user's session set. If the count exceeds a threshold (e.g., 5), revoke the oldest session.
3. Provide an API endpoint for users to list their active sessions (JTI, creation time, last-seen IP/user-agent).
4. Provide an API endpoint to revoke a specific session or all other sessions.
5. Optionally: notify the user (via WebSocket or email) when a new login occurs.

## Files to Modify
- `backend/internal/service/token_blacklist.go` — Add session tracking methods: `RegisterSession`, `ListSessions`, `RevokeSession`, and enforce max session count
- `backend/internal/handler/auth.go:270-275` — On login, register the session and enforce the limit
- `backend/internal/handler/auth.go:403-454` — On claim (invite accept), register the session
- `backend/internal/handler/users.go:388-441` — On password change, clear old sessions and register the new one
- `backend/internal/router/router.go` — Add new routes for session listing and individual revocation (e.g., `GET /api/users/me/sessions`, `DELETE /api/users/me/sessions/{jti}`)
- `backend/internal/ws/hub.go:44-73` — Add per-user connection count check in the register path

## Impact Analysis

### What This Affects
- Login flow — new logins may evict old sessions if the limit is reached
- Users with multiple devices (phone, laptop, tablet) need the limit to be high enough (5 is reasonable)
- Password change flow already revokes all sessions via `RevokeAllForUser`; session tracking should be cleared too
- WebSocket connections — per-user limit prevents memory exhaustion

### What Affects This
- Redis availability — session tracking depends on Redis; if Redis is down, session tracking fails open (consistent with token blacklist behavior)
- JWT expiry (7 days) — sessions auto-expire; the Redis set should use the same TTL

### Related PRDs
- PRD-019: Token Blacklist Fail-Open — Session tracking uses the same Redis infrastructure and has the same fail-open concerns
- PRD-024: Last Admin Protection — If the admin's account is compromised and has unlimited sessions, the attacker has persistent access
- PRD-026: Password+Redis Failure — Password change should clear session tracking, but if Redis is down, old sessions persist

## Implementation Notes
Session tracking in Redis can use a sorted set per user, with the JTI as the member and the creation timestamp as the score:

```go
// Register session
key := "sessions:" + userID
rdb.ZAdd(ctx, key, redis.Z{Score: float64(time.Now().Unix()), Member: jti})
rdb.Expire(ctx, key, 7*24*time.Hour)

// Enforce limit (remove oldest if over 5)
count, _ := rdb.ZCard(ctx, key).Result()
if count > 5 {
    rdb.ZRemRangeByRank(ctx, key, 0, count-6) // keep newest 5
    // Also add removed JTIs to the blacklist
}

// List sessions
sessions, _ := rdb.ZRangeWithScores(ctx, key, 0, -1).Result()

// Revoke one session
rdb.ZRem(ctx, key, jti)
// Also add to blacklist
```

For the WebSocket connection limit, add a count check in `hub.go:49-55`:

```go
case client := <-h.register:
    h.mu.Lock()
    if _, ok := h.clients[client.UserID]; !ok {
        h.clients[client.UserID] = make(map[*Client]bool)
    }
    if len(h.clients[client.UserID]) >= 5 {
        // reject or close oldest connection
    }
    h.clients[client.UserID][client] = true
    h.mu.Unlock()
```

This is a medium-complexity feature that can be implemented incrementally:
- Phase 1: Add per-user WebSocket connection limit (quick win)
- Phase 2: Add session tracking and max session enforcement on login
- Phase 3: Add session listing and individual revocation UI
