# PRD-041: Claim Name Validation

## Status
Done

## Priority
P2-Medium

## Category
Form Validation Mismatches

## Source
GAP-FINDER V4

---

## Problem Statement
The invite claim endpoint (`POST /api/auth/claim`) does not validate the name field length. The signup endpoint validates name length at `auth.go:94` with `validateLength(req.Name, "name", 255)`, but the claim endpoint at `auth.go:403-454` has no such check. A user accepting an invite can submit an arbitrarily long name, which could cause database issues or UI layout breakage.

## Current Behavior
- In `backend/internal/handler/auth.go:403-454`, the `Claim` handler validates `req.Token` and `req.Password` but does not call `validateLength(req.Name, "name", 255)`.
- In `backend/internal/handler/auth.go:94`, the `Signup` handler does call `validateLength(req.Name, "name", 255)`.
- The SQL at line 428 uses `CASE WHEN $2 = '' THEN name ELSE $2 END` to conditionally update the name, so a very long name will be stored directly.
- In `frontend/app/(auth)/claim/page.tsx:122-128`, the name input has no `maxLength` attribute, so the frontend imposes no limit.
- A user could paste a 10,000-character name string, which would be stored in the database and could break UI rendering in team lists, sidebar headers, and email display names.

## Expected Behavior
- The backend claim endpoint should validate name length with the same 255-character limit used by signup.
- The frontend claim form should enforce `maxLength={255}` on the name input.

## Files to Modify
- `backend/internal/handler/auth.go:412-416` -- Add `validateLength(req.Name, "name", 255)` check after the password validation block
- `frontend/app/(auth)/claim/page.tsx:122` -- Add `maxLength={255}` to the name `<Input>` element

## Impact Analysis

### What This Affects
- Invite claim flow for new team members
- Display name rendering in team lists, compose "From" display, thread participant lists

### What Affects This
- The signup handler's name validation at `auth.go:94` sets the precedent of 255 chars
- The database `users.name` column is TEXT type with no explicit length constraint, so the enforcement must come from application code

### Related PRDs
- PRD-039 (Password Rules) -- claim form already validates password, name validation is the gap
- PRD-040 (Setup Email Validation) -- same pattern of missing validation on auth endpoints

## Implementation Notes
- Backend change: add 3 lines after line 416 in `auth.go` (after `validatePassword` check):
  ```go
  if err := validateLength(req.Name, "name", 255); err != nil {
      writeError(w, http.StatusBadRequest, err.Error())
      return
  }
  ```
- Frontend change: add `maxLength={255}` to the Input at `claim/page.tsx:122`.
- The name field is not `required` in the claim form (it uses `CASE WHEN $2 = '' THEN name ELSE $2 END`), so empty names are fine -- we only need to cap the max length.

## User Notes
- Users accepting invites will be limited to 255-character names, matching the signup form behavior.
- This limit is generous enough to accommodate any reasonable real name.

## Acceptance Criteria

- [ ] Frontend claim form enforces `maxLength={255}` on the name input
- [ ] Names of exactly 255 characters are accepted
- [ ] Empty names are still accepted (the existing conditional update behavior is preserved)
- [ ] Signup and claim name validation limits are consistent (both 255)
- [ ] No regression in the invite claim flow for normal-length names
