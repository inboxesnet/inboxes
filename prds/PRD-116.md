# PRD-116: Dead Code — `/api/contacts/lookup` Endpoint Never Called

## Status
Draft

## Priority
P3-Low

## Category
Dead Code

## Source
GAP-FINDER O1

---

## Problem Statement
The `/api/contacts/lookup` endpoint is fully implemented in the backend with a registered route, but no frontend code ever calls it. This creates maintenance burden (the handler must be kept working through refactors), adds attack surface (4 unchecked `QueryRow.Scan()` calls on lines 78-95), and confuses developers who may assume it is actively used.

## Current Behavior
- `backend/internal/handler/contacts.go:62-108` — The `Lookup` method is fully implemented. It takes an `email` query parameter and returns sent/received counts plus the contact's name.
- `backend/internal/router/router.go:261` — The route is registered: `r.Get("/api/contacts/lookup", contacts.Lookup)`.
- Frontend only uses `/api/contacts/suggest` (called from `frontend/components/recipient-input.tsx` for autocomplete). A search of the entire `frontend/` directory for `contacts/lookup` returns zero results.
- The `Lookup` handler has 4 `QueryRow.Scan()` calls (lines 78-80, 82-84, 88-90, 92-95) whose errors are silently ignored, contributing to the unchecked-error findings in PRD-049.

## Expected Behavior
Two options:
1. **Remove** the endpoint: delete the `Lookup` method from `contacts.go` and the route from `router.go`. This is the cleanest approach if no future use is planned.
2. **Keep but mark as internal/future**: add a comment documenting that it is reserved for future contact card functionality and fix the unchecked error handling.

## Files to Modify
- `backend/internal/handler/contacts.go:62-108` — Remove the `Lookup` method (or fix error handling if keeping).
- `backend/internal/router/router.go:261` — Remove the route registration.

## Impact Analysis

### What This Affects
- No frontend functionality (the endpoint is unused).
- Reduces backend code surface area and potential security exposure.
- Removes 4 unchecked `QueryRow.Scan()` calls from the error handling audit scope.

### What Affects This
- If a future feature (e.g., contact card popup, contact detail panel) is planned, this endpoint may need to be recreated.

### Related PRDs
- PRD-049: Unchecked Queries — the `Lookup` handler contains 4 unchecked `.Scan()` calls.
- PRD-011: LIKE Wildcard Injection — the `Suggest` handler (same file, line 37) has the LIKE injection issue; `Lookup` uses exact match so is not affected.

## Implementation Notes
Before removing, verify there are no mobile apps, browser extensions, or third-party integrations calling this endpoint. Since the app is pre-launch, this is unlikely but worth confirming. If keeping for future use, at minimum fix the unchecked error handling on the 4 `QueryRow.Scan()` calls.
