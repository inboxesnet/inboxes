# PRD-071: Reply-To Header Ignored

## Status
Done

## Priority
P2-Medium

## Category
Email Threading & Rendering

## Source
GAP-FINDER T6

---

## Problem Statement
The `Reply-To` header is correctly parsed from inbound emails and stored in the database, but the frontend's reply logic completely ignores it. When a user clicks "Reply," the To field is populated with the `From` address instead of the `Reply-To` address. This is incorrect per RFC 5322 -- `Reply-To` exists specifically to redirect replies to a different address than the sender (e.g., mailing lists, shared inboxes, or "reply to this address instead" use cases).

## Current Behavior
In `backend/internal/queue/process_fetch.go`:
- Lines 158-161: `replyTo` is correctly extracted from either the Resend API response or the webhook data.
- Lines 204, 306-307: `replyToJSON` is marshalled and stored in the `reply_to_addresses` column of the `emails` table.

In `frontend/lib/types.ts`:
- Lines 73-98: The `Email` interface does not include a `reply_to_addresses` field. The data is stored in the backend but never sent to (or typed in) the frontend.

In `frontend/components/thread-view.tsx`:
- Lines 555-584: The `getReplyRecipients` function determines the To and CC addresses for a reply:
  - Line 583: For inbound emails, `defaultTo` is set to `[lastEmail.from_address]` -- the From address, not Reply-To.
  - The function never references `reply_to_addresses` or any Reply-To field.
- Lines 537-553: `getReplyFromAddress` determines which of the user's addresses to send from -- this is separate from the To-address issue.

## Expected Behavior
- The backend thread detail API should include `reply_to_addresses` in the email response.
- The `Email` TypeScript interface should include `reply_to_addresses?: string[]`.
- `getReplyRecipients` should check `lastEmail.reply_to_addresses` first: if it is a non-empty array, use `reply_to_addresses[0]` as the primary To address instead of `from_address`.
- For Reply All, `reply_to_addresses[0]` should replace `from_address` in To, and all other original To/CC recipients (minus the user's own address) go to CC.

## Files to Modify
- `backend/internal/handler/threads.go` — Ensure the thread detail query returns `reply_to_addresses` for each email in the thread. Check the `Get` handler's email scanning.
- `frontend/lib/types.ts:91` — Add `reply_to_addresses?: string[];` to the `Email` interface.
- `frontend/components/thread-view.tsx:555-584` — Update `getReplyRecipients` to check `lastEmail.reply_to_addresses` before falling back to `lastEmail.from_address`.

## Impact Analysis

### What This Affects
- Reply behavior for all inbound emails that include a Reply-To header.
- Mailing list replies will go to the correct list address instead of the individual sender.
- No change for emails without a Reply-To header (the common case).

### What Affects This
- The backend must include `reply_to_addresses` in the email response payload.
- The `reply_to_addresses` column already exists in the `emails` table (migration 001, line 110 confirms `delivered_via_alias` and the column is used at insert).

### Related PRDs
- PRD-066: References Header Threading — Both involve correct RFC 5322 compliance.
- PRD-070: Bounce Notifications — Bounce emails should not have a Reply button at all; Reply-To should not be applied to bounces.
- PRD-012: BCC Privacy Leak — Reply-To handling should not inadvertently expose BCC recipients.

## Implementation Notes
- Check whether the `Get` handler in `threads.go` already selects `reply_to_addresses` from the emails query. If not, add it to the SELECT list and scan it.
- On the frontend, `reply_to_addresses` is an array (multiple Reply-To addresses are allowed per RFC). Use the first element as the primary recipient.
- Show a subtle UI indicator when replying to a Reply-To address that differs from From, so the user knows where their reply is going (e.g., "Replying to list@example.com instead of sender@example.com").
