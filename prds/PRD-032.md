# PRD-032: Search Placeholder

## Status
Done

## Priority
P2-Medium

## Category
Search & Navigation

## Source
PRE-LAUNCH-AUDIT #22

---

## Problem Statement
The search input placeholder says "Search emails..." which does not communicate what fields are searchable or the scope of the search. Users do not know whether search covers subjects, email bodies, sender addresses, or all of the above. They also do not know whether the search is scoped to the current folder or searches across all folders for the domain. This leads to underuse of the search feature and confusion about unexpected results.

## Current Behavior
In `thread-list-page.tsx`, the search input (line 203) has a static placeholder:
```tsx
placeholder="Search emails..."
```
This appears identically across inbox, sent, archive, trash, spam, and starred views. It provides no hint about:
1. What fields are searched (subject? body? sender? all?)
2. Whether search is scoped to the current folder or spans all folders
3. What syntax is supported (e.g., exact phrases, OR operators)

## Expected Behavior
1. Change the placeholder to communicate scope and capability: "Search by subject, sender, or content..."
2. Optionally include the current folder context: "Search in Inbox..." or "Search in Sent..." depending on the active label.
3. Consider a small tooltip or info icon near the search bar that explains search behavior on hover.

## Files to Modify
- `frontend/components/thread-list-page.tsx:203` -- Change the `placeholder` prop from `"Search emails..."` to a more descriptive string. Use the `label` prop or `title` prop to dynamically generate a context-aware placeholder.
- `frontend/components/thread-list-page.tsx:47` -- The `ThreadListPageProps` already receives `label` and `title`, which can be used to build the placeholder dynamically.

## Impact Analysis

### What This Affects
- Search discoverability -- users are more likely to use search effectively when they understand its capabilities.
- User expectations -- clear placeholders prevent confusion about why certain results appear or do not appear.

### What Affects This
- The actual search backend implementation -- the placeholder should accurately reflect what the search API actually searches. The endpoint is `GET /api/emails/search?q=...&domain_id=...`, which likely searches across subject, body, and addresses.
- The `label` prop -- determines the current folder context for a dynamic placeholder.

### Related PRDs
- PRD-031 (Empty Search Results) -- improving search feedback pairs with improving search entry
- PRD-037 (Keyboard Shortcuts) -- the `/` and `Cmd+K` shortcuts focus the search input, so the placeholder is the first thing users see

## Implementation Notes
- Simple approach:
  ```tsx
  const searchPlaceholder = `Search ${title.toLowerCase()} by subject, sender, or content...`;
  // e.g., "Search inbox by subject, sender, or content..."
  ```
- Keep the placeholder short enough to fit in the input on mobile (max ~45 characters visible). Test with longer folder names like "Archive" to ensure it does not overflow.
- If the search actually searches across all folders (not just the current one), the placeholder should say "Search all emails..." rather than "Search inbox..." to avoid misleading users.
- Verify backend search behavior before finalizing the placeholder text -- check what `GET /api/emails/search` actually queries.

## User Notes
- This is a minor UX polish that sets correct expectations before the user types.
- Power users benefit from knowing they can search email bodies, not just subjects.

## Acceptance Criteria
- [ ] Search placeholder communicates what fields are searchable (subject, sender, content)
- [ ] Placeholder is context-aware, reflecting the current folder if search is scoped
- [ ] Placeholder text fits within the search input on both mobile and desktop without truncation
- [ ] Placeholder accurately reflects the backend search behavior (does not promise capabilities that do not exist)
