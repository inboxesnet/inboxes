# PRD-062: JSON Indexes

## Status
Done

## Priority
P2-Medium

## Category

## Source
GAP-FINDER D5

## Problem Statement
Several queries filter on JSONB columns (`to_addresses`, `cc_addresses`, `participant_emails`) using the `@>` containment operator, but these columns have no GIN indexes. This forces full table scans on the emails table and threads table, which are the two largest tables in the database.

## Current Behavior
Queries that filter on JSONB columns without GIN indexes:

**Onboarding GetAddresses** (`backend/internal/handler/onboarding.go:356-367`):
```sql
SELECT ... FROM discovered_addresses da
WHERE ... (e.to_addresses @> to_jsonb(da.address)
      OR e.cc_addresses @> to_jsonb(da.address))
```
This correlated subquery runs a full scan on `emails` for every discovered address.

**Contact Lookup** (`backend/internal/handler/contacts.go:82-84`):
```sql
SELECT COUNT(*) FROM emails WHERE org_id = $1 AND to_addresses @> $2::jsonb
```
Full table scan on `emails` for each lookup.

**Thread matching in fetch worker** (`backend/internal/queue/process_fetch.go:268-271`):
```sql
SELECT id FROM threads WHERE org_id = $1 AND domain_id = $2 AND subject = $3
 AND participant_emails @> $4::jsonb AND last_message_at > now() - interval '90 days'
```
Uses `@>` on `participant_emails` without a GIN index on threads.

**Emails table** has JSONB columns `to_addresses`, `cc_addresses`, `bcc_addresses` — none have GIN indexes.

**Threads table** has `participant_emails` JSONB column — no GIN index.

Existing indexes from migrations:
- `idx_emails_search` is a GIN index on `search_vector` (full-text search), NOT on the JSONB columns.
- No GIN indexes exist on any JSONB columns.

## Expected Behavior
Add GIN indexes on the JSONB columns used for filtering:
1. `emails.to_addresses` — used in contact lookup and onboarding address count
2. `emails.cc_addresses` — used in onboarding address count
3. `threads.participant_emails` — used in thread matching during email fetch

## Implementation Notes
- Migration file:
  ```sql
  -- +goose Up
  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_emails_to_addresses_gin
    ON emails USING GIN(to_addresses);

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_emails_cc_addresses_gin
    ON emails USING GIN(cc_addresses);

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_threads_participant_emails_gin
    ON threads USING GIN(participant_emails);

  -- +goose Down
  DROP INDEX IF EXISTS idx_emails_to_addresses_gin;
  DROP INDEX IF EXISTS idx_emails_cc_addresses_gin;
  DROP INDEX IF EXISTS idx_threads_participant_emails_gin;
  ```

- GIN indexes on JSONB columns support the `@>` (containment) operator used in all the affected queries.
- GIN indexes have higher write overhead than B-tree indexes. For the emails table (which has high write volume from inbound processing), monitor INSERT performance after adding these indexes.
- `bcc_addresses` is intentionally excluded — there are no queries filtering on BCC, and adding an unnecessary GIN index would slow inserts.
- Use `CONCURRENTLY` to avoid locking tables during index creation.
- Run `EXPLAIN ANALYZE` on the affected queries before and after to confirm index usage.

## User Notes
The onboarding "configure addresses" step will load much faster for organizations that have imported many emails. Contact lookup in the compose window will also be faster. Users will not notice any behavioral change.
