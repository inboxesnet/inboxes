# PRD-043: Recipient Limit

## Status
Done

## Priority
P2-Medium

## Category
Form Validation Mismatches

## Source
GAP-FINDER V6

---

## Problem Statement
There is no limit on the number of recipients (To, Cc, Bcc) that can be added to an email on either the frontend or backend. A user can add hundreds of recipients, which could exceed Resend's API limits, cause performance issues with JSON serialization of large recipient arrays, and potentially be used for spam abuse.

## Current Behavior
- In `frontend/components/recipient-input.tsx`, the `addRecipient` function at line 59-76 adds recipients without any count check. The `value` array grows unbounded.
- In `frontend/components/floating-compose-window.tsx`, the `to`, `cc`, and `bcc` state arrays have no maximum length enforcement.
- In `backend/internal/handler/emails.go:80-107`, recipient arrays are validated for email format but not for count. The Send handler iterates over `req.To`, `req.CC`, and `req.BCC` to validate each address, but never checks `len(req.To)`, `len(req.CC)`, or `len(req.BCC)`.
- In `backend/internal/handler/drafts.go:275-445`, the draft Send handler similarly has no recipient count check.
- Resend's API has a limit of 50 recipients per email (across To, Cc, Bcc combined). Exceeding this causes an API error that propagates back as a generic send failure.

## Expected Behavior
- The backend should enforce a maximum recipient count per field (e.g., 50 per field, or 50 total across To+Cc+Bcc).
- The frontend should prevent adding more recipients once the limit is reached.
- The RecipientInput component should show a clear message when the limit is hit.

## Files to Modify
- `backend/internal/handler/emails.go:80-107` -- Add recipient count validation after the existing empty checks
- `backend/internal/handler/drafts.go:309-312` -- Add recipient count validation for draft sends
- `frontend/components/recipient-input.tsx:59-76` -- Check count before adding; accept a `maxRecipients` prop
- `frontend/components/floating-compose-window.tsx` -- Pass `maxRecipients` prop to RecipientInput components

## Impact Analysis

### What This Affects
- Email composition flow (compose new, reply, forward)
- Draft save and send operations
- Large Cc/Bcc lists for announcements or team emails

### What Affects This
- Resend's API recipient limits (currently 50 per email)
- Email deliverability -- large recipient lists increase spam score
- JSON payload size for email jobs stored in the database

### Related PRDs
- PRD-029 (Double-Send) -- if a large-recipient email triggers a send failure, retry logic is complicated
- PRD-038 (Subject Length) -- same pattern of frontend/backend validation mismatch
- PRD-042 (Rate Limit Range) -- same pattern of missing numeric limit enforcement

## Implementation Notes
- Backend validation in `emails.go` (add after line 107):
  ```go
  totalRecipients := len(req.To) + len(req.CC) + len(req.BCC)
  if totalRecipients > 50 {
      writeError(w, http.StatusBadRequest, "too many recipients (max 50 total across To, Cc, and Bcc)")
      return
  }
  ```
- Same check in `drafts.go` after unmarshaling recipients.
- Frontend: add a `maxRecipients` prop to `RecipientInput`:
  ```typescript
  if (maxRecipients && value.length >= maxRecipients) {
    toast.error(`Maximum ${maxRecipients} recipients allowed`);
    return;
  }
  ```
- The combined limit of 50 aligns with Resend's documented limits. Check Resend's current docs to confirm the exact limit before implementing.
- Consider whether the limit should be per-field (50 each for To, Cc, Bcc) or combined (50 total). Resend uses combined, so combined is recommended.

## User Notes
- Users will see a clear message when they hit the recipient limit.
- The limit of 50 recipients total is generous for normal use. Mass emails should use proper mailing list tools.

## Acceptance Criteria
- [ ] Backend rejects emails with more than 50 total recipients (To + Cc + Bcc) with a clear 400 error
- [ ] Frontend RecipientInput prevents adding recipients beyond the limit
- [ ] User sees a toast or inline message when hitting the recipient limit
- [ ] Draft saves with any recipient count are still accepted (drafts are works-in-progress)
- [ ] Draft sends enforce the same recipient limit as direct sends
- [ ] Existing emails with large recipient counts in the database are unaffected
- [ ] The limit is consistent between the direct send endpoint and the draft send endpoint
