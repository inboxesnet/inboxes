# PRD-044: Label Transactions

## Status
Done

## Priority
P1-High

## Category
Race Conditions

## Source
GAP-FINDER R2

---

## Problem Statement
Bulk actions on threads (archive, trash, spam, star, mute, move, mark read/unread) perform multiple SQL statements without transactional wrapping. The `BulkAction` handler in `threads.go:368-540` calls helper functions like `bulkAddLabel()`, `bulkRemoveLabel()`, and `h.DB.Exec()` as separate, independent operations. Concurrent operations on the same threads can leave labels in an inconsistent state -- for example, a thread could end up with both "inbox" and "trash" labels, or have a trash label without the corresponding `trash_expires_at` timestamp.

## Current Behavior
- In `backend/internal/handler/threads.go:368-540`, the `BulkAction` handler dispatches to actions like `archive`, `trash`, `move`, etc.
- The "trash" action at lines 429-434 calls `bulkAddLabel()` followed by a separate `h.DB.Exec()` to set `trash_expires_at`. If the first succeeds and the second fails, the thread has a trash label but no expiry date.
- The "move:inbox" action at lines 463-469 calls `bulkAddLabel()`, `bulkRemoveLabel("trash")`, `bulkRemoveLabel("spam")`, and `h.DB.Exec()` to clear `trash_expires_at` -- four separate operations.
- Individual thread actions in `Move()` at lines 542-621, `Star()` at lines 700-742, `Trash()` at lines 816-845, `Archive()` at lines 788-814, and `Spam()` at lines 847-887 also perform multiple non-transactional label operations.
- In `labels.go:156-218`, the helper functions `addLabel()`, `removeLabel()`, `bulkAddLabel()`, `bulkRemoveLabel()` each execute a single SQL statement using the pool directly (not inside a transaction).
- Two users simultaneously archiving and moving the same thread to inbox creates a race where the final label state is unpredictable.

## Expected Behavior
- All multi-step label operations should be wrapped in a database transaction.
- If any step fails, the entire operation should be rolled back to prevent inconsistent state.
- The label helper functions already accept a `Querier` interface (line 150-154) which supports both `*pgxpool.Pool` and `pgx.Tx`, so they are already designed for transactional use.

## Files to Modify
- `backend/internal/handler/threads.go:368-540` -- Wrap each BulkAction case in a transaction using `h.DB.Begin()` + `tx.Commit()` / `tx.Rollback()`
- `backend/internal/handler/threads.go:542-621` -- Wrap Move handler operations in a transaction
- `backend/internal/handler/threads.go:700-742` -- Wrap Star handler in a transaction (check + toggle is a read-then-write race)
- `backend/internal/handler/threads.go:744-786` -- Wrap Mute handler in a transaction
- `backend/internal/handler/threads.go:816-845` -- Wrap Trash handler in a transaction
- `backend/internal/handler/threads.go:847-887` -- Wrap Spam handler in a transaction

## Impact Analysis

### What This Affects
- All thread label mutations: archive, trash, spam, star, mute, move, read/unread
- Bulk operations on multiple threads
- WebSocket event publishing (events should only fire after successful commit)

### What Affects This
- The `Querier` interface in `labels.go:150-154` already supports transactions -- no changes needed to label helpers
- WebSocket events in `h.Bus.Publish()` should be called after `tx.Commit()`, not inside the transaction
- Database connection pool sizing -- transactions hold connections longer

### Related PRDs
- PRD-045 (Duplicate Threads) -- another race condition in thread creation
- PRD-046 (Star/Unstar Sync) -- the Star handler's check-then-toggle pattern is a specific instance of this problem
- PRD-048 (Label Rename) -- label rename has a similar non-atomic multi-statement problem
- PRD-047 (Bulk Selection) -- bulk actions failing mid-way leave partial state

## Implementation Notes
- For `BulkAction`, begin a transaction at the top of the handler and pass `tx` to all label helper calls:
  ```go
  tx, err := h.DB.Begin(ctx)
  if err != nil {
      writeError(w, http.StatusInternalServerError, "failed to start transaction")
      return
  }
  defer tx.Rollback(ctx)
  // ... use tx instead of h.DB for all operations ...
  if err := tx.Commit(ctx); err != nil { ... }
  ```
- For the Star handler, the check-then-toggle at lines 719-724 should use `SELECT ... FOR UPDATE` or be wrapped in a serializable transaction to prevent the race where two concurrent star requests both see "not starred" and both try to add the label.
- Move the `h.Bus.Publish()` calls after `tx.Commit()` to ensure events are only emitted for committed state changes.
- Consider using `util.DBCtx()` for timeout control on transactions, consistent with the drafts and emails handlers.

## User Notes
- Users should not notice any behavior change under normal use.
- Under high concurrency (multiple users acting on the same threads simultaneously), operations will now be consistent instead of potentially leaving orphaned labels.

## Acceptance Criteria
- [ ] BulkAction handler wraps each action case in a database transaction
- [ ] Move, Star, Mute, Trash, Archive, Spam handlers use transactions for multi-step operations
- [ ] If any SQL step fails, the entire operation is rolled back
- [ ] WebSocket events are published only after successful transaction commit
- [ ] Star toggle is atomic (no race between check and insert/delete)
- [ ] Concurrent archive and move-to-inbox on the same thread produces a consistent final label state
- [ ] No regression in single-thread operations under normal load
- [ ] Bulk operations on 100+ threads complete within acceptable time (< 2 seconds)
