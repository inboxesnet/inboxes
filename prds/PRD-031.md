# PRD-031: Empty Search Results

## Status

## Priority
P2-Medium

## Category

## Problem Statement
When a search query returns zero results, the search results area shows a completely empty container with no feedback message. Users cannot distinguish between "search is still loading," "search found nothing," and "search broke." This creates a confusing experience where users may think the feature is broken or keep waiting for results that will never appear.

## Current Behavior
In `thread-list-page.tsx`, the search results rendering (lines 244-271) actually already handles the empty state correctly as of the latest code. Lines 251-254 render "No results found" when `searchResults.length === 0` and `searchFetching` is false. However, the implementation is minimal:
1. The message is generic ("No results found") and does not include the search query.
2. There is no suggestion to try a different search or clear the search.
3. The search error state (`isError`) from the useQuery hook is not checked (line 63-70) -- if the search API call fails, the spinner shows indefinitely because `searchFetching` remains true until the query settles, and the error path is never handled.

## Expected Behavior
1. When search returns 0 results: show "No results found for '[query]'" with a suggestion to "Try a different search or clear the filter."
2. When search fails with an error: show "Search failed. [Retry]" instead of an infinite spinner.
3. Include a "Clear search" button in the empty/error states.

## Files to Modify
- `frontend/components/thread-list-page.tsx:63-70` -- Add `isError` and `error` destructuring from the `useQuery` result for the search query.
- `frontend/components/thread-list-page.tsx:251-254` -- Enhance the empty state message to include the query string and a "Clear search" action.
- `frontend/components/thread-list-page.tsx:247-250` -- Add an error state check before the loading spinner check: if `isError` is true, show an error message with a retry button.

## Impact Analysis

### What This Affects
- All users who use the search feature and encounter zero-result queries or search failures.
- User confidence in the search functionality.

### What Affects This
- The search API endpoint (`GET /api/emails/search`) -- its availability and response format determine whether results, empty, or error states are shown.
- React Query's error handling -- the `useQuery` hook returns `isError` and `error` that are currently unused for search.
- The `searchQuery` state variable that drives whether search mode is active.

### Related PRDs
- PRD-032 (Search Placeholder) -- improving search discoverability pairs well with improving search feedback
- PRD-055 (Search Loading) -- the infinite loading state on search failure is the error-path counterpart

## Implementation Notes
- The `useQuery` call on lines 63-70 returns `{ data, isFetching, isError, error, refetch }`. Destructure `isError` and `refetch`.
- Rendering priority in search mode should be:
  1. `isError` -> Error message with retry button (`refetch()`)
  2. `searchFetching` -> Spinner
  3. `searchResults.length === 0` -> "No results" message
  4. Default -> Thread list
- The "Clear search" action should call `setSearchInput(""); setSearchQuery("");` -- this logic already exists on lines 210-213.
- Consider adding a small search icon in the empty state to visually reinforce that this is a search result, not a folder view.

## User Notes
- This is a polish improvement that makes search feel more reliable and informative.
- Users who search for misspelled terms or obscure queries will benefit from clear "no results" feedback.

## Acceptance Criteria
- [ ] Searching for a query with no matching results shows "No results found for '[query]'"
- [ ] The empty state includes a "Clear search" button that resets the search
- [ ] If the search API returns an error, an error message is displayed instead of an infinite spinner
- [ ] The error state includes a "Retry" button that re-executes the search query
- [ ] The loading spinner is only shown while the search is actively fetching
- [ ] Clearing the search returns the user to the normal thread list view
