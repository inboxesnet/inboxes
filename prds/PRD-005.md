# PRD-005: Add Content-Security-Policy Header

## Status
Done

## Priority
P0-Critical

## Source
PRE-LAUNCH-AUDIT #5

## Problem Statement
The application has no Content-Security-Policy (CSP) header. Email HTML content is rendered using `dangerouslySetInnerHTML` after DOMPurify sanitization, but if the sanitizer is ever bypassed (a known risk with evolving browser features), injected scripts could execute freely. A CSP header provides defense-in-depth against XSS by restricting what resources the browser will load and execute.

## Current Behavior
In `backend/internal/middleware/security.go` (line 9-19), the `SecurityHeaders` middleware sets:
- `X-Content-Type-Options: nosniff` (line 11)
- `X-Frame-Options: DENY` (line 12)
- `Referrer-Policy: strict-origin-when-cross-origin` (line 13)
- `Cache-Control: no-store` (line 14)
- `Strict-Transport-Security` (conditional, line 15-17)

**No `Content-Security-Policy` header is set.**

The frontend renders email HTML on `frontend/components/thread-view.tsx:446-449`:
```tsx
<div dangerouslySetInnerHTML={{ __html: sanitizedHtml }} />
```
DOMPurify allows `<img>` with `src` and `<*>` with `style` attributes (line 369-379), which means:
- Images load from any origin (tracking pixels, see PRD-013).
- Inline styles can be injected (malicious CSS, see PRD-014).
- If DOMPurify fails, inline scripts could execute with no browser-level restriction.

## Expected Behavior
- A strict CSP header should be added to all API responses from the backend.
- The CSP should:
  - Block inline scripts (`script-src 'self'`).
  - Allow inline styles (needed for email rendering: `style-src 'self' 'unsafe-inline'`).
  - Allow images from any origin (`img-src * data: blob:` — emails contain external images).
  - Allow WebSocket connections (`connect-src 'self' wss:`).
  - Block framing (`frame-ancestors 'none'`).
- Note: The frontend is served by Next.js (port 3000), not by the Go backend (port 8080). The CSP on API responses protects against XSS in API-driven content but the frontend itself needs CSP via Next.js config or meta tags. Both should be addressed.

## Files to Modify
- `backend/internal/middleware/security.go:10-18` — Add `Content-Security-Policy` header in the `SecurityHeaders` function:
  ```go
  w.Header().Set("Content-Security-Policy",
      "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; "+
      "img-src * data: blob:; connect-src 'self' wss:; frame-ancestors 'none'")
  ```
- `frontend/next.config.js` or `frontend/next.config.ts` — Add CSP headers in the Next.js security headers configuration for frontend-served pages.

## Impact Analysis

### What This Affects
- All HTTP responses from the backend will include the CSP header.
- Email HTML rendering — inline styles will work (`'unsafe-inline'` for style-src), but inline scripts will be blocked.
- External image loading in emails — allowed by `img-src *`.
- WebSocket connections — allowed by `connect-src 'self' wss:`.

### What Affects This
- DOMPurify configuration in `thread-view.tsx:369-379` — CSP is a second layer if DOMPurify is bypassed.
- Any future use of inline `<script>` tags or `eval()` would be blocked by the CSP. The app currently uses no inline scripts.

### Related PRDs
- PRD-013: Tracking Pixels — CSP's `img-src` directive could be tightened later to block tracking pixels, but for now `img-src *` is needed for legitimate email images.
- PRD-014: Malicious CSS — CSP allows `'unsafe-inline'` for styles, so CSS injection is not blocked by CSP. PRD-014 addresses this via DOMPurify configuration.
- PRD-004: Docker Root — Both are defense-in-depth measures reducing blast radius.

## Implementation Notes
- The backend serves API responses, not HTML pages. The CSP on API responses is belt-and-suspenders for any scenario where API JSON is rendered in a browser context.
- The primary CSP for the user-facing app should be set by Next.js. Check if `frontend/next.config.ts` has a `headers()` function; if not, add one.
- `'unsafe-inline'` for `style-src` is required because email HTML uses inline `style` attributes and DOMPurify allows them. This is an acceptable trade-off.
- If the app ever uses a nonce-based CSP for scripts, the Next.js middleware would need to generate nonces per request.
- Test that the Tiptap editor (used in compose) still works with the CSP — it may require `'unsafe-inline'` for scripts if it uses inline event handlers. If so, investigate using nonces instead.
