# PRD-019: Token Blacklist Fail-Open

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
PRE-LAUNCH-AUDIT #11

---

## Problem Statement
The token blacklist system explicitly fails open when Redis is unavailable. If Redis goes down, all previously revoked tokens (from logouts, password changes, and admin-disabled accounts) are silently accepted as valid. A user who logged out or had their account disabled can continue accessing the system with their old token for up to 7 days (the JWT expiry window). There is no warning or alerting mechanism for this condition.

## Current Behavior
In `backend/internal/service/token_blacklist.go:50-74`, the `IsRevoked` method returns `false` (not revoked) whenever Redis is unavailable:

```go
func (b *TokenBlacklist) IsRevoked(ctx context.Context, jti, userID string, issuedAt time.Time) bool {
    if b == nil || b.rdb == nil {
        return false  // fail open: nil client
    }
    // Check individual token revocation
    if jti != "" {
        exists, err := b.rdb.Exists(ctx, "token:bl:"+jti).Result()
        if err != nil {
            slog.Warn("token_blacklist: redis error on jti check, failing open", "error", err)
            return false  // fail open: Redis error
        }
        ...
    }
    // Check user-wide revocation
    val, err := b.rdb.Get(ctx, "token:rv:"+userID).Int64()
    if err != nil {
        // Key doesn't exist or Redis error — not revoked
        return false  // fail open: Redis error or key missing
    }
    ...
}
```

This is called from `AuthMiddleware` in `backend/internal/middleware/auth.go:48-56`:

```go
// Check token revocation (fail open if Redis is down)
if blacklist.IsRevoked(r.Context(), claims.ID, claims.UserID, issuedAt) {
    http.Error(w, `{"error":"unauthorized"}`, http.StatusUnauthorized)
    return
}
```

The comment on line 48 explicitly acknowledges the fail-open behavior but takes no additional action.

The `RevokeToken` and `RevokeAllForUser` methods (lines 23-46) also silently return when Redis is nil or errors occur — they log a warning but do not propagate the error to the caller.

## Expected Behavior
1. When Redis is unavailable, the system should log a prominent error (not just a warning) on every request that skips the blacklist check, using `slog.Error` instead of `slog.Warn`.
2. The health endpoint (`/api/health` at `router.go:70-89`) already checks Redis status — this should be sufficient for monitoring. But the fail-open behavior should be documented.
3. Consider adding a configurable behavior: `TOKEN_BLACKLIST_MODE=fail-open|fail-closed`. Default to `fail-open` for backward compatibility, but allow operators to switch to `fail-closed` for high-security deployments.
4. When `RevokeAllForUser` fails (password change, account disable), the calling code should be informed and should warn the user that session invalidation may not have worked.

## Files to Modify
- `backend/internal/service/token_blacklist.go:50-74` — Upgrade log level from Warn to Error; add configurable fail-closed mode
- `backend/internal/service/token_blacklist.go:23-46` — Return errors from `RevokeToken` and `RevokeAllForUser` instead of silently ignoring them
- `backend/internal/middleware/auth.go:48-56` — Add periodic error logging when Redis is down (avoid log spam with a cooldown)
- `backend/internal/handler/users.go:432-433` — Handle error from `RevokeAllForUser` on password change
- `backend/internal/handler/users.go:206,215,229` — Handle error from `RevokeAllForUser` on user disable
- `backend/internal/handler/auth.go:397-398` — Handle error from `RevokeAllForUser` on password reset

## Impact Analysis

### What This Affects
- Security posture during Redis outages — currently, all revocations are silently bypassed
- Password change flow — user is told "other sessions logged out" even when it fails (also covered in PRD-026)
- Account disable flow — disabled user's tokens remain valid during Redis outage
- Logout flow — logged-out tokens remain valid during Redis outage

### What Affects This
- Redis availability — this is entirely a Redis-dependency issue
- Redis deployment configuration (PRD-017) — unauthenticated Redis could be deliberately taken down

### Related PRDs
- PRD-017: Redis Auth — Unsecured Redis could be deliberately disrupted to exploit fail-open behavior
- PRD-026: Password+Redis Failure — Specific scenario where password change + Redis failure leads to misleading user feedback
- PRD-015: Admin Rate Limits — Rate limiting also fails open on Redis outage, compounding the exposure

## Implementation Notes
Changing `RevokeToken` and `RevokeAllForUser` to return errors is a breaking API change that will require updating all call sites. The recommended approach:

1. Change the return signature to `error`
2. At each call site, check the error. For logout, the token cookie is already cleared so the impact is limited. For password change and account disable, the calling handler should still succeed but include a caveat in logs.
3. For fail-closed mode, `IsRevoked` should return `true` (revoked) when Redis is unavailable, effectively forcing re-authentication. This is aggressive but correct for high-security deployments.
4. Use a rate-limited logger (e.g., log at most once per minute) to avoid flooding logs during sustained Redis outages.
