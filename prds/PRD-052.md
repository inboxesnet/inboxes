# PRD-052: Redis Webhook

## Status
Draft

## Priority
P1-High

## Category
Error Handling — Backend

## Source
GAP-FINDER E4

---

## Problem Statement
When Redis is down or the `LPUSH` to the job queue fails, the webhook handler at `webhooks.go:175-177` still returns HTTP 200 OK to Resend. Resend interprets this as successful delivery and will not retry. The email job record exists in Postgres (it was inserted before the Redis push) but has no corresponding queue entry, so it sits orphaned until the stale recovery cron picks it up 5+ minutes later. During this window, inbound emails are silently delayed. If Redis remains down, the recovery cron cannot re-enqueue either, causing permanent email loss.

## Current Behavior
In `backend/internal/handler/webhooks.go:174-178`:
```go
if err := h.RDB.LPush(ctx, "email:jobs", jobID).Err(); err != nil {
    slog.Error("webhook: redis lpush failed", "job_id", jobID, "error", err)
}
```
The error is logged but the function continues to `w.WriteHeader(http.StatusOK)` on line 125.

The same pattern exists in:
- `backend/internal/handler/emails.go:278-280` — send email endpoint, Redis push fails silently
- `backend/internal/handler/drafts.go:433-435` — draft send endpoint, Redis push fails silently

For the send endpoints (emails.go, drafts.go), the impact is less severe because the user initiated the action and will notice the email never sends. But for the webhook handler, there is no user watching.

## Expected Behavior
For the webhook handler specifically:
1. If Redis LPUSH fails, return HTTP 503 (Service Unavailable) so Resend retries the webhook delivery.
2. The orphaned job in Postgres acts as a safety net — the stale recovery cron will eventually pick it up.
3. This dual approach (immediate retry via Resend + eventual recovery via cron) ensures no email is lost.

For the send endpoints (emails.go, drafts.go):
1. Return HTTP 202 Accepted (as currently done) but include a warning flag in the response: `"queue_delayed": true`.
2. The frontend can optionally display this to the user.

## Files to Modify
- `backend/internal/handler/webhooks.go:174-178` — Return HTTP 503 on Redis LPUSH failure instead of 200
- `backend/internal/handler/emails.go:278-280` — Add warning flag to response on Redis failure
- `backend/internal/handler/drafts.go:433-435` — Add warning flag to response on Redis failure

## Impact Analysis

### What This Affects
- Inbound email delivery latency (currently 5+ minutes when Redis is down; will be near-zero with Resend retries)
- Outbound email queuing reliability
- Resend webhook retry behavior (Resend retries on 5xx status codes)

### What Affects This
- Redis availability and connectivity
- Stale recovery cron (backup mechanism, runs every 60 seconds in email_worker.go:214)
- Resend's webhook retry policy (exponential backoff over several hours)

### Related PRDs
- PRD-049 (Unchecked Queries) — the webhook secret query is another silent failure in the same handler
- PRD-019 (Token Blacklist Fail-Open) — another Redis failure scenario
- PRD-026 (Password+Redis Failure) — pattern of Redis failure causing silent degradation
- PRD-053 (Worker Panics) — if workers panic, the orphaned job recovery is the only safety net

## Implementation Notes
- Resend webhooks follow the Svix standard: they retry on 4xx (except 410) and 5xx with exponential backoff for up to 72 hours. Returning 503 is the safest choice.
- The DB insert happens before the Redis push (both in webhooks.go and emails.go), so the job record always exists in Postgres as a durable backup.
- The `recoverOrphanedJobs` function in `email_worker.go:270-291` already handles re-enqueuing jobs that are stuck in "pending" state for more than 5 minutes. This is the safety net.
- Do NOT roll back the Postgres insert on Redis failure — the job record is the durable reference, and the recovery cron depends on it.

## User Notes
For inbound email: users currently experience silent 5+ minute delays when Redis is down. After this fix, Resend will retry immediately, reducing the delay to seconds.

For outbound email: users will see normal behavior. The only change is a subtle warning if Redis fails during queuing, and the email will still be processed by the recovery cron.
