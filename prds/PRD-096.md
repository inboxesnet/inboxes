# PRD-096: ARIA Combobox Pattern on Recipient Input

## Status
Done

## Priority
P3-Low

## Category
Accessibility

## Source
GAP-FINDER A2

---

## Problem Statement
The recipient autocomplete input (`RecipientInput` component) does not implement the WAI-ARIA combobox pattern. Screen readers cannot navigate the suggestion dropdown, cannot announce how many suggestions are available, and do not know when suggestions appear or disappear. The input and dropdown have no ARIA relationship.

## Current Behavior
In `recipient-input.tsx`, the input element (line 184-196) is a plain `<input>` with no `role`, `aria-autocomplete`, `aria-expanded`, or `aria-activedescendant` attributes. The suggestion dropdown (lines 200-222) is a plain `<div>` with no `role="listbox"`. Individual suggestion buttons (line 203) have no `role="option"` or `aria-selected`. Arrow key navigation works visually (lines 118-123) but the selected index is not communicated to assistive technology.

## Expected Behavior
The component should follow the [WAI-ARIA Combobox pattern](https://www.w3.org/WAI/ARIA/apg/patterns/combobox/):
- Input: `role="combobox"`, `aria-autocomplete="list"`, `aria-expanded="true/false"`, `aria-controls="suggestions-listbox-id"`, `aria-activedescendant="selected-option-id"`
- Dropdown container: `role="listbox"`, `id="suggestions-listbox-id"`
- Each suggestion: `role="option"`, `id="option-{index}"`, `aria-selected="true/false"`

## Files to Modify
- `frontend/components/recipient-input.tsx:184-196` -- Add combobox ARIA attributes to the `<input>` element
- `frontend/components/recipient-input.tsx:200-201` -- Add `role="listbox"` and `id` to suggestion dropdown container
- `frontend/components/recipient-input.tsx:203-219` -- Add `role="option"`, `id`, and `aria-selected` to each suggestion button
- `frontend/components/recipient-input.tsx:31-33` -- Add an `id` ref for unique listbox/option IDs (use `useId()` or counter)

## Impact Analysis

### What This Affects
- Screen reader users' ability to use the autocomplete when composing emails
- WCAG 2.1 Level A compliance (Success Criterion 4.1.2: Name, Role, Value)
- Keyboard navigation experience (already partially working, needs ARIA wiring)

### What Affects This
- The debounced suggestion fetch (line 100-101) controls when `aria-expanded` should change
- The `selectedIndex` state (line 33) already tracks the active option -- needs to map to `aria-activedescendant`
- The blur handler (line 129-137) with its 150ms delay for click handling affects when suggestions close

### Related PRDs
- PRD-095 (Focus Trap) -- recipient input lives inside the compose dialog
- PRD-094 (ARIA Labels) -- remove button on chips needs `aria-label`
- PRD-043 (Recipient Limit) -- related recipient input enhancement

## Implementation Notes
- Use `useId()` from React 18+ to generate unique IDs for the listbox and options
- `aria-activedescendant` should point to the `id` of the currently highlighted option (or be absent when `selectedIndex === -1`)
- Suggestion items should use `<div role="option">` instead of `<button>` inside a listbox, or keep `<button>` with `role="option"` override
- Announce the number of results with an `aria-live="polite"` region: "5 suggestions available"
- The remove button on each chip (line 172-181) needs `aria-label="Remove {email}"`

## User Notes
Low priority -- app is 100% behind auth, no SEO or AI indexing impact. No legal risk at current scale. Address when there's traction and real screen reader users.

## Acceptance Criteria
- [ ] The input has `role="combobox"`, `aria-autocomplete="list"`, and `aria-expanded` that reflects dropdown visibility
- [ ] The suggestion dropdown has `role="listbox"` with a linked `id`
- [ ] Each suggestion has `role="option"` and `aria-selected` reflecting the focused state
- [ ] `aria-activedescendant` on the input tracks the highlighted suggestion
- [ ] Screen reader announces suggestion count when dropdown opens (e.g., "5 suggestions available")
- [ ] VoiceOver can navigate suggestions with arrow keys and hear each option announced
- [ ] Chip remove buttons have descriptive `aria-label` (e.g., "Remove user@example.com")
