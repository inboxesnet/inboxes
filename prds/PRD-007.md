# PRD-007: Case-Sensitive Email Addresses

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S1

---

## Problem Statement
Email addresses are stored and compared case-sensitively throughout the authentication system. A user who signs up as `User@Example.com` cannot log in with `user@example.com`. Worse, a second user could register `user@example.com` as a separate account, causing confusion and potential account takeover vectors. Per RFC 5321, the local part of an email is technically case-sensitive, but in practice all major email providers treat addresses as case-insensitive. The application should normalize to lowercase everywhere.

## Current Behavior
In `backend/internal/handler/auth.go`:
- **Signup** (line 140-143): `req.Email` is stored directly without lowercasing:
  ```go
  err = tx.QueryRow(dbCtx,
      `INSERT INTO users (org_id, email, name, password_hash, role, status, email_verified)
       VALUES ($1, $2, $3, $4, 'admin', 'active', $5) RETURNING id`,
      orgID, req.Email, name, string(hash), emailVerified,
  ).Scan(&userID)
  ```
- **Login** (line 240-243): `WHERE email = $1` is case-sensitive in PostgreSQL:
  ```go
  err := h.DB.QueryRow(ctx,
      `SELECT id, org_id, name, role, status, password_hash, email_verified FROM users WHERE email = $1`,
      req.Email,
  ).Scan(...)
  ```
- **ForgotPassword** (line 333-335): `WHERE email = $3` — case-sensitive.
- **VerifyEmail** (line 472-477): `WHERE email = $1` — case-sensitive.
- **ResendVerification** (line 518-522): `WHERE email = $3` — case-sensitive.
- **Claim** (line 427-433): Does not involve email lookup by user input.

**Inconsistency**: In `backend/internal/handler/helpers.go:88`, the `canSendAs` function uses `strings.EqualFold(userEmail, fromAddress)` for case-insensitive comparison, showing awareness of the issue but inconsistent application.

## Expected Behavior
- All email addresses should be normalized to lowercase before storage and before querying.
- Add `strings.ToLower(req.Email)` at the top of every handler that accepts an email input.
- Existing data should be migrated to lowercase.
- The `users` table UNIQUE constraint on `email` already prevents duplicates, but only for exact case matches. After normalization, any existing case-variant duplicates would conflict (unlikely in practice but should be handled in migration).

## Files to Modify

  - Apply in `Signup` (after line 86), `Login` (after line 227), `ForgotPassword` (after line 315), `VerifyEmail` (after line 465), `ResendVerification` (after line 509).
- `backend/internal/handler/users.go` — Normalize email in `Invite` and any other user creation endpoints.

## Impact Analysis

### What This Affects
- User registration and login flows.
- Password reset flow (email lookup).
- Email verification flow.
- User invitation (email stored for new users).
- Contact suggestion queries (may return mixed-case results from historical data).
- Alias matching in email routing (already lowercase in `process_fetch.go`).

### What Affects This
- The UNIQUE constraint on `users.email` — after lowercasing, duplicate case variants would conflict. The migration should handle this (e.g., keep the most recently active account).

### Related PRDs
- PRD-008: Verification Brute-Force — If email case is inconsistent, an attacker could try different casings to find accounts.
- PRD-009: Disabled Users Auth — Normalization ensures disabled user checks work regardless of email casing.
- PRD-011: LIKE Wildcard Injection — Contact search queries would benefit from consistent casing.

## Implementation Notes
- Add a helper function:
  ```go
  func normalizeEmail(email string) string {
      return strings.ToLower(strings.TrimSpace(email))
  }
  ```
- Call it immediately after `readJSON` in every auth handler, before validation.
- The `validateEmail` function in `helpers.go` could incorporate normalization, but it's cleaner to keep validation and normalization separate.
- For the migration, handle potential conflicts:
  ```sql
  -- First, check for conflicts
  SELECT LOWER(email), COUNT(*) FROM users GROUP BY LOWER(email) HAVING COUNT(*) > 1;
  -- If conflicts exist, manually resolve before running:
  UPDATE users SET email = LOWER(email);
  ```
- Also lowercase alias addresses if they aren't already.
