# PRD-073: Deleted Domains Still Receive Emails

## Status
Done

## Priority
P1-High

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L2

---

## Problem Statement
The webhook handler and email processing pipeline do not check domain status before accepting and processing inbound emails. Emails addressed to domains with `status='deleted'` are still queued as `email_jobs`, processed by the worker, and potentially create new threads and email records -- even though the domain has been intentionally removed by the admin.

## Current Behavior
In `backend/internal/handler/webhooks.go`:
- Lines 59-126: `HandleResend` validates the webhook signature and dispatches to `handleEmailReceived` based on `payload.Type`. There is no check on the org's `deleted_at` or any domain status.
- Lines 128-179: `handleEmailReceived` finds an admin user, inserts an `email_jobs` record, and pushes to Redis. No domain status check.

In `backend/internal/queue/process_fetch.go`:
- Lines 60-112: The `processFetch` function scans To/CC/BCC addresses and looks up matching domains via `SELECT id FROM domains WHERE org_id = $1 AND domain = $2 AND status = 'active'`. This query does filter for `status = 'active'`, which would correctly reject `status = 'deleted'` domains.
- However, the job was already created and queued (in webhooks.go) before this check runs. The worker discovers no matching active domain and returns an error on line 111, but the job was already created and will be retried by the stale job recovery worker.
- Wasted resources: the email_job sits in Redis, gets dequeued, fetches the full email from Resend API, then fails. The stale job recovery worker (`status_recovery.go`) may retry it repeatedly.

## Expected Behavior
- The webhook handler should check domain status (or at minimum, org deletion status) before creating an `email_jobs` record.
- For deleted domains: return `200 OK` to Resend (to prevent retries) but do not create a job. Log the rejection.
- For deleted orgs: return `410 Gone` to signal Resend to stop sending webhooks (see PRD-078).
- The `processFetch` domain lookup already correctly filters by `status = 'active'`, so it serves as a secondary guard.

## Files to Modify
- `backend/internal/handler/webhooks.go:128-179` — In `handleEmailReceived`, before inserting the email_job, check if the org has `deleted_at IS NOT NULL`:
  ```go
  var orgDeletedAt *time.Time
  h.DB.QueryRow(ctx, "SELECT deleted_at FROM orgs WHERE id = $1", orgID).Scan(&orgDeletedAt)
  if orgDeletedAt != nil {
      slog.Info("webhook: org deleted, ignoring email", "org_id", orgID)
      return
  }
  ```
- `backend/internal/handler/webhooks.go:128-179` — Additionally check if any active domain matches the recipient addresses:
  ```go
  // Quick check: does this org have any active domains for these recipients?
  ```
- `backend/internal/handler/webhooks.go:59-126` — In `HandleResend`, add an early org status check before signature verification (or immediately after) and return 410 for deleted orgs.

## Impact Analysis

### What This Affects
- Webhook processing for all inbound emails.
- Deleted domains will no longer accumulate failed email_jobs.
- Resend's retry queue will not be polluted with permanently failing deliveries.
- The status recovery worker will have fewer stale jobs to process.

### What Affects This
- Org deletion flow in `orgs.go:119-185` sets `deleted_at` but does not unregister webhooks from Resend.
- Domain deletion flow in `domains.go:434-453` sets `status='deleted'`.
- The `processFetch` domain lookup (process_fetch.go:60-112) is already a secondary guard.

### Related PRDs
- PRD-072: Domain Deletion Data Cleanup — Cleanup should prevent new data from accumulating; this PRD prevents new jobs from being created.
- PRD-078: Webhook Cleanup for Deleted Orgs — Org-level webhook unregistration is the permanent solution; this PRD provides the immediate guard.
- PRD-086: Emails After Org Deletion — Same concern at the org level.

## Implementation Notes
- Returning `200 OK` for deleted domain emails (instead of an error) is important because Resend will retry on non-2xx responses. We want to acknowledge receipt and silently discard.
- For deleted orgs, returning `410 Gone` is a stronger signal. Check whether Resend respects 410 as a "stop sending" signal -- if not, use 200 with logging.
- The org status check adds one DB query per webhook call. This is acceptable given webhook volume is bounded by actual email traffic.
- Consider caching org status in Redis with a short TTL (e.g., 60 seconds) if webhook volume is high.
