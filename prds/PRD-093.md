# PRD-093: Reconnect Jitter

## Status
Done

## Priority
P2-Medium

## Category
WebSocket & Real-Time

## Source
GAP-FINDER W6

## Problem Statement
The WebSocket reconnection logic has only 1 second of random jitter (`Math.random() * 1000`). When a server restarts or network blip affects many users simultaneously, all clients reconnect within approximately the same 1-second jitter window. This creates a "thundering herd" effect where hundreds of concurrent WebSocket upgrade requests, followed by token validation queries and alias lookups, spike server load and can cause cascading failures.

## Current Behavior
In `frontend/contexts/notification-context.tsx:99-105`:
```typescript
ws.onclose = () => {
    setConnected(false);
    const attempt = reconnectAttemptRef.current;
    reconnectAttemptRef.current = attempt + 1;
    const delay = Math.min(1000 * Math.pow(2, attempt), 30000) + Math.random() * 1000;
    reconnectTimeoutRef.current = setTimeout(connect, delay);
};
```

The exponential backoff is: `2^attempt * 1000ms + random(0-1000ms)`.

For the first reconnection attempt (`attempt = 0`):
- Base delay: `1000 * 2^0 = 1000ms` (1 second)
- Jitter: `0-1000ms`
- Total: 1000-2000ms

If 100 users disconnect simultaneously (server restart), all 100 will attempt to reconnect within a 1-2 second window. Each connection requires:
1. WebSocket upgrade (HTTP handshake)
2. JWT token validation
3. Database query for alias addresses (`loadUserAliases` in `client.go:117-135`)
4. Hub registration

This concentrated burst can overwhelm the database connection pool and cause further failures.

## Expected Behavior
1. Increase the jitter range to be proportional to the backoff delay (e.g., 50% of the base delay).
2. For the first reconnection attempt, use a jitter of 0-5 seconds (not 0-1 second).
3. For subsequent attempts, jitter should scale with the backoff to spread reconnections over a wider window.
4. The maximum backoff cap (30 seconds) is appropriate and should remain.

## Files to Modify
- `frontend/contexts/notification-context.tsx:103` -- Update the jitter calculation to use a wider range proportional to backoff

## Impact Analysis

### What This Affects
- Server load during mass reconnection events (server restarts, network recovery)
- Database connection pool utilization during reconnection spike
- User experience during reconnection (slightly longer average wait, but more reliable connection)

### What Affects This
- Number of concurrent users (determines severity of thundering herd)
- Server restart frequency
- Network stability

## Implementation Notes
- Replace the jitter calculation with a full-jitter approach (recommended by AWS for exponential backoff):
  ```typescript
  ws.onclose = () => {
      setConnected(false);
      const attempt = reconnectAttemptRef.current;
      reconnectAttemptRef.current = attempt + 1;
      // Full jitter: random value between 0 and the backoff cap
      const baseDelay = Math.min(1000 * Math.pow(2, attempt), 30000);
      const jitter = Math.random() * baseDelay;
      const delay = Math.max(1000, jitter); // minimum 1 second delay
      reconnectTimeoutRef.current = setTimeout(connect, delay);
  };
  ```

  This gives:
  - Attempt 0: random 1-1000ms (similar to current)
  - Attempt 1: random 1-2000ms
  - Attempt 2: random 1-4000ms
  - Attempt 3: random 1-8000ms
  - Attempt 4+: random 1-16000ms, capped at 30000ms

  The "full jitter" approach maximizes spread while maintaining the same average reconnection time.

- Alternative: use "equal jitter" for more predictable behavior:
  ```typescript
  const baseDelay = Math.min(1000 * Math.pow(2, attempt), 30000);
  const halfBase = baseDelay / 2;
  const delay = halfBase + Math.random() * halfBase;
  ```
  This gives a range of [baseDelay/2, baseDelay] for each attempt.

- The initial reconnection attempt (attempt 0) is the most critical for thundering herd. Consider using a minimum initial jitter of 2-5 seconds for the first attempt specifically:
  ```typescript
  const initialJitter = attempt === 0 ? 2000 + Math.random() * 3000 : 0;
  ```

## User Notes
- Reconnection after server restarts or network issues will be slightly slower on average (2-3 seconds instead of 1-2 seconds for the first attempt), but significantly more reliable.
- The maximum reconnection delay remains 30 seconds.
- Users will see the "Reconnecting..." state for slightly longer during mass disconnection events.

## Acceptance Criteria
- [ ] First reconnection attempt has jitter of at least 1-5 seconds (not 1-2 seconds)
- [ ] Subsequent reconnection attempts have jitter proportional to the backoff delay
- [ ] Maximum backoff remains capped at 30 seconds
- [ ] Minimum reconnection delay is 1 second (no instant reconnection)
- [ ] Exponential backoff pattern is preserved
- [ ] Simulated test: 100 concurrent disconnections result in reconnections spread over 5+ seconds (not clustered in 1-2 seconds)
- [ ] No regression in reconnection reliability for single-user scenarios
