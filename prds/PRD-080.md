# PRD-080: Stripe Dedup

## Status
Done

## Priority
P1-High

## Category
Billing & Stripe

## Source
GAP-FINDER B1

---

## Problem Statement
Stripe webhook events are not deduplicated. When Stripe retries a webhook delivery (due to timeout, 5xx response, or network issue), the same event is processed multiple times. This can cause duplicate plan changes, duplicate customer updates, and inconsistent org state.

## Current Behavior
`HandleStripeWebhook` in `backend/internal/handler/billing.go:213-362` processes every incoming event without checking whether the event ID has already been handled. The `event.ID` field from Stripe's payload is available but never recorded. Stripe retries failed deliveries up to 3 times over 72 hours, meaning a single `checkout.session.completed` event could upgrade an org multiple times or a `customer.subscription.deleted` event could reset grace period timestamps repeatedly.

## Expected Behavior
Each Stripe event should be processed exactly once. The system should:
1. Record processed event IDs (e.g., `evt_xxx`) in a database table or Redis set with a TTL.
2. On receiving an event, check if its ID has already been processed.
3. If already processed, return 200 OK immediately without re-executing business logic.
4. Use a database table (`stripe_events`) for durable dedup, with a cleanup job for events older than 7 days.

## Files to Modify
- `backend/internal/handler/billing.go:213-226` -- Add dedup check after signature verification, before the `switch` block
- `backend/internal/db/migrations/` -- New migration to create `stripe_events(event_id TEXT PRIMARY KEY, event_type TEXT, processed_at TIMESTAMPTZ DEFAULT now())`
- `backend/internal/worker/` -- Optional: add cleanup job for old stripe_events rows

## Impact Analysis

### What This Affects
- All Stripe webhook processing (checkout completed, subscription deleted/updated, invoice paid/failed)
- Org plan state consistency
- Grace period timestamp accuracy

### What Affects This
- Stripe retry behavior (up to 3 retries over 72 hours)
- Network reliability between Stripe and the backend
- Database availability for dedup table lookups

### Related PRDs
- PRD-081 (Stripe Event Handlers) -- more event types means more dedup surface
- PRD-084 (Grace Period) -- duplicate `subscription.deleted` events can reset grace period incorrectly
- PRD-085 (Past-Due) -- duplicate `subscription.updated` events can flip plan state

## Implementation Notes
- Use `INSERT ... ON CONFLICT DO NOTHING` to atomically check-and-insert the event ID.
- If the insert succeeds (rows affected = 1), process the event. If not, skip.
- Add an index on `processed_at` for the cleanup job.
- Consider using Redis `SETNX` with a 72-hour TTL as a fast-path check before the DB insert.

## User Notes
- No user-visible changes. This is a reliability improvement that prevents duplicate billing state changes.
