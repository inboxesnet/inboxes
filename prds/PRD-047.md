# PRD-047: Bulk Selection

## Status
Done

## Priority
P1-High

## Category
Race Conditions

## Source
GAP-FINDER R6

---

## Problem Statement
The bulk action flow in `thread-list-page.tsx:151-174` clears the user's thread selection before the bulk API call completes. If the API call fails, the selection is already gone and the user cannot retry the operation without manually re-selecting all threads. This creates a frustrating user experience, especially for large bulk operations.

## Current Behavior
- In `frontend/components/thread-list-page.tsx:151-174`, the `handleBulkAction` callback:
  1. Reads the selected IDs from `selection.selectedIds` (line 153)
  2. Calls `selection.clearSelection()` (line 170) -- clears selection immediately
  3. Calls `bulkMutation.mutate()` (line 171) -- fires the async API call
- The optimistic update in `use-threads.ts:344-401` (`useBulkAction.onMutate`) removes threads from the list cache immediately.
- If the API call fails, `onError` at line 472-477 shows a toast error and invalidates queries, but the selection has already been cleared.
- The user sees threads reappear in the list (from query refetch) but has no selection to retry with.
- For bulk archive/trash at `onSuccess` lines 478-498, the undo toast uses the thread IDs captured at mutation time, so undo works. But the user cannot initiate a new retry of the same bulk action.

## Expected Behavior
- The selection should be preserved until the bulk API call succeeds.
- On failure, the selection should be restored so the user can retry.
- On success, the selection should be cleared.

## Files to Modify
- `frontend/components/thread-list-page.tsx:151-174` -- Move `selection.clearSelection()` to the `onSuccess` callback or after mutation settles
- `frontend/hooks/use-threads.ts:326-500` -- Move selection clearing to `onSuccess` in `useBulkAction`, or accept a `clearSelection` callback

## Impact Analysis

### What This Affects
- All bulk actions: archive, trash, spam, read, unread, mute, unmute, delete, move, label, unlabel
- The toolbar checkbox state (reflects selection)
- The undo flow for archive/trash (currently works because it captures IDs, but selection-based retry does not work)

### What Affects This
- The `useThreadSelection` hook which manages `selectedIds`
- Optimistic updates in `useBulkAction.onMutate` -- these still remove threads from the cache optimistically, which is correct for the visual update
- The toolbar's display of "X selected" which reads from `selection.selectedIds`

### Related PRDs
- PRD-044 (Label Transactions) -- if the backend bulk action partially fails (no transaction), some threads may have been processed while others failed
- PRD-046 (Star/Unstar Sync) -- similar pattern of optimistic update needing error recovery

## Implementation Notes
- **Approach A: Clear on success.** Move `selection.clearSelection()` from `handleBulkAction` to a callback that fires on success:
  ```typescript
  const handleBulkAction = useCallback(
    (actionStr: string) => {
      const ids = Array.from(selection.selectedIds);
      if (ids.length === 0) return;
      // ... parse action ...
      bulkMutation.mutate(
        { threadIds: ids, action, label: moveFolder },
        {
          onSuccess: () => selection.clearSelection(),
          // onError: selection is preserved, user can retry
        }
      );
    },
    [selection.selectedIds, bulkMutation]
  );
  ```

- **Approach B: Restore on error.** Keep the current "clear immediately" behavior for visual smoothness, but save the previous selection and restore it on error:
  ```typescript
  const savedSelectionRef = useRef<Set<string>>(new Set());
  // In handleBulkAction:
  savedSelectionRef.current = new Set(selection.selectedIds);
  selection.clearSelection();
  bulkMutation.mutate(/* ... */, {
    onError: () => {
      selection.selectIds(Array.from(savedSelectionRef.current));
    },
  });
  ```

- Recommended: Approach A is simpler and clearer. The optimistic update already hides the selected threads from the list, so there is no visual penalty to keeping the selection "active" until success.

- Note: The toolbar's "Cancel" or "Deselect" action should still allow the user to manually clear the selection.

## User Notes
- If a bulk action fails, the selected threads will reappear in the list with their checkboxes still selected.
- Users can immediately retry the bulk action without re-selecting.
- The undo flow for archive/trash is unaffected.
