# PRD-092: Event Catchup

## Status
Done

## Priority
P2-Medium

## Category
WebSocket & Real-Time

## Source
GAP-FINDER W5

---

## Problem Statement
The event catchup endpoint (`GET /api/events?since=X`) has no age limit on the `since` parameter. A client that was offline for 30 days can request all events from the last 30 days, causing a massive database query that scans potentially millions of rows. This can degrade database performance for all users and deliver an overwhelming number of stale events to the reconnecting client.

## Current Behavior
In `backend/internal/handler/events.go:19-87`:
```go
func (h *EventHandler) Since(w http.ResponseWriter, r *http.Request) {
    sinceID, _ := strconv.ParseInt(r.URL.Query().Get("since"), 10, 64)
    limit, _ := strconv.Atoi(r.URL.Query().Get("limit"))
    if limit <= 0 || limit > 200 {
        limit = 100
    }

    rows, err := h.DB.Query(ctx,
        `SELECT id, event_type, domain_id, thread_id, payload, created_at
         FROM events
         WHERE org_id = $1 AND id > $2
         ORDER BY id ASC
         LIMIT $3`,
        claims.OrgID, sinceID, limit,
    )
```

The query has a `LIMIT` (max 200) but no time-based cutoff. If `sinceID` is very old:
1. The database must scan from the old ID through all newer rows, even though only `LIMIT` rows are returned.
2. The client receives 200 events and will make another request for the next 200, creating a long catchup loop.
3. The event pruner (`backend/internal/worker/event_pruner.go`) deletes old events, but only based on `RetentionDays`, so events within retention are all queryable.

On the frontend in `frontend/contexts/notification-context.tsx:59-71`:
```typescript
const data = await api.get<{ events: WSMessage[] }>(
    `/api/events?since=${lastEventIdRef.current}&limit=200`
);
for (const evt of data.events) {
    dispatch(evt);
}
```
Each event triggers handler callbacks, potentially causing 200 API refetches if handlers invalidate query caches.

## Expected Behavior
1. The catchup endpoint should enforce a maximum age for events (e.g., 24-48 hours).
2. If `since` points to an event older than the cutoff, return HTTP 410 Gone with a message indicating the client should perform a full data refetch instead.
3. The frontend should handle the 410 response by invalidating all query caches and refetching current data.

## Files to Modify
- `backend/internal/handler/events.go:34-41` -- Add a time-based cutoff to the query: `AND created_at > now() - interval '48 hours'`; return 410 if sinceID is too old
- `frontend/contexts/notification-context.tsx:59-71` -- Handle 410 response from catchup endpoint; trigger full refetch instead of event replay
- `frontend/lib/query-client.ts` -- Add a function to invalidate all queries for full refetch

## Impact Analysis

### What This Affects
- Database query performance during reconnection
- Client-side processing load during catchup
- Network bandwidth for long-offline clients

### What Affects This
- Event pruner retention period (determines maximum possible catchup range)
- Client offline duration (determines frequency of 410 responses)
- Event volume per org (determines cost of large catchup queries)

## Implementation Notes
- Add a time-based check to the `Since` handler:
  ```go
  // Check if sinceID is too old (>48 hours)
  if sinceID > 0 {
      var eventAge time.Time
      err := h.DB.QueryRow(ctx,
          "SELECT created_at FROM events WHERE id = $1", sinceID,
      ).Scan(&eventAge)
      if err != nil || time.Since(eventAge) > 48*time.Hour {
          writeError(w, http.StatusGone, "events too old, please refetch")
          return
      }
  }
  ```
- Alternatively, simply add `AND created_at > now() - interval '48 hours'` to the main query. If zero rows are returned and `sinceID > 0`, return 410.
- On the frontend, handle 410:
  ```typescript
  try {
      const data = await api.get<{ events: WSMessage[] }>(...);
      // ... process events
  } catch (err) {
      if (err instanceof ApiError && err.status === 410) {
          // Too old — invalidate all caches and refetch
          queryClient.invalidateQueries();
          lastEventIdRef.current = 0;
          return;
      }
  }
  ```
- Make the catchup age limit configurable: `EVENT_CATCHUP_MAX_AGE_HOURS` (default: 48).

## User Notes
- Clients that have been offline for less than 48 hours will catch up on missed events as before.
- Clients offline for more than 48 hours will see a brief loading state as all data is refreshed instead of event replay.
- No data loss -- the full refetch ensures the client has current data.

- [ ] Frontend handles 410 by invalidating caches and performing full refetch
