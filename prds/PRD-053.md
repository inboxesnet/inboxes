# PRD-053: Worker Panics

## Status
Done

## Priority
P1-High

## Category
Error Handling — Backend

## Source
GAP-FINDER E5

---

## Problem Statement
The background workers (email worker, sync worker, trash collector, event pruner, domain heartbeat, status recovery) have no panic recovery in their main processing loops. A single unexpected panic (nil pointer dereference, index out of range, type assertion failure) kills the worker goroutine permanently. No more jobs are processed, no alert fires, and the system silently degrades. Since workers run as goroutines within the main process, a dead worker does not cause a process restart.

## Current Behavior
Worker main loops:

- `backend/internal/queue/email_worker.go:31-57` — `Run()` loops with `for { select/default; BRPop; processJob }`. No `recover()`.
- `backend/internal/worker/sync_worker.go:30-57` — `Run()` loops similarly. No `recover()`.
- `backend/internal/worker/trash_collector.go:22-37` — `Run()` loops on a ticker. No `recover()`.
- `backend/internal/worker/event_pruner.go:20-45` — `Run()` loops on a ticker. No `recover()`.
- `backend/internal/worker/domain_heartbeat.go` — Runs on a ticker. No `recover()`.
- `backend/internal/worker/status_recovery.go` — Runs on a ticker. No `recover()`.

Additionally, the per-job processing functions (`processJob`, `processFetch`, `processSend`, `collect`, `prune`) have no `recover()`, so a panic inside any of them kills the entire worker loop.

The `util.SafeGo` helper exists and IS used for some goroutines (heartbeat goroutines, retry goroutines), but it is not wrapping the main worker loops or individual job processing.

## Expected Behavior
1. Each worker's main loop should have `defer recover()` that logs the panic, increments a metric counter, and continues processing. The loop should not exit on panic.
2. Individual job processing (`processJob`, `processFetch`, `processSend`) should be wrapped in a `recover()` that marks the specific job as failed with the panic message, then continues to the next job.
3. A panic counter/metric should be exposed so monitoring systems can alert on elevated panic rates.
4. After recovery, the worker should sleep briefly (1-2 seconds) before continuing, to avoid tight loops on persistent panics.

## Impact Analysis

### What This Affects
- Email processing reliability (inbound and outbound)
- Sync job processing
- Trash auto-deletion
- Event log cleanup
- Domain health monitoring
- Stale job recovery

### What Affects This
- Malformed data in the database causing unexpected nil values
- External API responses with unexpected formats
- Memory pressure causing allocation panics

### Related PRDs
- PRD-049 (Unchecked Queries) — nil pointer panics from uninitialized scan variables are the most likely panic source
- PRD-051 (JSON Errors) — nil json.RawMessage can cause panics in downstream processing
- PRD-052 (Redis Webhook) — if the email worker dies from a panic, orphaned jobs accumulate with no processor

## Implementation Notes
- Create a `recoverWorker(name string)` helper function that can be deferred at the top of each worker loop iteration:
  ```go
  func recoverWorker(name string) {
      if r := recover(); r != nil {
          slog.Error("worker panic recovered", "worker", name, "panic", r, "stack", string(debug.Stack()))
          time.Sleep(2 * time.Second)
      }
  }
  ```
- For per-job recovery, the job should be marked as "failed" with the panic message so operators can inspect it.
- The `util.SafeGo` helper already has panic recovery — use it as a reference pattern.
- Consider adding a simple panic counter (atomic int64) per worker that can be checked via a health endpoint.
- The sleep after recovery prevents tight panic loops (e.g., if the same corrupt job keeps being dequeued and panicking).

## User Notes
Users will not notice a difference in normal operation. In edge cases where a worker encounters a corrupt email or unexpected data, the individual job will fail and be retried (or permanently failed after max retries), but the worker will continue processing other jobs instead of silently dying.

## Acceptance Criteria
- [ ] Every worker's `Run()` loop has panic recovery that logs the panic with stack trace and continues
- [ ] Email worker's `processJob()` has panic recovery that marks the job as failed
- [ ] Sync worker's `processJob()` has panic recovery that marks the job as failed
- [ ] After a recovered panic, workers sleep 2 seconds before continuing
- [ ] A test verifies that a panicking job does not kill the worker loop
- [ ] Panic recovery logs include the full stack trace for debugging
- [ ] Panic count is logged periodically (or exposed via a metrics endpoint)
