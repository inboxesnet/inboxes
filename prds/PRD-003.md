# PRD-003: Open Redirect in PaymentWall (Stripe URL Validation)

## Status
Done

## Priority
P0-Critical

## Category

## Source

## Problem Statement
The PaymentWall component validates the Stripe checkout URL using `parsed.hostname.endsWith("stripe.com")`, which can be bypassed with a domain like `stripe.com.evil.com`. This could redirect users to a phishing site that mimics Stripe's checkout, stealing payment information.

## Current Behavior
In `frontend/components/payment-wall.tsx`, the `handleUpgrade` function (line 26-37):
- Line 30: `const parsed = new URL(res.url);`
- Line 31: `if (!parsed.hostname.endsWith("stripe.com")) throw new Error("Invalid checkout URL");`
- Line 32: `window.location.href = res.url;`
- The hostname check passes for any domain ending with the string "stripe.com", including:
  - `stripe.com.evil.com` (subdomain of evil.com)
  - `fakestripe.com` (different TLD prefix)
  - `notstripe.com` (unrelated domain)
- The actual Stripe checkout URL hostname is always `checkout.stripe.com`.

## Expected Behavior
- Use strict hostname equality: `parsed.hostname === "checkout.stripe.com"`.
- Optionally also accept `billing.stripe.com` for the portal redirect if billing portal URLs are ever validated similarly.
- The URL protocol should also be verified as `https:`.

## Files to Modify
- `frontend/components/payment-wall.tsx:31` — Change `parsed.hostname.endsWith("stripe.com")` to `parsed.hostname === "checkout.stripe.com" && parsed.protocol === "https:"`.

## Impact Analysis

### What This Affects
- The upgrade/checkout flow triggered from the PaymentWall dialog.
- No functional change for legitimate Stripe URLs.

### What Affects This
- The `/api/billing/checkout` endpoint (`backend/internal/handler/billing.go:32-119`) generates the checkout URL via the Stripe SDK. Legitimate URLs will always be `https://checkout.stripe.com/...`.
- PRD-006 (Billing URL Injection) addresses a related issue where the `firstDomainID` used in success/cancel URLs could be tampered with.

### Related PRDs
- PRD-006: Billing URL Injection — Both relate to Stripe checkout URL security. PRD-003 validates the redirect URL on the frontend; PRD-006 validates input used to construct callback URLs on the backend.

## Implementation Notes
- This is a single-line frontend fix.
- After the fix, verify that the Stripe SDK always returns URLs with `checkout.stripe.com` hostname by testing a real checkout session.
- The billing portal redirect (`billing.go:157`) returns a portal URL to the frontend, but `payment-wall.tsx` only handles the checkout flow. If portal URLs are also validated anywhere, use `parsed.hostname === "billing.stripe.com"`.

## User Notes
From FULL-SUMMARY.md: "The payment wall checks if a URL 'ends with stripe.com' -- but `stripe.com.evil.com` also passes that check. Use exact domain matching instead."
