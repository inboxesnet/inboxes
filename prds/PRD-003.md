# PRD-003: Email Lifecycle Status — End-to-End

## Problem

We don't verify send status. After sending via Resend API, we assume it worked. There's no confirmation that the email was actually delivered, and no systematic handling of the full status lifecycle: `queued → sent → delivered` (happy path) or `bounced / failed / complained / suppressed` (unhappy paths).

Some plumbing exists (`email.status_updated` WS event, webhook status handler in `webhooks.go`), but completeness is unknown. This PRD covers the full picture — it requires a thorough audit of what's built before implementation.

## Two Delivery Paths

### Path 1: Webhook-based (production, public URL)
Resend pushes status webhooks as events happen. We already receive some of these. Need to audit: which statuses are handled, what updates happen in DB, what surfaces to the user.

### Path 2: Polling-based (self-hosted, localhost, no webhook reachability)
Auto-sync heartbeat already exists for fetching new emails. Extend it to also poll Resend for outbound email status. Logic:
- After sending, email starts in "sent" (our local status)
- Poll Resend for actual status until we reach a terminal state (`delivered`, `bounced`, `failed`)
- Don't poll indefinitely — reasonable TTL (e.g. 48 hours, then assume delivered or flag as unknown)
- Don't poll every email every heartbeat — only emails in non-terminal states
- Bonus statuses (`opened`, `clicked`, `complained`) can be picked up opportunistically but aren't worth dedicated polling

### Both paths must coexist
Webhook path is primary when available. Polling path is fallback for environments where webhooks can't reach the server. Same DB updates and UI behavior regardless of source.

## Resend Status Lifecycle

```
Queued → Scheduled → Sent → Delivered
                         ↘ Bounced
                         ↘ Failed
                         ↘ Delivery Delayed → Delivered / Bounced / Failed

Post-delivery (bonus):
  Delivered → Opened → Clicked
  Delivered → Complained
  Delivered → Suppressed (future sends blocked by Resend)
```

## Required Audit (before implementation)

1. What does `webhooks.go` status handler currently do? Which statuses does it handle?
2. What DB column(s) store email send status? What values?
3. Does `email.status_updated` WS event actually update the frontend UI? How?
4. What happens on bounce/failure today — does the user ever find out?
5. Does `handleEmailStatusWithRetry` (the retry goroutine) handle the race where status webhook arrives before the email record is inserted?
6. What does the Resend GET endpoint return for a single email's status? Rate limits?

## Key Behaviors (post-implementation)

- **Send completes**: Email record created with status `sent`. Start tracking.
- **Terminal success** (`delivered`): Update status, stop tracking. No toast needed (this is the normal case).
- **Terminal failure** (`bounced`, `failed`): Update status, push WS event, toast the user, show error badge on the email in thread view.
- **Complained**: Update status, surface to user — this means recipient marked it as spam.
- **Suppressed**: Update status, warn user — future sends to this address will be blocked by Resend.
- **Delivery delayed**: Update status, no toast (yet). Keep tracking until terminal.
- **Opened/clicked**: Update status if we want read receipts. Nice-to-have, not critical.

## Retry Policy

Resend failures should NOT automatically retry sending the email. If Resend says the send failed, we surface it to the user and let them decide. Silent retries risk duplicate sends and mask real problems (bad address, domain issues, etc.).

## Priority

Medium. Core reliability gap — users currently have no way to know if their email actually arrived.
