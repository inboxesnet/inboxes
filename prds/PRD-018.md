# PRD-018: Replace context.Background() in Goroutines with App Context

## Context

Four places in the backend spawn goroutines or create timeouts using `context.Background()` instead of deriving from the app-level context (`ctx` from `main.go:40`). These operations survive server shutdown — they don't know the server is stopping and keep running until their own timeout expires or the process is killed.

Matters during deploys/restarts: a 60-second webhook retry goroutine can hold a DB connection and delay clean shutdown.

## Instances

| File | Line | Operation | Timeout |
|------|------|-----------|---------|
| `ws/client.go` | 117 | DB query — load user aliases during WS setup | 5s |
| `ws/client.go` | 194 | Redis lookup — token revocation check in writePump | 5s |
| `queue/email_worker.go` | 257 | Redis LPUSH — re-enqueue failed email job after backoff | 10s |
| `handler/webhooks.go` | 269 | Recursive retry — status webhook when email not yet inserted | 60s |

## Fix

Pass the app-level context into these components. Each timeout becomes derived from it instead of `Background()`:

```go
// Before
retryCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)

// After
retryCtx, cancel := context.WithTimeout(appCtx, 10*time.Second)
```

If the server shuts down, `appCtx` cancels, which cancels all derived contexts, which stops all in-flight goroutines.

### Plumbing

The app context already flows into workers (`EmailWorker`, `SyncWorker`, etc.) via their `Run(ctx)` methods. The changes:

- **ws/client.go**: `ServeWS` and `Client` need access to the app context. Pass it through the `Hub` (which is already created in `main.go` with the app context) or add it as a field on `Client`.
- **email_worker.go**: `failJob` spawns retry goroutine. The worker already has `ctx` from its `Run` method — capture it in the closure instead of using `Background()`.
- **webhooks.go**: `handleEmailStatusWithRetry` spawns retry goroutine. Store the app context on the handler struct (injected at startup) and use it here.

## Not Changing

- `main.go:40` — this IS the root context, `Background()` is correct
- `main.go:152` — shutdown timeout needs to outlive the app context by definition
- Test files — `Background()` is standard practice in tests

## Priority

Low-medium. Only matters during deploys, but easy to do and prevents the occasional slow shutdown.
