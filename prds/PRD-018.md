## Problem Statement
The application relies solely on `SameSite=Lax` cookies for CSRF protection. While this is sufficient today because all state-changing endpoints use POST/PATCH/DELETE methods (which `SameSite=Lax` blocks from cross-origin), this is a fragile defense. A future developer could add a GET endpoint with side effects and unknowingly create a CSRF vulnerability. There is no defense-in-depth mechanism and no documentation of the CSRF strategy.

## Current Behavior
In `backend/internal/middleware/auth.go:96-107`, the auth cookie is set with `SameSite=Lax`:

```go
func SetTokenCookie(w http.ResponseWriter, token, appURL string) {
    secure := strings.HasPrefix(appURL, "https")
    http.SetCookie(w, &http.Cookie{
        Name:     "token",
        Value:    token,
        Path:     "/",
        MaxAge:   7 * 24 * 60 * 60,
        HttpOnly: true,
        Secure:   secure,
        SameSite: http.SameSiteLaxMode,
    })
}
```

No additional CSRF protection exists. There is no `X-Requested-With` header check, no CSRF token, and no documentation of why `SameSite=Lax` alone is considered sufficient.

The `ValidateContentType` middleware in `backend/internal/middleware/security.go:25-39` rejects non-JSON content types for mutation methods, which provides some incidental CSRF protection (browsers cannot send `application/json` cross-origin without CORS preflight). However, it explicitly allows empty Content-Type headers (line 29: `if ct != ""`) to support webhook endpoints.

In `backend/internal/router/router.go:37`, CORS is configured via `middleware.CORSMiddleware(appURL)` which restricts allowed origins, but CORS protects against cross-origin JavaScript requests, not form-based CSRF.

## Expected Behavior
1. Add an `X-Requested-With` header check as defense-in-depth for all authenticated endpoints. This header cannot be set by cross-origin form submissions or simple requests — only JavaScript can set custom headers, which triggers a CORS preflight.
2. Document the CSRF protection strategy in code comments explaining why `SameSite=Lax` + `X-Requested-With` is sufficient.
3. Audit all GET endpoints to confirm none have side effects (state changes).

## Files to Modify
- `backend/internal/middleware/security.go` — Add `X-Requested-With` header validation for authenticated state-changing requests
- `backend/internal/middleware/auth.go:27-62` — Add `X-Requested-With` check inside `AuthMiddleware` for POST/PATCH/PUT/DELETE methods
- `frontend/lib/api.ts` — Ensure all API calls include `X-Requested-With: XMLHttpRequest` header

## Impact Analysis

### What This Affects
- All authenticated API calls from the frontend need to include the `X-Requested-With` header
- Webhook endpoints (`/api/webhooks/resend/{orgId}`, `/api/webhooks/stripe`) must be excluded from the check since they are called by external services
- API clients or integrations (if any) would need to include the header

### What Affects This
- CORS configuration — the `X-Requested-With` header must be listed in `Access-Control-Allow-Headers`
- Webhook routes are defined outside the authenticated group (`router.go:123-127`), so they would not be affected by a check inside `AuthMiddleware`

### Related PRDs
- PRD-015: Admin Rate Limits — Rate limiting adds another layer of protection against automated attacks
- PRD-019: Token Blacklist Fail-Open — CSRF protection becomes more important if token revocation is unreliable

## Implementation Notes
The simplest approach is to add the check inside `AuthMiddleware` (which only wraps authenticated routes). For POST/PATCH/PUT/DELETE requests, require `X-Requested-With: XMLHttpRequest`. GET requests are safe because `SameSite=Lax` cookies are only sent on top-level navigations (not embedded requests).

Webhook endpoints are outside the auth middleware group, so they are unaffected.

The frontend `api.ts` helper should be updated to include the header in all requests. Check if the existing fetch wrapper already sets custom headers.

Key exception: The WebSocket upgrade request (`/api/ws` at `router.go:130-132`) is a GET that occurs outside the auth middleware. The WebSocket upgrade handler in `ws/client.go:39-114` validates the cookie independently and also checks the Origin header via the upgrader's `CheckOrigin` function (lines 64-74).
