# PRD-114: Default Database Password is "inboxes"

## Status
Done

## Priority
P2-Medium

## Category
Infrastructure

## Source
PRE-LAUNCH-AUDIT #34

---

## Problem Statement
The default PostgreSQL password in `.env.example` and `docker-compose.yml` is the word "inboxes." Operators who deploy by copying `.env.example` without changing the password expose their database to trivial unauthorized access. Since the compose file also exposes PostgreSQL on the internal Docker network, any compromised container can connect to the database with the known default credentials.

## Current Behavior
In `.env.example` (lines 11-12):
```
POSTGRES_USER=inboxes
POSTGRES_PASSWORD=inboxes
```

In `docker-compose.yml` (lines 6-8):
```yaml
environment:
  POSTGRES_USER: ${POSTGRES_USER:-inboxes}
  POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-inboxes}
```

The `:-inboxes` default fallback means that even if the operator forgets to set `POSTGRES_PASSWORD`, the database starts with the known password "inboxes." There is no startup check or warning when the default password is in use.

## Expected Behavior
- `docker-compose.yml` should use `${POSTGRES_PASSWORD:?Set POSTGRES_PASSWORD in .env}` to require an explicit password, failing fast if unset.
- `.env.example` should contain `POSTGRES_PASSWORD=` (empty) with a comment explaining how to generate a secure password.
- Alternatively, keep the default for local development but add a backend startup warning when the database password is "inboxes" or less than 12 characters.

## Files to Modify
- `.env.example:12` — Change default to empty with a comment: `# Generate with: openssl rand -hex 16`.
- `docker-compose.yml:7` — Change to `${POSTGRES_PASSWORD:?Set POSTGRES_PASSWORD}` to require explicit setting.
- `docker-compose.yml:37` — Update `DATABASE_URL` to match the required variable pattern.
- `backend/internal/config/config.go` — Optionally add a startup warning if the DATABASE_URL contains `password=inboxes` or `:inboxes@`.

## Impact Analysis

### What This Affects
- All new deployments using `docker compose up` — will fail until password is set.
- Existing deployments that already have `POSTGRES_PASSWORD` set will be unaffected.
- Local development workflow — developers will need to set the password in their `.env`.

### What Affects This
- The `DATABASE_URL` connection string in the backend service, which interpolates the password.
- PostgreSQL healthcheck (uses `POSTGRES_USER` but not password directly).

### Related PRDs
- PRD-112: PUBLIC_URL — similar pattern of dangerous defaults in compose.
- PRD-113: Resource Limits — production-readiness of compose configuration.
- PRD-016: DB SSL — database connection security (password in cleartext over network).
- PRD-017: Redis Auth — similar default-no-auth issue for Redis.

## Implementation Notes
Deployment instructions should enforce changing this. The `${VAR:?message}` pattern is the simplest approach — it causes `docker compose up` to fail immediately with a clear error. For local development, a `docker-compose.override.yml` or `.env` file can set the password to "inboxes" for convenience.

Consider also adding a check in `scripts/setup.sh` that warns if the password is still the default.
