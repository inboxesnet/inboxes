# PRD-076: Disabled Users' Aliases Still Receive Emails

## Status
Done

## Priority
P2-Medium

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L5

---

## Problem Statement
When a user is disabled, their aliases are not deactivated. Inbound emails addressed to a disabled user's alias can route through the alias lookup in `processFetch`, find the alias, but then fail to find an active user via `alias_users` -- causing the email to fall through to the catch-all admin route. If no admin is active, the email fails silently. The design note says this is intentional (emails roll up to whoever the disabled user was merged into), but the code path is fragile and undocumented.

## Current Behavior
In `backend/internal/queue/process_fetch.go`:
- Lines 411-454: The `routeEmail` function performs 3-phase routing:
  1. Phase 1 (lines 417-423): Direct user match by email + `status = 'active'`. Fails for disabled users.
  2. Phase 2 (lines 426-442): Alias match. The alias lookup (line 428) does NOT check user status. Then `alias_users` JOIN to `users` (lines 432-438) filters by `u.status = 'active'`, so if the only user on the alias is disabled, no match is found.
  3. Phase 3 (lines 445-451): Catch-all org admin. If an active admin exists, the email routes to them.
- If the disabled user was reassigned (via `users.go` `reassignAndDisable` at lines 239-341), `alias_users` records are transferred to the target user, so Phase 2 would find the target user. However, if the user was disabled WITHOUT reassignment (lines 221-237), `alias_users` records are NOT cleaned up.

In `backend/internal/handler/users.go`:
- Lines 220-237: Simple disable without reassignment does not touch `alias_users`. The disabled user's alias associations remain, but since Phase 2 filters by `status = 'active'`, they are effectively dead.
- Lines 239-341: Disable with reassignment copies `alias_users` to the target and deletes the source's records. This works correctly.

## Expected Behavior
- When a user is disabled WITHOUT reassignment, their `alias_users` records should either:
  - Option A: Be deleted, causing emails to fall through to the admin catch-all (current implicit behavior, but make it explicit and documented).
  - Option B: Be transferred to the first active admin, similar to the reassignment flow but automatic.
- Add a log warning when an email routes via catch-all because alias-to-user resolution failed due to disabled users.
- Document the intended behavior so future developers understand the routing fallback.

## Files to Modify
- `backend/internal/handler/users.go:220-237` — After disabling the user, clean up their `alias_users` records:
  ```go
  h.DB.Exec(ctx, `DELETE FROM alias_users WHERE user_id = $1`, userID)
  ```
- `backend/internal/queue/process_fetch.go:426-442` — Add a log message when alias is found but no active user is associated:
  ```go
  slog.Warn("email worker: alias found but no active user", "alias", address, "alias_id", aliasID)
  ```

## Impact Analysis

### What This Affects
- Email routing for addresses associated with disabled users.
- Simple user disable (without reassignment) will now clean up alias associations, making the catch-all routing explicit.
- No impact on disable-with-reassignment, which already handles alias transfer.

### What Affects This
- The user enable flow (`users.go` `Enable`, lines 560-579) does not restore `alias_users` records. Re-enabling a user after this fix will require manual alias re-assignment by an admin.
- PRD-072 (Domain Deletion Cleanup) also needs to clean up `alias_users`.

### Related PRDs
- PRD-072: Domain Deletion Data Cleanup — Both involve cleaning up `alias_users` records as part of lifecycle management.
- PRD-009: Disabled Users Auth — Disabled users should not be able to authenticate; this PRD addresses the email routing side.
- PRD-024: Last Admin Protection — If all admins are disabled, the catch-all route (Phase 3) fails. PRD-024 prevents this scenario.

## Implementation Notes
- The simple approach: delete `alias_users` on user disable. This is consistent with the "disabled means gone" mental model.
- Alternative: keep `alias_users` but mark them inactive with a flag. This preserves the association for re-enable scenarios but adds complexity.
- The recommended approach is the simple delete, since admins can re-assign aliases after re-enabling a user.
- Consider adding a warning in the frontend when disabling a user who has active aliases: "This user has N active aliases. Emails to these aliases will be routed to an org admin."
