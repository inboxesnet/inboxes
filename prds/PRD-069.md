# PRD-069: RTL Text Support

## Status
Draft

## Priority
P2-Medium

## Category
Email Threading & Rendering

## Source
GAP-FINDER T4

---

## Problem Statement
Emails written in right-to-left (RTL) languages such as Hebrew, Arabic, and Persian display incorrectly because the email content container lacks `dir="auto"` or any other bidirectional text detection. All text renders left-to-right regardless of its actual script direction, making RTL emails unreadable.

## Current Behavior
In `frontend/components/thread-view.tsx`:
- Lines 446-449: The `dangerouslySetInnerHTML` div that renders sanitized email HTML has extensive Tailwind classes for styling but no `dir` attribute:
  ```tsx
  <div
    className="max-w-2xl text-[13px] leading-relaxed ..."
    dangerouslySetInnerHTML={{ __html: sanitizedHtml }}
  />
  ```
- Lines 451-453: The `<pre>` fallback for plain-text emails also lacks any `dir` attribute:
  ```tsx
  <pre className="text-[13px] whitespace-pre-wrap font-sans ...">
    {email.body_plain}
  </pre>
  ```
- The DOMPurify sanitizer config (lines 369-380) allows `style` but does not include `dir` in `ALLOWED_ATTR`, so even if the original email HTML contains `dir="rtl"`, it gets stripped.

## Expected Behavior
- Both the HTML container `<div>` and the plain-text `<pre>` element should include `dir="auto"`, which lets the browser detect the dominant text direction from the first strong character.
- The DOMPurify config should allow the `dir` attribute so that inline direction hints from the original email are preserved.
- Mixed-direction content (e.g., Arabic text with embedded English) should render correctly with the browser's native bidi algorithm.

## Files to Modify
- `frontend/components/thread-view.tsx:447` — Add `dir="auto"` to the HTML content div.
- `frontend/components/thread-view.tsx:451` — Add `dir="auto"` to the plain-text pre element.
- `frontend/components/thread-view.tsx:376-378` — Add `"dir"` to the `ALLOWED_ATTR` array in the DOMPurify config.

## Impact Analysis

### What This Affects
- Rendering of all email content in the thread view.
- Emails in LTR scripts are unaffected -- `dir="auto"` defaults to LTR when the first strong character is LTR.
- Quoted replies and forwarded content will also benefit from correct direction detection.

### What Affects This
- DOMPurify sanitization must allow the `dir` attribute (change included above).
- The TipTap editor in `frontend/components/tiptap-editor.tsx` may also need `dir="auto"` for composing RTL replies, but that is a separate enhancement.

### Related PRDs
- PRD-014: Malicious CSS — CSS sanitization is related to rendering security; `dir` is safe and does not create a vector.

## Implementation Notes
- `dir="auto"` is the correct choice over `dir="rtl"` because it handles mixed-direction content and does not force RTL on English emails.
- This is a three-line frontend change with no backend work.
- Verify that email subjects in the thread header also display correctly -- the subject text in the `<h2>` at the top of ThreadView may also need `dir="auto"`.
