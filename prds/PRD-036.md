# PRD-036: Trash Countdown

## Status
Done

## Priority
P2-Medium

## Category
Session & Auth UX

## Source

## Problem Statement
When viewing a trashed thread, a banner at the top says "This conversation will be permanently deleted in X days." However, this value is computed once on render and never updates. If a user leaves the tab open for hours or days, the countdown remains stale, potentially displaying "3 days" when the actual time remaining is 1 day or even negative (already deleted). This creates a false sense of safety -- the user thinks they have more time to recover the thread than they actually do.

## Current Behavior
In `thread-view.tsx` (lines 298-303), the trash countdown banner is rendered as:
```tsx
{thread.trash_expires_at && (
  <div className="bg-destructive/10 text-destructive text-xs text-center py-1.5 px-4 border-b shrink-0">
    This conversation will be permanently deleted in{" "}
    {Math.max(0, Math.ceil((new Date(thread.trash_expires_at).getTime() - Date.now()) / (1000 * 60 * 60 * 24)))} days
  </div>
)}
```
This calculation uses `Date.now()` at render time. The value is never recalculated because:
1. There is no `useEffect` + `setInterval` to periodically update.
2. The component does not re-render unless the thread data changes.
3. The `thread.trash_expires_at` value is static from the API response.

## Expected Behavior
1. The countdown should update periodically (every hour is sufficient -- days-level granularity does not need real-time updates).
2. When the countdown reaches 0 days, the text should change to "This conversation will be permanently deleted soon" or "This conversation is scheduled for deletion."
3. If the expiry time has passed, display "This conversation has been deleted" or hide the banner.

## Files to Modify
- `frontend/components/thread-view.tsx:298-303` -- Extract the countdown into a separate `TrashCountdown` component that uses `useState` + `useEffect` with a `setInterval` to recalculate the remaining time.

## Impact Analysis

### What This Affects
- Users viewing trashed threads, especially those who leave tabs open for extended periods.
- The urgency signal -- a stale "3 days" when only 1 day remains could cause users to miss the recovery window.

### What Affects This
- The `trash_expires_at` timestamp from the thread API response.
- The trash collector background job interval -- it runs periodically to delete expired trash, so the actual deletion may lag behind the timestamp.
- Browser tab visibility -- tabs in the background may have throttled timers.

### Related PRDs
- PRD-057 (N+1 Trash) -- trash expiry date fetching performance
- PRD-033 (Scroll Position Lost) -- users navigating through trash threads

## Implementation Notes
- **Simple interval approach**:
  ```tsx
  function TrashCountdown({ expiresAt }: { expiresAt: string }) {
    const [now, setNow] = useState(Date.now());

    useEffect(() => {
      const interval = setInterval(() => setNow(Date.now()), 1000 * 60 * 60); // every hour
      return () => clearInterval(interval);
    }, []);

    const msRemaining = new Date(expiresAt).getTime() - now;
    const daysRemaining = Math.max(0, Math.ceil(msRemaining / (1000 * 60 * 60 * 24)));

    if (msRemaining <= 0) {
      return (
        <div className="bg-destructive/10 text-destructive text-xs text-center py-1.5 px-4 border-b shrink-0">
          This conversation is scheduled for permanent deletion
        </div>
      );
    }

    return (
      <div className="bg-destructive/10 text-destructive text-xs text-center py-1.5 px-4 border-b shrink-0">
        This conversation will be permanently deleted in {daysRemaining} day{daysRemaining !== 1 ? "s" : ""}
      </div>
    );
  }
  ```
- An hourly interval is sufficient since the granularity is days. Minute-level or second-level updates would be wasteful.
- Consider using `requestAnimationFrame` or `document.visibilitychange` to recalculate immediately when the tab becomes visible again after being backgrounded.
- The singular/plural "day" vs "days" should be handled correctly.
- For countdowns less than 24 hours, consider showing "less than 1 day" or hours for more precision.

## User Notes
- This is a minor polish item. The stale countdown is unlikely to cause real harm in practice because most users do not leave trash threads open for days.
- The fix is low-effort and improves perceived quality of the trash management feature.

## Acceptance Criteria
- [ ] The trash countdown banner updates its displayed value at least once per hour
- [ ] When the expiry time has passed, the banner text changes to indicate imminent/completed deletion
- [ ] The countdown displays correct singular/plural for "day" vs "days"
- [ ] Switching tabs and returning recalculates the countdown immediately
- [ ] The countdown does not cause unnecessary re-renders of the entire ThreadView component
- [ ] The banner is not shown if `trash_expires_at` is null or undefined
