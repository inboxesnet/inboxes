# PRD-020: Attachment MIME Bypass

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
PRE-LAUNCH-AUDIT #12

---

## Problem Statement
The attachment upload handler blocks executable file types but trusts the client-provided `Content-Type` for `text/*` subtypes. A user could upload a malicious HTML or SVG file with `Content-Type: text/html`, and when another user downloads it, the browser may render the HTML (executing embedded scripts) instead of treating it as a safe download. Although the download endpoint sets `Content-Disposition: attachment`, not all browsers strictly enforce this for all content types, and the content type is still set to the attacker-controlled value.

## Current Behavior
In `backend/internal/handler/attachments.go:40-62`, the upload handler detects the actual MIME type using Go's `http.DetectContentType()` but then has a carve-out for `text/*` types:

```go
// Detect actual MIME type from file content to block dangerous uploads
detected := http.DetectContentType(data)
blockedTypes := map[string]bool{
    "application/x-executable":     true,
    "application/x-msdos-program":  true,
    "application/x-msdownload":     true,
    "application/x-dosexec":        true,
    "application/vnd.microsoft.portable-executable": true,
}
if blockedTypes[detected] {
    writeError(w, http.StatusBadRequest, "file type not allowed")
    return
}

contentType := header.Header.Get("Content-Type")
if contentType == "" {
    contentType = "application/octet-stream"
}
// Use detected type, but trust client for text/* subtypes (plain, html, csv, etc.)
if detected != "application/octet-stream" && (len(contentType) < 5 || contentType[:5] != "text/") {
    contentType = detected
}
```

This means `text/html`, `text/xml`, and `text/svg+xml` (which can all contain executable JavaScript) are stored with their dangerous content types.

In `backend/internal/handler/attachments.go:83-103`, the download handler serves the file with the stored content type:

```go
w.Header().Set("Content-Type", contentType)
w.Header().Set("Content-Disposition", fmt.Sprintf(`attachment; filename="%s"`, filename))
```

While `Content-Disposition: attachment` instructs browsers to download rather than render, this is belt-without-suspenders. Some browsers or browser extensions may still render certain content types inline, and the filename injection issue (PRD item #2) compounds the risk.

## Expected Behavior
1. For downloads, always force `Content-Type: application/octet-stream` regardless of stored content type. This ensures browsers treat all downloads as opaque binary files.
2. Alternatively, maintain a whitelist of known-safe content types that can be served with their real content type (e.g., `image/png`, `image/jpeg`, `image/gif`, `application/pdf`) and force everything else to `application/octet-stream`.
3. Add `X-Content-Type-Options: nosniff` header to download responses (already set globally in `security.go:11`, but explicitly setting it on the download response adds defense-in-depth).

## Files to Modify
- `backend/internal/handler/attachments.go:59-62` — During upload, do not trust client Content-Type for `text/*`; use detected type or force to `application/octet-stream`
- `backend/internal/handler/attachments.go:99` — During download, force safe content types; either always use `application/octet-stream` or whitelist safe MIME types
- `backend/internal/handler/attachments.go:100` — Ensure the `Content-Disposition` header also sanitizes the filename (related to PRD item #2, but relevant here as defense-in-depth)

## Impact Analysis

### What This Affects
- Attachment download behavior — browsers will always download files rather than attempting to render them
- Image previews — if images are ever served inline (not currently the case), they would need to use the whitelist approach
- Email attachments with `text/html` content type will be forced to safe download

### What Affects This
- The `X-Content-Type-Options: nosniff` header from `backend/internal/middleware/security.go:11` already prevents MIME sniffing
- Browser behavior varies; `Content-Disposition: attachment` is well-supported but not universally enforced

### Related PRDs
- PRD-018: CSRF — CSRF protection prevents cross-origin requests that could trick users into downloading malicious files
- PRD-019: Token Blacklist Fail-Open — A compromised session could be used to upload malicious attachments

## Implementation Notes
The safest approach is a whitelist of MIME types that are safe to serve with their real content type:

```go
safeTypes := map[string]bool{
    "image/png":  true,
    "image/jpeg": true,
    "image/gif":  true,
    "image/webp": true,
    "application/pdf": true,
    "text/plain": true,
    "text/csv":   true,
}
```

Everything not in the whitelist gets served as `application/octet-stream`. This preserves the user experience for common file types while blocking dangerous content.

For the upload path, consider storing the detected content type instead of the client-provided one. This makes the database an accurate reflection of what was actually uploaded.

Also consider adding `text/html` and `image/svg+xml` to the block list during upload if there is no legitimate use case for uploading HTML files as email attachments.

## Acceptance Criteria
- [ ] Uploading a file with `Content-Type: text/html` does not store `text/html` as the content type
- [ ] Downloading any attachment serves with either `application/octet-stream` or a whitelisted safe content type
- [ ] Image attachments (PNG, JPEG, GIF) can still be served with their correct content type
- [ ] `X-Content-Type-Options: nosniff` header is present on download responses
- [ ] Uploading an SVG file with embedded JavaScript either blocks the upload or forces a safe content type on download
- [ ] Existing attachments with dangerous content types are served safely (no migration needed — the download handler forces safe types)
