# PRD-115: Migration Rollback Documentation

## Status
Draft

## Priority
P2-Medium

## Category
Infrastructure

## Source
PRE-LAUNCH-AUDIT #35

---

## Problem Statement
There are 31 migration files in `backend/internal/db/migrations/` and migrations run automatically on server startup via goose. If a migration breaks production (e.g., a column rename that causes the running application to error, or a bad data backfill), there is no documented procedure for rolling back. The operations guide (`docs/operations.md`) documents schema management and backup/restore but does not cover migration rollback.

## Current Behavior
- `docs/operations.md` (line 98) states: "Migrations are managed by goose and run automatically on server startup." It lists tables and backup procedures but has no rollback section.
- Each migration file has a `-- +goose Down` section (e.g., `backend/internal/db/migrations/028_attachments.sql:17-22`) defining the rollback SQL. However, there is no documentation on how to invoke `goose down`, what order to follow, or what data loss to expect.
- The backend binary auto-runs `goose up` on startup (`backend/cmd/api/main.go`), so restarting the backend after a rollback would re-apply the migration unless the binary is also rolled back.

## Expected Behavior
- `docs/operations.md` should include a "Migration Rollback" section explaining:
  1. How to run `goose down` to roll back one migration.
  2. How to prevent auto-migration on restart (e.g., deploy the previous binary version first).
  3. Which migrations are safe to roll back (additive changes) vs. destructive (column drops, data transforms).
  4. A pre-rollback checklist: stop the backend, take a database backup, run the rollback, verify, restart with the matching binary.
- A table or annotations in the migration files indicating rollback safety.

## Files to Modify
- `docs/operations.md` — Add a "Migration Rollback" section after the "Schema Management" section (after line 98).

## Impact Analysis

### What This Affects
- Operations team confidence in deploying schema changes.
- Recovery time when a migration causes production issues.
- The decision to auto-migrate on startup vs. requiring explicit migration commands.

### What Affects This
- The goose migration tool and its `down` command syntax.
- Whether the Go binary version matches the expected schema version.
- Database backup freshness (rollback may cause data loss for rows created after the migration).

## Implementation Notes
Example rollback procedure to document:
```bash
# 1. Stop the backend to prevent auto-migration on restart
docker compose stop backend

# 2. Back up the database
docker exec $(docker ps -qf name=postgres) \
  pg_dump -U inboxes -Fc inboxes > pre_rollback_$(date +%Y%m%d_%H%M%S).dump

# 3. Check current migration version
docker exec $(docker ps -qf name=postgres) \
  psql -U inboxes -d inboxes -c "SELECT * FROM goose_db_version ORDER BY id DESC LIMIT 5;"

# 4. Roll back one migration
cd backend && goose -dir internal/db/migrations postgres "$DATABASE_URL" down

# 5. Deploy the previous backend version (matching the rolled-back schema)
# Then restart:
docker compose up -d backend
```

Also document which migrations have destructive `Down` sections (e.g., `DROP TABLE`, `DROP COLUMN`) so operators know the risk.
