# PRD-085: Past-Due

## Status
Done

## Priority
P1-High

## Category
Billing & Stripe

## Source
GAP-FINDER B5

---

## Problem Statement
When Stripe marks a subscription as `past_due` (payment failed but Stripe is still retrying), the app immediately sets the org plan to `cancelled`. This causes users to see "Cancelled" in the billing UI and lose access to paid features, even though Stripe may successfully retry the payment within hours or days. If the retry succeeds, the `customer.subscription.updated` webhook restores the plan, but users panic in the interim and may contact support or attempt to re-subscribe.

## Current Behavior
In `backend/internal/handler/billing.go:286-293`:
```go
case stripe.SubscriptionStatusPastDue, stripe.SubscriptionStatusUnpaid:
    plan = "cancelled"
```

And in `billing.go:343-347`, `invoice.payment_failed` also sets:
```go
`UPDATE orgs SET plan = 'cancelled', plan_expires_at = $1 ...`
```
with a 3-day grace period.

This means any payment failure immediately marks the org as "cancelled" with a grace period. Stripe's default retry schedule is 3 attempts over ~7 days. If the second or third retry succeeds, the `customer.subscription.updated` (status=active) and `invoice.payment_succeeded` events will restore the plan. But during the retry window, users see "Cancelled" and may lose access after the 3-day grace period expires, even though Stripe hasn't given up yet.

## Expected Behavior
1. Introduce a `past_due` plan status distinct from `cancelled`.
2. When `subscription.updated` arrives with `past_due` status, set `plan = 'past_due'` (not `cancelled`).
3. When `invoice.payment_failed` arrives, set `plan = 'past_due'` with a longer grace period (e.g., 14 days, matching Stripe's retry window).
4. `RequirePlan` middleware should treat `past_due` the same as `pro` (allow access while Stripe retries).
5. The billing UI should show "Past Due" with a warning banner instead of "Cancelled".
6. Only set `plan = 'cancelled'` when `customer.subscription.deleted` arrives (Stripe has given up or user explicitly cancelled).

## Files to Modify
- `backend/internal/handler/billing.go:286-293` -- Change `PastDue` mapping from `"cancelled"` to `"past_due"`
- `backend/internal/handler/billing.go:343-347` -- Change `invoice.payment_failed` to set `plan = 'past_due'`
- `backend/internal/middleware/auth.go:172-179` -- Add `past_due` as an allowed plan status in `RequirePlan`
- `frontend/components/settings-modal.tsx:1135-1136` -- Add `past_due` to the plan badge display logic
- `frontend/components/settings-modal.tsx` -- Add a warning banner for past_due plans
- `frontend/lib/types.ts` -- Update `BillingInfo` type if needed

## Impact Analysis

### What This Affects
- User access during payment retry window (currently blocked after 3-day grace; should remain accessible)
- Billing UI display (shows correct "Past Due" status instead of misleading "Cancelled")
- User anxiety and support load during payment failures
- Grace period worker (PRD-084) needs to handle `past_due` -> `free` transition too

### What Affects This
- Stripe's payment retry schedule (configurable in Stripe dashboard)
- `customer.subscription.updated` webhook accuracy
- `invoice.payment_failed` vs `customer.subscription.deleted` event ordering

### Related PRDs
- PRD-081 (Stripe Event Handlers) -- `past_due` is a new status that event handlers must support
- PRD-084 (Grace Period) -- grace period worker should also handle `past_due` -> `free` if subscription is eventually deleted
- PRD-082 (Paywall Flash) -- `past_due` status should not trigger the paywall
- PRD-088 (Cancellation UI) -- billing UI needs to distinguish past_due from cancelled

## Implementation Notes
- The `RequirePlan` middleware change is minimal: add `|| plan == "past_due"` to the `plan == "pro"` check at line 172.
- Consider showing a non-blocking banner in the app for `past_due` orgs: "Your last payment failed. We'll retry automatically. Please update your payment method."
- The billing portal (Stripe hosted) already lets users update payment methods. Link to it from the past_due banner.
- If a CHECK constraint exists on the `plan` column, add `'past_due'` to the allowed values via migration.

## User Notes
- Users with failed payments will see "Past Due" instead of "Cancelled", with a clear message about payment retry.
- Access to the app continues during the payment retry window.
- A link to update payment method is provided.
