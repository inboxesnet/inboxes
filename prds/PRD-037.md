## Problem Statement
The app has a comprehensive set of keyboard shortcuts (j/k for navigation, x for select, e for archive, # for trash, s for star, Cmd+N for compose, / for search, ? for help), but there is no visible hint anywhere in the UI that these shortcuts exist. Users must accidentally press `?` to discover the shortcuts dialog. The shortcuts are a significant productivity feature that goes entirely unnoticed by most users.

## Current Behavior
- `keyboard-shortcuts.tsx` (lines 1-197): Registers global keydown listeners for all shortcuts. Pressing `?` opens the `KeyboardShortcutsDialog`.
- `keyboard-shortcuts-dialog.tsx` (lines 1-71): A dialog component that lists all shortcuts organized by section (Navigation, Actions, Domains, Other).
- `domain-sidebar.tsx`: The sidebar has Settings, Theme toggle, and Logout buttons at the bottom, but no keyboard shortcut hint icon.
- There is no tooltip, badge, onboarding step, or visible UI element that tells users shortcuts are available.

## Expected Behavior
1. Add a small `?` icon button in the sidebar footer (near the Settings/Logout buttons) that opens the shortcuts dialog when clicked.
2. On first visit (or first 3 visits), show a subtle tooltip/popover near the `?` icon: "Press ? for keyboard shortcuts."
3. The `?` icon should be visible on both desktop and mobile sidebar layouts.
4. Consider adding a brief "Keyboard shortcuts: press ?" hint in the search bar area or toolbar on desktop.

## Files to Modify
- `frontend/components/domain-sidebar.tsx:296-315` -- Add a `?` icon button in the desktop sidebar's bottom icon strip (between the Settings and Logout buttons).
- `frontend/components/domain-sidebar.tsx:244-267` -- Add a keyboard shortcuts link in the mobile sidebar's bottom section.
- `frontend/components/domain-sidebar.tsx:48-52` -- Add `onOpenShortcuts` to the `DomainSidebarProps` interface, or manage the dialog state internally within the sidebar.
- `frontend/app/(app)/d/[domainId]/layout.tsx` -- Wire up the shortcuts dialog opening. Currently `KeyboardShortcuts` is rendered here and manages its own dialog state. The sidebar needs a way to open it.

## Impact Analysis

### What This Affects
- Feature discoverability -- shortcuts are a major productivity feature that most users will never find without a visual hint.
- User onboarding experience -- new users have no way to learn that shortcuts exist.
- Power users who switch from other email clients (Gmail shortcuts are very similar) will expect a shortcut help panel.

### What Affects This
- The `KeyboardShortcutsDialog` component -- already built and fully functional.
- The sidebar layout -- adding a button should not disrupt the existing icon layout.
- The `KeyboardShortcuts` component -- its `shortcutsOpen` state needs to be either lifted up or triggered via a custom event.

## Implementation Notes
- **Desktop sidebar** -- Add a `?` button to the icon strip at the bottom of the sidebar:
  ```tsx
  {/* Keyboard Shortcuts */}
  <button
    onClick={handleOpenShortcuts}
    className="flex items-center justify-center h-12 w-12 rounded-[24px] hover:rounded-2xl bg-muted text-muted-foreground hover:bg-accent transition-all duration-200"
    title="Keyboard shortcuts (?)"
  >
    <span className="text-lg font-bold">?</span>
  </button>
  ```
  Place it after the Theme toggle button and before the Settings button (around line 300).

- **Mobile sidebar** -- Add a row in the bottom section:
  ```tsx
  <button
    onClick={() => { handleOpenShortcuts(); onCloseSidebar?.(); }}
    className="flex items-center gap-3 w-full rounded-md px-3 py-2.5 text-sm text-muted-foreground hover:bg-accent/50 hover:text-foreground transition-colors"
  >
    <span className="h-4 w-4 flex items-center justify-center font-bold">?</span>
    Keyboard shortcuts
  </button>
  ```

- **State management options**:
  1. **Custom event**: `window.dispatchEvent(new Event("open-shortcuts"))` -- the `KeyboardShortcuts` component already manages its own state, just add an event listener.
  2. **Lift state**: Pass `onOpenShortcuts` as a prop through the layout hierarchy.
  3. **Internal state**: Render the `KeyboardShortcutsDialog` directly inside the sidebar, separate from the global `KeyboardShortcuts` component.

  Option 1 (custom event) is the simplest and avoids prop drilling.

- **First-visit tooltip** (nice-to-have):
  Use `localStorage.getItem("shortcuts-hint-seen")` to show a tooltip on the first 3 sessions. After that, hide it. This can be implemented as a follow-up.
