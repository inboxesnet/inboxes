# PRD-095: Focus Trap in Compose Window

## Status
Done

## Priority
P3-Low

## Category
Accessibility

## Source
GAP-FINDER A1

---

## Problem Statement
The floating compose window behaves like a modal dialog (overlays content, captures user attention) but lacks proper ARIA dialog semantics and focus trapping. Keyboard users can Tab out of the compose window into the elements behind it, which is disorienting. There is no `role="dialog"`, `aria-modal="true"`, or programmatic focus management when the window opens or closes.

## Current Behavior
In `floating-compose-window.tsx`, the compose window renders as a plain `<div>` (line 408 for mobile, line 546 for desktop). Neither has `role="dialog"` or `aria-modal`. Pressing Tab while in the compose form cycles focus out to the sidebar and main content behind the compose window. When the window opens, focus is not explicitly moved to the first field. When closed, focus does not return to the element that triggered the compose action.

## Expected Behavior
- The compose window should have `role="dialog"` and `aria-modal="true"` on both mobile and desktop containers
- An `aria-label="Compose email"` or `aria-labelledby` pointing to the "New Message" title
- Tab key should cycle only within the compose window (focus trap)
- On open, focus should move to the "To" input (or subject if replying with pre-filled recipients)
- On close, focus should return to the element that opened the compose window (compose button, reply button, etc.)

## Files to Modify
- `frontend/components/floating-compose-window.tsx:408` -- Add `role="dialog"`, `aria-modal="true"`, `aria-label="Compose email"` to mobile compose container
- `frontend/components/floating-compose-window.tsx:546` -- Add `role="dialog"`, `aria-modal="true"`, `aria-label="Compose email"` to desktop compose container
- `frontend/components/floating-compose-window.tsx:39-42` -- Add focus management: save trigger element ref, restore focus on close
- `frontend/contexts/email-window-context.tsx` -- Optionally pass the trigger element ref through context so focus can be restored

## Impact Analysis

### What This Affects
- Keyboard navigation flow -- users will no longer accidentally tab out of compose
- Screen reader announcements -- compose window will be announced as a dialog
- Mobile full-screen compose behavior (acts as a page overlay)

### What Affects This
- The minimized state (line 349-381) does not need focus trapping since it is not a modal
- `TipTapEditor` has its own focus management that must cooperate with the focus trap
- `RecipientInput` autocomplete dropdown focus behavior needs to work within the trap

### Related PRDs
- PRD-094 (ARIA Labels) -- compose window button labels
- PRD-096 (Combobox) -- recipient input accessibility within the trapped focus
- PRD-027 (Compose Draft Loss) -- related close behavior
- PRD-105 (Form Errors) -- error announcements in compose form

## Implementation Notes
- Use a focus-trap library like `focus-trap-react` or implement manually with `MutationObserver` on focusable elements
- The desktop compose window is not a true modal (no backdrop) but still benefits from focus trapping since it floats over content
- Make sure Escape key closes the compose window (already handled by TipTap editor potentially, verify no conflicts)
- The minimized state should NOT trap focus -- only the open/expanded state

## User Notes
Low priority -- app is 100% behind auth, no SEO or AI indexing impact. No legal risk at current scale. Address when there's traction and real screen reader users.

## Acceptance Criteria
- [ ] Both mobile and desktop compose containers have `role="dialog"` and `aria-modal="true"`
- [ ] Pressing Tab while in the compose window cycles focus only within the compose window's focusable elements
- [ ] Opening the compose window moves focus to the first interactive element (To input or subject)
- [ ] Closing the compose window returns focus to the element that triggered it
- [ ] Escape key closes the compose window
- [ ] VoiceOver announces "Compose email, dialog" when the compose window opens
