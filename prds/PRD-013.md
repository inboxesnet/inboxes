# PRD-013: Email Tracking Pixels Load Automatically

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Problem Statement
Every image in every email loads automatically when the user views a thread, including invisible 1x1 tracking pixels. This allows senders to know exactly when, where, and on what device the recipient opened their email. There is no "Load external images" button, no per-sender trust setting, and no global privacy preference. This is a significant privacy concern, especially for a product that handles business email.

## Expected Behavior
- By default, external images in emails should NOT load automatically.
- Replace external `<img src>` URLs with a placeholder or proxy URL.
- Show a "Load external images" button/banner at the top of each email that has blocked images.
- When clicked, replace placeholders with actual URLs and re-render.
- Optionally, add a user preference: "Always load images" / "Always load images from [sender]".
- Inline images (data: URLs, CID references) should always display.

  - Option A: Use a DOMPurify hook to rewrite `src` attributes:
    ```tsx
    DOMPurify.addHook('afterSanitizeAttributes', (node) => {
        if (node.tagName === 'IMG' && node.getAttribute('src')) {
            const src = node.getAttribute('src');
            if (src && !src.startsWith('data:') && !src.startsWith('cid:')) {
                node.setAttribute('data-original-src', src);
                node.removeAttribute('src');
    ```

## Impact Analysis

### What This Affects
- All email rendering in thread view.
- Emails with legitimate images (newsletters, product updates, marketing) will show placeholders until the user clicks "Load images."
- User experience trade-off: privacy vs. convenience. Most modern email clients (Gmail, Apple Mail, Hey) default to blocking tracking pixels.

### What Affects This
- The DOMPurify configuration — adding hooks changes global DOMPurify behavior if not scoped.
- Inline/embedded images (data: URIs) should be exempt from blocking.
- CSS background images (`style="background-image: url(...)"`) can also be tracking vectors — these should also be blocked.

### Related PRDs
- PRD-014: Malicious CSS — CSS can contain `background-image: url()` which is also a tracking vector. Both PRDs address untrusted content in emails.
- PRD-005: CSP Header — CSP's `img-src` directive could provide an additional layer, but would also block legitimate images.
- PRD-012: BCC Privacy Leak — Both relate to privacy protection in email handling.

## Implementation Notes
- The DOMPurify hook approach is cleanest — it runs during sanitization, so the HTML stored in the render is already safe.
- Use a component-scoped DOMPurify instance to avoid global side effects:
  ```tsx
  const purify = DOMPurify(window);
  purify.addHook('afterSanitizeAttributes', ...);
  ```
- For the "Load images" toggle, it can be per-email or per-thread. Per-email is more granular but more complex. Start with per-thread.
- CSS `background-image` in inline styles is a separate tracking vector. DOMPurify allows `style` attributes (see PRD-014). Consider stripping `url()` from style values or use a CSS parser.
- Performance: DOMPurify with hooks is fast. The re-render on "Load images" is just a state change.
- Future enhancement: "Always load images from [sender]" preference stored in user settings.
- Consider proxying images through the backend instead of blocking them. This prevents the sender from learning the user's IP, while still displaying images. However, this requires backend infrastructure and bandwidth.
