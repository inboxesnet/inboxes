# PRD-021: Login Lockout

## Problem Statement
The login endpoint has IP-based rate limiting (10 requests per 15 minutes) but no per-account lockout after repeated failed attempts. An attacker using a botnet or rotating proxies can bypass IP-based rate limiting entirely and brute-force any account's password with unlimited attempts across different source IPs. There is no mechanism to detect or prevent credential stuffing attacks targeting specific email addresses.

## Current Behavior
In `backend/internal/router/router.go:115`, the login route has only IP-based rate limiting:

```go
r.With(middleware.RateLimitByIP(rdb, 10, 15*60)).Post("/api/auth/login", auth.Login)
```

In `backend/internal/handler/auth.go:217-292`, the login handler validates credentials but performs no per-account tracking of failed attempts:

```go
func (h *AuthHandler) Login(w http.ResponseWriter, r *http.Request) {
    ...
    err := h.DB.QueryRow(ctx,
        `SELECT id, org_id, name, role, status, password_hash, email_verified FROM users WHERE email = $1`,
        req.Email,
    ).Scan(&userID, &orgID, &name, &role, &status, &passwordHash, &emailVerified)
    if err != nil {
        bcrypt.CompareHashAndPassword(dummyHash, []byte(req.Password)) // constant-time
        writeError(w, http.StatusUnauthorized, "invalid credentials")
        return
    }
    ...
    if err := bcrypt.CompareHashAndPassword([]byte(passwordHash), []byte(req.Password)); err != nil {
        writeError(w, http.StatusUnauthorized, "invalid credentials")
        return
    }
    ...
}
```

The handler uses constant-time comparison with a dummy hash for non-existent accounts (line 246) to prevent timing-based account enumeration — this is good. But there is no tracking of failed login attempts for existing accounts.

The `RateLimitByIP` middleware in `backend/internal/middleware/ratelimit.go:20-41` uses Redis to count requests per IP per endpoint, but this does nothing against distributed attacks.

## Expected Behavior
1. Track failed login attempts per email address in Redis using a counter with a TTL.
2. After N failed attempts (e.g., 10) within a time window (e.g., 15 minutes), temporarily lock the account for a cooldown period (e.g., 15 minutes).
3. During lockout, return a generic "invalid credentials" response (do not reveal that the account is locked to prevent account enumeration).
4. Successful login resets the failed attempt counter.
5. The lockout check should happen before the password comparison to save CPU time on bcrypt operations.

## Files to Modify
- `backend/internal/handler/auth.go:217-292` — Add per-email failed attempt tracking and lockout check in the Login handler
- `backend/internal/router/router.go:115` — Consider also adding `RateLimitByBodyField(rdb, "email", 10, 15*60)` as an additional layer (similar to forgot-password on line 116)

## Impact Analysis

### What This Affects
- Login flow — legitimate users who mistype their password 10+ times will be temporarily locked out
- Account security — significantly harder to brute-force passwords
- Redis storage — adds a key per email address that has failed logins (auto-expires)

### What Affects This
- Redis availability — if Redis is down, the lockout mechanism won't work (consistent with other rate limiting behavior)
- PRD-019 (Token Blacklist Fail-Open) — same Redis dependency for security features

## Implementation Notes
Use `RateLimitByBodyField` middleware (already exists in `backend/internal/middleware/ratelimit.go:46-81`) as a quick win. Adding it to the login route like it is done for forgot-password (`router.go:116`) would provide per-email rate limiting with no handler changes:

```go
r.With(
    middleware.RateLimitByIP(rdb, 10, 15*60),
    middleware.RateLimitByBodyField(rdb, "email", 10, 15*60),
).Post("/api/auth/login", auth.Login)
```

For a more complete solution, implement lockout logic inside the login handler:
1. Before bcrypt comparison: check `INCR` on `login:fail:{email}` key
2. If count > threshold, return "invalid credentials" immediately
3. On failed password: increment counter, set TTL
4. On successful login: delete the counter

The lockout response should be identical to "invalid credentials" to prevent account enumeration. Optionally, log the lockout event at a higher severity for monitoring.

Consider also adding exponential backoff to the lockout duration (15 min, 30 min, 1 hour) for repeated lockouts.
