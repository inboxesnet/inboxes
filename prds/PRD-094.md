# PRD-094: ARIA Labels on Icon-Only Buttons

## Status
Done

## Priority
P3-Low

## Category
Accessibility

## Source
PRE-LAUNCH-AUDIT #24 / GAP-FINDER (general ARIA gap)

---

## Problem Statement
Icon-only action buttons throughout the app (star, archive, trash, mute, mark read/unread, refresh, compose, etc.) lack `aria-label` attributes. Screen readers announce these as just "button" with no indication of what action they perform. This affects both the thread list hover actions and the thread view toolbar.

## Current Behavior
Buttons use `title` attributes in some places (thread-toolbar.tsx, thread-view.tsx) but not `aria-label`. In thread-list.tsx, the star button at line 112 has no `aria-label` at all. The thread list hover action buttons (archive at line 196, trash at line 209, read/unread at line 221) use `title` but not `aria-label`. The sidebar icon buttons (settings, theme toggle, logout, add domain) also rely on `title` only. Screen readers may not consistently read `title` as accessible names.

## Expected Behavior
Every icon-only button should have an `aria-label` attribute that clearly describes its action. For toggle buttons (star/unstar, mute/unmute, read/unread), the label should reflect the current state (e.g., `aria-label="Unstar thread"` when already starred).

## Impact Analysis

### What This Affects
- Screen reader users' ability to understand and interact with all action buttons
- Automated accessibility testing scores (axe, Lighthouse)
- WCAG 2.1 Level A compliance (Success Criterion 4.1.2: Name, Role, Value)

### What Affects This
- Any new icon-only buttons added to the UI will need the same treatment
- Button labels should be kept in sync with `title` attributes where both exist

### Related PRDs
- PRD-095 (Focus Trap) -- compose window accessibility
- PRD-101 (Settings Tabs) -- settings modal accessibility
- PRD-102 (aria-expanded) -- collapsible email message accessibility
- PRD-104 (Drag-Drop) -- drag-and-drop accessibility

## Implementation Notes
- Prefer `aria-label` over `title` for screen reader support, but keep both for sighted keyboard users who see tooltips
- For toggle buttons, use dynamic labels: `aria-label={hasLabel(thread, "starred") ? "Unstar thread" : "Star thread"}`
- Consider adding `aria-pressed` to toggle buttons (star, mute) for semantic completeness
- Can be done incrementally -- start with the most-used buttons (thread list, toolbar)

## User Notes
Low priority -- app is 100% behind auth, no SEO or AI indexing impact. No legal risk at current scale. Address when there's traction and real screen reader users.
