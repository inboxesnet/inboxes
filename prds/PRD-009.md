# PRD-009: Disabled Users Can Continue Using the App

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S6

---

## Problem Statement
The auth middleware only validates the JWT token's signature, expiry, and blacklist status. It does not check whether the user is still active or whether their org has been deleted. A disabled user's JWT remains valid for up to 7 days (the token TTL), meaning they retain full access to the application even after an admin disables their account. Combined with the token blacklist failing open when Redis is down, this creates a window where disabled users have unrestricted access.

## Current Behavior
In `backend/internal/middleware/auth.go`, the `AuthMiddleware` function (line 27-62):
- Line 31-35: Extracts the JWT from the `token` cookie.
- Line 36-46: Parses and validates the JWT signature and expiry.
- Line 48-56: Checks token revocation via Redis blacklist (fails open if Redis is down).
- Line 58-59: Sets claims in context and proceeds.

**What is NOT checked:**
- Whether `users.status = 'active'` for the `claims.UserID`.
- Whether the user's org (`orgs.deleted_at IS NULL`) still exists.
- Whether the user's role has changed since the token was issued.

The JWT token contains `user_id`, `org_id`, and `role` — all are trusted without re-verification against the database.

In `backend/internal/handler/auth.go:251-254`, the `Login` handler does check status:
```go
if status != "active" {
    writeError(w, http.StatusUnauthorized, "account is not active")
    return
}
```
But this only runs at login time. A user who was active at login and later disabled retains their token.

Token TTL is 7 days (line 86 in `auth.go` middleware: `ExpiresAt: jwt.NewNumericDate(now.Add(7 * 24 * time.Hour))`).

## Expected Behavior
- The auth middleware should periodically verify the user's active status against the database.
- This check should NOT run on every request (performance concern), but should run often enough to limit the exposure window.
- Recommended approach: Cache the user's status in Redis with a short TTL (e.g., 1-2 minutes). On each request, check Redis first; if not cached, query the database.
- If the user is disabled or the org is deleted, reject the request with 401.
- When an admin disables a user, also revoke all their tokens immediately (call `RevokeAllForUser`).

## Files to Modify
- `backend/internal/middleware/auth.go:27-62` — Add a user status check after the blacklist check:
  ```go
  // Check user status (cached in Redis, refreshed every 2 minutes)
  statusKey := "user:status:" + claims.UserID
  status, err := rdb.Get(r.Context(), statusKey).Result()
  if err != nil {
      // Cache miss — query database
      var dbStatus string
      var orgDeletedAt *time.Time
      err := db.QueryRow(r.Context(),
          "SELECT u.status, o.deleted_at FROM users u JOIN orgs o ON o.id = u.org_id WHERE u.id = $1",
          claims.UserID,
      ).Scan(&dbStatus, &orgDeletedAt)
      if err != nil || dbStatus != "active" || orgDeletedAt != nil {
          http.Error(w, `{"error":"unauthorized"}`, http.StatusUnauthorized)
          return
      }
      rdb.Set(r.Context(), statusKey, dbStatus, 2*time.Minute)
  } else if status != "active" {
      http.Error(w, `{"error":"unauthorized"}`, http.StatusUnauthorized)
      return
  }
  ```
- `backend/internal/handler/users.go` — In the `Disable` handler, add `RevokeAllForUser` and clear the status cache:
  ```go
  blacklist.RevokeAllForUser(ctx, userID)
  rdb.Del(ctx, "user:status:"+userID)
  ```
- `backend/internal/handler/orgs.go` — In the `Delete` handler, revoke tokens for all org users and clear their status caches.

## Impact Analysis

### What This Affects
- Every authenticated request adds a Redis lookup (fast, ~0.1ms).
- Database query only on cache miss (every 2 minutes per user).
- User disable now has immediate effect (within 2 minutes, or instant if token revocation works).

### What Affects This
- Redis availability — if Redis is down, the status check falls through to a database query on every request (acceptable degradation).
- Database availability — if both Redis and DB are down, the middleware should reject the request (fail closed for this check).
- The `AuthMiddleware` function signature needs to accept `db *pgxpool.Pool` in addition to `secret` and `rdb`.

### Related PRDs
- PRD-001: Sync Job IDOR — Disabled users could exploit the sync endpoint if their tokens are still valid.
- PRD-007: Case-Sensitive Emails — Ensures user lookups are consistent.
- PRD-008: Verification Brute-Force — A disabled user could potentially brute-force a verification code to re-enable their account if the disable doesn't clear verification state.

## Implementation Notes
- The `AuthMiddleware` function currently accepts `secret string` and `rdb *redis.Client`. It needs to also accept `db *pgxpool.Pool` for the database fallback check.
- Update the call site in `router.go:136`:
  ```go
  r.Use(middleware.AuthMiddleware(secret, rdb, db))
  ```
- The 2-minute cache TTL is a trade-off between security (shorter = faster disable) and performance (shorter = more DB queries). 1-2 minutes is reasonable.
- For the status cache, consider also caching the user's current role. If an admin demotes a user, the role change takes effect within the cache TTL.
- An alternative approach is to only check on "important" endpoints (send email, admin actions), but this adds complexity and is harder to reason about. The Redis-cached approach is simpler and provides consistent security.

## Acceptance Criteria
- [ ] Auth middleware checks user status against database/cache on every authenticated request.
- [ ] A disabled user is rejected within 2 minutes of being disabled (cache TTL).
- [ ] Disabling a user also revokes all their tokens immediately.
- [ ] Deleting an org revokes tokens for all org users.
- [ ] If Redis is down, the status check falls back to a direct database query.
- [ ] If both Redis and DB are unavailable, the request is rejected (fail closed).
- [ ] The status cache is cleared when a user is disabled or re-enabled.
- [ ] Unit test: disable a user, verify their next API request within cache TTL is rejected.
- [ ] Performance: the Redis lookup adds less than 1ms to request latency.
