# PRD-118: Dead Code — Attachment Download Endpoint Appears Unused

## Status
Draft

## Priority
P3-Low

## Category
Dead Code

## Source
GAP-FINDER O3

---

## Problem Statement
The `GET /api/attachments/{id}` download endpoint is registered and fully implemented, but no frontend code calls it. The frontend only uses the upload endpoint (`POST /api/attachments/upload`). Attachment data in emails is displayed via inline metadata from the email's `attachments` JSON field, not via this download endpoint. However, this endpoint may be needed for future attachment download functionality.

## Current Behavior
- `backend/internal/handler/attachments.go:83-103` — The `Download` method is fully implemented. It fetches the attachment by ID and org, sets `Content-Type`, `Content-Disposition`, and `Content-Length` headers, and writes the binary data to the response.
- `backend/internal/router/router.go:265` — The route is registered: `r.Get("/api/attachments/{id}", attachments.Download)`.
- Frontend attachment upload is at `frontend/components/floating-compose-window.tsx:174`, using `fetch("/api/attachments/upload", ...)`. A search of the frontend for `attachments/` only finds this upload call — no download call exists.
- The `Download` handler has a security issue: `Content-Disposition` header on line 100 interpolates the filename without sanitization (see PRD-002), and `Content-Type` on line 99 trusts the stored value which could serve malicious HTML (see PRD-020).

## Expected Behavior
**Keep the endpoint** but mark it as reserved for future use. Attachment download is a fundamental feature that will be needed when:
- Users want to download email attachments from the thread view.
- Draft attachments need to be previewed before sending.
- A future "download all attachments" feature is built.

Fix the security issues (PRD-002: filename sanitization, PRD-020: MIME type bypass) while keeping the endpoint.

## Files to Modify
- `backend/internal/handler/attachments.go:99-100` — Fix Content-Type to force `application/octet-stream` for downloads, and sanitize the filename in the Content-Disposition header.
- `backend/internal/router/router.go:265` — Keep the route but add a comment noting it is not yet used by the frontend.

## Impact Analysis

### What This Affects
- No current frontend functionality (endpoint is unused).
- Security posture — the endpoint exists as an attack surface even though it is not called by the UI.

### What Affects This
- PRD-002 (Header Injection) directly affects this endpoint's `Content-Disposition` header.
- PRD-020 (Attachment MIME Bypass) directly affects this endpoint's `Content-Type` header.
- PRD-122 (BYTEA Attachments) — this endpoint loads the full binary into memory, which will become a scaling issue.

### Related PRDs
- PRD-002: Header Injection — filename in `Content-Disposition` header is unsanitized on line 100.
- PRD-020: Attachment MIME Bypass — `Content-Type` on line 99 trusts stored value, could serve malicious HTML.
- PRD-122: BYTEA Attachments — loading full binary data into memory via `SELECT ... data` is the scalability concern.
- PRD-126: Draft Attachments — frontend needs this endpoint to display previously uploaded attachments.

## Implementation Notes
May be a stub for future use -- keep for now. When the frontend attachment display is built, this endpoint will be the natural backend for it. In the meantime, fix the two security issues:

1. Line 99: Replace `w.Header().Set("Content-Type", contentType)` with `w.Header().Set("Content-Type", "application/octet-stream")` to prevent serving malicious HTML.
2. Line 100: Sanitize filename by stripping quotes and non-ASCII characters, or use RFC 5987 encoding.
