# PRD-001: Standardize WebSocket Cache Sync to Surgical Updates

## Problem

The frontend WebSocket sync layer (`use-ws-sync.ts`) uses two different cache update strategies inconsistently:

- **`setQueryData` / `setQueriesData`** — surgical, instant, no network call
- **`invalidateQueries`** — throws away cache, triggers re-fetch from API

Many handlers use both: surgical update for thread lists, then `invalidateQueries` for domain unread counts. Some events skip surgical updates entirely and just invalidate everything. This creates unnecessary API calls after every WS event and inconsistent UI update speeds.

## Current State

| Event | Thread Lists | Thread Detail | Unread Counts |
|-------|-------------|---------------|---------------|
| email.received | surgical | surgical | **invalidate** |
| email.sent | surgical | surgical | **invalidate** |
| thread.starred | surgical | — | — |
| thread.unstarred | surgical | — | — |
| thread.muted | surgical | surgical | — |
| thread.unmuted | surgical | surgical | — |
| thread.read | surgical | — | **invalidate** |
| thread.unread | surgical | — | **invalidate** |
| thread.archived | surgical | — | **invalidate** |
| thread.trashed | surgical | — | **invalidate** |
| thread.spammed | surgical | — | **invalidate** |
| thread.moved | surgical | — | **invalidate** |
| thread.deleted | surgical | — | **invalidate** |
| thread.bulk_action | **invalidate** | — | **invalidate** |
| email.status_updated | — | **invalidate** | — |

| domain.disconnected | — | — | **invalidate** (domains.list) |
| domain.reconnected | — | — | **invalidate** (domains.list) |

**"invalidate" = unnecessary API round-trip.**

## Goal

Convert all `invalidateQueries` calls to `setQueryData` surgical updates. The WS payload already contains enough data to compute the new state locally. Zero re-fetches after WS events.

## Changes

### 1. Unread Counts — Surgical Increment/Decrement

The biggest win. Currently 8 event types invalidate `domains.unreadCounts`, causing a GET on every single thread action.

**Fix:** The WS payload includes `domain_id`. The handler knows the action (read/unread/archive/trash/new email). Compute the delta and apply it:

```ts
// Instead of: invalidateQueries({ queryKey: queryKeys.domains.unreadCounts })
// Do:
setQueryData(queryKeys.domains.unreadCounts, (old) => {
  if (!old) return old
  return old.map(d =>
    d.domain_id === domainId
      ? { ...d, unread_count: Math.max(0, d.unread_count + delta) }
      : d
  )
})
```

Delta logic:
- `email.received` where thread was unread or new: **+1**
- `thread.read`, `thread.archived`, `thread.trashed`, `thread.spammed`, `thread.deleted`: **-1** (if thread was unread)
- `thread.unread`: **+1**
- `thread.moved` (domain change): **-1** on old domain, **+1** on new domain (if unread)

The WS payload should include `was_unread: boolean` so the frontend knows whether to adjust the count. If the backend doesn't send this today, add it.

### 2. Bulk Action — Surgical Update

`thread.bulk_action` currently does full invalidation. The backend already knows which threads were affected.

**Fix:** Include `thread_ids` and `action` in the WS payload. Apply the same per-thread surgical logic used by individual thread events, just in a loop:

```ts
setQueriesData({ queryKey: queryKeys.threads.lists }, (old) => {
  if (!old) return old
  return {
    ...old,
    threads: old.threads.filter(t => !threadIds.includes(t.id))
  }
})
```

Unread count delta = count of affected threads that were unread.

### 3. email.status_updated — Surgical Update

Currently invalidates the full thread detail. The payload has `email_id` and new `status`.

**Fix:**
```ts
setQueryData(queryKeys.threads.detail(threadId), (old) => {
  if (!old) return old
  return {
    ...old,
    emails: old.emails.map(e =>
      e.id === emailId ? { ...e, status } : e
    )
  }
})
```

### 4. sync.completed — Keep as Invalidation

This is the one exception. After a full mailbox sync, the data delta is unknown — full invalidation is correct here. No change needed.

### 5. domain.disconnected / domain.reconnected — Surgical Update

Currently invalidates `domains.list`. The payload has `domain_id` and the new status.

**Fix:**
```ts
setQueryData(queryKeys.domains.list, (old) => {
  if (!old) return old
  return old.map(d =>
    d.id === domainId ? { ...d, status: newStatus } : d
  )
})
```

## Backend Changes

Add `was_unread` boolean to WS payloads for events that affect unread counts. This tells the frontend whether to adjust the count without needing to inspect stale cache state.

Add `thread_ids` array to `thread.bulk_action` WS payload (may already be there — verify).

## Out of Scope

- Optimistic mutation updates (already surgical, working well)
- Cross-tab BroadcastChannel sync (uses invalidation by design — tabs don't share WS payload data, so invalidation is correct there)
- `sync.completed` (full invalidation is appropriate)

## Verification

After each conversion:
1. Open Network tab, filter to API calls
2. Trigger the WS event (receive email, star thread, etc.)
3. Confirm **zero** GET requests fire after the WS update
4. Confirm UI state matches — counts, thread lists, thread detail all correct
5. Test reconnection catchup still works (disconnect WiFi, reconnect, verify state)
