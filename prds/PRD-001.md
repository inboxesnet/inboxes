# PRD-001: Sync Job IDOR

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source

## Problem Statement
The `GET /api/sync/{id}` endpoint fetches a sync job by its ID but originally did not verify that the requesting user's org owns the job. Although a post-fetch `orgID != claims.OrgID` check was added on line 120, the query itself still selects by ID alone (line 108-111), leaking the existence of job IDs across orgs via timing differences and potentially returning data before the guard executes if future refactors move code around. The fix should push the org check into the SQL query itself for defense-in-depth.

## Current Behavior
In `backend/internal/handler/sync.go`, the `GetSync` method (line 93-143):
- Line 108-111: Queries `SELECT ... FROM sync_jobs WHERE id = $1` with only the job ID parameter.
- Line 120-122: Performs an application-level check `if orgID != claims.OrgID` after the query returns.
- The query returns a row for any valid job ID regardless of org, meaning the DB round-trip and scan complete before the ownership check. If the job doesn't exist, a 404 is returned; if it belongs to another org, a different 404 path runs. This timing difference can be used for enumeration.

## Expected Behavior
- The SQL query should include `AND org_id = $2` with `claims.OrgID` as the second parameter, so the database never returns rows from other orgs.
- The application-level guard on line 120 should remain as a secondary defense but will never trigger since the query already filters by org.
- A non-existent job and a job belonging to another org should be indistinguishable (same timing, same error).

## Files to Modify
- `backend/internal/handler/sync.go:108-111` — Add `AND org_id = $2` to the WHERE clause and pass `claims.OrgID` as the second query parameter.

## Impact Analysis

### What This Affects
- Sync job polling from the frontend (`/hooks/use-sync-job.ts`) — no functional change, just tighter security.
- Any admin triggering a sync via the onboarding flow.

### What Affects This
- Auth middleware must correctly populate `claims.OrgID` (already does).

### Related PRDs
- PRD-009: Disabled Users Auth — If a disabled user's token is still valid, they could exploit this endpoint. Fixing PRD-009 (checking user status in middleware) provides an additional layer.

## Implementation Notes
- This is a one-line SQL change. The application-level check on line 120-122 can remain as belt-and-suspenders.
- The `StartSync` method (line 22-89) already correctly scopes by `org_id = $1` on line 42, so only `GetSync` needs the fix.
- No migration needed.
