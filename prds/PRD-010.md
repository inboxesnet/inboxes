# PRD-010: Webhook Secret Lookup Failure Bypass

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S8

---

## Problem Statement
The webhook handler's database lookup for the org's webhook signing secret silently ignores errors. If the database query fails (connection timeout, transient error), the `webhookSecret` variable remains empty. While the current code does reject empty secrets on line 93-96, the original concern was that a failed database lookup could allow unsigned webhooks through. The actual vulnerability is more nuanced: the `Scan()` error on line 78-80 is not checked, meaning any database error is silently swallowed, and the handler relies on the empty-string check downstream rather than failing explicitly with a clear error.

## Current Behavior
In `backend/internal/handler/webhooks.go`, the `HandleResend` function (line 59-126):
- Line 75-80: The webhook secret lookup:
  ```go
  var webhookSecret string
  var encSecret, encIV, encTag *string
  var plainSecret *string
  h.DB.QueryRow(r.Context(),
      `SELECT resend_webhook_secret, resend_webhook_secret_encrypted, resend_webhook_secret_iv, resend_webhook_secret_tag FROM orgs WHERE id = $1`, orgID,
  ).Scan(&plainSecret, &encSecret, &encIV, &encTag)
  ```
  - **The error from `Scan()` is completely ignored.** If the query fails, all four pointer variables remain `nil`.

- Line 82-92: Decryption and fallback logic:
  ```go
  if encSecret != nil && *encSecret != "" && h.EncSvc != nil {
      decrypted, decErr := h.EncSvc.Decrypt(*encSecret, *encIV, *encTag)
      // ...
  }
  if webhookSecret == "" && plainSecret != nil {
      webhookSecret = *plainSecret
  }
  ```
  - If `Scan()` failed, `plainSecret` is nil and `encSecret` is nil, so `webhookSecret` stays empty.

- Line 93-97: Empty secret rejection:
  ```go
  if webhookSecret == "" {
      slog.Warn("webhook: no webhook secret configured, rejecting", "org_id", orgID)
      http.Error(w, "webhook secret not configured", http.StatusUnauthorized)
      return
  }
  ```
  - This catch-all does prevent bypass, but it conflates "no secret configured" with "database error."

**The risk**: A transient database error returns a `401 Unauthorized` with a misleading "not configured" message. Resend will retry the webhook (good), but operators have no visibility into whether the failure was a configuration issue or a database problem. More critically, if the empty-string check were ever removed or the logic refactored, the bypass would re-emerge.

## Expected Behavior
- The `Scan()` error on line 78-80 should be explicitly checked.
- If the error is "no rows" (org not found), return `404` or `410 Gone`.
- If the error is any other database error, return `500` and log the specific error, so Resend retries.
- The downstream empty-string check on line 93 should remain as defense-in-depth.

## Files to Modify
- `backend/internal/handler/webhooks.go:78-80` — Check the `Scan()` error:
  ```go
  err := h.DB.QueryRow(r.Context(),
      `SELECT resend_webhook_secret, resend_webhook_secret_encrypted, resend_webhook_secret_iv, resend_webhook_secret_tag FROM orgs WHERE id = $1`, orgID,
  ).Scan(&plainSecret, &encSecret, &encIV, &encTag)
  if err != nil {

      }
      slog.Error("webhook: failed to lookup webhook secret", "org_id", orgID, "error", err)
      http.Error(w, "internal error", http.StatusInternalServerError)
      return
  }
  ```
- `backend/internal/handler/webhooks.go:82-97` — Keep the existing decryption and empty-string check logic as-is for defense-in-depth.

## Impact Analysis

### What This Affects
- Inbound email processing — webhook requests from Resend for all orgs.
- Error visibility — operators can now distinguish "not configured" from "database down."
- Resend's retry behavior — returning 500 causes Resend to retry (correct for transient DB errors), while the current 401 may cause Resend to give up sooner.

### What Affects This
- Database availability — this is the critical path for inbound email delivery.
- The encryption service (`h.EncSvc`) — if decryption fails, the handler already logs the error and falls back to plaintext (line 84-86), which is correct.

### Related PRDs
- PRD-009: Disabled Users Auth — Deleted orgs should not have their webhooks processed. Currently the webhook handler does not check `org.deleted_at` (see also GAP-FINDER L7). This PRD focuses on the secret lookup error; a separate effort should check org status.

## Implementation Notes
- Import `"errors"` and `"github.com/jackc/pgx/v5"` for `pgx.ErrNoRows`.
- The `pgx.ErrNoRows` check ensures we distinguish "org doesn't exist" from "database error."
- The HTTP status codes matter for Resend's retry logic:
  - `401/403`: Resend may stop retrying (authentication failure).
  - `500`: Resend will retry with backoff (transient failure).
  - `200`: Resend considers delivery successful.
- Returning 500 on database error is the correct choice — it tells Resend to retry.
- Also consider adding org status check (`deleted_at IS NULL`) to the query WHERE clause, which prevents processing webhooks for deleted orgs. This overlaps with GAP-FINDER L7 but is a natural addition here.
