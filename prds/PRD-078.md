# PRD-078: Webhook Cleanup for Deleted Orgs

## Status
Done

## Priority
P2-Medium

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L7

---

## Problem Statement
When an org is soft-deleted, its Resend webhook remains registered and continues to fire. Incoming webhook payloads for deleted orgs are processed through signature verification, job creation, and worker execution -- all of which fail silently or waste resources. The webhook should be unregistered from Resend on org deletion, and the webhook handler should return an appropriate HTTP status code for payloads targeting deleted orgs.

## Current Behavior
In `backend/internal/handler/orgs.go`:
- Lines 119-185: Org deletion cancels the Stripe subscription, sets `deleted_at`, disables users, and marks domains deleted. But the Resend webhook (identified by `orgs.resend_webhook_id`) is never unregistered.
- The `resend_webhook_id` is set to NULL (`stripe_subscription_id = NULL` is set, but not `resend_webhook_id`), though the webhook still exists in Resend's system.

In `backend/internal/handler/webhooks.go`:
- Lines 59-102: `HandleResend` verifies the webhook signature using the org's stored secret. For a deleted org:
  - Line 78-80: The `QueryRow` for the webhook secret queries `orgs WHERE id = $1` -- it does not check `deleted_at`. So the secret is returned, signature verification passes, and processing continues.
  - Lines 112-126: Events are dispatched normally.
- There is no check for `orgs.deleted_at IS NOT NULL` anywhere in the webhook handler.

In `backend/internal/handler/domains.go`:
- Lines 455-513: `ReregisterWebhook` creates new webhooks via the Resend API but does not clean up old ones.

## Expected Behavior
- On org deletion: call the Resend API to delete the webhook (`DELETE /webhooks/{id}`) using the stored `resend_webhook_id`.
- In the webhook handler: check `org.deleted_at` early in `HandleResend`. If the org is deleted, return `410 Gone` to signal Resend to stop sending webhooks.
- Clear `resend_webhook_id` and `resend_webhook_secret*` columns when the org is deleted.

## Files to Modify
- `backend/internal/handler/orgs.go:119-185` — After cancelling the Stripe subscription, add webhook unregistration:
  ```go
  var webhookID *string
  h.DB.QueryRow(ctx, "SELECT resend_webhook_id FROM orgs WHERE id = $1", claims.OrgID).Scan(&webhookID)
  if webhookID != nil && *webhookID != "" {
      h.ResendSvc.Fetch(ctx, claims.OrgID, "DELETE", "/webhooks/"+*webhookID, nil)
  }
  ```
- `backend/internal/handler/orgs.go:151-153` — Add cleanup of webhook columns:
  ```sql
  UPDATE orgs SET deleted_at = now(), resend_webhook_id = NULL, resend_webhook_secret = NULL,
  resend_webhook_secret_encrypted = NULL, resend_webhook_secret_iv = NULL, resend_webhook_secret_tag = NULL, ...
  ```
- `backend/internal/handler/webhooks.go:59-102` — Add an early check in `HandleResend`:
  ```go
  var orgDeletedAt *time.Time
  h.DB.QueryRow(ctx, "SELECT deleted_at FROM orgs WHERE id = $1", orgID).Scan(&orgDeletedAt)
  if orgDeletedAt != nil {
      w.WriteHeader(http.StatusGone) // 410
      return
  }
  ```

## Impact Analysis

### What This Affects
- Org deletion workflow: adds webhook unregistration step.
- Webhook processing: deleted orgs' webhooks return 410 instead of processing.
- Resend's retry behavior: 410 should signal Resend to stop retrying (verify Resend's behavior on 410).
- Resource usage: eliminates wasted DB queries, Redis pushes, and worker cycles for deleted orgs.

### What Affects This
- The Resend API must support `DELETE /webhooks/{id}`. If the API call fails (e.g., webhook already deleted on Resend's side), the deletion should continue.
- The `ResendSvc.Fetch` method must have a valid API key for the org. If the key was already removed, the unregistration will fail -- handle gracefully.

### Related PRDs
- PRD-073: Deleted Domains Still Receive Emails — Both address the problem of deleted entities continuing to receive traffic.
- PRD-072: Domain Deletion Data Cleanup — Domain cleanup should also consider webhook implications.
- PRD-086: Emails After Org Deletion — Draining the Redis queue is a complement to webhook unregistration.
- PRD-010: Webhook Secret Bypass — If the webhook secret is cleared on deletion, the bypass is no longer possible for deleted orgs.

## Implementation Notes
- The Resend API call to delete the webhook may fail if the API key is invalid or the webhook was already deleted. Wrap in a try/catch and log the failure without blocking org deletion.
- Consider a defensive approach: even if unregistration fails, the 410 response in the webhook handler provides a fallback.
- If Resend does not respect 410 as "stop sending," fall back to returning 200 with no processing (same approach as PRD-073).
- The webhook secret columns should be NULLed out on org deletion to prevent any future signature verification from succeeding.
