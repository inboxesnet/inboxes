# PRD-029: Double-Send

## Status
Done

## Priority
P1-High

## Category
Drafts & Compose Issues

## Source
GAP-FINDER R1

---

## Problem Statement
Double-clicking the Send button or pressing Cmd+Enter twice quickly can send the same email twice. Neither the frontend nor the backend has idempotency protection on the send path. The frontend disables the button via the `sending` state, but React state updates are asynchronous, leaving a window where two clicks both enter the `handleSend` function before `setSending(true)` takes effect. On the backend, the draft send endpoint (`POST /api/drafts/:id/send`) and direct send endpoint (`POST /api/emails/send`) both create a new `email_jobs` row on every call with no idempotency key or duplicate check.

## Current Behavior
- **Frontend** (`floating-compose-window.tsx`, lines 284-344): `handleSend` sets `setSending(true)` on line 293, but the state update is async. Two rapid clicks can both pass the `if (sending) return` check that does not exist -- there is no early guard, only a `disabled={sending}` prop on the button which depends on a re-render.
- **Backend draft send** (`drafts.go`, lines 275-445): `Send()` fetches the draft, creates an email row, creates an `email_jobs` row, and pushes to Redis. There is no check for whether this draft has already been sent. The draft is not deleted during the transaction -- it is deleted later by the worker after successful send.
- **Backend direct send** (`emails.go`, lines 68-290): Similarly creates email + job rows without any idempotency key.

## Expected Behavior
1. Frontend: Immediately prevent duplicate submissions using a ref-based guard (not state-based).
2. Backend: Accept an optional `Idempotency-Key` header. If a key is provided and already seen (stored in Redis with a short TTL), return the original response instead of creating a duplicate.
3. Backend draft send: Check if the draft already has an associated `email_jobs` row before creating another one.

## Files to Modify
- `frontend/components/floating-compose-window.tsx:284-344` -- Add a `sendingRef` (useRef<boolean>) set to `true` synchronously at the top of `handleSend`. Return early if already true. Reset in `finally`.
- `backend/internal/handler/drafts.go:275-445` -- In `Send()`, add a check: `SELECT id FROM email_jobs WHERE draft_id = $1` before creating a new job. If a job already exists, return the existing email_id/thread_id/job_id instead of creating duplicates.
- `backend/internal/handler/emails.go:68-290` -- Add optional `Idempotency-Key` header support. Store the key in Redis with a 5-minute TTL after successful processing. On duplicate key, return the cached response.
- `backend/internal/middleware/` -- Consider a reusable idempotency middleware that can be applied to any POST endpoint.

## Impact Analysis

### What This Affects
- All email recipients -- they would receive the same email twice, which is unprofessional and confusing.
- Thread state -- duplicate sends create duplicate emails within the same thread, inflating message counts.
- Send quotas -- duplicate sends consume Resend API rate limits and email quotas unnecessarily.

### What Affects This
- Network latency -- slower networks increase the window for double-clicks.
- The `disabled={sending}` button prop -- effective only after React re-renders, which is too slow for sub-100ms double-clicks.
- Redis availability -- the idempotency key storage depends on Redis.

### Related PRDs
- PRD-028 (Duplicate Drafts) -- analogous race condition on the save path
- PRD-027 (Compose Draft Loss) -- the save/send lifecycle in compose is intertwined
- PRD-045 (Duplicate Threads) -- two simultaneous draft sends without a thread_id create duplicate threads

## Implementation Notes
- **Frontend quick fix** (highest priority):
  ```tsx
  const sendingRef = useRef(false);
  async function handleSend(e: React.FormEvent) {
    e.preventDefault();
    if (sendingRef.current) return;
    sendingRef.current = true;
    // ... existing logic ...
    // in finally:
    sendingRef.current = false;
  }
  ```
- **Backend draft send guard**: The draft `Send()` handler already fetches the draft by `id + user_id`. Add a subquery in the same transaction: `SELECT id FROM email_jobs WHERE draft_id = $1 AND status != 'failed'`. If found, return the existing result.
- **Idempotency middleware** (nice-to-have): `X-Idempotency-Key` header, stored as `idempotency:{key}` in Redis with 5-minute TTL. The middleware wraps the handler, caches the response body and status code, and replays on duplicate keys.
- The existing `Cmd+Enter` shortcut handler (line 278-281) calls `handleSend` directly -- it also needs the ref guard.

## User Notes
- Users with accessibility devices or touch-based input may trigger double-submit more frequently.
- The fix should be invisible to users -- they should just see the email sent once.

## Acceptance Criteria
- [ ] Double-clicking Send within 100ms sends exactly one email
- [ ] Pressing Cmd+Enter twice rapidly sends exactly one email
- [ ] Backend returns the same email_id/job_id if the same draft is sent twice concurrently
- [ ] No duplicate `email_jobs` rows are created for the same draft_id
- [ ] The Send button is visually disabled (spinner shown) immediately on first click
- [ ] Direct send endpoint (`POST /api/emails/send`) supports optional `Idempotency-Key` header
- [ ] If `Idempotency-Key` is provided and already processed, backend returns 200 with the cached result
