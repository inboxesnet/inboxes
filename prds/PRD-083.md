# PRD-083: Double Checkout

## Status
Done

## Priority
P1-High

## Category
Billing & Stripe

## Source

## Problem Statement
Double-clicking the "Upgrade to Pro" button in the PaymentWall component creates two separate Stripe checkout sessions. While only one can be completed by the user, the second session remains open in Stripe's system and creates unnecessary API calls, potential confusion if the user sees two browser tabs, and wasted Stripe resources.

## Current Behavior
In `frontend/components/payment-wall.tsx:26-37`:
```typescript
async function handleUpgrade() {
    setLoading(true);
    try {
      const res = await api.post<{ url: string }>("/api/billing/checkout");
      const parsed = new URL(res.url);
      if (!parsed.hostname.endsWith("stripe.com")) throw new Error("Invalid checkout URL");
      window.location.href = res.url;
    } catch {
      toast.error("Failed to start checkout. Please try again.");
      setLoading(false);
    }
  }
```

The `setLoading(true)` on line 27 disables the button via the `disabled={loading}` prop on line 52, but there is a race window: the user can click twice before React re-renders the disabled state. Each click fires `api.post("/api/billing/checkout")` which creates a new Stripe checkout session on the backend (`billing.go:99-116`). The second click's redirect may override the first, or both may complete depending on timing.

The same issue exists in the settings modal billing tab at `frontend/components/settings-modal.tsx:466-475` where the checkout button also lacks dedup.

## Expected Behavior
1. Only one checkout session should be created per upgrade attempt.
2. Rapid clicks should be debounced or blocked at the request level.
3. If a checkout session was recently created (within the last 5 minutes), reuse it instead of creating a new one.

## Files to Modify
- `frontend/components/payment-wall.tsx:26-37` -- Add a ref-based guard to prevent concurrent calls; use `useRef` to track in-flight state
- `frontend/components/settings-modal.tsx:466-475` -- Same fix for the billing tab checkout button
- `backend/internal/handler/billing.go:32-119` -- Optionally: store the last checkout session URL in the org record and reuse if unexpired

## Implementation Notes
- Use a `useRef<boolean>` as a mutex guard, not just `useState`, because refs update synchronously while state updates are batched.
- Example pattern:
  ```typescript
  const checkoutInFlight = useRef(false);
  async function handleUpgrade() {
    if (checkoutInFlight.current) return;
    checkoutInFlight.current = true;
    // ... existing logic ...
    // Reset on error only (on success, page navigates away)
  }
  ```
- Backend-side dedup: optionally store `last_checkout_session_id` and `last_checkout_at` on the org. If a session was created in the last 5 minutes, return the existing session URL.
