# PRD-065: Onboarding LIMIT

## Status
Draft

## Priority
P2-Medium

## Category
Database Performance

## Source
GAP-FINDER D8

---

## Problem Statement
The `GetAddresses()` endpoint in the onboarding handler fetches ALL discovered addresses for an organization with no LIMIT clause. For organizations that have imported large mailboxes, the `discovered_addresses` table can contain tens of thousands (or even millions) of rows. The query also runs a correlated subquery per row that counts matching emails using JSONB containment operators (`@>`) without GIN indexes. This combination — unbounded result set plus expensive per-row subquery — makes the endpoint extremely slow for large organizations and risks timing out or exhausting database resources.

## Current Behavior
In `backend/internal/handler/onboarding.go:356-369`:
```go
rows, err := h.DB.Query(ctx,
    `SELECT da.id, da.address, da.local_part, da.type,
            (SELECT COUNT(*) FROM emails e
             WHERE e.domain_id = da.domain_id
               AND (e.from_address = da.address
                    OR e.to_addresses @> to_jsonb(da.address)
                    OR e.cc_addresses @> to_jsonb(da.address))
            ) as email_count
     FROM discovered_addresses da
     JOIN domains d ON d.id = da.domain_id
     WHERE d.org_id = $1 AND d.hidden = false
     ORDER BY email_count DESC`,
    claims.OrgID,
)
```

Problems:
1. **No LIMIT clause** — fetches every discovered address for the org. An org that imported a mailbox with 50,000 unique addresses gets all 50,000 rows.
2. **Correlated subquery per row** — for each discovered address, a `COUNT(*)` subquery scans the `emails` table checking three conditions: `from_address` match OR `to_addresses @>` OR `cc_addresses @>`. Without GIN indexes on `to_addresses` and `cc_addresses`, each subquery execution does a sequential scan of emails.
3. **ORDER BY email_count DESC** — since `email_count` is computed by the subquery, PostgreSQL must materialize the entire result set (all rows with their counts) before sorting. No index can help with this sort.
4. **No pagination** — the frontend receives the entire list in a single response. The onboarding UI only shows addresses for user review, and users rarely need to see more than a few hundred.

The result set is iterated at lines 377-388 and accumulated into a `[]map[string]interface{}` slice with no cap hint, so memory allocation also scales linearly with the result size.

## Expected Behavior
1. Add a `LIMIT` clause to cap the number of addresses returned (e.g., 500 most active addresses).
2. Add pagination support so the frontend can request additional pages if needed.
3. Optimize the email count subquery to avoid per-row sequential scans (partially addressed by PRD-062's GIN indexes).
4. Consider pre-computing or caching email counts during the sync/import phase rather than computing them on every request.

## Files to Modify
- `backend/internal/handler/onboarding.go:356-369` — Add LIMIT clause and optional pagination parameters
- `frontend/app/(app)/onboarding/page.tsx` — Handle paginated address list (if pagination is added)

## Impact Analysis

### What This Affects
- Onboarding page load time for organizations with many discovered addresses
- Database CPU and I/O during onboarding (correlated subquery runs per row)
- Memory usage in the Go backend (unbounded slice allocation)
- Database connection pool (long-running query holds a connection)

### What Affects This
- Number of discovered addresses per organization (grows with mailbox size)
- Number of emails per organization (affects subquery scan time)
- Whether GIN indexes exist on `to_addresses` and `cc_addresses` (PRD-062)
- Number of domains per organization

### Related PRDs
- PRD-062 (JSON Indexes) — GIN indexes on `to_addresses` and `cc_addresses` would dramatically speed up the correlated subquery
- PRD-049 (Unchecked Queries) — the `rows.Scan()` at line 380 does not check its error return value
- PRD-063 (Composite Index) — part of the overall index improvement effort
- PRD-064 (FK Indexes) — `discovered_addresses.user_id` index is added there

## Implementation Notes
- Immediate fix — add LIMIT:
  ```go
  rows, err := h.DB.Query(ctx,
      `SELECT da.id, da.address, da.local_part, da.type,
              (SELECT COUNT(*) FROM emails e
               WHERE e.domain_id = da.domain_id
                 AND (e.from_address = da.address
                      OR e.to_addresses @> to_jsonb(da.address)
                      OR e.cc_addresses @> to_jsonb(da.address))
              ) as email_count
       FROM discovered_addresses da
       JOIN domains d ON d.id = da.domain_id
       WHERE d.org_id = $1 AND d.hidden = false
       ORDER BY email_count DESC
       LIMIT 500`,
      claims.OrgID,
  )
  ```

- Better approach — use pagination with cursor:
  ```go
  limit := 100
  offset := 0
  if q := r.URL.Query().Get("limit"); q != "" {
      if n, err := strconv.Atoi(q); err == nil && n > 0 && n <= 500 {
          limit = n
      }
  }
  if q := r.URL.Query().Get("offset"); q != "" {
      if n, err := strconv.Atoi(q); err == nil && n >= 0 {
          offset = n
      }
  }

  rows, err := h.DB.Query(ctx,
      `SELECT da.id, da.address, da.local_part, da.type,
              (SELECT COUNT(*) FROM emails e
               WHERE e.domain_id = da.domain_id
                 AND (e.from_address = da.address
                      OR e.to_addresses @> to_jsonb(da.address)
                      OR e.cc_addresses @> to_jsonb(da.address))
              ) as email_count
       FROM discovered_addresses da
       JOIN domains d ON d.id = da.domain_id
       WHERE d.org_id = $1 AND d.hidden = false
       ORDER BY email_count DESC
       LIMIT $2 OFFSET $3`,
      claims.OrgID, limit, offset,
  )
  ```
  Return a response wrapper with total count and pagination metadata:
  ```json
  {
    "addresses": [...],
    "total": 12345,
    "limit": 100,
    "offset": 0
  }
  ```

- Even better long-term — pre-compute email counts during sync:
  - Add an `email_count` column to `discovered_addresses`
  - Update the count incrementally when new emails arrive (in `process_fetch.go`)
  - The `GetAddresses` query becomes a simple `SELECT ... ORDER BY email_count DESC LIMIT $2 OFFSET $3` with no subquery at all
  - This approach eliminates the expensive correlated subquery entirely

- The `LIMIT 500` value is chosen because onboarding address review is a one-time setup step. Users typically have fewer than 100 meaningful addresses (real people vs. newsletters/noreply). Showing 500 covers even large organizations.

- Also fix the unchecked `rows.Scan()` at line 380 (cross-reference with PRD-049):
  ```go
  if err := rows.Scan(&id, &address, &localPart, &addrType, &emailCount); err != nil {
      slog.Error("scan discovered address", "error", err)
      continue
  }
  ```

- Pre-allocate the addresses slice with a reasonable capacity:
  ```go
  addresses := make([]map[string]interface{}, 0, limit)
  ```

## User Notes
The onboarding "configure addresses" step will load much faster for organizations that have imported large mailboxes. Previously, organizations with thousands of discovered addresses might see the page take 30+ seconds to load or time out entirely. With the LIMIT in place, the page will load in under a second. The most frequently emailed addresses are shown first, so users can configure the most important ones without waiting for the full list.
