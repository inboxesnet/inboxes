# PRD-059: Labels Subquery

## Status
Done

## Priority
P2-Medium

## Category

## Source

## Problem Statement
The `labelsSubquery` constant is a correlated subquery that runs once per row in every thread list query. For a page of 50 threads, this executes 50 separate subqueries to aggregate labels from `thread_labels`. While `thread_labels` has an index on `(thread_id, label)`, the repeated subquery execution adds significant overhead, especially as the number of labels per thread and total threads grows.

## Current Behavior
In `backend/internal/handler/threads.go:22`:
```go
const labelsSubquery = `(SELECT COALESCE(array_agg(tl2.label ORDER BY tl2.label), ARRAY[]::text[]) FROM thread_labels tl2 WHERE tl2.thread_id = t.id)`
```

This subquery is included in every thread list SELECT:
- Archive query (line 95)
- Trash/spam query (line 110)
- Default label query (line 125)
- Search query in `emails.go:308,314`
- Thread detail query (line 244)
- Thread summary for events (`threads.go:1013`, `process_fetch.go:525`)
- Unread count queries do NOT use it (they don't need labels)

The `thread_labels` table has an index `idx_thread_labels_thread` on `(thread_id, label)` (from migration 014), which helps the subquery but does not eliminate the per-row execution cost.

The same subquery is duplicated in `backend/internal/queue/process_fetch.go:511`:
```go
const labelsSubquery = `(SELECT COALESCE(array_agg(tl2.label ORDER BY tl2.label), ARRAY[]::text[]) FROM thread_labels tl2 WHERE tl2.thread_id = t.id)`
```

## Expected Behavior
Replace the correlated subquery with a `LEFT JOIN` + `GROUP BY` approach or a lateral join. This allows PostgreSQL to use a hash join or merge join strategy instead of executing a subquery per row.

Alternatively, use a CTE (Common Table Expression) to pre-aggregate labels for the set of thread IDs being returned, then join back.

## Implementation Notes
- **Option A: LEFT JOIN + GROUP BY** — Restructure the query to use `LEFT JOIN thread_labels tl2 ON tl2.thread_id = t.id` and `GROUP BY t.id`. The labels become `COALESCE(array_agg(tl2.label ORDER BY tl2.label) FILTER (WHERE tl2.label IS NOT NULL), ARRAY[]::text[])`. This is the most straightforward approach but requires GROUP BY on all selected columns.

- **Option B: Post-query batch fetch** — Execute the main thread query without labels, collect the thread IDs, then run a single `SELECT thread_id, array_agg(label) FROM thread_labels WHERE thread_id = ANY($1) GROUP BY thread_id` query. Merge the results in Go code. This is the cleanest separation and avoids changing the main query structure.

- **Option C: LATERAL JOIN** — Use `LEFT JOIN LATERAL (SELECT array_agg(...) FROM thread_labels WHERE thread_id = t.id) labels ON true`. This is functionally equivalent to the correlated subquery but may allow PostgreSQL to optimize differently.

- Option B is recommended for simplicity and testability. It adds one query (batch labels) instead of modifying the complex main query.

- The `thread_labels` table already has `idx_thread_labels_thread` on `(thread_id, label)`, which is an Index Only Scan candidate for the batch query.

## User Notes
Thread list pages will load faster. The improvement scales with the number of threads per page and the number of labels per thread. Users with many custom labels will see the biggest improvement.
