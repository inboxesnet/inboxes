# PRD-057: N+1 Trash

## Status
Done

## Priority
P2-Medium

## Category
Database Performance

## Source
PRE-LAUNCH-AUDIT #25, GAP-FINDER (related to D1)

---

## Problem Statement
When listing threads in the trash view, the handler runs a separate database query for each thread to fetch its `trash_expires_at` value. For a page of 50 threads, this is 50 extra queries inside the `for rows.Next()` loop. This N+1 query pattern causes significant latency on the trash folder page, especially as the number of trashed threads grows.

## Current Behavior
In `backend/internal/handler/threads.go:202-211`:
```go
if label == "trash" {
    var trashExpiresAt *time.Time
    h.DB.QueryRow(ctx,
        "SELECT trash_expires_at FROM threads WHERE id = $1", id,
    ).Scan(&trashExpiresAt)
    if trashExpiresAt != nil {
        t["trash_expires_at"] = *trashExpiresAt
    }
}
```

This runs inside the `for rows.Next()` loop (line 171), which iterates once per thread. The `threads` table already has the `trash_expires_at` column on the same row being iterated, but it is not included in the SELECT clause of the main query.

The same pattern exists in `threads.go:358-363` (the Get handler), where `trash_expires_at` is fetched separately after the main thread query:
```go
var trashExpiresAt *time.Time
h.DB.QueryRow(ctx, "SELECT trash_expires_at FROM threads WHERE id = $1", threadID).Scan(&trashExpiresAt)
```

## Expected Behavior
Include `t.trash_expires_at` in the main list query's SELECT clause. The column is already on the `threads` table and requires no join. For the Get handler, include it in the existing SELECT.

## Files to Modify
- `backend/internal/handler/threads.go:93-99` — Add `t.trash_expires_at` to the archive query SELECT
- `backend/internal/handler/threads.go:108-113` — Add `t.trash_expires_at` to the trash/spam query SELECT
- `backend/internal/handler/threads.go:123-130` — Add `t.trash_expires_at` to the default (inbox/sent/starred) query SELECT
- `backend/internal/handler/threads.go:171-212` — Update scan variables to include trash_expires_at; remove the inner query loop
- `backend/internal/handler/threads.go:241-248` — Add `t.trash_expires_at` to the Get query SELECT
- `backend/internal/handler/threads.go:358-363` — Remove the separate trash_expires_at query in Get

## Impact Analysis

### What This Affects
- Trash folder page load time (eliminates N extra queries per page)
- Thread detail view load time (eliminates 1 extra query)
- Database load and connection pool utilization

### What Affects This
- Number of threads per page (currently 50-200)
- Database query plan caching

## Implementation Notes
- Add `t.trash_expires_at` as a nullable column to all three query variants (archive, trash/spam, default).
- Update the scan in the row loop to include a `*time.Time` variable for `trash_expires_at`.
- Only include `trash_expires_at` in the response map when it is non-nil (same as current behavior).
- Remove the inner `QueryRow` at line 205-207 and the separate query at line 360.
- For the Get handler, add `t.trash_expires_at` to the main SELECT and scan it inline.
- Since this column is always NULL for non-trash threads, the overhead of including it in all queries is negligible (just reading a NULL from the already-fetched row).

## User Notes
Users viewing the trash folder will see faster page loads. The countdown banner showing "Deleted in X days" will still work exactly the same way — the data source just becomes the main query instead of a per-thread query.
