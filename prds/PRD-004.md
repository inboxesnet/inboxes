# PRD-004: Docker Containers Run as Root

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
PRE-LAUNCH-AUDIT #4

---

## Problem Statement
Neither the backend nor frontend Dockerfiles create or switch to a non-root user. If an attacker achieves code execution inside the container (e.g., through a dependency vulnerability), they have root privileges, which significantly increases the blast radius of a container escape.

## Current Behavior
**Backend Dockerfile** (`backend/Dockerfile`, 13 lines):
- Line 1: Build stage uses `golang:1.23-alpine`.
- Line 8: Runtime stage uses `alpine:3.20`.
- Line 10: `COPY --from=builder /inboxes /inboxes` — binary owned by root.
- Line 13: `CMD ["/inboxes"]` — runs as root (PID 1).
- No `USER` directive anywhere.

**Frontend Dockerfile** (`frontend/Dockerfile`, 16 lines):
- Line 1: Build stage uses `node:20-alpine`.
- Line 8: Runtime stage uses `node:20-alpine`.
- Line 11-12: Copies `.next/standalone` and `.next/static` — files owned by root.
- Line 16: `CMD ["node", "server.js"]` — runs as root.
- No `USER` directive anywhere.

## Expected Behavior
- Both containers should run as a non-root user (e.g., `appuser` with UID 1001).
- The binary/application files should be owned by or readable by the non-root user.
- The health check should still work (curl runs as the non-root user against localhost).
- No functionality should be affected since neither service needs root privileges at runtime.

## Files to Modify
- `backend/Dockerfile:8-13` — After the `FROM alpine:3.20` line, add a non-root user and switch to it before CMD:
  ```dockerfile
  RUN addgroup -S appgroup && adduser -S appuser -G appgroup
  COPY --from=builder /inboxes /inboxes
  USER appuser
  ```
- `frontend/Dockerfile:8-16` — After the `FROM node:20-alpine` line, add a non-root user and switch to it:
  ```dockerfile
  RUN addgroup -S appgroup && adduser -S appuser -G appgroup
  WORKDIR /app
  COPY --chown=appuser:appgroup --from=builder /app/.next/standalone ./
  COPY --chown=appuser:appgroup --from=builder /app/.next/static ./.next/static
  USER appuser
  ```

## Impact Analysis

### What This Affects
- Container runtime security posture.
- Docker image builds and deployment pipeline.
- Health check commands (curl) — must still work as non-root user.

### What Affects This
- The backend binary listens on port 8080 (>1024, so no root needed).
- The frontend listens on port 3000 (>1024, so no root needed).
- File permissions on copied artifacts must allow the non-root user to read/execute.

### Related PRDs
- PRD-005: CSP Header — Both are defense-in-depth measures. Docker hardening limits container escape impact; CSP limits script injection impact.

## Implementation Notes
- Use `addgroup -S` and `adduser -S` (Alpine-specific syntax for system users/groups).
- `--chown=appuser:appgroup` on COPY directives ensures the non-root user owns the files.
- For the backend, the binary just needs execute permission (default from Go build).
- For the frontend, the entire `/app` directory needs read access; `node` needs write access to nothing at runtime (Next.js standalone doesn't write to disk).
- Test by running `docker exec <container> whoami` — should return `appuser`, not `root`.
- If the app writes temp files, ensure the working directory is writable by the non-root user.
