# PRD-075: DNS Record Change Detection

## Status
Draft

## Priority
P2-Medium

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L4

---

## Problem Statement
The domain heartbeat worker checks whether domains still exist in Resend but never verifies whether their DNS records (MX, SPF, DKIM) have changed. If a domain's DNS records become misconfigured after initial verification, the system continues to treat the domain as fully verified. Emails may silently fail to deliver or be marked as spam by recipients because SPF/DKIM authentication fails, but the app gives no indication that DNS configuration has degraded.

## Current Behavior
In `backend/internal/worker/domain_heartbeat.go`:
- Lines 103-118: The Resend API response is parsed to extract `Data[].ID` and `Data[].Name` only. The response likely also includes verification status fields, but they are not captured.
- Lines 114-118: A set of domain names (`resendDomains`) is built for presence checking only.
- Lines 120-154: Local domains are compared against the Resend set for existence. If a domain is missing from Resend, it is marked `disconnected`. If it reappears, it is marked `active`. No verification status is checked.
- The `domains` table has columns `mx_verified`, `spf_verified`, `dkim_verified` (migration 001, lines 54-56), but these are never updated after initial domain creation.

In `backend/internal/handler/domains.go`:
- Lines 170-209: The `Verify` handler calls Resend's verify endpoint and updates `status` and `dns_records`, but this is only triggered manually by the user.

## Expected Behavior
- The domain heartbeat should fetch full domain details (including verification status) from Resend.
- Compare the returned verification fields against the stored `mx_verified`, `spf_verified`, `dkim_verified` values.
- If any verification field changes from `true` to `false`, update the local record and emit a warning event (log + optional notification).
- If all verifications are restored, clear the warning.
- This provides ongoing monitoring of DNS health, not just domain existence.

## Files to Modify
- `backend/internal/worker/domain_heartbeat.go:103-118` — Expand the `resendResp` struct to include verification fields:
  ```go
  Data []struct {
      ID          string `json:"id"`
      Name        string `json:"name"`
      Status      string `json:"status"`
      Records     []struct {
          Type   string `json:"type"`
          Status string `json:"status"`
      } `json:"records"`
  } `json:"data"`
  ```
- `backend/internal/worker/domain_heartbeat.go:129-154` — After the existence check, fetch individual domain details from Resend (if needed) and update `mx_verified`, `spf_verified`, `dkim_verified` in the database.
- `backend/internal/worker/domain_heartbeat.go:75-101` — Consider calling `GET /domains/{id}` for each active domain to get full verification status, or check if the list endpoint includes sufficient detail.

## Impact Analysis

### What This Affects
- Domain health monitoring.
- The `domains` table columns `mx_verified`, `spf_verified`, `dkim_verified` will be kept up-to-date.
- Frontend domain settings can show current verification status.
- Admins will be alerted to DNS misconfigurations before they cause delivery issues.

### What Affects This
- Resend API rate limits: if there are many domains, fetching individual domain details could hit rate limits. Use the org-level rate limiter.
- The heartbeat interval (currently 6 hours) determines how quickly DNS changes are detected.

### Related PRDs
- PRD-079: Domain Disconnect Notification — DNS degradation is a type of "disconnection" that should trigger notifications.
- PRD-072: Domain Deletion Data Cleanup — Deleted domains should be excluded from heartbeat checks.
- PRD-109: Worker Intervals — The heartbeat interval may need to be configurable for DNS-sensitive deployments.

## Implementation Notes
- The Resend API `GET /domains` list endpoint may not return full DNS record status. If not, each domain needs an individual `GET /domains/{id}` call. Rate-limit these calls per org.
- Compare the fetched status against stored values before writing to DB to avoid unnecessary updates.
- For the first iteration, updating the three boolean columns and logging changes may be sufficient. Notification (email/WS event) can be added as part of PRD-079.
- Consider adding a `dns_last_checked_at` timestamp column to track when DNS was last verified.
