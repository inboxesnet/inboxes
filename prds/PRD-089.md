# PRD-089: WS Connection Limit

## Status
Done

## Priority
P2-Medium

## Category
WebSocket & Real-Time

## Source
GAP-FINDER W1

---

## Problem Statement
There is no limit on the number of WebSocket connections a single user or org can open. The `clients` map in the Hub (`backend/internal/ws/hub.go:16`) is unbounded. A compromised account or misbehaving client (e.g., multiple browser tabs, automated scripts) can open hundreds or thousands of connections, consuming server memory, goroutines, and database connection pool slots (each connection periodically queries aliases and checks token blacklist).

## Current Behavior
In `backend/internal/ws/hub.go:15-22`:
```go
type Hub struct {
    clients    map[string]map[*Client]bool // userID -> set of clients
    register   chan *Client
    unregister chan *Client
    // ...
}
```

The `register` case in `Hub.Run()` (line 49-55) unconditionally adds new clients:
```go
case client := <-h.register:
    h.mu.Lock()
    if _, ok := h.clients[client.UserID]; !ok {
        h.clients[client.UserID] = make(map[*Client]bool)
    }
    h.clients[client.UserID][client] = true
    h.mu.Unlock()
```

No check is performed on the number of existing connections for the user. Each client spawns two goroutines (`readPump` and `writePump` in `client.go:112-113`), each with periodic tickers. The `writePump` queries the database for alias addresses every 5 minutes (line 204-206) and checks the token blacklist (line 195-202).

In `backend/internal/ws/client.go:39-114`, `ServeWS` creates a new client for every WebSocket upgrade request without any connection counting.

## Expected Behavior
1. Limit WebSocket connections to 5 per user (covers multiple tabs/devices).
2. When a user exceeds the limit, close the oldest connection with a WebSocket close message indicating the reason.
3. Optionally: limit total connections per org to prevent a single org from overwhelming the server.
4. Log when connections are rejected or evicted due to limits.

## Files to Modify
- `backend/internal/ws/hub.go:49-55` -- Add connection count check in the `register` case; evict oldest connection if over limit
- `backend/internal/ws/client.go:39-114` -- Optionally: reject connection before upgrade if count check is done early
- `backend/internal/ws/hub.go:15-22` -- Add a `maxConnectionsPerUser` field to Hub config

## Impact Analysis

### What This Affects
- Server memory and goroutine usage under connection pressure
- Database connection pool usage (each WS client queries DB periodically)
- User experience with many tabs open (oldest tab's WS connection will be closed)

### What Affects This
- Number of browser tabs/devices per user
- Token validation interval (5 minutes per client)
- Alias address refresh interval (5 minutes per client)

### Related PRDs
- PRD-090 (Token Expiry) -- token checks per client consume resources proportional to connection count
- PRD-091 (Alias Filter Fail) -- alias queries per client scale with connections
- PRD-093 (Reconnect Jitter) -- connection limits prevent thundering herd from creating unbounded connections

## Implementation Notes
- In the `register` case of `Hub.Run()`:
  ```go
  case client := <-h.register:
      h.mu.Lock()
      if _, ok := h.clients[client.UserID]; !ok {
          h.clients[client.UserID] = make(map[*Client]bool)
      }
      userClients := h.clients[client.UserID]
      if len(userClients) >= maxConnectionsPerUser {
          // Find and evict the oldest connection
          var oldest *Client
          for c := range userClients {
              if oldest == nil || c.IssuedAt.Before(oldest.IssuedAt) {
                  oldest = c
              }
          }
          if oldest != nil {
              delete(userClients, oldest)
              close(oldest.send)
              slog.Info("ws: evicted oldest connection", "user_id", client.UserID)
          }
      }
      userClients[client] = true
      h.mu.Unlock()
  ```
- Set `maxConnectionsPerUser` to 5 initially. Make it configurable via environment variable `WS_MAX_CONNECTIONS_PER_USER`.
- Send a close message with reason before evicting: `oldest.conn.WriteMessage(websocket.CloseMessage, websocket.FormatCloseMessage(4001, "connection limit exceeded"))`.
- The frontend reconnection logic in `notification-context.tsx` will automatically attempt to reconnect, but the limit will evict the oldest connection again, so the user settles at the limit.

## User Notes
- Users with more than 5 open tabs will have the oldest tab's real-time updates disconnected. The tab will still work but won't receive live updates until refreshed.
- Normal usage (1-3 tabs) is unaffected.

## Acceptance Criteria
- [ ] A per-user WebSocket connection limit is enforced (default: 5)
- [ ] When the limit is exceeded, the oldest connection is closed with a descriptive close message
- [ ] The limit is configurable via `WS_MAX_CONNECTIONS_PER_USER` environment variable
- [ ] Server memory usage remains bounded under connection pressure
- [ ] Log message is emitted when a connection is evicted
- [ ] Normal multi-tab usage (up to 5 tabs) works without interruption
- [ ] Unit test: opening 6 connections for the same user results in the first being evicted
