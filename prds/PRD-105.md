# PRD-105: Form Error Announcements for Screen Readers

## Status
Done

## Priority
P3-Low

## Category
Accessibility

## Source
GAP-FINDER A11

---

## Problem Statement
Error messages on authentication forms (login, signup, forgot password, reset password, claim, verify email) and the compose window are not announced to screen readers. Error `<div>` elements lack `role="alert"`, `aria-live`, and `aria-atomic` attributes. Form inputs do not have `aria-invalid` or `aria-describedby` linking them to their error messages. Users relying on assistive technology may not notice that their form submission failed.

## Current Behavior
In `login/page.tsx:79-82`, the error message is rendered as:
```tsx
<div className="text-sm text-destructive bg-destructive/10 p-3 rounded-md">{error}</div>
```
This same pattern is used in:
- `signup/page.tsx:117-120` -- signup form errors
- `forgot-password/page.tsx` -- password reset request errors
- `reset-password/page.tsx` -- password reset errors
- `claim/page.tsx` -- invite claim errors
- `verify-email/page.tsx` -- verification code errors
- `floating-compose-window.tsx:435-438` (mobile) and `578-581` (desktop) -- compose form errors

None of these error containers have `role="alert"` or `aria-live` attributes. The form inputs do not have `aria-invalid="true"` when errors are present. There is no `aria-describedby` linking inputs to their error messages.

## Expected Behavior
- Error message containers should have `role="alert"` so they are immediately announced when they appear
- Form inputs should have `aria-invalid="true"` when validation fails
- Error messages should be linked to relevant inputs via `aria-describedby`
- Error messages should include `aria-atomic="true"` so the full message is read, not just the changed part

## Implementation Notes
- Simplest fix: add `role="alert"` to all error divs:
  ```tsx
  {error && (
    <div role="alert" className="text-sm text-destructive bg-destructive/10 p-3 rounded-md">
      {error}
    </div>
  )}
  ```
- `role="alert"` implies `aria-live="assertive"` and `aria-atomic="true"`, so no need to add those separately
- For input-level validation, add `aria-invalid`:
  ```tsx
  <Input
    id="email"
    type="email"
    aria-invalid={!!error}
    aria-describedby={error ? "form-error" : undefined}
    ...
  />
  ```
- Consider creating a reusable `FormError` component to standardize the pattern across all forms
- The compose window has two error divs (mobile and desktop) -- both need the fix
