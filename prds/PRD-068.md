# PRD-068: Cross-Domain Thread Splitting

## Status
Done

## Priority
P1-High

## Category
Email Threading & Rendering

## Source
GAP-FINDER T3

## Problem Statement
Thread matching in Step 2 filters by `domain_id`, which means a reply from domain B to a conversation that originated on domain A creates a new thread instead of continuing the existing one. Organizations using multiple domains see fragmented conversations when contacts reply to different domain addresses within the same conversation.

## Current Behavior
In `backend/internal/queue/process_fetch.go`:
- Lines 267-272: Step 2 query includes `WHERE org_id = $1 AND domain_id = $2 AND subject = $3`. The `domain_id` is derived from the recipient address of the current inbound email (lines 60-112).
- If the original thread was created under domain A (e.g., `alice@company.com`), and the external sender replies but the email is delivered to domain B (e.g., `support@company.io`), the `domain_id` for domain B does not match domain A. Step 2 fails to find the existing thread.
- Step 1 (In-Reply-To matching, line 253-261) also scopes by `org_id` only, so it would find the thread -- but only if the email client includes a correct `In-Reply-To` header. Many scenarios lack this header.
- The result: cross-domain replies create duplicate threads when `In-Reply-To` is missing.

## Expected Behavior
- Step 2 (subject+counterparty matching) should match by `org_id` instead of `domain_id`, allowing threads to span multiple domains within the same organization.
- The thread's `domain_id` on the `threads` table represents the domain the thread was first created under and should remain unchanged.
- Emails within the thread will still have their correct `domain_id` from the email record itself.

## Files to Modify
- `backend/internal/queue/process_fetch.go:268-271` — Remove `AND domain_id = $2` from the Step 2 query, or replace with `AND org_id = $1` (already present). Adjust the parameter numbering accordingly.

## Impact Analysis

### What This Affects
- Thread grouping for organizations using multiple domains.
- Frontend thread list: threads may now group emails from different domains, which is the desired behavior.
- The thread's `domain_id` remains the originating domain; individual emails keep their own `domain_id`.
- Unread count queries in `backend/internal/handler/domains.go:251-277` group by `t.domain_id` -- cross-domain threads will count toward the originating domain's unread badge.

### What Affects This
- Step 1 (In-Reply-To) already scopes by `org_id` only, so this change aligns Step 2.
- PRD-066 (References matching) should also scope by `org_id`.

### Related PRDs
- PRD-066: References Header Threading — Both modify thread matching logic; References match should also be org-scoped.
- PRD-067: Empty Subject Threading — Adjacent modification to Step 2 logic.
- PRD-045: Duplicate Threads — Cross-domain replies are a source of duplicate threads.

## Implementation Notes
- Remove `AND domain_id = $2` from the Step 2 query at line 268 and shift parameter indices.
- The counterparty match (`participant_emails @> $N::jsonb`) combined with subject match within the org is sufficient to prevent false-positive merges across unrelated conversations.
- Consider adding a comment explaining why `domain_id` is intentionally excluded from Step 2.
- The 90-day window on `last_message_at` provides additional protection against stale matches.
