# PRD-027: Compose Draft Loss

## Status
Done

## Priority
P1-High

## Category
Drafts & Compose Issues

## Source
PRE-LAUNCH-AUDIT #13

---

## Problem Statement
When a user closes the compose window, the `handleClose` function attempts to save the draft if there are unsaved changes. However, if that save fails, the `catch` block swallows the error and closes the window anyway, permanently losing the user's work. This is particularly dangerous because the user receives no indication that their draft was not saved.

## Current Behavior
In `floating-compose-window.tsx`, the `handleClose` callback (lines 265-275) wraps `saveDraft()` in a try/catch. On failure, the catch block is empty (just a comment: "Save failed -- still close to avoid trapping the user"), and `closeCompose()` is called unconditionally. The user's draft disappears with no error message, no toast, and no opportunity to retry or copy their work.

## Expected Behavior
When a save fails on close:
1. Show an error toast: "Failed to save draft. Please try again or copy your work."
2. Keep the compose window open so the user retains their content.
3. Provide a "Discard anyway" option if the user explicitly wants to close despite the failure.
4. Only auto-close on successful save or explicit user discard.

## Files to Modify
- `frontend/components/floating-compose-window.tsx:265-275` -- Change `handleClose` to show an error toast and keep the window open on save failure. Add a `closeFailedSave` state to conditionally show a "Discard unsaved changes?" prompt.
- `frontend/components/floating-compose-window.tsx:370-375` -- Update the minimized-state close button to also respect the save-failure guard.

## Impact Analysis

### What This Affects
- All users composing emails -- any time they close the compose window with unsaved changes and the network is unreliable or the backend is down, they risk data loss.
- The `handleClose` function is called from the X button in both mobile (line 411) and desktop (line 565) compose views, plus the minimized state (line 370).

### What Affects This
- The `saveDraft` function (lines 196-234) -- its error handling determines whether the close proceeds.
- Network connectivity and backend availability.
- The `dirtyRef` flag that gates whether a save is attempted on close.

### Related PRDs
- PRD-028 (Duplicate Drafts) -- rapid saves affect the same save flow
- PRD-030 (Auto-Save Invisible) -- the silent failure on auto-save is the root cause of data appearing "saved" when it is not
- PRD-035 (Session Expiry) -- session expiry can cause saves to fail with 401, compounding this issue

## Implementation Notes
- Add a `saveError` state variable. In `handleClose`, set it on catch instead of proceeding to `closeCompose()`.
- When `saveError` is true, render a small alert banner in the compose window: "Draft could not be saved. [Try again] [Discard]".
- The "Try again" button should call `saveDraft()` again and close on success.
- The "Discard" button should call `closeCompose()` directly without saving.
- Consider adding `toast.error("Failed to save draft")` in the catch block as an immediate visual cue.

## User Notes
- Users on slow or unreliable connections are most affected.
- This matches Gmail's behavior: Gmail warns "Your message has not been sent" and keeps the compose window open if a save/send fails.

## Acceptance Criteria
- [ ] Closing compose with unsaved changes when the backend returns an error keeps the window open
- [ ] An error toast or inline error message is displayed when save-on-close fails
- [ ] User can retry the save from the error state
- [ ] User can explicitly discard the draft from the error state
- [ ] The beforeunload warning still fires if the window has unsaved changes and the user tries to leave the page
- [ ] Mobile and desktop close buttons both respect the save-failure guard
- [ ] Minimized-state close button also respects the save-failure guard
