# PRD-060: Bulk Delete Queries

## Status
Draft

## Priority
P2-Medium

## Category
Database Performance

## Source
GAP-FINDER D2

---

## Problem Statement
The bulk delete action in the thread handler runs 3+ separate queries per thread in a loop: `hasLabel()` to check for trash label, `removeAllLabels()` to delete labels, and an UPDATE to set `deleted_at`. For 100 threads, this is 300+ database round trips. This causes significant latency for bulk delete operations and monopolizes database connections.

## Current Behavior
In `backend/internal/handler/threads.go:508-518`:
```go
case "delete":
    for _, tid := range req.ThreadIDs {
        if hasLabel(ctx, h.DB, tid, "trash") {
            removeAllLabels(ctx, h.DB, tid)
            h.DB.Exec(ctx,
                `UPDATE threads SET deleted_at = now(), updated_at = now() WHERE id = $1 AND org_id = $2`,
                tid, claims.OrgID)
            affected++
        }
    }
```

For each thread:
1. `hasLabel()` runs: `SELECT EXISTS(SELECT 1 FROM thread_labels WHERE thread_id = $1 AND label = $2)` — 1 query
2. `removeAllLabels()` runs: `DELETE FROM thread_labels WHERE thread_id = $1` — 1 query
3. `UPDATE threads SET deleted_at = now()` — 1 query

Total: 3 queries per thread, or 300 queries for 100 threads.

The same pattern exists in the single-thread Delete handler (`threads.go:889-915`) but is less impactful since it only processes one thread.

## Expected Behavior
Replace the per-thread loop with batch operations:
1. A single query to filter thread IDs that have the "trash" label.
2. A single `DELETE FROM thread_labels WHERE thread_id = ANY($1)` for all filtered IDs.
3. A single `UPDATE threads SET deleted_at = now() WHERE id = ANY($1) AND org_id = $2` for all filtered IDs.

Total: 3 queries regardless of thread count.

## Files to Modify
- `backend/internal/handler/threads.go:508-518` — Replace per-thread loop with batch SQL operations

## Impact Analysis

### What This Affects
- Bulk delete operation latency (100x reduction in query count)
- Database connection pool utilization during bulk operations
- API response time for the bulk action endpoint

### What Affects This
- Number of threads being deleted (impact proportional to count)
- Database connection pool size
- Transaction isolation level

### Related PRDs
- PRD-050 (Fire-and-Forget Writes) — the current loop also ignores errors from all three operations
- PRD-044 (Label Transactions) — the batch operations should be wrapped in a transaction
- PRD-057 (N+1 Trash) — same N+1 pattern in a different context

## Implementation Notes
- Batch implementation:
  ```go
  case "delete":
      // Step 1: Filter to only threads that have "trash" label
      var trashIDs []string
      rows, err := h.DB.Query(ctx,
          `SELECT DISTINCT thread_id FROM thread_labels
           WHERE thread_id = ANY($1::uuid[]) AND label = 'trash'`,
          req.ThreadIDs)
      if err == nil {
          defer rows.Close()
          for rows.Next() {
              var tid string
              if rows.Scan(&tid) == nil {
                  trashIDs = append(trashIDs, tid)
              }
          }
      }
      if len(trashIDs) > 0 {
          // Step 2: Batch delete all labels
          h.DB.Exec(ctx,
              `DELETE FROM thread_labels WHERE thread_id = ANY($1::uuid[])`,
              trashIDs)
          // Step 3: Batch soft-delete threads
          tag, _ := h.DB.Exec(ctx,
              `UPDATE threads SET deleted_at = now(), updated_at = now()
               WHERE id = ANY($1::uuid[]) AND org_id = $2`,
              trashIDs, claims.OrgID)
          affected = tag.RowsAffected()
      }
  ```
- Wrap all three steps in a transaction for atomicity (overlaps with PRD-044).
- Check and handle errors from each step (overlaps with PRD-050).
- The trash collector (`trash_collector.go:70-82`) already uses a batch approach — use it as a reference.

## User Notes
Bulk deleting threads from the trash will be significantly faster. Currently, deleting 100 threads takes several seconds; after this fix, it should complete in under a second.

## Acceptance Criteria
- [ ] Bulk delete of 100 threads executes at most 4 SQL queries (filter + delete labels + update threads + optional count)
- [ ] Only threads with the "trash" label are deleted (same guard as current behavior)
- [ ] Thread labels are removed for all deleted threads
- [ ] The `affected` count in the response is accurate
- [ ] Errors from batch operations are checked and logged
