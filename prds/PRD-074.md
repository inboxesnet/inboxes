# PRD-074: Domain Uniqueness is Global

## Status
Done

## Priority
P2-Medium

## Category
Domain & User Lifecycle

## Source
GAP-FINDER L3

---

## Problem Statement
The `domains` table has a global `UNIQUE` constraint on the `domain` column (`domain TEXT NOT NULL UNIQUE` in migration 001, line 51). This means a soft-deleted domain in one org blocks any other org from adding the same domain. In a multi-tenant environment, this creates a permanent lock on domain names after deletion, with no way to release them.

## Current Behavior
In `backend/internal/db/migrations/001_initial.sql`:
- Line 51: `domain TEXT NOT NULL UNIQUE` -- the constraint is table-wide, not scoped to org or status.

In `backend/internal/handler/domains.go`:
- Lines 148-157: Domain creation INSERT uses `INSERT INTO domains (org_id, domain, ...) VALUES (...)`. If another org (or the same org after deletion) already has a row with that domain name, the INSERT fails with a unique constraint violation.
- Lines 356-365: Domain sync uses `ON CONFLICT (domain) DO UPDATE`, which would update the existing row regardless of which org owns it. This is potentially dangerous -- if org B syncs and org A has a deleted domain with that name, org B's sync could update org A's row.

## Expected Behavior
- Option A (recommended for now): Change the UNIQUE constraint to a partial unique index: `CREATE UNIQUE INDEX idx_domains_unique_active ON domains(domain) WHERE status NOT IN ('deleted')`. This allows re-adding a domain once the previous one is deleted.
- Option B (future multi-org): Change to a composite unique constraint `UNIQUE(org_id, domain)` -- but this allows different orgs to claim the same domain simultaneously, which has security implications (domain claiming).
- Option A is safer: only one active domain record per domain name, but deleted records do not block re-creation.
- The `ON CONFLICT (domain)` clause in the Sync handler must be updated to match the new index.

## Files to Modify
- New migration file (e.g., `backend/internal/db/migrations/032_domain_unique_active.sql`) — Drop the existing UNIQUE constraint and create a partial unique index:
  ```sql
  ALTER TABLE domains DROP CONSTRAINT domains_domain_key;
  CREATE UNIQUE INDEX idx_domains_unique_active ON domains(domain) WHERE status NOT IN ('deleted');
  ```
- `backend/internal/handler/domains.go:358-362` — Update the `ON CONFLICT (domain)` in the Sync handler to use `ON CONFLICT ON CONSTRAINT idx_domains_unique_active` or adjust the conflict target.

## Impact Analysis

### What This Affects
- Domain creation: previously deleted domains can be re-added by the same or different org.
- Domain sync: the ON CONFLICT clause needs adjustment.
- Domain uniqueness enforcement: only active/pending/verified domains hold the uniqueness lock.

### What Affects This
- Migration ordering: this must run after migration 025 (domain deleted status) and 019 (disconnected status).
- The Sync handler's ON CONFLICT behavior must be tested carefully.

- PRD-075: DNS Changes Detection — Domain re-addition may involve new DNS records.

## Implementation Notes
- PostgreSQL partial unique indexes work with `ON CONFLICT` only if the conflict target matches the index exactly. Use `ON CONFLICT (domain) WHERE status NOT IN ('deleted')` or restructure the upsert.
- Test the edge case: org A has `example.com` (deleted), org B adds `example.com` -- should succeed. Org B deletes `example.com`, org A re-adds it -- should also succeed.
- The FULL-SUMMARY notes this may be intentional ("prevents domain claiming. Domain transfers = support ticket like Gmail"). The recommended Option A is a compromise: deleted domains can be re-claimed, but active domains cannot.
