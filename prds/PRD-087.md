# PRD-087: Plan Check at Send

## Status
Done

## Priority
P1-High

## Category
Billing & Stripe

## Source
GAP-FINDER B7

---

## Problem Statement
The plan/subscription check is performed when the email is queued (`RequirePlan` middleware on the `POST /api/emails/send` route), but not when the email is actually sent by the worker. If a subscription lapses between queue time and send time (which could be minutes or hours if the queue is backed up or Redis had issues), the email still goes out. This allows sending emails without an active plan.

## Current Behavior
The email sending flow has two stages:

**Stage 1 - Queue time** (`backend/internal/router/router.go:182-183`):
```go
r.Use(middleware.RequirePlan(stripeKey, db))
// ...
r.Post("/api/emails/send", emails.Send)
```
`RequirePlan` in `backend/internal/middleware/auth.go:148-186` checks `plan` and `plan_expires_at` before allowing the request.

**Stage 2 - Send time** (`backend/internal/queue/process_send.go:23-101`):
The `processSend` function checks domain status (line 37-47) but does NOT check the org's plan status. It proceeds to call the Resend API regardless of whether the org still has an active subscription.

The gap is:
- User queues email at 2:00 PM (plan is `pro`)
- Subscription expires at 2:05 PM (webhook sets plan to `cancelled`)
- Worker picks up job at 2:10 PM and sends the email despite no active plan

## Expected Behavior
The email worker should re-verify the org's plan status before making the Resend API call. If the plan is no longer active (not `pro`, not `past_due`, not in grace period), the job should be marked as `failed` with a clear error message indicating the subscription lapsed.

## Files to Modify
- `backend/internal/queue/process_send.go:23-47` -- Add plan check after loading the job and before the domain status check
- `backend/internal/queue/email_worker.go:59-119` -- Alternative: add plan check in `processJob` before dispatching to `processSend`

## Impact Analysis

### What This Affects
- Email delivery for orgs with lapsed subscriptions
- Revenue protection (prevents free usage via queued emails)
- Email worker error handling and job failure messages

### What Affects This

## Implementation Notes
- Extract the plan-check logic from `RequirePlan` middleware into a shared function (e.g., `service.IsPlanActive(ctx, db, orgID) bool`) to avoid duplication.
- The check should be: `plan IN ('pro', 'past_due') OR (plan = 'cancelled' AND plan_expires_at > now())`.
- Place the check in `processSend` right after loading the job, before any Resend API calls:
  ```go
  // Check plan status at send time
  var plan string
  var planExpiresAt *time.Time
  if err := w.pool.QueryRow(ctx,
      "SELECT plan, plan_expires_at FROM orgs WHERE id = $1 AND deleted_at IS NULL",
      orgID).Scan(&plan, &planExpiresAt); err != nil {
      return fmt.Errorf("org not found or deleted: %w", err)
  }
  if !isPlanActive(plan, planExpiresAt) {
      return fmt.Errorf("subscription inactive (plan=%s), email not sent", plan)
  }
  ```
- This should be a non-retryable failure (permanently fail the job, don't retry).
- Only apply this check when `stripeKey` is configured (self-hosted instances skip it).

## User Notes
- If a user's subscription lapses while emails are queued, those emails will fail with a clear error message visible in the admin jobs panel.
- Users can re-send the emails after reactivating their subscription.

## Acceptance Criteria

- [ ] Grace period is respected (emails within grace period still send)
- [ ] `past_due` orgs can still send emails
