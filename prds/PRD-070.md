# PRD-070: Scope users.email Uniqueness to Org

## Problem

`users.email` has a global UNIQUE constraint. If two separate orgs (Resend accounts) are hosted on the same instance, a user can't exist in both — the second INSERT fails with a duplicate key error.

## Context

The app is designed for single-tenant use (one org per instance). This constraint works fine today. But the schema already has `org_id` on users, implying multi-org was at least considered. If a consultant or agency wants to manage multiple Resend accounts from one instance, this blocks them.

## Priority

Low. Only matters if/when someone double-dips across orgs on the same instance.

## Change

Single migration:

```sql
ALTER TABLE users DROP CONSTRAINT users_email_key;
ALTER TABLE users ADD CONSTRAINT users_org_email_unique UNIQUE (org_id, email);
```

No code changes expected — queries already filter by org_id in practice.

## Risks

- If there's any login/auth path that looks up users by email alone (without org_id), it would need updating. Audit `WHERE email = $1` queries before migrating.
- Effectively harmless for single-org instances — the constraint is just narrower.
