# PRD-070: Bounce Notification Filtering

## Status
Done

## Priority
P2-Medium

## Category
Email Threading & Rendering

## Source
GAP-FINDER T5

---

## Problem Statement
Bounce notification emails from `mailer-daemon@` addresses are processed as normal inbound emails, creating user-visible threads in the inbox. These automated bounce notifications clutter the user's thread list and should either be filtered into a system folder, suppressed, or handled as status metadata on the original outbound email.

## Current Behavior
In `backend/internal/handler/webhooks.go`:
- Lines 112-123: The `HandleResend` switch processes `email.received` events by calling `handleEmailReceived`. There is no check on the sender address before queuing the job.
- Lines 128-179: `handleEmailReceived` creates an `email_jobs` record with `job_type='fetch'` regardless of the email's origin.

In `backend/internal/queue/process_fetch.go`:
- Lines 35-408: The `processFetch` function processes the email normally regardless of the sender. A bounce from `mailer-daemon@resend.dev` gets a thread created, labels applied, and an event published just like any legitimate email.
- Lines 184-189: The spam classifier (`service.ClassifySpam`) gives `mailer-daemon@` addresses a mild 0.05 spam score bump (see `backend/internal/service/spam.go:61-70`), which is far too low to classify the email as spam. The email passes classification and appears in the inbox.

In `backend/internal/service/spam.go`:
- Lines 59-70: `mailer-daemon@` is listed among suspicious patterns but only adds 0.05 to the spam score, treating it as "not necessarily spam."

## Expected Behavior
- Bounce notifications (emails from addresses matching `mailer-daemon@*` or with specific bounce-related headers like `X-Failed-Recipients`) should be detected during `processFetch`.
- Option A (recommended): Create the email record but skip thread creation. Instead, find the original outbound email by matching the bounce's `In-Reply-To` or `References` header, and update its status to `bounced`. This provides bounce information without creating noise.
- Option B: If a matching outbound email cannot be found, create the thread but auto-label it as `system` or `bounced` (a new label) instead of `inbox`, keeping it out of the default view.
- The webhook status handler already processes `email.bounced` events (line 119-120), but those are for outbound emails sent via Resend. Inbound bounce notifications from external MTAs are a different path.

## Files to Modify
- `backend/internal/queue/process_fetch.go:184-189` — After spam classification, add bounce detection logic that checks `emailData.From` for `mailer-daemon@` patterns and inspects headers for bounce indicators.
- `backend/internal/queue/process_fetch.go:288-293` — Modify the label assignment for new threads: if the email is a detected bounce, assign a `bounced` or `system` label instead of `inbox`.
- `backend/internal/service/spam.go:59-70` — Optionally export a helper like `IsBounceNotification(from string, headers map[string]string) bool` for reuse.

## Impact Analysis

### What This Affects
- Inbound email processing for bounce notifications.
- Users will see fewer noise threads in their inbox.
- If Option A is implemented, bounce status information flows to the original outbound email, improving visibility.

### What Affects This
- The Resend webhook for `email.bounced` (webhooks.go:119-120) handles the outbound bounce path; this PRD addresses the inbound path for bounces from external MTAs.
- Spam classifier scoring in `spam.go`.

### Related PRDs
- PRD-066: References Header Threading — Bounce emails often include References headers pointing to the original message; threading improvements help correlate bounces.
- PRD-071: Reply-To Header — Bounce emails typically have no Reply-To; users should not be able to "reply" to a bounce.
- PRD-067: Empty Subject Threading — Bounces sometimes have empty or generic subjects.

## Implementation Notes
- Common bounce sender patterns: `mailer-daemon@*`, `postmaster@*`, `MAILER-DAEMON@*` (case-insensitive).
- Common bounce headers: `X-Failed-Recipients`, `Auto-Submitted: auto-replied`, DSN (Delivery Status Notification) content types.
- Start with From-address detection as the simplest approach. Header-based detection can be added later.
- A new label value like `bounced` or `system` would need to be recognized by the frontend to either hide these threads or show them in a dedicated view.

## Acceptance Criteria
- [ ] Emails from `mailer-daemon@*` addresses do not appear in the inbox label.
- [ ] Bounce emails are either correlated with the original outbound email or placed in a non-inbox label.
- [ ] Legitimate emails from addresses that are not bounce senders continue to appear in the inbox.
- [ ] The spam classifier is not the sole mechanism for handling bounces (they receive dedicated handling).
- [ ] Unit test: inbound email from `mailer-daemon@resend.dev` does not create a thread with the `inbox` label.
