# PRD-108: Stripe Variables Missing from docker-compose.yml

## Status
Done

## Priority
P2-Medium

## Category
Configuration & Environment

## Source
GAP-FINDER C3

---

## Problem Statement
The `docker-compose.yml` backend service does not pass through Stripe-related environment variables (`STRIPE_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_PRICE_ID`, `SYSTEM_FROM_ADDRESS`). Commercial mode (billing, subscriptions, payment wall) cannot work in Docker deployments because these variables are never forwarded to the backend container, even if set in the host environment or `.env` file.

## Current Behavior
In `docker-compose.yml:36-44`, the backend service environment section includes:
- `DATABASE_URL`, `REDIS_URL`, `ENCRYPTION_KEY`, `SESSION_SECRET`, `APP_URL`, `PUBLIC_URL`, `API_PORT`, `RESEND_SYSTEM_API_KEY`
- Missing: `STRIPE_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_PRICE_ID`, `SYSTEM_FROM_ADDRESS`
- Also missing: `TRASH_COLLECTOR_ENABLED`, `EVENT_RETENTION_DAYS`

In `config.go:41-45`, these Stripe variables are loaded from `os.Getenv()`. Without the docker-compose mapping, they will always be empty inside the container, and the config Stripe cross-validation (lines 58-71) will never trigger because `STRIPE_KEY` will be empty (disabling commercial mode silently).

## Expected Behavior
All Stripe-related variables should be passed through in `docker-compose.yml` with empty defaults (so they are opt-in). When a user sets these in their `.env` file or Coolify environment, they should be forwarded to the container.

## Files to Modify
- `docker-compose.yml:36-44` -- Add Stripe environment variable mappings to the backend service:
  ```yaml
  STRIPE_KEY: ${STRIPE_KEY:-}
  STRIPE_WEBHOOK_SECRET: ${STRIPE_WEBHOOK_SECRET:-}
  STRIPE_PRICE_ID: ${STRIPE_PRICE_ID:-}
  SYSTEM_FROM_ADDRESS: ${SYSTEM_FROM_ADDRESS:-}
  TRASH_COLLECTOR_ENABLED: ${TRASH_COLLECTOR_ENABLED:-false}
  EVENT_RETENTION_DAYS: ${EVENT_RETENTION_DAYS:-90}
  ```

## Impact Analysis

### What This Affects
- Docker-based deployments with commercial mode (billing/subscriptions)
- Coolify deployments that inject environment variables
- Self-hosted operators who want to use billing features

### What Affects This
- The `.env` file -- operators need to set these values for commercial mode
- Coolify's environment variable injection -- may bypass docker-compose entirely (injected directly)
- Config validation (PRD-107) -- Stripe cross-validation depends on `STRIPE_KEY` being present

## Implementation Notes
- Use `${VAR:-}` syntax for empty defaults (self-hosted mode = no billing)
- Use `${VAR:-default}` for variables with sensible defaults
- Group additions with a comment:
  ```yaml
  # ── Commercial mode (leave empty for self-hosted) ──
  STRIPE_KEY: ${STRIPE_KEY:-}
  STRIPE_WEBHOOK_SECRET: ${STRIPE_WEBHOOK_SECRET:-}
  STRIPE_PRICE_ID: ${STRIPE_PRICE_ID:-}
  SYSTEM_FROM_ADDRESS: ${SYSTEM_FROM_ADDRESS:-}

  # ── Workers ──
  TRASH_COLLECTOR_ENABLED: ${TRASH_COLLECTOR_ENABLED:-false}
  EVENT_RETENTION_DAYS: ${EVENT_RETENTION_DAYS:-90}
  ```
- Also consider adding `BACKEND_URL` for frontend if it's needed for server-side rendering
- Note: Coolify may inject vars directly without docker-compose mapping. Document this in deployment docs.

## User Notes
This is a deployment configuration fix. Without it, commercial mode silently fails in Docker deployments.
