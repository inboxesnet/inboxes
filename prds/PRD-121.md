# PRD-121: No Cascade Deletes on Foreign Keys

## Status
Done

## Priority
P2-Medium

## Category
Data Model Concerns

## Source
PRE-LAUNCH-AUDIT #28

---

## Problem Statement
Most foreign keys in the schema do not have `ON DELETE CASCADE` or `ON DELETE SET NULL`. When an org, user, or domain is deleted (whether soft-delete via the application or hard-delete via manual SQL), child rows in threads, emails, aliases, discovered_addresses, and other tables are orphaned. The application's soft-delete handlers do not clean up all related data, leaving inconsistent state.

## Current Behavior
In `backend/internal/db/migrations/001_initial.sql`:
- Line 29: `users.org_id` references `orgs(id)` — no cascade. Deleting an org leaves orphaned users.
- Line 51: `domains.org_id` references `orgs(id)` — no cascade. Deleting an org leaves orphaned domains.
- Line 69-71: `threads` references `orgs(id)`, `users(id)`, `domains(id)` — no cascade on any FK.
- Line 92: `emails.thread_id` references `threads(id) ON DELETE CASCADE` — this is the only cascade in the schema.
- Line 93-95: `emails` references `users(id)`, `orgs(id)`, `domains(id)` — no cascade.
- Lines 128-129: `aliases` references `orgs(id)`, `domains(id)` — no cascade.
- Line 141: `alias_users.alias_id` references `aliases(id) ON DELETE CASCADE` — cascade exists.
- Lines 151-155: `discovered_addresses` references `domains(id)`, `users(id)`, `aliases(id)` — no cascade.

The application's domain delete handler (`backend/internal/handler/domains.go:434-453`) only sets `status='deleted'` but does not clean up aliases, threads, or emails. The org delete handler (`backend/internal/handler/orgs.go:119-185`) cancels the Stripe subscription and soft-deletes the org but does not cascade to child tables.

## Expected Behavior
Two approaches, both valid:
1. **Add CASCADE deletes** to all FK relationships — clean but conflicts with the soft-delete pattern (CASCADE would hard-delete child rows even when the parent is only soft-deleted).
2. **Ensure soft-delete handlers clean up properly** — the application-level approach, where each delete handler explicitly marks or cleans up child records.

Option 2 is more appropriate for this codebase since soft-delete is the primary deletion pattern. The handlers should be updated to properly cascade soft-deletes through the entity hierarchy.

## Files to Modify
- `backend/internal/handler/domains.go:434-453` — Add cleanup of aliases, threads, emails, and discovered_addresses when a domain is soft-deleted.
- `backend/internal/handler/orgs.go:119-185` — Add cleanup of users, domains, and all descendant entities when an org is deleted.
- `backend/internal/handler/users.go` — Consider cleanup of user-specific data (alias_users, drafts) when a user is disabled.
- New migration file (optional) — Add `ON DELETE SET NULL` to nullable FKs and `ON DELETE CASCADE` to non-nullable FKs for defense-in-depth against manual SQL deletes.

## Impact Analysis

### What This Affects
- Domain deletion flow — currently orphans all domain data.
- Org deletion flow — currently orphans all org data.
- Database integrity — orphaned rows waste space and can cause unexpected query results.
- Orphaned personal data persists after "deletion."

### What Affects This
- The soft-delete pattern used throughout the codebase.
- Background workers that query across tables (e.g., trash collector, domain heartbeat) — must handle orphaned state gracefully.
- The `emails.thread_id ON DELETE CASCADE` already exists and works correctly.

### Related PRDs
- PRD-072: Domain Cleanup — domain deletion should cascade to aliases, threads, emails.
- PRD-078: Webhook Cleanup — deleted orgs' webhooks should be unregistered.
- PRD-086: Emails After Deletion — queued emails for deleted orgs still send.
- PRD-120: Email Uniqueness — related schema concern about user deletion and email slot release.

## Implementation Notes
CASCADE would conflict with the soft-delete pattern. The real fix is ensuring soft-delete handlers clean up properly (see PRD-072 for domain cleanup). For defense-in-depth, consider adding `ON DELETE SET NULL` to nullable FK columns so that manual SQL deletes don't fail with FK violations but instead leave a traceable null reference.

A comprehensive approach:
1. Domain soft-delete: set `status='deleted'`, then `UPDATE aliases SET deleted_at = now() WHERE domain_id = $1`, cascade to threads and emails.
2. Org soft-delete: set `deleted_at = now()`, then cascade to all domains (which cascades further), users, labels, webhooks.
3. Wrap each cascade in a transaction.
