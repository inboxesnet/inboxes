# PRD-048: Label Rename

## Status
Draft

## Priority
P1-High

## Category
Race Conditions

## Source
GAP-FINDER R5

---

## Problem Statement
The label rename operation in `labels.go:84-126` updates `org_labels` and `thread_labels` in two separate SQL statements without a transaction. If a concurrent operation creates a thread label with the old name between these two statements, that label will be orphaned -- it will reference a name that no longer exists in `org_labels`. Additionally, the two UPDATE statements can partially fail, leaving `org_labels` renamed but `thread_labels` still referencing the old name.

## Current Behavior
- In `backend/internal/handler/labels.go:84-126`, the `Rename` handler:
  1. Reads the old name at line 106-108: `SELECT name FROM org_labels WHERE id = $1 AND org_id = $2`
  2. Updates `org_labels` at line 116-118: `UPDATE org_labels SET name = $1 WHERE id = $2 AND org_id = $3`
  3. Updates `thread_labels` at line 121-123: `UPDATE thread_labels SET label = $1 WHERE org_id = $2 AND label = $3`
- These are three separate queries on the pool (`h.DB.Exec`), not on a transaction.
- Race condition: Between step 2 and step 3, a concurrent request could:
  - Create a new thread label with the old name (e.g., via `addLabel()` or webhook processing), which would not be renamed by step 3.
  - Create a new org_label with the old name, since the old name is now "free" after step 2.
- Partial failure: If step 2 succeeds but step 3 fails (e.g., database timeout), `org_labels` has the new name but `thread_labels` still reference the old name. Threads would lose their label association.

## Expected Behavior
- The label rename should be atomic: either both `org_labels` and `thread_labels` are updated, or neither is.
- No concurrent operations should be able to insert labels with the old name between the rename steps.
- The rename should use a database transaction with appropriate locking.

## Files to Modify
- `backend/internal/handler/labels.go:84-126` -- Wrap the rename operation in a transaction, and use `SELECT ... FOR UPDATE` on the org_label row to prevent concurrent modifications

## Impact Analysis

### What This Affects
- Custom label rename in the Settings > Labels panel
- All thread_labels referencing the renamed label
- Thread list filtering by label name (if thread_labels reference the old name, threads will not appear under the new label)

### What Affects This
- The `addLabel()` function in `labels.go:156-161` which can concurrently insert labels with any name
- Webhook processing in `process_fetch.go` which may add labels to threads
- The `bulkAddLabel()` function in `labels.go:205-212`
- The label Delete handler at `labels.go:128-147` has a similar non-atomic pattern (DELETE from org_labels then DELETE from thread_labels)

### Related PRDs
- PRD-044 (Label Transactions) -- the broader issue of non-transactional label operations
- PRD-045 (Duplicate Threads) -- another race condition requiring transactional protection

## Implementation Notes
- Wrap the rename in a transaction with row-level locking:
  ```go
  func (h *LabelHandler) Rename(w http.ResponseWriter, r *http.Request) {
      claims := middleware.GetCurrentUser(r.Context())
      labelID := chi.URLParam(r, "id")

      var req struct { Name string `json:"name"` }
      if err := readJSON(r, &req); err != nil || req.Name == "" { ... }
      // ... validation ...

      ctx := r.Context()
      tx, err := h.DB.Begin(ctx)
      if err != nil {
          writeError(w, http.StatusInternalServerError, "failed to start transaction")
          return
      }
      defer tx.Rollback(ctx)

      // Lock the org_label row to prevent concurrent renames
      var oldName string
      err = tx.QueryRow(ctx,
          `SELECT name FROM org_labels WHERE id = $1 AND org_id = $2 FOR UPDATE`,
          labelID, claims.OrgID).Scan(&oldName)
      if err != nil {
          writeError(w, http.StatusNotFound, "label not found")
          return
      }

      // Update org_labels
      tx.Exec(ctx,
          `UPDATE org_labels SET name = $1 WHERE id = $2 AND org_id = $3`,
          req.Name, labelID, claims.OrgID)

      // Rename in thread_labels
      tx.Exec(ctx,
          `UPDATE thread_labels SET label = $1 WHERE org_id = $2 AND label = $3`,
          req.Name, claims.OrgID, oldName)

      if err := tx.Commit(ctx); err != nil {
          writeError(w, http.StatusInternalServerError, "failed to rename label")
          return
      }

      writeJSON(w, http.StatusOK, map[string]string{"id": labelID, "name": req.Name})
  }
  ```
- The `FOR UPDATE` lock on `org_labels` prevents two concurrent renames of the same label.
- The transaction ensures both tables are updated atomically.
- Apply the same transactional pattern to the `Delete` handler at lines 128-147 which has an identical non-atomic pattern.
- Consider adding error checking on the `tx.Exec` calls inside the transaction (currently the `h.DB.Exec` return values are ignored).

## User Notes
- Label rename will work the same way from the user's perspective.
- The fix prevents a rare but possible data inconsistency where threads lose their label during concurrent operations.

## Acceptance Criteria
- [ ] Label rename wraps all SQL operations in a single transaction
- [ ] The `org_labels` row is locked with `FOR UPDATE` during the rename to prevent concurrent modifications
- [ ] If any step fails, the entire rename is rolled back
- [ ] Concurrent label creation with the old name during a rename does not create orphaned labels
- [ ] Concurrent rename of the same label by two admins is serialized (one succeeds, one waits or gets conflict)
- [ ] The label Delete handler is also wrapped in a transaction (same pattern)
- [ ] Thread list filtering by the renamed label works correctly after rename
- [ ] No regression in label rename for normal (non-concurrent) operations
