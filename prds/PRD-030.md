# PRD-030: Auto-Save Invisible

## Status
Done

## Priority
P1-High

## Category
Drafts & Compose Issues

## Source
GAP-FINDER E6

---

## Problem Statement
When auto-save fails during compose, the save status indicator silently clears instead of showing an error. The user continues typing, believing their draft is being saved, when in reality the last successful save may have been minutes ago. If they close the compose window or their session expires, all work since the last successful save is lost.

## Current Behavior
In `floating-compose-window.tsx`, the `saveDraft` callback (lines 231-233) has a catch block that:
1. Checks for AbortError (line 232) -- correctly ignoring aborted requests.
2. For all other errors, sets `setSaveStatus("")` (line 233) -- clearing the status to blank.

The blank status shows nothing in the UI. The desktop title bar (lines 551-556) only renders text for "saving" and "saved" states. The mobile title bar (line 415) similarly only shows "Saving..." or "Saved". There is no visual state for "save failed" -- the indicator simply disappears, leaving the user with no feedback.

## Expected Behavior
1. When auto-save fails, display a visible error indicator: a red "Save failed" badge in the compose title bar.
2. Show a toast notification on the first failure: "Draft auto-save failed. Your changes may not be saved."
3. Continue retrying on the next debounce cycle (which already happens because `dirtyRef` remains true).
4. If 3+ consecutive saves fail, show a persistent warning banner in the compose window.

## Files to Modify
- `frontend/components/floating-compose-window.tsx:55` -- Extend `saveStatus` type to include `"error"`: `useState<"" | "saving" | "saved" | "error">("")`
- `frontend/components/floating-compose-window.tsx:231-233` -- In the catch block, set `setSaveStatus("error")` and call `toast.error("Draft save failed")` on the first failure.
- `frontend/components/floating-compose-window.tsx:551-556` -- Add rendering for the `"error"` state in the desktop title bar: `{saveStatus === "error" && <span className="text-xs text-destructive mr-1">Save failed</span>}`
- `frontend/components/floating-compose-window.tsx:415` -- Add rendering for the `"error"` state in the mobile title bar.
- `frontend/components/floating-compose-window.tsx:238-248` -- Ensure `scheduleSave` still schedules a retry when `saveStatus` is `"error"` (it does, since `dirtyRef` stays true, but verify this is not accidentally cleared).

## Impact Analysis

### What This Affects
- All compose users -- when the backend is temporarily unavailable, they have no way to know their work is not being saved.
- The trust relationship between the user and the app's auto-save feature.
- Data loss scenarios are silently created rather than surfaced to the user.

### What Affects This
- Backend availability -- any downtime causes save failures.
- Network connectivity -- intermittent connections trigger this path.
- The `saveAbortRef` abort mechanism -- aborted saves should NOT show an error (already handled correctly on line 232).
- Session expiry (PRD-035) -- expired sessions cause 401 errors on save, which currently also get silently swallowed.

### Related PRDs
- PRD-027 (Compose Draft Loss) -- the close-on-failure bug is worsened by the user not knowing saves are failing
- PRD-028 (Duplicate Drafts) -- the save mechanism is shared
- PRD-035 (Session Expiry) -- session expiry causes all saves to fail silently

## Implementation Notes
- Add a `saveFailCountRef` to track consecutive failures. Show a toast on the first failure, and escalate to a persistent banner after 3 failures.
- The error state should auto-clear when a subsequent save succeeds (which the existing `setSaveStatus("saved")` on line 229 already handles).
- Consider adding a manual "Retry save" button when in error state, which calls `saveDraft()` immediately instead of waiting for the next debounce cycle.
- The save status indicator should use a consistent color scheme:
  - "saving" = muted/gray text
  - "saved" = muted/gray text (or brief green)
  - "error" = red/destructive text
- Be careful not to spam toast notifications on every retry failure -- only show the first toast per error sequence.

## User Notes
- This is a quality-of-life improvement that prevents silent data loss. Users who compose long emails on unreliable connections will benefit most.
- The fix aligns with how Google Docs shows "Trying to connect..." when auto-save fails.

## Acceptance Criteria
- [ ] When auto-save fails, the compose title bar shows "Save failed" in red text
- [ ] A toast notification appears on the first save failure
- [ ] The error indicator clears automatically when the next save succeeds
- [ ] Aborted saves (from debounce cancellation) do NOT trigger the error state
- [ ] The `dirtyRef` remains true after a save failure, ensuring the next debounce cycle retries
- [ ] After 3+ consecutive failures, a persistent warning banner appears in the compose body area
- [ ] Both mobile and desktop compose layouts show the error state
