# PRD-015: Admin Rate Limits

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
PRE-LAUNCH-AUDIT #7

---

## Problem Statement
The admin-only and cron endpoints (`/api/cron/purge-trash`, `/api/cron/cleanup-webhooks`, `/api/admin/jobs`) have no rate limiting applied. A compromised admin token could trigger unlimited trash purge cycles, mass webhook cleanup operations, or repeated job listing queries, causing excessive database load and potential data loss through rapid purge operations.

## Current Behavior
In `backend/internal/router/router.go:173-179`, the admin group is protected only by `middleware.RequireAdmin` but has no rate-limiting middleware:

```go
r.Group(func(r chi.Router) {
    r.Use(middleware.RequireAdmin)
    r.Post("/api/cron/purge-trash", cron.PurgeTrash)
    r.Post("/api/cron/cleanup-webhooks", cron.CleanupWebhooks)
    r.Get("/api/admin/jobs", emails.AdminJobs)
})
```

By contrast, all public auth endpoints in `router.go:113-121` are wrapped with `middleware.RateLimitByIP()`. The rate limiter implementation exists in `backend/internal/middleware/ratelimit.go:20-41` and is ready to use — it just is not applied to this route group.

The `PurgeTrash` handler (`backend/internal/handler/cron.go:18-43`) soft-deletes all threads past trash expiry and then runs a label cleanup query. The `CleanupWebhooks` handler (`cron.go:47-110`) iterates all orgs and makes external API calls to Resend to list and delete webhooks. Both are expensive operations that should not be callable at high frequency.

## Expected Behavior
Admin and cron endpoints should have a rate limit of approximately 5 requests per minute per IP. This prevents automated abuse from a compromised admin token while still allowing legitimate manual operations. When the rate limit is exceeded, the endpoint should return HTTP 429 with a `Retry-After` header, consistent with existing rate-limited endpoints.

## Files to Modify
- `backend/internal/router/router.go:173-179` — Add `middleware.RateLimitByIP(rdb, 5, 60)` to the admin route group

## Impact Analysis

### What This Affects
- Admin operations will be throttled if called too frequently
- Cron job triggers (if automated via external scheduler) need to respect the rate limit window
- No impact on normal user flows — only admin-only endpoints are affected

### What Affects This
- Redis availability — rate limiting depends on Redis (`backend/internal/middleware/ratelimit.go:86-90` fails open when Redis is down)
- PRD-019 (Token Blacklist Fail-Open) — if Redis is down, both rate limiting and token revocation fail open simultaneously

### Related PRDs
- PRD-019: Token Blacklist Fail-Open — Both features depend on Redis; a Redis outage disables rate limits and token revocation at the same time
- PRD-022: Change-Password Rate Limit — Same pattern of adding rate limiting to an unprotected endpoint
- PRD-023: Resend-Verification Rate Limit — Same pattern of adding rate limiting to an unprotected endpoint

## Implementation Notes
The rate limiter middleware `RateLimitByIP` already exists and is well-tested on public endpoints. The `rdb` (Redis client) parameter needs to be passed into the admin route group, which requires accessing it from the `New()` function's parameters — it is already available as the `rdb` parameter on line 28.

A suggested limit of 5 requests per 60 seconds balances protection against abuse with the ability to manually re-trigger operations during incident response.
