# PRD-022: Change-Password Rate Limit

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
GAP-FINDER S4

---

## Problem Statement
The change-password endpoint (`PATCH /api/users/me/password`) has no rate limiting. An attacker with a valid session (e.g., from a stolen cookie or XSS) can brute-force the current password by calling this endpoint at high speed. Since the endpoint accepts the current password for verification before allowing a change, unlimited attempts effectively give the attacker an oracle to discover the password.

## Current Behavior
In `backend/internal/router/router.go:148`, the change-password route is inside the authenticated group but has no rate-limiting middleware:

```go
r.Patch("/api/users/me/password", users.UpdatePassword)
```

In `backend/internal/handler/users.go:388-441`, the handler verifies the current password via bcrypt comparison before proceeding:

```go
func (h *UserHandler) UpdatePassword(w http.ResponseWriter, r *http.Request) {
    claims := middleware.GetCurrentUser(r.Context())
    ...
    var hash string
    err := h.DB.QueryRow(r.Context(),
        `SELECT password_hash FROM users WHERE id = $1`, claims.UserID).Scan(&hash)
    ...
    if bcrypt.CompareHashAndPassword([]byte(hash), []byte(req.CurrentPassword)) != nil {
        writeError(w, http.StatusUnauthorized, "current password is incorrect")
        return
    }
    ...
}
```

The response "current password is incorrect" confirms when a guess is wrong, making brute-force straightforward. There is no delay, no attempt counter, and no lockout after repeated failures.

By contrast, public auth endpoints like login (`router.go:115`) and forgot-password (`router.go:116`) have both IP-based and field-based rate limiting. The change-password endpoint is behind authentication, but being authenticated does not mean the attacker should get unlimited password guesses.

## Expected Behavior
1. Add rate limiting to the change-password endpoint: approximately 5 attempts per 15 minutes per user.
2. Rate limit by authenticated user ID (not just IP), since the attacker may be using the victim's session from the same IP.
3. After exceeding the rate limit, return HTTP 429 with a `Retry-After` header.
4. Log excessive failed attempts for monitoring.

## Files to Modify
- `backend/internal/router/router.go:148` — Add rate limiting middleware to the change-password route
- `backend/internal/handler/users.go:388-441` — Optionally add per-user rate limiting inside the handler using Redis (for user-ID-based limiting rather than IP-based)

## Impact Analysis

### What This Affects
- Password change flow — legitimate users who mistype their current password more than 5 times in 15 minutes will be rate-limited
- Users can still change their password normally (typical usage is 1-2 attempts)

### What Affects This
- Redis availability — rate limiting depends on Redis (fails open per `ratelimit.go:87-90`)
- Authentication middleware — the endpoint requires a valid session before rate limiting applies

## Implementation Notes
There are two approaches:

**Approach A: Router-level rate limiting (quick)**
Add `middleware.RateLimitByIP(rdb, 5, 15*60)` to the route. This provides IP-based rate limiting, which is sufficient for most scenarios:

```go
r.With(middleware.RateLimitByIP(rdb, 5, 15*60)).Patch("/api/users/me/password", users.UpdatePassword)
```

**Approach B: Handler-level rate limiting (thorough)**
Add a per-user-ID rate limit inside the handler using Redis. This protects against distributed attacks using a stolen session:

```go
key := fmt.Sprintf("rl:pw-change:%s", claims.UserID)
// INCR + check count + EXPIRE
```

Approach A is the minimum viable fix. Approach B is recommended for better security but requires more code.

Note: The `rdb` parameter is not currently available in the authenticated route group scope. It would need to be passed through, or the rate limiting should be done at the handler level where `h.RDB` is available on the `UserHandler` struct (`users.go:23`).
