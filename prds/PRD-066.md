# PRD-066: References Header Threading

## Status
Done

## Priority
P1-High

## Category
Email Threading & Rendering

## Source
GAP-FINDER T1

---

## Problem Statement
The `References` header from inbound emails is parsed, stored in the `references_header` JSONB column in the `emails` table, but never used for thread matching. RFC 5322 defines `References` as the most reliable way to thread emails, especially when `In-Reply-To` is missing or when mail clients strip it. Ignoring this header causes thread fragmentation for legitimate conversation chains.

## Current Behavior
In `backend/internal/queue/process_fetch.go`:
- Lines 218-226: The `References` header is extracted from email headers, split by whitespace, and marshalled to JSON as `refsJSON`.
- Lines 298-310: `refsJSON` is stored in the `references_header` column during the email INSERT.
- Lines 248-276: Thread matching uses a 3-step approach:
  1. Step 1 (line 253-261): Match by `In-Reply-To` header against `emails.message_id`.
  2. Step 2 (line 264-276): Match by cleaned subject + counterparty within a 90-day window, scoped to `domain_id`.
  3. Step 3 (line 279-294): Create a new thread if no match found.
- The `References` header (which may contain multiple Message-IDs from the full conversation chain) is never consulted in any of these steps.

## Expected Behavior
- Add a Step 1.5 between the current In-Reply-To check and subject matching: query `emails.message_id` against ALL Message-IDs in the parsed `References` header array.
- If any `References` Message-ID matches an existing email, thread to that email's `thread_id`.
- This catches cases where `In-Reply-To` is stripped but `References` is preserved (common with mailing lists and some corporate mail gateways).
- The match should be scoped to `org_id` (not `domain_id`) to allow cross-domain threading (see PRD-068).

## Files to Modify
- `backend/internal/queue/process_fetch.go:261-264` — Insert a new matching step between Step 1 and Step 2 that queries `SELECT thread_id FROM emails WHERE message_id = ANY($1::text[]) AND org_id = $2 ORDER BY created_at DESC LIMIT 1` using the parsed references array.

## Impact Analysis

### What This Affects
- Thread grouping for all inbound emails.
- Frontend thread view will show more correctly grouped conversations.
- Thread list counts and snippets may change if previously-split conversations merge.

### What Affects This
- Email headers must be correctly normalized to lowercase (already done at lines 150-157).
- The `References` header must be properly parsed (already done at lines 218-226).

### Related PRDs
- PRD-067: Empty Subject Threading — Both address thread fragmentation.
- PRD-068: Cross-Domain Threading — References-based matching should be org-scoped, not domain-scoped.
- PRD-071: Reply-To Header — Another email header that needs proper handling.

## Implementation Notes
- Parse the `refsJSON` back into a `[]string` for the SQL `ANY()` query, or use the already-parsed `strings.Fields(refs)` result directly.
- Index consideration: `emails.message_id` already has an index (`idx_emails_message_id`), so the `ANY()` query against it should perform well.
- Prefer the most recent match (`ORDER BY created_at DESC LIMIT 1`) in case References span multiple threads.
