# PRD-063: Composite Index

## Status
Done

## Priority
P2-Medium

## Category
Database Performance

## Source
GAP-FINDER D6

---

## Problem Statement
The most common query pattern in the application — listing threads by org, filtering out deleted, ordered by date — has no composite index that covers all three conditions. Every thread list request must combine multiple indexes or fall back to a sequential scan. This is the single highest-impact missing index in the database.

## Current Behavior
Every thread list query in `backend/internal/handler/threads.go` follows this pattern:
```sql
SELECT ... FROM threads t
JOIN thread_labels tl ON tl.thread_id = t.id
WHERE tl.org_id = $1 AND tl.label = $2 AND t.deleted_at IS NULL
ORDER BY t.last_message_at DESC
LIMIT $3 OFFSET $4
```

The query planner must satisfy:
1. `tl.org_id = $1` — filter by organization
2. `tl.label = $2` — filter by folder label
3. `t.deleted_at IS NULL` — exclude soft-deleted threads
4. `ORDER BY t.last_message_at DESC` — sort by most recent message

Current relevant indexes:
- `idx_thread_labels_org_label` on `thread_labels(org_id, label, thread_id)` — covers the label join
- `idx_threads_last_message` on `threads(last_message_at DESC)` — covers the sort but not the filter
- `idx_threads_user_domain_active` on `threads(user_id, domain_id)` — does not help

There is no index on `threads` that combines `org_id + deleted_at IS NULL + last_message_at DESC`. The planner likely does a hash join between the thread_labels index and threads, then sorts the result, which becomes expensive as thread count grows.

## Expected Behavior
Add a composite partial index that covers the most common query pattern:
```sql
CREATE INDEX idx_threads_org_active_date
ON threads(org_id, last_message_at DESC)
WHERE deleted_at IS NULL;
```

This index allows PostgreSQL to:
1. Filter by `org_id` using the leading column
2. Skip deleted threads via the partial index condition
3. Return results pre-sorted by `last_message_at DESC`
4. Combine efficiently with the thread_labels join via thread ID

## Impact Analysis

### What This Affects
- All thread list views (inbox, sent, archive, trash, spam, starred, custom labels)
- Thread count queries (same filter pattern without ORDER BY)
- Unread count queries (similar filter pattern)
- Overall database CPU usage (sort operations are expensive)

### What Affects This
- Total number of threads per org
- Ratio of deleted to active threads
- Index maintenance overhead on thread INSERT/UPDATE/soft-DELETE

### Related PRDs
- PRD-058 (deleted_at Index) — overlaps with the partial index condition; should be consolidated
- PRD-059 (Labels Subquery) — improving the labels fetch after the main query is faster
- PRD-057 (N+1 Trash) — trash list uses the same query pattern

## Implementation Notes
- If PRD-058 is implemented first, its `idx_threads_not_deleted` index may already partially cover this. Coordinate the two migrations to avoid redundant indexes.
- The recommended consolidated index:
  ```sql
  -- +goose Up
  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_threads_org_active_date
    ON threads(org_id, last_message_at DESC)
    WHERE deleted_at IS NULL;

  -- The count query variant (no ORDER BY needed, just filter)
  -- The above index also serves count queries since org_id is the leading column.

  -- If PRD-058's separate deleted_at index was already created, consider dropping it
  -- since this index subsumes its use case for list queries.

  -- +goose Down
  DROP INDEX IF EXISTS idx_threads_org_active_date;
  ```

- For the archive query (which does NOT join thread_labels but uses `NOT EXISTS` subqueries), this index helps filter by org_id + deleted_at efficiently, and the sort is pre-computed.
- For label-based queries (inbox, sent, etc.), the planner can do a nested loop: scan thread_labels for matching (org_id, label) pairs, then probe the threads index for each thread_id to verify deleted_at IS NULL and get last_message_at for sorting.
- Include `INCLUDE (domain_id)` if the query also commonly filters by domain_id — but this can be added later if needed.
- Use `CONCURRENTLY` for production safety.

## User Notes
All thread list pages will load faster, with the improvement most noticeable for organizations with large numbers of threads (10,000+). The sort-by-date ordering will be served directly from the index instead of requiring a sort operation.
