# PRD-122: BYTEA Attachments Won't Scale

## Status
Done

## Priority
P2-Medium

## Category
Data Model Concerns

## Source
PRE-LAUNCH-AUDIT #29

---

## Problem Statement
Attachment binary data is stored inline as PostgreSQL `BYTEA` columns. Every query that touches the `attachments` table loads the full binary content into memory, including listing operations that only need metadata. Database backups grow proportionally to attachment volume, and large attachments (up to 10MB each) can cause out-of-memory conditions during query processing.

## Current Behavior
- `backend/internal/db/migrations/028_attachments.sql:1-11` — The `attachments` table stores `data BYTEA NOT NULL` inline with metadata columns.
- `backend/internal/handler/attachments.go:34` — Upload reads the entire file into memory with `io.ReadAll(file)`.
- `backend/internal/handler/attachments.go:65-68` — Insert stores the full binary: `INSERT INTO attachments (..., data, ...) VALUES (..., $7, ...)`.
- `backend/internal/handler/attachments.go:90-93` — Download loads the full binary: `SELECT filename, content_type, size, data FROM attachments WHERE id = $1`.
- The upload handler limits files to 10MB (`r.ParseMultipartForm(10 << 20)` on line 22), but there is no database-level CHECK constraint enforcing this limit.
- For draft attachments, the attachment IDs are stored in `drafts.attachment_ids` (added by migration 028, line 15), meaning the binary data is not duplicated in the drafts table.

## Expected Behavior
Short-term (pre-launch):
1. Add a `CHECK (size <= 10485760)` constraint to the `attachments` table to enforce the 10MB limit at the database level.
2. Ensure `SELECT` queries that only need metadata (for listing, validation) do not include the `data` column.

Long-term (post-launch):
1. Move attachment storage to S3-compatible object storage (MinIO for self-hosted, AWS S3 for cloud).
2. Store only a reference (bucket + key) in the database.
3. Use presigned URLs for direct browser uploads/downloads, bypassing the backend for large files.

## Files to Modify
- New migration file (e.g., `032_attachment_size_check.sql`) — Add `ALTER TABLE attachments ADD CONSTRAINT attachments_max_size CHECK (size <= 10485760);`
- `backend/internal/handler/attachments.go:90-93` — The download query is the only one that needs `data`; this is acceptable. Ensure no listing queries accidentally select `data`.
- `backend/internal/handler/drafts.go` — Verify that draft list queries do not join to the `attachments` table and load binary data.

## Impact Analysis

### What This Affects
- Database backup size — each attachment adds up to 10MB to the backup.
- Query performance — any accidental `SELECT *` from the attachments table loads binaries.
- PostgreSQL memory usage — `BYTEA` values are detoasted into memory during query execution.
- Self-hosted operators with limited disk space.

### What Affects This
- Email volume and attachment frequency.
- PostgreSQL `work_mem` setting (affects how large values are handled in memory).
- Backup strategy (pg_dump includes BYTEA data, making backups slow).

### Related PRDs
- PRD-118: Attachment Download — the download endpoint loads the full binary from the database, which compounds this scalability issue.
- PRD-126: Draft Attachments — draft attachment restoration will need to fetch attachment metadata (not binary data).
- PRD-113: Resource Limits — container memory limits interact with large BYTEA query processing.

## Implementation Notes
The long-term fix (S3 migration) is a significant architectural change. For launch:
1. Add the CHECK constraint to prevent oversized attachments at the DB level.
2. Document the limitation: "Attachments are stored in PostgreSQL. For high-volume deployments, consider migrating to object storage."
3. Monitor database backup size and query latency on the attachments table.

For the S3 migration (post-launch), the schema change would be:
```sql
ALTER TABLE attachments DROP COLUMN data;
ALTER TABLE attachments ADD COLUMN storage_key TEXT NOT NULL;
ALTER TABLE attachments ADD COLUMN storage_bucket TEXT NOT NULL DEFAULT 'attachments';
```
