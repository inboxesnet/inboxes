## Problem Statement
The Stripe webhook handler now handles 5 event types (`checkout.session.completed`, `customer.subscription.deleted`, `customer.subscription.updated`, `invoice.payment_succeeded`, `invoice.payment_failed`), but several additional event types are needed for a production-grade billing integration. Missing handlers for events like `customer.subscription.paused`, `customer.subscription.resumed`, `invoice.payment_action_required`, and dispute-related events leave gaps in plan state management and user communication.

## Current Behavior
The `HandleStripeWebhook` method in `backend/internal/handler/billing.go:229-359` has a `switch` on `event.Type` covering 5 cases. Unhandled event types fall through to the bottom where `w.WriteHeader(http.StatusOK)` is returned at line 361, silently acknowledging events the system doesn't understand. Notably:
- `invoice.payment_failed` (line 336) sets plan to `cancelled` with a 3-day grace -- but the semantics should be `past_due` (see PRD-085).
- `customer.subscription.updated` (line 278) maps `PastDue` and `Unpaid` both to `cancelled`, which is incorrect.
- No handler for `customer.subscription.paused`/`resumed` (Stripe Billing feature).
- No handler for `charge.dispute.created` to alert admins of chargebacks.
- No handler for `invoice.upcoming` to notify users before billing.

## Expected Behavior
The webhook handler should:
1. Handle all lifecycle events for the subscription billing model.
2. Map Stripe statuses to internal plan states correctly: `active`/`trialing` -> `pro`, `past_due` -> `past_due` (new status), `canceled`/`unpaid` -> `cancelled`.
3. Log unhandled event types at `slog.Warn` level instead of silently ignoring them.
4. Emit WebSocket events for plan state changes so the frontend updates in real-time.

Full event list to handle:
- `checkout.session.completed` (exists)
- `customer.subscription.created` (new -- initial subscription creation via API)
- `customer.subscription.updated` (exists, needs fix)
- `customer.subscription.deleted` (exists)
- `customer.subscription.paused` (new)
- `customer.subscription.resumed` (new)
- `invoice.payment_succeeded` (exists)
- `invoice.payment_failed` (exists, needs fix)
- `invoice.upcoming` (new -- notification only)

## Files to Modify

## Implementation Notes
- Add a `default` case in the switch that logs `slog.Warn("stripe: unhandled event type", "type", event.Type)`.
- For `invoice.upcoming`, send a notification (WebSocket event or email) but don't change plan state.
- Consider adding a `plan_status` column to `orgs` that mirrors Stripe subscription status more closely, separate from the billing `plan` column.
- Update the Stripe dashboard webhook settings to subscribe to all handled event types.

## User Notes
- Users will see more accurate plan status in the billing section of settings.
- Users with payment failures will see "Past Due" instead of "Cancelled", reducing confusion.
