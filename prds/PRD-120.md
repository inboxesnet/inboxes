# PRD-120: Email Uniqueness is Global — Blocks Multi-Org

## Status
Done

## Priority
P2-Medium

## Category
Data Model Concerns

## Source
PRE-LAUNCH-AUDIT #27

---

## Problem Statement
The `users` table has a globally unique constraint on the `email` column (`email TEXT NOT NULL UNIQUE` in migration 001). This means the same email address cannot exist in two different organizations. While this is fine for the current single-org model, it creates a hard blocker for any future multi-org support and also prevents re-inviting a user whose account was soft-deleted (their email remains locked by the uniqueness constraint).

## Current Behavior
- `backend/internal/db/migrations/001_initial.sql:30` — The users table is defined with `email TEXT NOT NULL UNIQUE`.
- `backend/internal/db/migrations/001_initial.sql:44` — An additional index exists: `CREATE INDEX idx_users_email ON users(email)`.
- When a user is disabled (soft-deleted), their row remains with the email still occupying the unique slot. The email address cannot be reused for a new invitation in the same or different org.
- All auth queries (`backend/internal/handler/auth.go`) use `WHERE email = $1` without org scoping, which works correctly under the global uniqueness assumption.
- The signup flow (`auth.go:335`) checks for existing users by email globally, preventing the same person from joining a different org.

## Expected Behavior
For the current launch (single-org), this is acceptable. The fix should be planned for post-launch:
1. Change the uniqueness constraint from `UNIQUE(email)` to `UNIQUE(org_id, email)` — same email can exist in different orgs.
2. Update all auth queries to include `org_id` in the WHERE clause.
3. Add an org-selection step to the login flow (or use subdomain-based routing to determine the org).
4. Handle the re-invite case by properly deleting disabled users or releasing their email slot.

## Files to Modify
- `backend/internal/db/migrations/001_initial.sql:30` — The source constraint (for reference; changes go in a new migration).
- New migration file (e.g., `032_email_unique_per_org.sql`) — `ALTER TABLE users DROP CONSTRAINT users_email_key; ALTER TABLE users ADD CONSTRAINT users_email_org_unique UNIQUE(org_id, email);`
- `backend/internal/handler/auth.go` — All queries using `WHERE email = $1` need `AND org_id = $2` or an org-resolution step.
- `backend/internal/handler/users.go` — Invite and user management queries.
- `backend/internal/handler/setup.go` — Initial admin creation.

## Impact Analysis

### What This Affects
- Login flow — needs org resolution before email lookup.
- Signup/invite flow — must determine target org first.
- Password reset — must identify which org's user to reset.
- The UNIQUE constraint change is a schema migration that must be coordinated with application code changes.

### What Affects This
- Multi-org feature decision — this change only makes sense if multi-org is on the roadmap.
- Releasing email slots is an alternative for the re-invite case.

### Related PRDs
- PRD-121: Cascade Deletes — related schema concern about FK relationships during org deletion.
- PRD-007: Case-Sensitive Emails — email normalization must happen before the uniqueness constraint can work correctly.
- PRD-074: Domain Uniqueness — similar global-vs-per-org uniqueness question for domains.

## Implementation Notes
This is schema debt, not a bug. For launch, document this as a known limitation. Post-launch, the migration requires:
1. Application code deployed first with org-scoped queries (backward-compatible with global uniqueness).
2. Migration to change the constraint.
3. Login flow updated to handle org selection.

Consider using email + org subdomain as the compound key for authentication rather than requiring users to pick an org from a dropdown.
