# PRD-040: Setup Email Validation

## Status
Done

## Priority
P2-Medium

## Category
Form Validation Mismatches

## Source
GAP-FINDER V3

---

## Problem Statement
The initial setup form (`/setup`) does not validate the admin email address before submitting to the backend. Unlike the signup and login forms which use `validateEmail()` server-side, the setup endpoint at `backend/internal/handler/setup.go` only checks that email is non-empty (line 131: `req.Email == ""`). This means the first admin account can be created with a malformed email address, which would break password reset flows and any system emails.

## Current Behavior
- In `frontend/app/(auth)/setup/page.tsx:308-314`, the email input has `type="email"` and `required` but no explicit format validation beyond browser defaults (which vary by browser and are lenient).
- In `backend/internal/handler/setup.go:131`, validation only checks `req.Email == ""`. There is no call to `validateEmail()` like in signup (`auth.go:86`) and login (`auth.go:227`).
- The setup form accepts entries like "notanemail" or "admin@" if the browser's built-in validation is lenient or bypassed.
- The `UpdateSystemEmail` handler at `setup.go:290-294` does call `validateEmail()` for the system from-address, showing the inconsistency.

## Expected Behavior
- The backend setup endpoint should call `validateEmail(req.Email)` before creating the admin account, consistent with signup and login.
- The frontend setup form should add client-side email format validation before submission.

## Files to Modify
- `backend/internal/handler/setup.go:131-134` -- Add `validateEmail(req.Email)` check after the empty-string check, before `validatePassword()`
- `frontend/app/(auth)/setup/page.tsx:109-131` -- Add client-side email validation in `handleSubmit` before API call

## Impact Analysis

### What This Affects
- The initial instance setup flow (only runs once per deployment)
- First admin account creation
- Downstream features that depend on valid admin email: password reset, system email delivery, verification codes

### What Affects This
- `validateEmail()` in `helpers.go:32-41` uses `mail.ParseAddress()` from Go's standard library for RFC 5322 validation
- This only affects new deployments where setup hasn't been completed yet

### Related PRDs
- PRD-039 (Password Rules) -- same pattern of incomplete validation on the setup form
- PRD-041 (Claim Name Validation) -- same pattern of missing validation on auth forms

## Implementation Notes
- Backend change is a 3-line addition after line 134 in `setup.go`:
  ```go
  if err := validateEmail(req.Email); err != nil {
      writeError(w, http.StatusBadRequest, err.Error())
      return
  }
  ```
- Frontend change: validate email format with a regex or `new URL("mailto:" + email)` pattern before calling the API.
- The HTML `type="email"` attribute provides basic browser validation but is not sufficient -- some browsers accept incomplete addresses.

## User Notes
- This only affects the initial setup flow which runs once per instance. Existing deployments with completed setup are not affected.
- If an admin was already created with a malformed email, they would need to manually update the users table in the database.

## Acceptance Criteria
- [ ] Backend setup endpoint validates email format using `validateEmail()` and returns 400 for invalid emails
- [ ] Frontend setup form validates email format before API call
- [ ] Valid emails (e.g., "admin@example.com") pass validation and complete setup
- [ ] Invalid emails (e.g., "notanemail", "admin@", "@example.com") are rejected with a clear error
- [ ] No regression in the setup flow for valid inputs
- [ ] The `validateEmail()` call in setup is consistent with signup (`auth.go:86`)
