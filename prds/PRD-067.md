# PRD-067: Empty Subject Thread Merging

## Status
Done

## Priority
P1-High

## Category
Email Threading & Rendering

## Source
GAP-FINDER T2

---

## Problem Statement
When an inbound email has no subject, the cleaned subject is an empty string. The thread-matching query (Step 2) searches for `WHERE subject = ''`, which matches ALL no-subject threads in the same domain and org. This causes completely unrelated conversations to merge into a single thread if they both lack a subject line.

## Current Behavior
In `backend/internal/queue/process_fetch.go`:
- Line 192: `cleanSubject := util.CleanSubjectLine(emailData.Subject)` -- if the original subject is empty or just whitespace/prefixes, this returns `""`.
- Lines 264-276 (Step 2): The query `WHERE org_id = $1 AND domain_id = $2 AND subject = $3 AND participant_emails @> $4::jsonb` runs with `$3 = ""`.
- Any existing thread with an empty subject and a matching counterparty in the same domain will be matched, even if the conversations are entirely unrelated.
- The 90-day window on `last_message_at` limits the blast radius but does not prevent incorrect merges within that window.

In `backend/internal/util/strings.go`:
- Lines 19-33: `CleanSubjectLine` strips `Re:`/`Fwd:` prefixes and collapses whitespace. An input of `""`, `"Re: "`, or whitespace-only results in `""`.

## Expected Behavior
- When `cleanSubject` is empty, skip Step 2 (subject-based matching) entirely.
- Fall through directly to Step 3 (create new thread) unless `In-Reply-To` or `References` headers provide a match (Steps 1 and 1.5).
- This prevents unrelated no-subject emails from merging based solely on having the same counterparty.

## Files to Modify
- `backend/internal/queue/process_fetch.go:264` — Wrap the Step 2 block in `if !found && cleanSubject != "" {` to skip subject matching for empty subjects.

## Impact Analysis

### What This Affects
- Inbound email processing for emails with no subject.
- Previously merged threads will not retroactively split, but new no-subject emails will create separate threads.
- Thread list may show more individual threads for users who receive many no-subject emails.

### What Affects This
- The `CleanSubjectLine` function in `backend/internal/util/strings.go` determines what constitutes an "empty" subject.
- PRD-066 (References Header Threading) -- if References matching is implemented, no-subject emails can still be correctly threaded via References.

## Implementation Notes
- This is a one-line conditional change. The guard `if !found && cleanSubject != ""` before the Step 2 block prevents the empty-subject query.
- Consider also logging when subject-based matching is skipped for debugging: `slog.Debug("email worker: skipping subject match for empty subject")`.
- No migration needed.
