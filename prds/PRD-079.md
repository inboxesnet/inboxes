# PRD-079: Domain Disconnect Notification

## Status
Draft

## Priority
P2-Medium

## Category
Domain & User Lifecycle

## Source

## Problem Statement
When the domain heartbeat worker detects that a domain has been disconnected from Resend (either removed from Resend or the API key revoked), it only logs the event. No email alert, no in-app notification, and no WebSocket event is sent to org admins. Admins discover domain disconnections by accident -- typically when users report that emails have stopped working -- rather than being proactively alerted.

## Current Behavior
In `backend/internal/worker/domain_heartbeat.go`:
- Lines 137-144: When a domain is missing from Resend and currently not disconnected:
  ```go
  slog.Warn("domain heartbeat: domain disconnected",
      "org_id", orgID, "domain_id", id, "domain", domain)
  ```
  The status is updated to `'disconnected'` in the database, but the only signal is a log line.
- Lines 86-97: When the API key is revoked (403), all domains are marked disconnected with a log warning but no notification.
- Lines 145-153: When a domain reconnects (reappears in Resend), only a log line is emitted.

In `backend/internal/event/event.go`:
- Lines 13-33: The event types include `EmailReceived`, `ThreadStarred`, etc., but there is no `DomainDisconnected` or `DomainStatusChanged` event type.

The `DomainHeartbeat` struct (lines 14-18) has `DB` and `ResendSvc` but no reference to the event bus (`*event.Bus`) or the Resend service for sending alert emails.

## Expected Behavior
- When a domain status changes to `'disconnected'`:
  1. Publish a WebSocket event (`domain.disconnected`) so the frontend can show an in-app banner or toast.
  2. Send an email alert to all org admins informing them that the domain is disconnected and requires attention.
- When a domain status changes back to `'active'` (reconnected):
  1. Publish a WebSocket event (`domain.reconnected`) to clear any in-app warnings.
  2. Optionally send a recovery email to admins.
- When the API key is revoked (all domains disconnected), send a single alert email rather than one per domain.

## Files to Modify
- `backend/internal/worker/domain_heartbeat.go:14-18` — Add `Bus *event.Bus` and `ResendSvc *service.ResendService` to the `DomainHeartbeat` struct (ResendSvc is already there for API calls; Bus needs to be added).
- `backend/internal/worker/domain_heartbeat.go:20-26` — Update `NewDomainHeartbeat` to accept and store the event bus.
- `backend/internal/worker/domain_heartbeat.go:137-144` — After marking a domain disconnected, publish an event:
  ```go
  dh.Bus.Publish(ctx, event.Event{
      EventType: "domain.disconnected",
      OrgID:     orgID,
      DomainID:  id,
      Payload:   map[string]interface{}{"domain": domain},
  })
  ```
- `backend/internal/worker/domain_heartbeat.go:145-153` — After marking a domain reconnected, publish a `domain.reconnected` event.
- `backend/internal/event/event.go:13-33` — Add `DomainDisconnected` and `DomainReconnected` event type constants.
- `backend/cmd/api/main.go` — Pass the event bus to `NewDomainHeartbeat`.
- `frontend/hooks/use-ws-sync.ts` or `frontend/contexts/notification-context.tsx` — Handle the new `domain.disconnected` event type and display a toast/banner.

## Impact Analysis

### What This Affects
- Domain health monitoring and alerting.
- Frontend will display real-time notifications when domains disconnect.
- Org admins will receive email alerts for domain issues.
- The WebSocket hub will broadcast domain status events to all org members.

### What Affects This
- The event bus must be available to the heartbeat worker (dependency injection change).
- Email alerts require a working system email sender (`ResendSvc.SystemFetch`). If no system Resend key is configured, fall back to log-only.
- The WebSocket hub's event filtering: domain events should broadcast to all org members, not filtered by alias visibility.

### Related PRDs
- PRD-075: DNS Changes Detection — DNS degradation is another form of domain health issue that should trigger notifications.
- PRD-072: Domain Deletion Data Cleanup — Deleted domains should not trigger disconnect notifications.
- PRD-073: Deleted Domains Still Receive Emails — Disconnect notifications complement the guard against processing deleted domain emails.
- PRD-109: Worker Intervals — The heartbeat interval affects notification timeliness.

## Implementation Notes
- For email alerts, use the system Resend key (`ResendSvc.SystemFetch`). Build a simple HTML email template inline (no external template engine needed for a single notification type).
- Batch notifications: if multiple domains disconnect at once (API key revoked), send a single email listing all affected domains rather than one email per domain.
- On the frontend, consider a persistent banner at the top of the domain sidebar (similar to "Reconnecting...") rather than a dismissible toast, since domain disconnection is a critical ongoing issue.
- Add the `domain.disconnected` and `domain.reconnected` events to the `broadcastEvents` map in `ws/hub.go` so they are sent to all org members without alias filtering.
