## Problem Statement
All background worker intervals are hardcoded with no way to tune them via environment variables. Operators cannot adjust timing for their specific deployment scale, database performance, or Resend API rate limits. This forces a one-size-fits-all configuration that may be too aggressive for small deployments or too slow for large ones.

## Current Behavior
The following intervals are hardcoded:

1. **Domain Heartbeat** (`backend/internal/worker/domain_heartbeat.go:24`): `6 * time.Hour` -- checks Resend API for domain status every 6 hours
2. **Trash Collector** (`backend/internal/worker/trash_collector.go:27`): `1 * time.Hour` -- purges expired trash every hour
3. **Event Pruner** (`backend/internal/worker/event_pruner.go:35`): `6 * time.Hour` -- cleans old events every 6 hours
4. **Status Recovery** (`backend/internal/worker/status_recovery.go:38`): `5 * time.Minute` -- polls Resend for stale email statuses every 5 minutes
5. **HTTP Timeouts** (`backend/cmd/api/main.go:130-133`): `ReadTimeout: 15s`, `WriteTimeout: 15s`, `IdleTimeout: 60s`, `ReadHeaderTimeout: 5s` -- not configurable

None of these are exposed via `config.go` or environment variables. Changing any interval requires a code change and redeployment.

## Expected Behavior
Worker intervals should be configurable via environment variables with sensible defaults matching current hardcoded values:
- `DOMAIN_HEARTBEAT_INTERVAL` -- default: `6h`
- `TRASH_COLLECTOR_INTERVAL` -- default: `1h`
- `EVENT_PRUNER_INTERVAL` -- default: `6h`
- `STATUS_RECOVERY_INTERVAL` -- default: `5m`

HTTP timeouts could remain hardcoded (they are less likely to need tuning) or be exposed as well.

## Files to Modify
- `backend/internal/config/config.go` -- Add interval fields to `Config` struct and parse from env vars using a `getEnvDuration` helper
- `backend/internal/worker/domain_heartbeat.go:20-25` -- Accept interval from config instead of hardcoding
- `backend/internal/worker/trash_collector.go:18-19,27` -- Accept interval from config
- `backend/internal/worker/event_pruner.go:16-17,35` -- Accept interval from config
- `backend/internal/worker/status_recovery.go:22-23,38` -- Accept interval from config
- `backend/cmd/api/main.go:97-110` -- Pass interval values from config to worker constructors

## Impact Analysis

## Implementation Notes
- Add a `getEnvDuration` helper to `config.go`:
  ```go
  func getEnvDuration(key string, fallback time.Duration) time.Duration {
      if val := os.Getenv(key); val != "" {
          if d, err := time.ParseDuration(val); err == nil && d > 0 {
              return d
          }
          slog.Warn("invalid duration for env var, using default", "key", key, "value", val, "default", fallback)
      }
      return fallback
  }
  ```
- Add to Config struct:
  ```go
  DomainHeartbeatInterval  time.Duration
  TrashCollectorInterval   time.Duration
  EventPrunerInterval      time.Duration
  StatusRecoveryInterval   time.Duration
  ```
- Update worker constructors to accept interval parameter:
  ```go
  func NewDomainHeartbeat(db *pgxpool.Pool, resendSvc *service.ResendService, interval time.Duration) *DomainHeartbeat {
      return &DomainHeartbeat{DB: db, ResendSvc: resendSvc, Interval: interval}
  }
  ```
- The `DomainHeartbeat` struct already has an `Interval` field (line 17) -- just need to populate it from config
- Add minimum interval validation (e.g., heartbeat >= 1m, status recovery >= 30s) to prevent accidental self-DoS
- Document format in .env.example: `DOMAIN_HEARTBEAT_INTERVAL=6h` (Go duration string format)

## User Notes
This is a deployment flexibility improvement. Operators running at different scales need different worker frequencies.
