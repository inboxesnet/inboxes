# PRD-038: Subject Length

## Status
Done

## Priority
P2-Medium

## Category
Form Validation Mismatches

## Source
GAP-FINDER V1

---

## Problem Statement
The frontend compose window allows subjects up to 998 characters (`maxLength={998}`) while the backend enforces a 500-character limit via `validateLength(req.Subject, "subject", 500)`. Users can type a 600-character subject that passes frontend validation but triggers a cryptic backend error on send or draft save.

## Current Behavior
- In `frontend/components/floating-compose-window.tsx`, lines 507 and 653 set `maxLength={998}` on the subject `<Input>` (both mobile and desktop compose views).
- In `backend/internal/handler/emails.go`, line 84 calls `validateLength(req.Subject, "subject", 500)`, rejecting anything over 500 chars.
- In `backend/internal/handler/drafts.go`, line 122 also calls `validateLength(req.Subject, "subject", 500)` for draft creation.
- When the user enters a subject between 501-998 characters, the frontend accepts it. On send/save, the backend returns a 400 error with the message "subject must be 500 characters or fewer". The user sees a generic error and has no idea why.

## Expected Behavior
- The frontend should enforce the same 500-character limit on the subject field.
- A character counter or hint should appear near the subject input when nearing the limit.
- If the user somehow bypasses the frontend limit, the backend error message should remain clear.

## Files to Modify
- `frontend/components/floating-compose-window.tsx:507` -- Change `maxLength={998}` to `maxLength={500}` (mobile compose subject input)
- `frontend/components/floating-compose-window.tsx:653` -- Change `maxLength={998}` to `maxLength={500}` (desktop compose subject input)

## Impact Analysis

### What This Affects
- Compose window behavior for all email sends and draft saves
- Any user typing long subjects will now be prevented at 500 characters instead of 998

### What Affects This
- Backend validation in `emails.go:84` and `drafts.go:122` -- if the backend limit ever changes, the frontend must be updated to match
- The 500-character limit was chosen to align with RFC 2822 recommendations (998-byte line length including header name)

### Related PRDs
- PRD-027 (Compose Draft Loss) -- draft save failures from this mismatch can cause data loss
- PRD-030 (Auto-Save Invisible) -- auto-save failures caused by subject length rejection would show no error

## Implementation Notes
- This is a one-line change on two occurrences in the same file.
- Consider adding a small character count indicator (e.g., "487/500") that appears when the subject exceeds 400 characters, similar to Twitter's approach.
- The backend already returns a clear error message, so no backend changes are needed.

## User Notes
- Users who previously typed very long subjects will now be stopped at 500 characters.
- No data migration needed -- existing drafts with long subjects will fail on next save attempt, which is the same behavior they would have had on send.

## Acceptance Criteria
- [ ] Both mobile and desktop subject inputs enforce `maxLength={500}`
- [ ] Typing beyond 500 characters is prevented by the browser
- [ ] Drafts with subjects of exactly 500 characters save successfully
- [ ] Subjects of 501+ characters are rejected by the backend with a clear error message (unchanged)
- [ ] No regression in compose window behavior for normal-length subjects
