# PRD-124: Select All Across Pages

## Problem Statement
Bulk selection in the thread list only applies to threads on the current page. There is no way to select all threads matching a filter across all pages. Users who need to bulk-archive, bulk-trash, or bulk-label hundreds of threads must manually repeat the operation page by page, which is tedious and error-prone.

## Current Behavior
- `frontend/hooks/use-thread-selection.ts:5-52` — The `useThreadSelection` hook manages selection state based on `threadIds`, which is the array of thread IDs visible on the current page.
- `frontend/hooks/use-thread-selection.ts:20-27` — `toggleSelectAll` selects/deselects only the `threadIds` array passed to the hook.
- `frontend/components/thread-list-page.tsx:76` — `threadIds` is computed from `activeThreads`, which is limited to the current page's data: `const threadIds = useMemo(() => activeThreads.map((t) => t.id), [activeThreads])`.
- `frontend/components/thread-list-page.tsx:19` — `LIMIT` is set to 100 threads per page.
- `frontend/components/thread-toolbar.tsx` — The toolbar shows a checkbox for "select all" but it only toggles current-page selection.
- The backend `PATCH /api/threads/bulk` endpoint (`backend/internal/handler/threads.go:368-540`) accepts an array of `thread_ids` and has no limit on array size, so it could handle server-side "select all" if the frontend provided matching thread IDs.

## Expected Behavior
When a user clicks "select all" and all current-page threads are selected, a banner should appear (Gmail-style): "All 100 conversations on this page are selected. **Select all N conversations in [folder].**" Clicking the link triggers a server-side bulk operation that applies to all matching threads, not just the currently loaded ones.

Implementation approach:
1. Add a `select_all` mode to the bulk action API that accepts a filter (label, domain_id) instead of explicit thread_ids.
2. The backend queries all matching thread IDs and applies the action in a single transaction.
3. The frontend shows the banner and sends the filter-based request.

## Files to Modify
- `frontend/hooks/use-thread-selection.ts` — Add a `selectAllPages` state and expose it.
- `frontend/components/thread-toolbar.tsx` — Add the "Select all N conversations" banner when all current-page threads are selected.
- `frontend/components/thread-list-page.tsx` — Pass `total` count to the toolbar for the banner text.
- `backend/internal/handler/threads.go:368-540` — Extend `BulkAction` to accept a filter-based mode: `{"action": "archive", "select_all": true, "domain_id": "...", "label": "inbox"}`.
- `frontend/hooks/use-threads.ts` — Update `useBulkAction` to support the filter-based request.

## Impact Analysis

### What This Affects
- Bulk operations across large thread sets (hundreds or thousands of threads).
- Database performance — a "select all" operation could touch thousands of rows in a single transaction.
- WebSocket events — bulk operations fire a single event with potentially thousands of thread IDs.

### What Affects This
- Pagination implementation — the total count is already returned by the list API (`data.total`).
- Database performance indexes (PRD-063) — the composite index for thread list queries is critical for server-side select-all performance.
- Transaction handling (PRD-044) — bulk operations need proper transaction wrapping.

### Related PRDs
- PRD-044: Label Transactions — bulk operations must be transactional.
- PRD-047: Bulk Selection — selection clearing before API completion; this feature extends that interaction pattern.
- PRD-060: Bulk Delete Queries — performance concern for large batch operations.
- PRD-063: Composite Index — the `threads(org_id, deleted_at, last_message_at)` index is critical for the server-side query.

## Implementation Notes
Gmail's approach:
1. User clicks "select all" checkbox -> selects current page threads.
2. Banner appears: "All 100 conversations on this page are selected. Select all 1,234 conversations in Inbox."
3. Clicking the link sets `selectAllPages = true` in state.
4. When a bulk action is triggered with `selectAllPages`, the frontend sends a filter instead of IDs.
5. Backend runs the action with `WHERE org_id = $1 AND label = $2 AND deleted_at IS NULL` instead of `WHERE id = ANY($1)`.

For safety, add a confirmation dialog when operating on more than 100 threads: "This will affect N conversations. Continue?"
