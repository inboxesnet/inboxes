# PRD-008: Verification Code Brute-Force

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
GAP-FINDER S2

---

## Problem Statement
The email verification endpoint accepts a 6-digit numeric code with a 15-minute expiry window. With only 1,000,000 possible codes and no per-endpoint rate limiting specific to the verification attempt, an attacker can enumerate all codes within the window. The existing IP-based rate limit on `/api/auth/verify-email` (5 requests per 15 minutes per IP, set in `router.go:120`) is insufficient because an attacker with multiple IPs or a botnet can trivially bypass it.

## Current Behavior
In `backend/internal/handler/auth.go`:
- **Code generation** (line 545-548): Generates a 6-digit code using crypto/rand:
  ```go
  func generateVerificationCode() string {
      n, _ := rand.Int(rand.Reader, big.NewInt(1000000))
      return fmt.Sprintf("%06d", n.Int64())
  }
  ```
- **Verification** (line 456-499): The `VerifyEmail` handler:
  ```go
  err := h.DB.QueryRow(ctx,
      `UPDATE users SET email_verified = true, verification_code = NULL, verification_expires_at = NULL
       WHERE email = $1 AND verification_code = $2 AND verification_expires_at > now()
       RETURNING id, org_id, name, role`,
      req.Email, req.Code,
  ).Scan(...)
  ```
  - No attempt counter or lockout mechanism.
  - No per-email rate limiting on verification attempts.
  - Failed attempts have no consequence.

- **Router rate limit** (`backend/internal/router/router.go:120`):
  ```go
  r.With(middleware.RateLimitByIP(rdb, 5, 15*60)).Post("/api/auth/verify-email", auth.VerifyEmail)
  ```
  - 5 attempts per 15 minutes per IP — but no per-email limit.

## Expected Behavior
- Add a per-email rate limit on verification attempts (e.g., 5 attempts per code, then the code is invalidated and user must request a new one).
- After N failed attempts for a given email, increment a counter in the database or Redis.
- After exceeding the limit, invalidate the current code and require the user to request a new one.
- Optionally increase code length to 8 digits or add an alpha component for higher entropy.

## Files to Modify
- `backend/internal/handler/auth.go:456-499` — Add attempt tracking:
  1. Before checking the code, increment a Redis counter keyed by email: `verify:attempts:{email}`.
  2. If the counter exceeds 5, return an error and invalidate the code in the database.
  3. On successful verification, delete the counter.
- `backend/internal/router/router.go:120` — Add `RateLimitByBodyField(rdb, "email", 5, 15*60)` as an additional middleware layer:
  ```go
  r.With(
      middleware.RateLimitByIP(rdb, 5, 15*60),
      middleware.RateLimitByBodyField(rdb, "email", 5, 15*60),
  ).Post("/api/auth/verify-email", auth.VerifyEmail)
  ```
- `backend/internal/handler/auth.go:545-548` — Optionally increase code entropy (e.g., 8-digit code for 100M possibilities).

## Impact Analysis

### What This Affects
- Email verification flow during signup in hosted mode (when `StripeKey != ""`).
- Legitimate users who mistype their code 5+ times will need to request a new code.
- The `ResendVerification` endpoint would need to clear the attempt counter.

### What Affects This
- Redis availability — attempt counters are stored in Redis. If Redis is down, the rate limiter already fails open (line 88-89 in `ratelimit.go`). For verification specifically, consider failing closed.
- The `ResendVerification` handler (line 501-542) generates a new code, which should also reset the attempt counter.

### Related PRDs
- PRD-007: Case-Sensitive Emails — Email normalization ensures rate limits work correctly regardless of casing (the rate limit key should use the lowercased email).
- PRD-009: Disabled Users Auth — If a disabled user's account still has an unverified email, brute-forcing the code could re-enable access.

## Implementation Notes
- Use a Redis key like `verify:attempts:{normalized_email}` with a TTL of 15 minutes (matching the code expiry).
- The per-email rate limit via `RateLimitByBodyField` middleware handles the common case. For extra safety, add an explicit attempt counter in the handler that invalidates the code after N failures.
- After invalidating a code, the user must call `/api/auth/resend-verification` to get a new one.
- Ensure `ResendVerification` (line 501-542) deletes the old attempt counter when generating a new code:
  ```go
  h.RDB.Del(ctx, "verify:attempts:"+normalizedEmail)
  ```
- Consider rate-limiting `ResendVerification` per email more tightly to prevent attackers from resetting the counter repeatedly (currently 3/hour per IP, which is already good — line 121).
