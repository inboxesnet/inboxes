# PRD-113: Container Resource Limits

## Status
Done

## Priority
P2-Medium

## Category
Infrastructure

## Source
PRE-LAUNCH-AUDIT #33

---

## Problem Statement
The `docker-compose.yml` has no memory or CPU limits on any service. A runaway container (e.g., the backend hitting an infinite loop, a memory leak, or PostgreSQL consuming excessive memory for a large query) can consume all host resources, taking down all other services including the database and frontend.

## Current Behavior
In `docker-compose.yml`, four services are defined: `postgres` (lines 2-15), `redis` (lines 17-26), `backend` (lines 28-44), and `frontend` (lines 46-53). None of them have `deploy.resources.limits` configured. On a typical VPS with 2-4 GB RAM, a single misbehaving container can trigger the Linux OOM killer, potentially terminating the database process and causing data corruption.

## Expected Behavior
Each service should have appropriate resource limits that prevent any single container from consuming all host resources while still allowing normal operation:
- `postgres`: 512MB memory, 0.5 CPU (handles typical email workload)
- `redis`: 128MB memory, 0.25 CPU (primarily caching and queues)
- `backend`: 256MB memory, 0.5 CPU (Go binary, low memory footprint)
- `frontend`: 256MB memory, 0.5 CPU (Next.js standalone server)

These should be tunable via environment variables for operators with different hardware.

## Files to Modify
- `docker-compose.yml:2-15` — Add `deploy.resources.limits` to postgres service.
- `docker-compose.yml:17-26` — Add `deploy.resources.limits` to redis service.
- `docker-compose.yml:28-44` — Add `deploy.resources.limits` to backend service.
- `docker-compose.yml:46-53` — Add `deploy.resources.limits` to frontend service.
- `docs/deployment.md` — Document resource limits and how to tune them.

## Impact Analysis

### What This Affects
- Container stability on resource-constrained VPS instances.
- PostgreSQL performance under heavy query load (may need limit increase for large orgs).
- Docker Compose compatibility — `deploy` key requires `docker compose` (v2) or `docker-compose --compatibility` flag.

### What Affects This
- Host machine RAM and CPU (limits must fit within available resources).
- Number of concurrent users and email volume (higher load may require higher limits).
- PostgreSQL `shared_buffers` and `work_mem` settings (should fit within the memory limit).

## Implementation Notes
Example addition to the backend service:
```yaml
backend:
  build: ./backend
  restart: unless-stopped
  deploy:
    resources:
      limits:
        memory: 256M
        cpus: '0.5'
```
Note: `deploy.resources` in Compose v2 works natively. For Compose v1, the `--compatibility` flag is needed. Add a comment in the compose file noting this.

For Redis, also consider adding `maxmemory 100mb` and `maxmemory-policy allkeys-lru` to prevent Redis itself from growing unbounded within its container limit.
