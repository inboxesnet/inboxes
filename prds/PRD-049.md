# PRD-049: Unchecked Queries

## Status
Done

## Priority
P1-High

## Category

## Problem Statement
Over 20 `QueryRow().Scan()` calls across the backend silently ignore errors. When a database query fails, the code continues using zero-valued or undefined variables, leading to incorrect behavior: wrong counts, missing data in responses, and potential null-pointer-like issues in downstream logic. This is especially dangerous in security-sensitive paths like webhook secret lookups and billing queries.

## Expected Behavior
Every `QueryRow().Scan()` call should check the returned error and either:
1. Return an appropriate HTTP error to the client, or
2. Log a warning and use a safe default value with explicit documentation, or
3. Skip the operation with a logged warning when the value is truly optional (e.g., domain_id for event payloads).

For critical paths (webhook secret, billing, auth), errors must return an HTTP error to prevent silent data corruption.

## Impact Analysis

### What This Affects
- All thread list and thread detail API responses (wrong counts, missing data)
- Setup and onboarding flow (may skip steps or repeat them)
- Billing checkout URLs (may generate broken redirects)
- Contact card data (incorrect email counts)
- WebSocket event payloads (missing domain_id means events may not route correctly)

### What Affects This
- Database connectivity and query performance
- Schema changes that alter column types or constraints

### Related PRDs
- PRD-050 (Fire-and-Forget Writes) — same pattern but for Exec() calls
- PRD-051 (JSON Errors) — related silent failure pattern
- PRD-052 (Redis Webhook) — webhook secret lookup is the most critical unchecked query
- PRD-010 (Webhook Secret Bypass) — the unchecked query at webhooks.go:78-80 enables this bypass

## Implementation Notes
- Prioritize the webhook secret lookup (webhooks.go:78-80) — this is also covered by PRD-010 but the fix here is checking the Scan error and returning 500 instead of proceeding with an empty secret.
- For domain_id lookups used only for event payloads (threads.go:587, 633, 670, etc.), a missing domain_id is non-fatal. Log a warning and publish the event with an empty domain_id rather than failing the entire request.
- For total count queries (threads.go:157), default to 0 but log the error so operators can diagnose.
- Consider creating a helper like `scanOrDefault(row, dest, defaultVal)` to reduce boilerplate for truly optional lookups.

## User Notes
Users currently receive incorrect data silently (e.g., thread count = 0 when the count query fails, but threads are still returned). After this fix, users will see proper error responses when critical queries fail, rather than silently degraded data.
