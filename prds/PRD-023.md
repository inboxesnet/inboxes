# PRD-023: Resend-Verification Rate Limit

## Status

## Priority
P1-High

## Problem Statement
The resend-verification endpoint (`POST /api/auth/resend-verification`) has IP-based rate limiting (3 per hour) but no per-email rate limiting. An attacker with access to multiple IPs (e.g., a botnet or cloud instances) can flood a user's inbox with hundreds of verification emails by calling this endpoint repeatedly from different IP addresses. This constitutes an email bombing attack and can also be used to cause the target's email provider to mark the sender domain as spam.

## Current Behavior
In `backend/internal/router/router.go:121`, the resend-verification route has only IP-based rate limiting:

```go
r.With(middleware.RateLimitByIP(rdb, 3, 1*60*60)).Post("/api/auth/resend-verification", auth.ResendVerification)
```

In `backend/internal/handler/auth.go:501-542`, the handler generates a new verification code and sends an email each time it is called:

```go
func (h *AuthHandler) ResendVerification(w http.ResponseWriter, r *http.Request) {
    ...
    code := generateVerificationCode()
    expires := time.Now().Add(15 * time.Minute)

    tag, err := h.DB.Exec(ctx,
        `UPDATE users SET verification_code = $1, verification_expires_at = $2
         WHERE email = $3 AND email_verified = false AND status = 'active'`,
        code, expires, req.Email,
    )
    if err != nil || tag.RowsAffected() == 0 {
        // Don't reveal whether user exists
        writeJSON(w, http.StatusOK, ...)
        return
    }

    // Send verification email (no per-email rate check)
    h.ResendSvc.SystemFetch(ctx, "POST", "/emails", map[string]interface{}{
        "from":    from2,
        "to":      []string{req.Email},
        "subject": "Verify your email",
        "html":    fmt.Sprintf("...%s...", code),
    })
    ...
}
```

The handler correctly avoids revealing whether the email exists (returning the same response either way), but it does not limit how many verification emails can be sent to a single address. The IP-based limit of 3 per hour is easily bypassed with multiple IPs.

Compare with the forgot-password endpoint (`router.go:116`), which has BOTH IP-based and email-based rate limiting:

```go
r.With(
    middleware.RateLimitByIP(rdb, 3, 1*60*60),
    middleware.RateLimitByBodyField(rdb, "email", 3, 1*60*60),
).Post("/api/auth/forgot-password", auth.ForgotPassword)
```

## Expected Behavior
1. Add per-email rate limiting to the resend-verification endpoint, matching the pattern already used by forgot-password.
2. Limit to 3 resend attempts per email address per hour.
3. Return the same generic response when rate-limited (do not reveal account existence).

## Files to Modify
- `backend/internal/router/router.go:121` — Add `middleware.RateLimitByBodyField(rdb, "email", 3, 1*60*60)` alongside the existing IP-based rate limit

## Impact Analysis

### What This Affects
- Verification email flow — legitimate users who need to resend verification are limited to 3 attempts per hour per email address
- Email sending costs — prevents abuse of the system email sending capability
- Sender domain reputation — prevents the domain from being flagged as spam due to email bombing

### What Affects This
- Redis availability — rate limiting depends on Redis (fails open per `ratelimit.go:87-90`)
- `RateLimitByBodyField` middleware already exists and is battle-tested on the forgot-password endpoint

### Related PRDs
- PRD-021: Login Lockout — Same pattern of adding per-account abuse prevention
- PRD-022: Change-Password Rate Limit — Same pattern of adding rate limiting to prevent abuse
- PRD-015: Admin Rate Limits — Broader pattern of adding rate limits to unprotected endpoints

## Implementation Notes
This is a one-line fix. Add `middleware.RateLimitByBodyField(rdb, "email", 3, 1*60*60)` to the existing middleware chain:

```go
r.With(
    middleware.RateLimitByIP(rdb, 3, 1*60*60),
    middleware.RateLimitByBodyField(rdb, "email", 3, 1*60*60),
).Post("/api/auth/resend-verification", auth.ResendVerification)
```

The `RateLimitByBodyField` middleware (`ratelimit.go:46-81`) reads the JSON body, extracts the `email` field, lowercases it, and uses Redis to track request count per email. It transparently re-buffers the request body so the handler can still read it.

The rate-limit response is a generic HTTP 429 (`ratelimit.go:31-35`), which does not reveal whether the email exists — this is consistent with the handler's existing behavior.

Consider also applying the same treatment to the `verify-email` endpoint (`router.go:120`) which currently has `RateLimitByIP(rdb, 5, 15*60)` but no per-email limit. This would be relevant for PRD item #8 (Verification Brute-Force).
