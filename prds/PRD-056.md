# PRD-056: Upload Errors

## Status
Done

## Priority
P2-Medium

## Category
Error Handling — Frontend

## Source
GAP-FINDER E9

---

## Problem Statement
The file upload function in the compose window uses raw `fetch()` instead of the app's `api` wrapper, which means it bypasses the centralized error handling (401 redirect, structured error parsing, etc.). While the function does have a try/catch block that sets an error message, network-level errors (DNS failure, CORS issues, connection refused) and non-JSON error responses are not handled with the same granularity as other API calls. The error message shown is always the generic "Failed to upload file" regardless of the actual cause.

## Current Behavior
In `frontend/components/floating-compose-window.tsx:156-189`:
```tsx
async function handleFileUpload(e: React.ChangeEvent<HTMLInputElement>) {
    // ... file validation ...
    setUploading(true);
    setError("");
    try {
      const form = new FormData();
      form.append("file", file);
      const res = await fetch("/api/attachments/upload", {
        method: "POST",
        credentials: "include",
        body: form,
      });
      if (!res.ok) throw new Error("Upload failed");
      const data = await res.json();
      setAttachments((prev) => [...prev, { id: data.id, filename: data.filename, size: data.size }]);
      scheduleSave();
    } catch {
      setError("Failed to upload file");
    } finally {
      setUploading(false);
      if (fileInputRef.current) fileInputRef.current.value = "";
    }
}
```

Issues:
1. `fetch()` does not go through the `api` wrapper in `lib/api.ts`, bypassing 401 session expiry handling.
2. If the response is not JSON (e.g., nginx 502 HTML page), `res.json()` throws and the actual error is lost.
3. All errors show the same "Failed to upload file" message — no distinction between "file too large," "unauthorized," "server error," or "network error."
4. A 401 response does not trigger the session expiry redirect (unlike all other API calls).

## Expected Behavior
1. Use the `api` wrapper or a similar pattern that handles 401 responses consistently.
2. Parse the server's error message from the response body when possible.
3. Show specific error messages:
   - "File too large" for 413 responses
   - "Session expired" for 401 responses (and trigger redirect)
   - "Upload failed: [server message]" for 4xx/5xx with a JSON error body
   - "Network error — check your connection" for fetch failures
4. A toast notification should also be shown for upload errors (in addition to the inline error) since the error message area is small and easy to miss.

## Files to Modify
- `frontend/components/floating-compose-window.tsx:156-189` — Rewrite handleFileUpload to use better error handling
- `frontend/lib/api.ts` — Optionally add a `uploadFile()` helper that handles FormData uploads through the same error-handling pipeline

## Implementation Notes
- Option A: Create an `api.upload(path, formData)` method in `lib/api.ts` that:
  - Uses `fetch()` with `credentials: "include"` (same as current)
  - Checks for 401 and triggers session expiry redirect
  - Parses JSON error bodies for 4xx/5xx responses
  - Returns the parsed response data
  - This keeps the FormData handling separate from the JSON-based `api.post`.

- Option B: Refactor `handleFileUpload` inline to handle errors more granularly:
  ```tsx
  const res = await fetch("/api/attachments/upload", { method: "POST", credentials: "include", body: form });
  if (!res.ok) {
    if (res.status === 401) { window.location.href = "/login"; return; }
    let msg = "Upload failed";
    try { const body = await res.json(); if (body.error) msg = body.error; } catch {}
    throw new Error(msg);
  }
  ```

- Option A is preferred for consistency, as other parts of the app may also need file upload support in the future (e.g., PRD-126 draft attachments).
- Show a toast in addition to the inline error: `toast.error("Upload failed: " + msg)`.

## User Notes
Currently, all upload errors show the generic "Failed to upload file" message. After this fix, users will see specific messages explaining why the upload failed (too large, unauthorized, server error, network issue) and can take appropriate action.
