# PRD-035: Session Expiry

## Status
Done

## Priority
P2-Medium

## Category
Session & Auth UX

## Source
PRE-LAUNCH-AUDIT #15

---

## Problem Statement
When a user's JWT session expires while they are actively using the app (especially while composing an email), the global 401 interceptor in `api.ts` immediately redirects to `/login` via `window.location.href = "/login"`. This hard redirect is destructive: any unsaved compose content, form data, or in-progress actions are instantly and irrecoverably lost. There is no warning, no modal, and no opportunity for the user to save their work.

## Current Behavior
In `api.ts` (lines 29-31), the `request()` function handles 401 responses with:
```typescript
if (res.status === 401) {
  window.location.href = "/login";
}
```
This executes immediately on any 401 response from any API call, including:
- Auto-save draft calls (every 3 seconds while composing)
- Thread list background refetches
- Search queries
- Mark-as-read calls when viewing a thread

The JWT has a 7-day expiry (configured in the backend). When it expires, the very next API call triggers the redirect, which:
1. Destroys all React state (compose content, selections, scroll positions).
2. Navigates away from the current page without any user interaction.
3. Does not fire the `beforeunload` event (since it is a programmatic navigation, not a page close).

## Expected Behavior
1. On the first 401 response, fire a custom event (`session-expired`) instead of redirecting immediately.
2. Show a modal dialog: "Your session has expired. Please log in again to continue."
3. Give the user time to copy any unsaved work from the compose window.
4. Include a "Log in" button in the modal that redirects to `/login`.
5. While the modal is open, suppress further API calls to avoid cascading 401 errors.
6. The compose window should remain visible behind the modal.

## Files to Modify
- `frontend/lib/api.ts:29-31` -- Replace the hard redirect with a custom event dispatch: `window.dispatchEvent(new CustomEvent("session-expired"))`. Only redirect if no listener handles the event.
- `frontend/lib/api.ts:68-88` -- Apply the same change to `uploadFile()` which has its own 401 handler on line 77.
- `frontend/app/(app)/layout.tsx` -- Add a `SessionExpiredModal` component that listens for the `session-expired` event and shows a dialog.
- New file: `frontend/components/session-expired-modal.tsx` -- A modal component with "Your session has expired" message and a "Log in" button.
- `frontend/components/floating-compose-window.tsx` -- The compose window should remain visible (not unmounted) while the session expired modal is showing.

## Impact Analysis

### What This Affects
- All authenticated users whose sessions expire while the app is open.
- Compose window users are the most impacted -- losing a half-written email is the worst case.
- Settings modal users editing team/org configurations.
- Any user with the app open in a background tab for extended periods.

### What Affects This
- JWT expiry duration (7 days) -- longer expiry reduces frequency but does not eliminate the issue.
- Background API calls (auto-save, refetches, WebSocket reconnects) -- these trigger the 401 before any user-initiated action.
- The 402 handler (line 35-37) for payment walls -- uses the event dispatch pattern already, which is the model to follow.

### Related PRDs
- PRD-027 (Compose Draft Loss) -- session expiry is a cause of draft loss
- PRD-030 (Auto-Save Invisible) -- session expiry causes auto-save to fail silently before the redirect
- PRD-019 (Token Blacklist Fail-Open) -- related session/token lifecycle concern

## Implementation Notes
- **Event-based approach** (matching the existing 402 pattern):
  ```typescript
  // In api.ts request():
  if (res.status === 401) {
    window.dispatchEvent(new CustomEvent("session-expired"));
    throw new ApiError(401, "Session expired");
  }
  ```
- **Session expired modal component**:
  ```tsx
  function SessionExpiredModal() {
    const [open, setOpen] = useState(false);
    useEffect(() => {
      function handler() { setOpen(true); }
      window.addEventListener("session-expired", handler);
      return () => window.removeEventListener("session-expired", handler);
    }, []);
    if (!open) return null;
    return (
      <Dialog open={open}>
        <DialogContent>
          <h2>Session Expired</h2>
          <p>Your session has expired. Please log in again.</p>
          <Button onClick={() => window.location.href = "/login"}>Log in</Button>
        </DialogContent>
      </Dialog>
    );
  }
  ```
- Place the modal in `app/(app)/layout.tsx` so it renders above all app content.
- Add a global `sessionExpired` flag (e.g., in a context or module-level variable) that the `request()` function checks before making API calls. Once set, skip the call and throw immediately to prevent cascading 401s.
- Do NOT auto-redirect after a timeout -- let the user explicitly click "Log in" so they have time to copy their compose content.

## User Notes
- Users composing long emails are the primary beneficiaries.
- The modal gives users a chance to manually copy their email text before logging in.
- After logging in, the user returns to the app root -- the compose content is still lost, but at least they had a chance to save it externally.

## Acceptance Criteria
- [ ] When a session expires, a modal dialog appears instead of an immediate redirect to /login
- [ ] The modal displays "Your session has expired. Please log in again."
- [ ] The modal includes a "Log in" button that redirects to /login
- [ ] The compose window remains visible behind the modal (not unmounted)
- [ ] No cascading 401 error toasts appear while the modal is open
- [ ] The session-expired event fires only once, even if multiple API calls return 401
- [ ] The `uploadFile` function (api.ts:68-88) also uses the event-based approach instead of hard redirect
- [ ] Background API calls (auto-save, refetches) are suppressed while the session-expired modal is open
