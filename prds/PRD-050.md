# PRD-050: Fire-and-Forget Writes

## Status
Done

## Priority
P1-High

## Category
Error Handling — Backend

## Source
GAP-FINDER E2

---

## Problem Statement
Over 15 database write operations (`Exec()` calls) silently discard errors. When these writes fail, the data becomes inconsistent: trash expiry dates are not set, labels are not applied or removed, read/unread state is stale, and emails are not properly marked. The caller receives a success response while the underlying operation failed.

## Current Behavior
The following `Exec()` calls ignore errors (using `_` for the error return or not checking at all):

**Thread label operations (threads.go):**
- `threads.go:426` — `bulkRemoveLabel()` for archive action, no error check
- `threads.go:430` — `bulkAddLabel()` for trash action, no error check
- `threads.go:431-433` — trash_expires_at UPDATE, error ignored
- `threads.go:437` — `bulkAddLabel()` for spam, no error check
- `threads.go:441-444` — bulk mark read, error captured but `_` discards it on line 441
- `threads.go:447-450` — bulk mark unread, same pattern
- `threads.go:464-469` — move to inbox: multiple label ops + trash_expires_at clear, all unchecked
- `threads.go:471-474` — move to trash, unchecked
- `threads.go:476-480` — move to spam/archive, unchecked
- `threads.go:489-497` — label/unlabel/mute/unmute actions, unchecked

**Single-thread operations (threads.go):**
- `threads.go:591-595` — Move handler: addLabel, removeLabel, UPDATE trash_expires_at, all unchecked
- `threads.go:597-599` — Move to trash, unchecked
- `threads.go:601-606` — Move to spam/archive/custom, unchecked
- `threads.go:644` — Mark all emails as read, error ignored
- `threads.go:681-684` — Mark latest email as unread, error ignored
- `threads.go:800` — Archive removeLabel, unchecked
- `threads.go:828-831` — Trash addLabel + UPDATE, unchecked
- `threads.go:868-874` — Spam addLabel/removeLabel, unchecked
- `threads.go:907` — Delete removeAllLabels, unchecked

**Other handlers:**
- `backend/internal/handler/users.go:281` — Delete source's alias_users in reassignment, unchecked
- `backend/internal/handler/users.go:322-325` — Audit row insert in reassignment, unchecked

## Expected Behavior
All write operations should check for errors. The approach depends on the operation's criticality:

1. **Label operations in bulk actions**: Log errors and continue processing remaining threads. Return a partial success indicator showing how many succeeded vs. failed.
2. **Single-thread label operations**: Return HTTP 500 if any write fails, with the specific operation that failed.
3. **Audit/logging writes**: Log errors but do not fail the parent operation (these are best-effort by nature).
4. **Thread state updates (trash_expires_at, unread_count)**: Return HTTP 500 on failure — these affect visible state.

## Files to Modify

## Impact Analysis

### What This Affects
- Thread label consistency (inbox/trash/spam/starred state)
- Trash expiry dates (threads may never auto-delete or may auto-delete when they shouldn't)
- Read/unread state (badge counts may drift from actual state)
- User reassignment audit trail

### What Affects This
- Database connectivity and transaction timeouts
- Concurrent operations on the same threads (race conditions from PRD-044)

### Related PRDs
- PRD-049 (Unchecked Queries) — read-side of the same pattern
- PRD-044 (Label Transactions) — these fire-and-forget writes compound the no-transaction problem
- PRD-060 (Bulk Delete Queries) — bulk delete is a specific instance of fire-and-forget in a loop

## Implementation Notes
- The `addLabel`, `removeLabel`, `bulkAddLabel`, `bulkRemoveLabel`, `hasLabel`, and `removeAllLabels` functions in the handler package should be updated to return errors instead of swallowing them internally.
- For bulk operations, consider accumulating errors and returning a summary: `{"affected": 95, "failed": 5, "errors": ["..."]}`.
- The `Move`, `MarkRead`, `MarkUnread` handlers should wrap their multi-step label changes in a transaction (this overlaps with PRD-044 but the error-checking part is independent).

## User Notes
Currently, users may see success toasts for operations that partially failed (e.g., "Moved to trash" when the label was added but the expiry date was not set). After this fix, failed operations will return errors so the frontend can show appropriate feedback.
