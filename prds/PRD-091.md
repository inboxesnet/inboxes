# PRD-091: Alias Filter Fail

## Status
Done

## Priority
P2-Medium

## Category
WebSocket & Real-Time

## Source
GAP-FINDER W3

---

## Problem Statement
When the database query for alias-based event filtering fails, the Hub broadcasts the event to ALL org members instead of only the authorized subset. This is a fail-open security issue: a database error (timeout, connection exhaustion, network blip) causes private thread events to leak to users who should not see them.

## Current Behavior
In `backend/internal/ws/hub.go:107-116`:
```go
// For thread-specific events, filter by alias permissions
if evt.ThreadID != "" && !broadcastEvents[evt.Event] {
    aliasLabels := h.getThreadAliasLabels(ctx, evt.ThreadID)
    if len(aliasLabels) > 0 {
        h.sendToOrgFiltered(evt.OrgID, aliasLabels, string(frontendMsg))
        continue
    }
}

// Broadcast events or threads without alias labels go to all
h.sendToOrg(evt.OrgID, string(frontendMsg))
```

The `getThreadAliasLabels` function (line 144-161) returns `nil` on database error:
```go
func (h *Hub) getThreadAliasLabels(ctx context.Context, threadID string) []string {
    rows, err := h.pool.Query(ctx, ...)
    if err != nil {
        return nil  // <-- fail-open: returns nil on error
    }
    // ...
}
```

When `getThreadAliasLabels` returns `nil` (empty slice), `len(aliasLabels) > 0` is false, so the code falls through to `sendToOrg()`, which broadcasts to ALL org members. This means:
- A thread assigned to alias `finance@company.com` should only notify users with that alias.
- If the DB query fails, ALL org members (including marketing, engineering, etc.) receive the event.
- The leaked event contains `thread_id` and `domain_id`, allowing unauthorized users to fetch the full thread via the API (though API-level alias filtering would block this).

## Expected Behavior
1. When alias filtering fails due to a database error, the event should be DROPPED (fail-closed), not broadcast to everyone.
2. The dropped event should be logged at error level with enough context to debug.
3. Clients that missed the event will catch up via the `/api/events` endpoint on their next reconnection, which has proper alias filtering at the API level.

## Files to Modify
- `backend/internal/ws/hub.go:144-161` -- Change `getThreadAliasLabels` to return an error alongside the labels
- `backend/internal/ws/hub.go:107-116` -- Check the error from `getThreadAliasLabels`; on error, drop the event and log

## Impact Analysis

### What This Affects
- Privacy of thread events during database errors
- Real-time notification reliability (dropped events are recovered via catchup)
- Security posture of the alias-based filtering system

### What Affects This
- Database availability and connection pool health
- Query performance on `thread_labels` table
- Event volume (more events = more alias lookups)

### Related PRDs
- PRD-089 (WS Connection Limit) -- fewer connections means less DB pressure, reducing chance of query failures
- PRD-092 (Event Catchup) -- catchup mechanism recovers dropped events on reconnection
- PRD-090 (Token Expiry) -- alias queries happen alongside token checks, sharing DB load

## Implementation Notes
- Change `getThreadAliasLabels` to return `([]string, error)`:
  ```go
  func (h *Hub) getThreadAliasLabels(ctx context.Context, threadID string) ([]string, error) {
      rows, err := h.pool.Query(ctx,
          `SELECT label FROM thread_labels WHERE thread_id = $1 AND label LIKE 'alias:%'`,
          threadID)
      if err != nil {
          return nil, err  // Return error instead of nil
      }
      // ...
      return labels, nil
  }
  ```
- Update the caller:
  ```go
  if evt.ThreadID != "" && !broadcastEvents[evt.Event] {
      aliasLabels, err := h.getThreadAliasLabels(ctx, evt.ThreadID)
      if err != nil {
          slog.Error("ws: alias filter query failed, dropping event",
              "thread_id", evt.ThreadID, "event", evt.Event, "error", err)
          continue  // DROP the event — fail closed
      }
      if len(aliasLabels) > 0 {
          h.sendToOrgFiltered(evt.OrgID, aliasLabels, string(frontendMsg))
          continue
      }
  }
  ```
- The dropped event will be recovered by clients via the `/api/events?since=X` catchup endpoint, which performs alias filtering at the handler level with proper error handling.
- Consider adding a short-lived cache (e.g., 30-second TTL) for alias labels per thread to reduce DB pressure and avoid repeated queries for the same thread during burst events.

## User Notes
- No user-visible change under normal conditions.
- During database issues, users may experience a brief delay in receiving thread-specific notifications. Notifications will be recovered automatically on reconnection.
