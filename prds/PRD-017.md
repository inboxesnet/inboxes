# PRD-017: Consolidate SetupAddresses into a Single Transaction

## Context

`SetupAddresses` in `onboarding.go:388-406` loops over addresses, calling `setupOneAddress` for each. Each address creates its own internal transaction. This works — all operations are idempotent (`ON CONFLICT DO UPDATE/NOTHING`), so failures in practice don't happen. The only theoretical failure is a DB connection drop mid-loop.

This isn't a bug fix. It's a minor cleanup: one outer transaction instead of N inner ones means fewer DB round-trips, simpler code in `setupOneAddress` (drops its own tx management), and slightly more correct semantics for free.

## Change

Hoist the transaction to `SetupAddresses`. Pass `tx` into `setupOneAddress` instead of `h.DB`. Remove the per-address `Begin`/`Commit` from `setupOneAddress`.

Matches the pattern already used by `SelectDomains` and `Connect` in the same file.

## Scope

- `backend/internal/handler/onboarding.go` — `SetupAddresses` and `setupOneAddress` only
- No frontend changes

## Priority

Low — cleanup, not a fix.
