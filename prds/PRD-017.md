# PRD-017: Redis Auth

## Status
Done

## Priority
P1-High

## Category
Security — Fix Soon After Launch

## Source
PRE-LAUNCH-AUDIT #9

---

## Problem Statement
The default Redis connection URL in both the config and `.env.example` uses `redis://localhost:6379` with no password. In deployments where Redis is accessible over the network (not just localhost), anyone on the network can connect and read/write data including token blacklist entries, rate limit counters, email job queues, and WebSocket event channels. This could allow an attacker to bypass rate limits, un-revoke tokens, or inject malicious job IDs.

## Current Behavior
In `backend/internal/config/config.go:33`, the default Redis URL has no authentication:

```go
RedisURL: getEnv("REDIS_URL", "redis://localhost:6379"),
```

In `.env.example:17`, the Redis URL is commented out with no guidance on adding a password:

```
# REDIS_URL=redis://redis:6379
```

In `docker-compose.yml:19-26`, the Redis service has no password configured:

```yaml
redis:
  image: redis:7-alpine
  restart: unless-stopped
  volumes:
    - redis_data:/data
```

There is no `--requirepass` flag in the Redis command, no `redis.conf` mounted, and no documentation about securing Redis for production.

The Redis client is used throughout the backend for security-critical operations:
- Token blacklisting in `backend/internal/service/token_blacklist.go:31,43,56,67`
- Rate limiting in `backend/internal/middleware/ratelimit.go:86`
- Email job queue in `backend/internal/handler/emails.go:278`
- WebSocket pub/sub in `backend/internal/ws/hub.go:77`

## Expected Behavior
1. The `.env.example` should document the password-authenticated Redis URL format: `redis://:yourpassword@redis:6379`
2. The `docker-compose.yml` should either set a Redis password via the `command` directive or document that the default is for development only
3. The config loader should log a warning at startup if the Redis URL has no password and the host is not localhost
4. Deployment documentation should include Redis authentication setup

## Files to Modify
- `.env.example:16-17` — Add documented example with password: `REDIS_URL=redis://:yourpassword@redis:6379`
- `docker-compose.yml:19-26` — Add `command: redis-server --requirepass ${REDIS_PASSWORD:-}` or document the security implication
- `backend/internal/config/config.go:33` — Add startup warning for unauthenticated non-local Redis

## Impact Analysis

### What This Affects
- All Redis-dependent features: rate limiting, token revocation, email queue, WebSocket events
- Deployment procedures for self-hosted and commercial instances
- Existing deployments will need to update their Redis URL if they add a password

### What Affects This
- Network topology — Docker-internal connections are less exposed but still vulnerable to container escape
- PRD-019 (Token Blacklist Fail-Open) — If Redis is compromised, an attacker could delete blacklist entries

### Related PRDs
- PRD-016: DB SSL — Same pattern of insecure default infrastructure configuration
- PRD-019: Token Blacklist Fail-Open — Redis compromise would allow clearing the token blacklist
- PRD-026: Password+Redis Failure — Redis availability directly affects session security

## Implementation Notes
For `docker-compose.yml`, the recommended approach is to use a `REDIS_PASSWORD` environment variable:

```yaml
redis:
  image: redis:7-alpine
  command: redis-server --requirepass ${REDIS_PASSWORD:-}
  environment:
    - REDIS_PASSWORD=${REDIS_PASSWORD:-}
```

And update the backend's REDIS_URL accordingly:
```yaml
REDIS_URL: redis://:${REDIS_PASSWORD:-}@redis:6379
```

When `REDIS_PASSWORD` is empty, Redis runs without auth (development mode). When set, authentication is required.

The config loader warning should parse the Redis URL and check for the presence of a password component. If absent and host is not `localhost`/`127.0.0.1`/`redis` (Docker service), log a warning.
