# PRD-006: Billing Checkout URL Injection

## Status
Done

## Priority
P0-Critical

## Category
Security — Fix Before Launch

## Source
PRE-LAUNCH-AUDIT #6

---

## Problem Statement
The billing checkout handler constructs success and cancel URLs by concatenating the `firstDomainID` from a database query directly into URL path segments without validation. While this value comes from the database (not user input), it originates from the Resend API during domain creation. A malicious or corrupted domain ID containing path traversal characters (e.g., `../`, `?`, `#`) could alter the redirect URL sent to Stripe, potentially redirecting users to unintended pages after checkout.

## Current Behavior
In `backend/internal/handler/billing.go`, the `Checkout` method (line 32-119):
- Line 83-86: Queries the first domain ID:
  ```go
  h.DB.QueryRow(ctx,
      "SELECT id FROM domains WHERE org_id = $1 AND hidden = false ORDER BY display_order LIMIT 1",
      claims.OrgID,
  ).Scan(&firstDomainID)
  ```
  - Note: The `QueryRow` error is not checked (the result of `Scan` is silently ignored).
- Line 88-93: Constructs success URL:
  ```go
  successURL := h.AppURL
  if firstDomainID != "" {
      successURL += "/d/" + firstDomainID + "/inbox?billing=success"
  }
  ```
- Line 94-97: Constructs cancel URL similarly.
- The same pattern repeats in the `Portal` method (line 143-150):
  ```go
  returnURL += "/d/" + firstDomainID + "/inbox"
  ```

The `firstDomainID` is a UUID generated by PostgreSQL during domain creation, so under normal circumstances it is safe. However:
1. There is no validation that it is actually a UUID.
2. If the domains table is corrupted or if the ID format ever changes, the URL could be malformed.
3. The `Scan()` error is unchecked, meaning if the query fails, `firstDomainID` remains an empty string, which is handled but could mask errors.

## Expected Behavior
- Validate that `firstDomainID` is a valid UUID before interpolating into URLs.
- Check the error from `QueryRow().Scan()` and handle it explicitly.
- Use `url.PathEscape()` or UUID validation to ensure the domain ID cannot inject path segments.

## Files to Modify
- `backend/internal/handler/billing.go:83-86` — Check the error from `Scan()`:
  ```go
  err := h.DB.QueryRow(ctx, ...).Scan(&firstDomainID)
  if err != nil {
      // firstDomainID stays empty, will use fallback URL
      slog.Warn("billing: no domain found for checkout redirect", "org_id", claims.OrgID)
  }
  ```
- `backend/internal/handler/billing.go:89-93` — Validate UUID before use:
  ```go
  if firstDomainID != "" {
      if _, err := uuid.Parse(firstDomainID); err != nil {
          slog.Error("billing: invalid domain ID", "domain_id", firstDomainID)
          firstDomainID = "" // fall through to default URL
      }
  }
  ```
- `backend/internal/handler/billing.go:143-150` — Same fix in the `Portal` method.

## Impact Analysis

### What This Affects
- Stripe checkout success/cancel redirect URLs.
- Stripe billing portal return URL.
- User experience after completing a Stripe checkout (they land on the right page).

### What Affects This
- Domain creation in `onboarding.go` — this is where domain IDs are first created.
- The `uuid` package is already imported in the project (used in `attachments.go`).

### Related PRDs
- PRD-003: Open Redirect in PaymentWall — Both address URL injection in the billing flow. PRD-003 validates the Stripe URL on the frontend; PRD-006 validates the redirect URLs on the backend.

## Implementation Notes
- Import `"github.com/google/uuid"` (already a dependency).
- The same `firstDomainID` query pattern appears in both `Checkout` (line 83) and `Portal` (line 144). Extract to a helper if desired, but the fix is small enough to apply inline.
- Also fix the unchecked `Scan()` on line 59-61 (email query) — while less critical (it's only used for the Stripe customer email), it follows the same pattern.
- The `h.DB.QueryRow(ctx, ...).Scan(...)` on line 59-61 for the admin email is also fire-and-forget — if it fails, an empty email is passed to Stripe. This should also be checked.
